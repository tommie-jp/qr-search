# ブラウザログの表示 (Ver2.x)

サーバログは `/logs` で見られるようになった (docs/21) が、**クライアントで
起きた失敗はどこにも残らない**。iPhone の Safari は Web インスペクタを
繋ぐのに Mac が要り、手元の環境 (Windows + WSL) からは開けない。
画像検索 (docs/25)・OCR (docs/24)・パスキー (docs/29) と、クライアントで
完結する機能が増えた今、「iPhone でだけ動かない」の調査手段が無い。

実例: 画像検索のモデルが iPhone で読み込めなかったとき、原因の手掛かりは
ブラウザの console と network にしか出ていなかった。PC では再現せず、
iPhone では見る術が無く、あたりを付けては デプロイして試す、を繰り返した。

対処は 2 本立てで、役割が違う。

- **転送 (§1)**: ブラウザの warn / error / 未捕捉例外をサーバへ送り、
  既存の `/logs` に合流させる。**事後に PC からも**見られる恒常の仕組み
- **eruda (§2)**: `?debug=1` で DevTools 相当のパネルをその場に出す。
  console だけでなく **network** (fetch の失敗・ステータス) まで見える

転送は「いつでも拾えているが warn/error だけ」、eruda は「出している間
だけだが何でも見える」。片方では埋まらないので併用する。

## 1. 転送: ブラウザ → /api/client-logs → /logs

```text
root layout に client component を 1 つ置く
  → console.warn / console.error を包む (サーバの logBuffer と同じ発想)
  → window の 'error' / 'unhandledrejection' も拾う
  → 数秒バッファして POST /api/client-logs
サーバは browser 用リングバッファ (200 件) に控える
  → /logs がサーバ分とマージして表示 (バッジで見分け)
```

### 拾う

- **`console.warn` / `console.error` のラップ**。既存コードは 1 行も
  変えない (docs/21 と同じ理由)。包んだかどうかは `globalThis` に持ち、
  dev の HMR や StrictMode の二重実行でも包みは 1 重
- **`window.addEventListener('error')` と `'unhandledrejection'`**。
  console を経由しない失敗こそ拾いたい本命 — 誰も catch しなかった例外は
  `console.error` のラップには掛からない
- **info / log は拾わない** (docs/21 と同じ。ノイズで 200 件が流れる)

### 送る

- 数秒 (3 秒程度) 溜めてまとめて POST。ページを離れるとき
  (`pagehide`) は `navigator.sendBeacon` で送り切る — 遷移直前の
  エラーが一番失いたくないログで、fetch では間に合わない
- 上限で自衛する: 1 件 2000 文字 (`LOG_TEXT_LIMIT` と同じ)、1 回の
  送信 20 件、送信待ち 50 件超は古い方から捨てる。エラーが無限ループ
  しても回線とサーバを潰さない
- **送信の失敗はリトライせず、console.error にも出さない**。出すと
  「送れない → エラー → 送る」の無限ループになる。失敗しても元の
  console には残っているので、その場なら eruda (§2) で見られる

### 受ける

- `/api/client-logs` (POST)。`publicPaths` に載せない = proxy の既定
  クローズでログイン必須。handler 内でも `denyUnlessLoggedIn()` +
  `denyCrossSite()` をもう一度 (uploads.ts と同じ流儀 — proxy は
  楽観的な検査であって唯一の砦にしない)
- 中身は手で検証する (validation.ts の流儀)。level が warn / error、
  text が文字列、件数が上限内。形が違えば 400 で全部断る
- **時刻はサーバの受領時刻を使う**。クライアントの時計は信じない
  (ずれ・改ざんを考えなくて済む)。バッファ数秒の誤差は診断に効かない
- User-Agent から端末の印 (iPhone / Android / PC) を短く採る。
  複数端末で使うので「どの端末の悲鳴か」は要る

### 貯める・見せる

- バッファは**サーバ分と別に** browser 用を 200 件持つ。同じバッファに
  混ぜると、多弁な側が寡黙な側を押し流す (クライアントの暴走ループが
  サーバの肝心の 1 行を消す)
- `/logs` は両方を時刻順にマージし、「サーバ」「ブラウザ (iPhone)」の
  バッジで見分ける。ページの題は「サーバログ」→「ログ」
- 空文言もソース別に出す —「壊れて出ない」と見分けが付くように
  (docs/21 §3 と同じ)

### WASM の騒音を止める (本番で判明)

本番に出した直後、`/logs` が **onnxruntime の警告だけで埋まった**。OCR を
1 回走らせるたびに

```text
[W:onnxruntime:, graph.cc:5457 CleanUnusedInitializersAndNodeArgs]
Removing initializer 'p2o.pd_op.full_int_array.39.0'. …
```

が 27 件 (実測)。PP-OCRv5 のモデルは Paddle2ONNX 変換の名残で未参照の
初期化子を大量に含み、ORT がセッションを作るたび 1 件ずつ吐く。
**バッファを分けた意味が無くなる** — サーバとブラウザは分けたのに、
ブラウザ側 200 件がこの騒音で埋まり、本物の失敗を押し出していた。

2 つ直した。

- **ANSI エスケープを落とす** (`logText.ts`)。WASM は端末に出す前提で色を
  付けるので、画面には `[0;93m` という文字列として出て本文が読めない
- **セッションの警告を止める** (`lib/ort/quietOrtLogs.ts`)。**`env.logLevel`
  では効かない** — onnxruntime-web はセッション側で
  `sessionOptions.logSeverityLevel ?? 2` と既定 2 (warning) をハードコード
  しており、環境ロガーの閾値とは別物だから。PaddleOCR SDK は
  `logSeverityLevel` を渡す口を持たない (`OrtOptions` の型で確認) ので、
  `InferenceSession.create` を包んで既定値を足す。呼び出し側が明示した値は
  尊重する

なお **W レベルを捨てても失われる情報は無い** (error 以上は残る)。
ブラウザの console 自体が静かになるので、eruda 側も読めるようになる。

### dev では同じログが 2 行出る (実装後に判明)

`next dev` はブラウザの console をサーバの端末へ転送する。転送された行は
サーバ側の `console.warn/error` を通るので、logBuffer が**サーバのログとしても**
控える。結果、dev の `/logs` には同じ失敗が

- 「ブラウザ (PC)」… この計画の転送で届いたもの
- 「サーバ」… `[browser]` で始まる Next.js の転送

の 2 行で出る。本番のビルドにこの転送は無いので出ない。dev で二重に見えても
壊れてはいない。

### 秘密が映らないこと

docs/21 §2 と同じ方針: 鍵・トークンをログに**書かない側**で担保し、
ここでは何もフィルタしない。クライアントの console にはそもそも秘密を
出す口が無い (セッションは HttpOnly Cookie、API 鍵はサーバにしか無い)。
表示はログイン済み本人だけ。

## 2. eruda: `?debug=1` で DevTools 相当をその場に出す

[eruda](https://github.com/liriliri/eruda) はモバイルブラウザに
DevTools 風のオーバーレイを出す定番ライブラリ。console のほか
**network・storage・画面情報**が見える。自前で作る物ではない
(枯れた実装を使う)。

- **依存に足すが、`?debug=1` のときだけ dynamic import する**。
  普段のバンドルには乗らず、既存ページの重さは変わらない
- 発火は URL の `?debug=1`。root layout の client component (§1 と
  同じ場所) が検知して import → `eruda.init()`。**sessionStorage に
  印を覚え**、SPA 遷移でクエリが消えても・再読み込みしても出続ける。
  `?debug=0` で印を消して `eruda.destroy()`。タブを閉じれば消える
  (localStorage にはしない — 常用の道具ではなく、消し忘れて画面の隅に
  居座るのを避ける)
- **ログイン前でも動く** (クライアントで完結し、サーバに何も置かない)。
  §1 の転送は「ログインできない不具合」のログを運べない — その穴を
  ここが埋める。パスキーのログイン失敗 (docs/29) はこれで見る
- ハンバーガーメニューに「デバッグ」を足し、押すとトグルする。
  iPhone のアドレス欄で `?debug=1` を手打ちするのは苦行なので。
  メニューはログイン中しか出ない (layout.tsx) ため、ログイン前だけ
  手打ちになる — それは頻度が低いので許す
- eruda が拾えるのは **init 以降**だけ。init 前のログは §1 の転送が
  拾っている (capture はページ読み込み直後に入る) ので、過去分は
  /logs で見る

## 3. Worker のログは別手当て (Phase 3)

画像検索の embedWorker のような **Worker 内の console と例外は、
window のラップにも eruda にも乗らない** (Worker は別スレッドで
console も別)。§1 の capture を window に依存させず書いておき
(`globalThis` / `self` — 'error' も 'unhandledrejection' も Worker に
ある)、Worker の起動冒頭でも同じ install を 1 行呼ぶ。fetch は
Worker からも使えるので送る側はそのまま動く。

主スレッドへ postMessage で上げてから console に出す案は、Worker を
書くたびに配管が要るのでやめた。install 1 行のほうが忘れにくい。

## 4. 段階と見送り

- **Phase 1 (実装済み)**: クライアント capture + `/api/client-logs` + `/logs` 合流
- **Phase 2 (実装済み)**: eruda (`?debug=1` + メニューの「デバッグ」)
- **Phase 3 (実装済み)**: Worker 内 capture (embedWorker から)
- **見送り: info / log の収集** (docs/21 と同じ)
- **見送り: 自動更新・永続化・レベル切り替え** (docs/21 §4 と同じ理由)
- **見送り: ネットワークログの転送**。network はその場の eruda で見る。
  全 fetch をサーバに送るのは量が桁で違い、200 件がすぐ流れる
- **見送り: Sentry 等の外部サービス**。個人アプリに過剰で、ログを
  外部に出さない方針にも合わない
- **見送り: vConsole**。eruda と同役。二つは要らない

## 5. テスト

- ログの整形 (`logText`): ANSI エスケープを落とす。`[W:onnxruntime:]` の
  ような「エスケープでない角括弧」を巻き添えにしない
- ORT の警告抑止 (`quietOrtLogs`): 偽の ort を渡し、`logSeverityLevel=3` が
  足されること・呼び出し側の指定が尊重されること・`(buffer, offset, length)`
  のオーバーロードを壊さないこと・二重に掛けても 1 重であること
- クライアント capture (vitest):
  - warn / error を控え、元の console にも流れる
  - 'error' / 'unhandledrejection' を拾う
  - 2000 文字で切る / 送信待ち 50 件超は古い順に捨てる
  - 二重 install しても包みが 1 重のまま
  - flush が送る body の形 (level・text・件数上限)
  - 送信失敗で例外を漏らさず、console.error も呼ばない (無限ループ防止)
- `/api/client-logs` (vitest): 未ログイン 401 / cross-site 拒否 /
  形の悪い body は 400 / 正しい body がバッファに入り 200 件で回る
- `/logs` ページ: ブラウザ項目がバッジ付きで出る・サーバ分と時刻順に
  混ざる (renderToStaticMarkup、docs/21 §5 と同じやり方)
- eruda: 出し入れの**判断だけ** (`debugEnabledFor`) を試す。本体の読み込みは
  ブラウザが要るので単体テストでは追わない (このリポジトリは jsdom を持たず、
  部品テストは静的描画だけ。docs/09 §7 と同じ方針)
- ブラウザ (ローカルで一巡。2026-07-19 実施):
  `?debug=1` でパネルが出る → `console.error` と未処理の Promise 拒否を起こす
  → `/logs` に「ブラウザ (PC)」で 3 件出る → メニューの「デバッグ」で
  出し入れが切り替わる、まで確認済み
- 実機 (iPhone) に残っているのは同じ手順の追試。端末の印が「iPhone」に
  なることと、パネルが小さい画面で使えることを見る

## 6. 診断イベント (info レベル) — メモリ調査用 (2026-07-19)

OCR / 画像検索のメモリ不足 (docs/24 §9-1) を実機で追うため、失敗ではない
「何がいつ起きたか」を info レベルで /logs に流す (`src/lib/diagLog.ts` の
`logDiagEvent`)。console は包まない — 明示的に送ったものだけが info になる。

仕込んであるイベント:

- `[OCR] Worker 起動` / `[OCR] モデル準備完了 (X.X秒)` /
  `[OCR] Worker 破棄 (理由)` — 理由は「編集画面を離脱」「画像検索を開く」。
  **「画像検索の前に OCR は本当に死んでいたか」を実機ログで証明する**ための印
- `[画像検索] Worker 起動 (デバイス自動選択 / WASM 強制)` /
  `[画像検索] モデル準備完了 (X.X秒)` — WASM 強制が出たら 1 回目が落ちた証拠
- 読み込みの失敗は従来どおり warn (console.warn を capture が拾う)

- `[環境] ブラウザ 32bit x86 (WOW64) / RAM ~4GB / CPUコア 8` — OCR /
  画像検索の Worker を最初に起こすとき 1 度だけ送る環境情報。ビット数は
  UA Client Hints (`getHighEntropyValues(['bitness','architecture','wow64'])`)
  で、**ブラウザ本体**のビット数が取れる (32bit Chrome の特定に丸一日
  かかった教訓。WOW64 = 64bit OS 上の 32bit ブラウザなので「64bit 版に
  入れ替えれば直る」とまで言い切れる)。WebKit は userAgentData が無いので
  「ビット数不明」と出る。RAM は navigator.deviceMemory (8 は「8 以上」)

ヒープの数値は `[JSヒープ 使用123 確保256 上限4096MB]` の 3 つ組
(used/total/limit)。**上限は環境の指紋**: 通常の x64 Chrome は ~4GB、
~1120MB なら 32bit Chrome か低メモリ端末。

### メモリの数値は iPhone では取れない

iOS は全ブラウザが WebKit で、`performance.memory` (Blink 独自)・
`navigator.deviceMemory`・`measureUserAgentSpecificMemory` のどれも無く、
残量は OS が隠している (jetsam は予告なく殺す)。よって:

- 取れる環境 (PC Chrome) でだけ `[JSヒープ 使用/上限MB]` をイベントに添える
  (wasm の確保も JS ヒープの外部バッファとして載るので傾向は見える)
- iPhone では**イベントの順序と経過時間**が診断の本体。それで「どの段階で・
  何秒後に・何が生きているときに」落ちたかまでは特定できる
- 「大きな ArrayBuffer を試しに確保して残量を推定する」プローブは**やらない**。
  その確保自体が jetsam の引き金になり、調べたいものを調べる前に殺される

## 7. ログのクリア (2026-07-19)

/logs にクリアボタンを置いた (`ClearLogsButton` → POST `/api/logs/clear` →
`clearLogBuffer()`)。実機調査では「一度消してから再現操作をする」と、並ぶのが
今回の再現ぶんだけになる。口の門番は他と同じ二段 (未ログイン 401・
cross-site 403) — 開けっ放しだと第三者のページから調査中の証拠を消せてしまう。
