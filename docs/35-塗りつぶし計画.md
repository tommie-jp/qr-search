# お絵かきの塗りつぶし計画 (Ver2.x)

お絵かき (docs/34) にバケツ塗りを足した記録。実装済み (2026-07-20)。

クリックした点と「同じ色の連続した領域」を選択中の色で塗る、ペイント系の
定番の道具。ペンで囲った領域への色入れと、白紙での簡単な図解が狙い。

## 1. 方式の選定

fabric はベクタ、バケツ塗りはラスタの操作なので、どこかで橋を架ける必要がある。

- **(採用) その場でラスタを読んで flood fill し、結果を画像オブジェクトとして
  足す**。いまの canvas の画素から塗るべき領域を求め、その領域だけを持つ
  透過 PNG を作って `FabricImage` として追加する。オブジェクトが増えるだけ
  なので、undo (object:added → 履歴) も消しゴム (erasable) も既存機構が
  そのまま効く
- (却下) 塗り領域のベクタ化 — 輪郭追跡 (marching squares) で Path に起こす案。
  履歴 JSON は小さくなるが、穴あき領域・斜め接続の扱いが複雑で KISS に反する
- (却下) クリックしたオブジェクトの `fill` を設定 — 実装は一番安いが、
  ペンの線は**開いた** Path なので fill の結果が意図と合わない。複数の線で
  囲った領域も塗れない。バケツに期待される動きではない

## 2. アルゴリズム (純粋ロジック)

`src/lib/draw/floodFill.ts` に DOM 非依存の純関数で置く (TDD 対象)。

- **scanline flood fill**。クリック点の色を基準に、4 近傍で走査する。
  1600×1200 ≒ 190 万画素の走査は数十 ms 想定で、クリック 1 発の操作なので
  同期実行でよい (重ければ実測してから考える)
- **許容差**: ペンの線はアンチエイリアスされていて、境界の画素は基準色と
  微妙に違う。色距離のしきい値 (定数、実機で調整) を持たせる
- **1px 膨張**: 許容差だけでは塗りと線の間に 1px の白い縁 (ハロー) が残り
  がちなので、求めた領域を 1px 膨らませてから塗る
- **bbox**: 塗った領域の外接矩形も返す。画像はこの矩形で切り出して作る
  (§4 のメモリ対策)

## 3. fabric への組み込み

- **読み取り元は `fc.lowerCanvasEl` を直接 `getImageData`**。下段 canvas には
  背景色・下敷き画像・全オブジェクトが合成済みで、選択枠などの UI は上段に
  分かれているので写り込まない。`enableRetinaScaling: false` (docs/34 §3-2)
  のおかげで実バッファ = 論理サイズとなり、座標変換も追加レンダも要らない
- 塗り結果は bbox で切り出した canvas → **dataURL (PNG) → `FabricImage`**。
  `src` が dataURL なので `toObject()` → `loadFromJSON()` の履歴復元が
  そのまま効く (canvas 要素のままでも `getSrc()` は `toDataURL()` に落ちるが、
  履歴に載る形を最初から持たせて曖昧さを消す)
- 追加したオブジェクトは `object:added` の既定処理で `erasable: true` が立ち、
  スナップショットも積まれる。**新規コードで履歴・消しゴムを触る必要は無い**
- 道具 `fill` は文字道具と同じ「`mouse:down` で 1 発」型。`isDrawingMode` は
  false、`getScenePoint` でクリック位置を取る

## 4. 履歴とメモリ

dataURL は履歴 JSON の**全スナップショット**に乗るので、塗るほど履歴が重くなる。

- bbox 切り出し + PNG 圧縮で抑える。単色領域の PNG は小さい (白紙の全面
  塗りでも単色 1600×1200 で数 KB)
- 写真の上を塗ると領域が斑になり PNG が肥えやすい。実機で実測し、必要なら
  「履歴に積む前にスナップショット長で HISTORY_MAX を絞る」を検討する
  (最初からは作り込まない)

## 5. UI

- ツールバーの並びは「ペン・**塗る**・消しゴム・選択・文字」。色は既存の
  色選択を共用する。太さは意味を持たないので無効化 (disabled) する
- 塗りは 1 クリック完結なので、進行表示は要らない想定。実測で 100ms を
  超えるようなら考える

## 6. 実装したもの

- `src/lib/draw/floodFill.ts` (+ 8 件のテスト) — scanline flood fill・許容差・
  膨張・bbox。許容差は 32、膨張は 1px (`rasterTool.ts` の定数)
- `src/components/draw/rasterTool.ts` — `lowerCanvasEl` から画素を読み、
  bbox で切り出した透過 PNG を `FabricImage` にして返す。モザイク
  (docs/36 §3) もこの基盤に相乗りしている
- `useDrawCanvas` の `mouse:down` に道具 `fill` の分岐
- `DrawToolbar` に「塗る」ボタン。太さが効かない道具では太さの選択を伏せる

**未確認**: 実機での速度・ハロー・履歴の重さは見ていない (§7 のリスク参照)。

## 7. リスク

- **写真の上の塗り**: グラデーションだと許容差の内側だけが斑に塗れて
  見栄えが悪い。用途としては「囲った領域を塗る」が主なので許容し、
  しきい値の調整だけで粘る
- **ハロー**: 1px 膨張で消え切らない場合がある (太い線の半透明端)。
  膨張幅を定数で持ち、実機で詰める
- iOS のメモリ: 一時 canvas (bbox サイズ) と ImageData (フル 1 枚 ≒ 7.7MB)
  は塗り 1 回ごとに使い捨て。確保しっぱなしにしない
