# 回路図 (CircuiTikZ) 計画

memo 欄に電子回路図を描けるようにする計画 (2026-07-15 起案)。
CircuiTikZ (LaTeX) 記法をそのまま書けることを目標にする。

## 方針

`node-tikzjax` (TeX の WebAssembly 移植、circuitikz 同梱) を
**サーバー側で子プロセスとして実行**し、出力 SVG を
**Postgres にハッシュキャッシュ**する。

```text
memo 本文の ```circuitikz フェンス
  → sha256(source + レンダラのバージョン)
  → circuit_svgs キャッシュ hit → SVG をそのまま埋め込む
  → miss → 子プロセスで tex2svg() → 許可リストで検査 → 保存 → SVG
```

- クライアント JS の追加はゼロ (mermaid と違い動的 import しない)
- Docker イメージの増加は npm パッケージ 8.4MB のみ
- TeX Live のインストールは不要

## 検討した代替案

| 案 | 追加コスト | 判断 |
| --- | --- | --- |
| **A. サーバー側 node-tikzjax + SVG キャッシュ** | npm 8.4MB | **採用** |
| B. クライアント側 isomorphic-tikzjax | DL 4.7MB + タブ 72MB | 次点 |
| C. TeX Live サイドカー | イメージ +420MB | 却下 |
| D. JS 製の代替ライブラリ | — | 該当なし |

### C (TeX Live) を却下した理由

`doDeploy.sh` の通り vps2 はメモリ不足でビルドできず、イメージを
`docker save`/`load` で転送する運用のため、+420MB は転送のたびに効く。
Alpine ベースの TeX Live は 934MB と Debian (422MB) の 2.2 倍で、
`node:24-alpine` ベースとの相性も悪い。

### D (JS 代替) が無い理由

- mermaid v11 に回路図の図種は無く、サードパーティ製プラグインも無い
- netlistsvg は npm が 2022-05 で停滞。入力も Yosys の netlist JSON
- schemdraw は Python。Pyodide 経由で動かした前例が見つからない
- schemascii は記法こそ理想的だが 10 スター・GPL-3.0・2025-08 で停止

CircuiTikZ を置き換えられる JS ライブラリは現時点で存在しない。

> [!WARNING]
> npm の `circuit-svg` は用途に一致する説明を持つが、公開が 2026-07-15
> ・履歴なし・リポジトリ未記載・DL 数ゼロで、typosquat か LLM 生成物の
> 形をしている。使わないこと。

## 設計判断と根拠

### 描画はサーバー側 (KaTeX と同じ位置)

`MarkdownView` は Server Component で、クライアント側のライブプレビューが
存在しない (`MemoPanel` は描画済みの ReactNode を prop で受け取るだけ)。
よって mermaid のような client component + API route は不要で、
`MermaidDiagram` より単純になる。保存後の再描画時にだけ約 1 秒かかり、
以降はキャッシュヒットで即時。

### キャッシュは volume ではなく Postgres

`schema.prisma` が画像を `Bytes` で DB に持っているのと同じ理由
(`pg_dump` だけでメモと画像が揃い、両者の整合が壊れない)。
Ver2.4 で uploads volume を撤去した方針とも一致する。

ハッシュには**レンダラのバージョンも混ぜる**。Wikimedia の Math 拡張は
ここを入れ忘れており、レンダラ更新時にキャッシュが無効化されない。

### フォントは self-host

npm パッケージに Bakoma CM の TTF 140 個 (計 3.7MB) と `fonts.css` が
同梱されており、`fontCssUrl` オプションで参照先を差し替えられる
(既定値は jsDelivr の CDN)。`@font-face` はブラウザが実際に使う書体だけを
取りに行くため、実転送は 1〜3 ファイル (各 30KB 前後) で済む。

TikZJax の出力は輪郭パスではなく `<text font-family="cmmi10">` なので、
フォントを配らないと文字が化ける。

### 子プロセス隔離 (worker_threads ではなく child_process)

Phase 0 のスパイクで **worker_threads ではなく子プロセス**を採ることにした。
描画中のピーク RSS が 399MB に達するため、スレッドだとその分が本体プロセスに
居座る。子プロセスなら終了時に OS がすべて回収し、実測でも親は
44MB → 47MB のまま動かなかった。

なお `fork()` は使えず `spawn` + `'ipc'` になった (後述)。

- **timeout がない**: TikZJax に timeout オプションは存在せず、
  `\def\x{\x}\x` は 90 秒経っても返らないことを実測で確認した。
  親から `kill('SIGKILL')` で殺すのが唯一の停止手段 (5 秒で kill 確認済み)
- **直列化が要る**: README が "Don't run multiple instances at the same
  time" と明記 (モジュールレベルの共有状態)。1 本のキューで順に流す
- **常駐させない**: 描画のたびに spawn → exit。キャッシュのおかげで
  発火頻度は低い

### エラーログは子プロセスの stdout から拾う

`tex2svg()` が throw する例外の文言は
`TeX engine render failed. Set 'options.showConsole' to 'true' to see logs.`
だけで役に立たない。原因 (`! Package pgfkeys Error: ...` と該当行 `l.6`) は
`showConsole: true` のときに **stdout へ出る**。子を
`stdio: ['ignore', 'pipe', 'pipe', 'ipc']` で起動し、親がその出力を拾って
赤枠 UI に載せる (出し続ける TeX に備えて 64KB で打ち切る)。

### SVG は許可リストで検査し、想定外なら描画エラーにする

`uploads.ts` が「SVG はスクリプトを埋め込めるため対応しない」として
アップロードを明示的に拒否している以上、生成物とはいえ
`dangerouslySetInnerHTML` に流す SVG を無検査で通すのは既存方針と矛盾する。

**これは実際に必要だった。** `\special{dvisvgm:raw ...}` は TeX の素の命令で
(パッケージ不要)、中身をエスケープせずそのまま SVG に流し込む
(`@prinsss/dvi2html` の `specials/svg.js`)。つまりフェンスに
`<a xlink:href='javascript:...'>` を書けば任意のスクリプトを仕込めた。
web で拾った回路スニペットを貼る使い方を考えると、現実的な経路。

当初の「危険なものを消す」実装 (`<script>` と `on*` を正規表現で除去) は
**穴だらけだった**。実際に取り逃がしていたもの:

- `javascript:alert(1)` — 正規表現が `javascript:text/html` しか見ていなかった
- `<script xlink:href="data:,alert(1)"/>` — 自己閉じタグに閉じタグが無い
- `<set attributeName="onload" to="alert(1)"/>` — SMIL 経由の指定

そこで**許可リストで検査し、想定外なら丸ごと捨てる** (fail-closed) 方式に
変えた (`assertSafeCircuitSvg`)。TikZJax が実際に出す要素は
svg / g / defs / style / path / text の 6 種類しかないので、許可リストは現実的。
上記 3 つと `\special` 注入の実物は回帰テストとして残してある。

キャッシュから読んだ SVG も毎回検査する。検査を直したときに
`RENDERER_VERSION` を上げ忘れても、古い行が素通りしないようにするため。

なお `\write18` (シェル実行) は WASM サンドボックス内で無効であることを
実測確認済み。サーバー側 LaTeX 最大の懸念である任意コード実行は起きない。

memo を書くのは Basic 認証済みの本人だけだが、上記のとおり「自分で書いた
ものしか出てこない」とは言い切れないため、検査は入れる。

## Phase 1〜4 で分かったこと (実装メモ)

計画時の想定と違った点。

### `child_process.fork` は使えない — `spawn` + `'ipc'` を使う

Turbopack は `fork()` の第 1 引数を「バンドルすべきモジュール」として
静的解析するため、バンドル対象外の描画スクリプトを渡すとビルドが
`Module not found: Can't resolve './ROOT/scripts/renderCircuit.cjs'` で落ちる。
`spawn(process.execPath, [script], { stdio: [..., 'ipc'] })` は fork と等価に
IPC が張れて、解析対象にならない。

### tracing のキーは picomatch なので `[itemNo]` をエスケープする

`outputFileTracingIncludes` のキーは**ルートに対する picomatch の glob**。
`"/item/[itemNo]"` と書くと `[...]` が文字クラスと解釈されてどのルートにも
マッチせず、**何も同梱されないまま黙ってビルドが成功する**。
`"/item/\\[itemNo\\]"` と書く。

### node_modules は include glob では運べない

`"./node_modules/node-tikzjax/**/*"` を include に書いても無視される。
代わりに `serverExternalPackages` + **呼ばれない動的 import** で nft に
追跡させると、依存の閉包ごと standalone に入る。
親で普通に `import` すると 87MB / 228ms を無駄に抱えるため、
実行時に評価されない `() => import('node-tikzjax')` にしてある。
