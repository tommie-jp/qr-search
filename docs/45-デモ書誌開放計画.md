# 45 デモ書誌開放計画 (キー不要の書籍情報をデモでも取得)

デモインスタンスで、ISBN スキャン時の**書誌 (書名・著者) の事前入力**を
動かせるようにする計画。書誌は openBD / NDL サーチのキー不要 API で引けるのに、
いまは一律で塞いでいる。

関連: [39-デモ公開計画.md](39-デモ公開計画.md) §5 /
[44-アクセスキー方式検討.md](44-アクセスキー方式検討.md) §5 /
[13-書誌自動取得計画.md](13-書誌自動取得計画.md)

## 1. 現状 (なぜ動かないか)

`/api/books/[isbn]` はルート冒頭で `isDemoMode()` を見て `demoDisabled` を即返す
([api/books/[isbn]/route.ts:38](../src/app/api/books/[isbn]/route.ts))。
このゲートが `lookupBook` より**手前**にあるため、キー不要の openBD/NDL 書誌取得まで
巻き添えで止まっている。JAN (`/api/products`) は Yahoo キーが本質的に要るので
これでよいが、書籍側はキー不要で開けられる。

- 書誌: openBD → NDL サーチ ([bookLookup.ts](../src/lib/bookLookup.ts))。**キー不要**。
- 書影: openBD cover (キー不要) → 楽天 (キー要)。楽天は `fetchCoverUrl` が
  キー未設定を warn して null にするため throw しない
  ([coverLookup.ts](../src/lib/coverLookup.ts))。

## 2. 方針

**Phase 1: 書誌のみデモで開放 (書影は出さない)** を推奨。理由は §4。

- `/api/books/[isbn]`: デモでも `lookupBook` を走らせ、書名・著者を返す。
- 書影 (`saveCoverImage`) は**デモでは呼ばない**。openBD cover はキー不要で
  引けるが、DB (images) に書き込む副作用があり、デモの容量・画像 GC と
  絡むため Phase 1 の対象外にする (§4-2)。
- `/api/products/[jan]` (JAN): **変更なし**。Yahoo キーが無いので `demoDisabled`
  のまま。

## 3. 変更点 (実装時の当たり所)

### 3-1. サーバ: `/api/books/[isbn]/route.ts`

- 冒頭の `if (isDemoMode()) { return demoDisabled }` を**削除**。
- `isJan`/`isIsbn` の検算・`denyUnlessLoggedIn`+`denyCrossSite` は現状維持
  (デモユーザは共有資格情報でログイン済みなので通る。踏み台対策として必須)。
- 書影の呼び出しを分岐: デモのときは `coverImageUrl` を付けない。

  ```ts
  const skipCover = isDemoMode() // デモは書影を保存しない (docs/45 §4-2)
  const data = book
    ? {
        ...book,
        coverUrl: undefined,
        coverImageUrl: skipCover
          ? undefined
          : await saveCoverImage(isbn, book.coverUrl),
      }
    : null
  ```

### 3-2. クライアント: `usePrefill.ts` / `MemoEditor.tsx`

- **コード変更は基本不要**。書籍取得が demoDisabled ではなく通常の
  `loaded` / `notFound` / `error` を返すようになるだけで、既存の分岐がそのまま効く。
- `PREFILL_DEMO_MESSAGE.book` ("デモ版では書籍情報を取得できません") は
  **書籍では発火しなくなる**。product 用は残す。使われなくなる book 側の文言は
  消すか、将来のフォールバックに備え残すか判断 (消すなら型を product 専用に絞る)。

### 3-3. 濫用対策 (踏み台の考慮)

デモ開放で `/api/books` は NDL/openBD への**代理踏み台**として開く。既に
`denyUnlessLoggedIn` + `denyCrossSite` の下にあり、デモは共有ログインなので
無制限開放ではないが、外部 API を叩く口である点は変わらない。

- 既存の `demoQuota.ts` は保存容量・ノート数の門番で、**外部 API 呼び出しは
  対象外**。書誌取得にレート制限を足すかは Phase 1 では見送り可 (書影を
  保存しないので副作用は外部 GET のみ、`sourceTimeout` で 1 回あたりも有界)。
- 監視だけ入れておくと安全: デモで書誌取得が過剰なら後追いで IP/セッション
  単位の軽いレート制限を検討 (別途)。

## 4. Phase 分けの理由

### 4-1. なぜ書誌を先に開けるか

「書籍情報取得」の主目的は**書名・著者の事前入力**で、これはキー不要で
完結する。デモの価値 (スキャンして本が登録できる体験) の大半をこれで満たせる。

### 4-2. なぜ書影 (openBD cover) を Phase 1 で外すか

- `saveCoverImage` は **images テーブルへ書き込む**。デモは容量クォータ
  (`checkDemoUploadQuota`) を持つが、書影保存は**このクォータを経由しない**
  ため、開けると容量計算の外で画像が増える。
- ノートを保存せず離脱すると参照されない書影が残る (docs/19 §4)。デモの
  定期リセットで消えるとはいえ、開放と同時に増やすと画像 GC の観測が濁る。
- openBD cover の実測ヒット率は低い (1/11)。デモ体験への寄与が小さく、
  副作用 (DB 書き込み) との釣り合いが悪い。

### 4-3. Phase 2 (任意): デモでも openBD 書影を出す

やるなら、書影保存もデモ容量クォータの対象に含める
(`saveCoverImage` 前に `checkDemoUploadQuota` 相当を通す) 改修とセットにする。
Phase 1 の運用を見てから判断。

## 5. テスト観点

- `/api/books/[isbn]` route: `isDemoMode()=true` で書誌が返り、`coverImageUrl` が
  付かないこと。JAN route は demo で `demoDisabled` のままであること。
- `usePrefill`: 書籍で `loaded` / `notFound` に落ちること (demoDisabled にならない)。
- 既存のデモ関連テスト (`logs`・`settings` の 404、product の demoDisabled) が
  壊れないこと。
