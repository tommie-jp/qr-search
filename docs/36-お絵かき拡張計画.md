# お絵かき拡張計画 — 矢印・枠 / マーカー / モザイク / ズーム (Ver2.x)

お絵かき (docs/34) に足した 4 つの記録。実装済み (2026-07-20)。
塗りつぶしは docs/35 に分離。

## 1. 矢印・枠 (図形ツール)

写真注釈の定番。「ここ」を指す矢印と、範囲を示す枠。手描きの丸より速くて
読みやすい。

### 1-1. 道具と操作

- 道具は **矢印・矩形・楕円** の 3 つ。ドラッグで始点→終点を決める
  (down で仮置き、move で形を更新、up で確定)
- `isDrawingMode` は false で、文字道具と同じく `mouse:down` から自前で
  組み立てる。数 px 未満のドラッグは捨てる (タップとの区別)
- 色・太さは既存の選択を共用。枠 (矩形・楕円) は `fill` なしの線だけ、
  矢印は軸線と鏃 (やじり) を持つ

### 1-2. 実装の形

- **矢印は Path 1 本**にする (軸線 + 鏃の三角を 1 つの path データで)。
  Group (Line + Triangle) より、消しゴム・直列化・選択の扱いが素直。
  鏃の大きさは太さに比例させる
- 矩形は `Rect`、楕円は `Ellipse`。負方向へのドラッグは left/top を
  入れ替えて正規化する
- **ドラッグ中は履歴を止める** (`suppressRef`)。仮オブジェクトの add で
  スナップショットが積まれると、中途半端な形が履歴に残る。up で解除して
  1 手として積む
- 純粋ロジックとして切るもの (TDD 対象、`src/lib/draw/shapes.ts`):
  始点・終点・太さ → 矢印の path データ文字列 / ドラッグ矩形の正規化

## 2. マーカー (半透明ペン)

蛍光ペン。文字や写真の上をなぞっても下が透ける。**4 候補で最安**。

- `PencilBrush` の色を `rgba()` にするだけ。fabric のブラシ色は
  `ctx.strokeStyle` にそのまま渡るので alpha が効く (ソースで確認済み)。
  既存の hex 色 + 固定 alpha (0.35 目安) を合成する純関数を 1 つ書く
- 太さは選択値の 3 倍程度 (マーカーらしい幅)。消しゴムの `ERASER_SCALE` と
  同じ流儀の係数で持つ
- 1 ストロークは 1 つの Path なので、自己交差しても濃くならない。
  ストローク同士の重なりは濃くなる — 本物のマーカーと同じ挙動なので許容
- **性能の前提**: alpha < 1 のブラシは `needsFullRender()` が true になり、
  消しゴムと同じ「毎フレーム全再描画」経路に入る (`BaseBrush` で確認)。
  docs/34 §3-2 の対策 (retina 無効・1600px 上限) が効いている前提の機能で、
  上限を緩めるときはマーカーの描き味も再確認すること

## 3. モザイク

公開ノート (docs/22) に載せる写真の住所・顔・シリアル番号を隠す。

### 3-1. 黒塗りとの違い (誠実な整理)

書き出しは画像への焼き込み (docs/34 §3) なので、**黒ペンで塗っても書き出し後
の画像から下の画素は復元できない**。それでもモザイクを別に持つ価値は:

- **半透明マーカーで塗って透けていた**という事故を防ぐ (モザイクは必ず読めなくなる)
- 「隠したが、そこに何かある」ことは伝わる見た目
- ブロックを大きめに取れば、拡大やシャープ化で読み戻される心配も減る
  (ブロックサイズは定数で大きめに。小さすぎるモザイクは復元研究の的)

### 3-2. 実装の形

- 操作は**矩形ドラッグ**で範囲指定 (§1 のドラッグ基盤を流用)
- 加工は塗りつぶし (docs/35 §3) と同じ「`lowerCanvasEl` から `getImageData`
  → 加工 → bbox の canvas → dataURL → `FabricImage`」基盤。違いは加工が
  flood fill ではなく**ピクセル化** (ブロック平均色) なだけ
- 純粋ロジック: `pixelate(imageData, blockSize)` (TDD 対象)
- 置いたモザイクはオブジェクトなので undo・消しゴも既存機構のまま。
  下のオブジェクトを後から動かすとズレるが、注釈全般が同じ性質なので許容

## 4. ピンチでズーム / パン

大きい写真の細部に描き込むための拡大。**4 候補で唯一、実装コスト中〜高**。

### 4-1. 方式

**CSS の表示倍率 (displayScale) を可変にする** 案を採る。fabric の
`viewportTransform` でズームする案は、描画バッファが論理サイズのままなので
拡大してもぼけ方は同じで、座標系だけが複雑になる。現行の
「論理サイズ固定 + CSS 拡縮」は fabric がポインタ座標を
`getBoundingClientRect` 比で補正してくれる (docs/34 §3) ため、
**倍率を動かしても座標変換の追加実装が要らない**。

- ピンチで `displayScale` を上げ下げ (fit 倍率〜4 倍程度)
- はみ出しはステージの `overflow: scroll` でパン
- ズーム中心の維持: ピンチ中心が動かないようにスクロール位置を補正する
- **ホイール (PC) はどの道具でも効く**。ピンチと違い 1 本指の描画と
  取り合いにならないので「移動」道具に限定する理由が無い。ポインタ位置を
  軸に拡大し、係数は `exp(-deltaY × 0.002)` — 上下に同じだけ回すと正確に
  元の倍率へ戻る。Firefox の行単位 (deltaMode = 1) も px へ換算して受ける

### 4-2. 難所

- **1 本指 = 描画、2 本指 = ズーム/パン** の振り分け。描画中に 2 本目の指が
  着いたら、描きかけのストロークを捨てる必要がある (fabric に指 2 本の概念は
  無い)。ここがこの機能の実装コストの大半
- iOS Safari のネイティブピンチ (ページ全体の拡大) との競合。ステージに
  `touch-action: none` を敷いて自前処理に寄せる
- 太さ・文字サイズの基準は当初「その時の画面上の見た目で一定」(displayScale で
  割る = 拡大中に描くと論理的に細くなる) にしていたが、**実際に使うと拡大中に
  置いた文字が 100% に戻すと他より小さく、論理的に食い違う**という指摘を受けて
  改めた。いまは **fitScale (100% 時の縮み) で割り、拡大は純粋な虫めがね** ——
  道具の論理サイズは倍率に依らず一定で、Photoshop / Excalidraw と同じ意味論。
  画面基準のまま残しているのは、内容ではないもの: 選択ハンドルの大きさと
  「タップかドラッグか」の判定しきい値だけ (これらは displayScale で割る)

### 4-3. 送りを scroll から transform へ変えた (不具合対応)

**iPhone で「モザイクの範囲表示と実際に処理される範囲がズレる」**という報告から
調べ、2 つの不具合が出た。座標変換そのものは健全だった (Playwright で
DrawModal と同じ DOM を組んで実測。拡大・スクロール状態でも
`getScenePoint` は期待値と完全に一致した)。

**(1) 仮表示が焼き込まれた画素を読んでいた** — これが報告の本体。
`fc.remove()` が呼ぶのは `requestRenderAll` で、実際の描き直しは rAF まで
待たされる。モザイクは「範囲を見せていた黒い仮表示を消した直後」に
`getImageData` するため、待たずに読むと**仮表示ごと平均されて平坦な黒い板**が
出来ていた (実測: 赤 255 の場所が 153 で読めた)。`readCanvas` の先頭で
同期の `renderAll()` を呼んで直した。

**(2) スクロールできる親が座標をずらす** — 拡大時に出る潜在不具合。
fabric の `getPointer` は **イベントの target の祖先**のスクロール量を足し、
基準位置 `getElementOffset` は **canvas の祖先**のスクロール量を引く。
`getScrollLeftTop` は要素自身を数えないので、指が canvas の外 (= 枠の上) へ
出て target が枠自身になった瞬間、両者の対象がずれて**スクロール量ぶん
ちょうど座標が飛ぶ** (実測: 220px 送った状態で 220px の食い違い)。

→ 送りを `overflow: auto` + `scrollLeft` から**中身の `transform: translate`**に
変え、canvas の上からスクロールできる親を無くした。あわせて、枠に
`scroll` ハンドラを付けてスクロール量を常に 0 に戻している ——
`overflow: hidden` でも、文字道具が fabric の隠し textarea に焦点を当てると
ブラウザが「見える位置まで送る」ためにスクロールさせるため。

### 4-4. fabric v7 の originX/originY の既定は center (不具合対応)

**Windows Chrome で「ペン・マーカー・塗る・矢印は合うのに、四角・丸・モザイク
(の範囲表示) だけマウスとズレる」**という報告。実測すると Rect が指定した
left/top から幅・高さの半分だけ左上に描かれていた (−262px / −212px)。

原因は fabric v7 の破壊的変更 —— **originX/originY の既定が left/top から
center/center に変わった** (CHANGELOG「BREAKING chore(): Deprecate originX
and originY and change their default to center/center #10715」)。origin を
明示していない Rect / Ellipse / IText は、left/top が「中心」と解釈される。

道具ごとに症状が割れた理由:

- **Path (矢印・ペンの線)** は座標を path データから決めるので origin の
  影響を受けない → 合う
- **塗りつぶし・モザイクの処理結果** は `RASTER_DEFAULTS` で origin を
  明示していた → 合う
- **Rect / Ellipse (四角・丸・モザイクの範囲表示)** は origin 未指定 → ズレる。
  iPhone で「モザイクの範囲表示が実際の処理範囲とズレる」と見えていた
  真因もこれ (ズレていたのは処理ではなく**プレビューの方**)

対処:

- `SHAPE_DEFAULTS` と文字道具の IText に `originX: "left", originY: "top"` を明示
- あわせて Rect / Ellipse は左上を**半太さぶん戻す** (`strokeCenteredRect`)。
  fabric の left/top は「ストロークを含む見た目の箱」の角なので、そのままだと
  線の外縁が角に来て、ペン・矢印 (線の中心がポインタ) と意味が食い違う
- モザイクの範囲表示は `strokeWidth: 0` を明示 (stroke が無くても既定の 1 が
  寸法に数えられ、処理範囲と 1px ずれる)

修正は Playwright + 実 fabric の画素実測で確認した (四角・丸・矢印の描画位置が
すべて「線の中心 = ドラッグ枠」の ±半太さに一致)。

### 4-5. 2 本指ジェスチャを全道具に開放 + ジェスチャが 1 目盛りで死ぬバグ

「拡大したときに移動が動かない」「ピンチが効かない」という報告への対応。

**バグ**: ピンチ・ドラッグの進行状態を effect のローカル変数に持ちながら、
effect の依存に zoom / pan (state) を入れていた。1 目盛り動くと state が
変わり → effect が張り直され → 進行状態が破棄される。つまり**どの
ジェスチャも 1 目盛りだけ動いて止まる**。対処として、ハンドラは state を
閉じ込めず ref (zoomRef / panRef) を読む形にし、effect の依存から
変わり続ける値を外した。書き込みは commit() の 1 口に集約して
ref と state の食い違いを作らない。

**2 本指の開放**: 従来ピンチは「移動」道具限定だったが、スマホの自然な
期待に合わせて**どの道具でも 2 本指 = 拡大 + 送り** (開けば拡大、つまんだ
まま動かせば送り。`panForPinch` が 1 つの式で両方を出す) にした。
「1 本指は描画」と衝突する問題は、2 本目の指が着いた瞬間に
`cancelActiveInput` で描きかけを捨てることで解いた:

- ペン・マーカー・消しゴム: `_isCurrentlyDrawing` を折る (中断の公開 API は
  無い。防御的に try で包み、名前が変わっても描画自体は壊さない)。
  消しゴムはさらに `active` を折ってから `onMouseUp` を呼び、確定を跳ばして
  after:render リスナの後始末だけを走らせる (@erase2d の実装で確認)
- 図形のドラッグ: shapeTool の cancel() で仮表示ごと捨てる
- 文字・塗りつぶし: 発火を mouse:down から **mouse:up** に移した。down 発火の
  ままだとピンチの 1 本目の指で発火してしまう。down〜up の間に 2 本目が
  着いたら見送る (suppressTapRef)

1 本指ドラッグでの送りは従来どおり「移動」道具だけ (1 本指は描画のもの)。

## 5. 実装の記録

提案した順 (矢印・枠 → マーカー → 塗りつぶし → モザイク → 拡大) に入れた。
狙いどおり、§1 のドラッグ基盤と docs/35 のラスタ基盤の両方を §3 が流用して
いる (モザイクの新規コードは `pixelate.ts` と切り出しだけ)。

**§4 は計画から方式を変えた。** 「1 本指は描画、2 本指は拡大」の振り分けは
やめ、**「移動」道具を選んでいるときだけジェスチャを受ける**形にした。
振り分け方式だと 2 本目の指が着いた瞬間に描きかけのストロークを捨てる必要が
あり、fabric に中断の API が無いため内部状態 (`_isCurrentlyDrawing` /
`PencilBrush._reset`) に手を入れることになる。道具で分ければその難所が
まるごと消え、内部 API に依存しないで済む。§4-2 に挙げた難所のうち
「振り分け」は、こうして問題ごと無くした。

道具は 10 個 (ペン・マーカー・塗る・矢印・四角・丸・モザイク・消しゴム・
選択・文字・移動) になったので、ツールバーは横スクロールのまま、
効かない道具では色・太さの選択を伏せている。

**未確認**: 実機での操作感は見ていない。特にモザイクの粗さ (§3-1) と
拡大中の描き味 (§4-2) は、実物で詰める余地がある。
