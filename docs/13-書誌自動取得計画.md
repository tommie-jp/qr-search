# ISBN からの書誌自動取得の設計 (Ver2.x)

本のバーコード (ISBN) をスキャンして新規登録すると、書名も著者も自分で
打つことになる。これを openBD から自動で取り込み、**書名・著者・出版社が
入った状態でエディタが開く**ようにする。ここでは設計判断を残す。

前提: スキャン本体は [09-スキャン計画.md](./09-スキャン計画.md)、
新規登録の導線は [10-スキャン新規登録計画.md](./10-スキャン新規登録計画.md)。
10 の §6 で「見送り」としていた項目を、ここで実装に変えた。

## 1. 方針: 自分のサーバが openBD → NDL サーチの順に引く

導線の全体像 (太字が今回足したところ):

```text
ISBN をスキャン
  → /?q=<ISBN> が 0 件 + 「#<ISBN> を新規登録」ボタン
  → /edit/<次番号>?code=<ISBN>
  → **ブラウザが /api/books/<ISBN> を引く**
  → **サーバが openBD → NDL サーチの順に試す**
  → **書名・著者・出版社が入ったエディタ** (取れなければ従来どおり空)
  → 「更新」で作成 (それまで DB には何も書かない)
```

- 取得は**編集ページの 1 箇所に閉じる**。検索ページ (0 件の画面) で引いて
  ボタンに書名を出す案もあるが、書名を URL で持ち回すか二重に取得するかに
  なる。編集ページなら「ISBN で開いた新規ノート」という条件がその場で揃う
- **待たない**。ページはいつもどおり即座に開き、書誌は届いたら後から載る。
  取れなくても導線は止まらない (下の §4)

### 1 つの API では網羅できない (実測)

初版は openBD だけをブラウザから直接引いていたが、実機で 4 冊スキャンして
書名が取れたのは 1 冊だけだった。原因を実測で確かめた。

NDL から採った実在の ISBN に対する **openBD の収録率**:

| 刊行年 | openBD の収録率 |
| --- | --- |
| 1995-1999 | **1/12** |
| 2003-2007 | 2/3 |
| 2012-2016 | 25/25 |
| 2022-2025 | 18/21 |

openBD は新刊・近刊に強い一方、**古い本をほとんど持っていない**。
一方 NDL サーチは納本制度により和書をほぼ網羅していて、上の未収録分も
すべて引けた。そこで **openBD → NDL サーチ** の順に試す。

順番が逆だと、NDL の目録が追いつく前の新刊を落とす。

### サーバを経由する理由 (NDL の口が 2 つあるため)

NDL には口が 2 つあり、**速いほうは CORS ヘッダを返さない**。

| NDL の口 | 形式 | CORS | 未キャッシュ ISBN の応答 |
| --- | --- | --- | --- |
| SRU | XML | `*` | **14〜42 秒** (実測) |
| OpenSearch (RSS) | XML | **無し** | 30ms〜3 秒 (実測) |

ブラウザから直接引けるのは SRU だけだが、その遅さでは使えない
(4 秒で打ち切ると毎回間に合わない)。速い OpenSearch を使うには
サーバを挟むしかないので、`/api/books/[isbn]` を置いた。

openBD も同じルートに寄せた。CORS 的にはブラウザから引けるが、2 つの
API を 2 通りの経路で呼ぶと、失敗の扱いも取得順もクライアントとサーバに
散らばる。ルートは 1 本にして、クライアントは `/api/books/<ISBN>` を
1 回引くだけにする。

### 他の API を採らなかった理由

| サービス | 形式 | キー | 判断 |
| --- | --- | --- | --- |
| openBD | JSON | 不要 | **採用** (新刊に強い) |
| NDL サーチ | XML | 不要 | **採用** (和書をほぼ網羅・古い本に強い) |
| Google Books | JSON | 実質必要 | 見送り (§6) |
| OpenLibrary | JSON | 不要 | 不採用 (和書がほぼ無い。実測で `{}`) |

- **Google Books** は匿名アクセスが 429 になり、実質 API キー運用が要る。
  上の 2 つで洋書も引けている (NDL 所蔵分) ので、取りこぼしが実際に
  苦になってから (§6)

### Next.js の作法から外れる点

Next.js が案内するクライアント側のデータ取得は `use` API か SWR 等だが、
どちらも採らない。`use` API はページの描画を書誌の取得まで待たせる形
(promise を渡して Suspense で待つ) で、**待たない**方針と合わない。
SWR はこの 1 箇所のために依存を増やす価値がない。素の `useEffect` +
`fetch` で引く (実装は
[src/components/useBookPrefill.ts](../src/components/useBookPrefill.ts))。

## 2. 取得した書誌はノート本文にコピーする

表示のたびに引くのではなく、**登録時に 1 回だけ本文へ書き込む**。

| | 本文にコピー (採用) | 表示時に都度取得 |
| --- | --- | --- |
| 全文検索 | **書名・著者で探せる** | ISBN でしか探せない |
| オフライン / API 停止 | 影響なし | 読めない |
| 誤マッチの修正 | 保存前にその場で直せる | 直せない |
| 正本の所在 | メモ本文 (このアプリの流儀) | 外部に分裂する |

このアプリの値打ちは「スキャン → 検索」なので、**書名・著者が全文検索に
乗ること自体が本命の便益**になる。10 で決めた「正本はメモ本文」とも揃う。

## 3. 事前入力の形

```text
リーダブルコード : より良いコードを書くためのシンプルで実践的なテクニック
Boswell, Dustin / Foucher, Trevor / 角, 征典
オーム社 (2012.06)

#9784873115658 #book
```

- **1 行目が書名**。一覧の要約 (`memoSummary`) は 1 行目から取るので、
  要約が自動で書名になる。10 の「タイトルはスキャンしたコード」が
  「タイトルは書名」に格上げされ、そのための実装は要らない
- 空行の下がタグ。空行は自分のメモ (置き場所・感想) を書く場所
- **欠けた項目は行ごと落とす** (著者不明なら著者行を作らない)。
  出版社と刊行年月は片方だけでも成立させる
- 書誌が取れなければ従来どおり `\n\n#<ISBN> #book` のまま

### 著者は連結された文字列から取らない

両方の API に「著者を 1 本の文字列に連結した項目」があるが、どちらも使わない。

- openBD の `summary.author` … `"Boswell,Dustin Foucher,Trevor 角,征典"`
- NDL の `<author>` … `"大野, 浩,大野浩 著"`

**区切り位置を復元できない** (名前自体に空白や読点が入りうる)。代わりに
1 人 1 要素で返る項目を使う。openBD は ONIX の `Contributor`、NDL は
`dc:creator`。Contributor が無いときだけ `summary.author` を 1 人分として使う。

NDL は典拠の生没年を付ける (`"尾田, 栄一郎, 1975-"`) ので、生没年の形を
したものだけ落として openBD 側の見た目 (`"角, 征典"`) に揃える。
`"大野, 浩, テクニカルライター"` のような肩書きは残す。

### XML の値を数値として読ませない

NDL の応答は RSS (XML) で、`fast-xml-parser` は既定で値を数値に変換する。
刊行年月 `1996.10` が数値 `1996.1` になり、**10 月が 1 月に化ける**。
`parseTagValue: false` で必ず文字列として読む (テストで固定した)。

### 書名が `#` で始まるとタグになる (直さない)

`#Twitter社会論` のような書名は 1 行目に置くとタグ記法に一致し、
`items.tags` にノイズのタグが入る。エスケープはしない。

- 保存前のエディタに見えていて、その場で消せる (作成は「更新」を押すまで
  起きない)。本文に `\#` と書く方が読みにくい
- 語中の `#` (`C#`) は直前が非空白なのでタグにならない
  ([06-タグ計画.md](./06-タグ計画.md))。実害が出るのは行頭 `#` の書名だけ

### プロパティ (`キー=値`) にはしない

プロパティ記法は値に空白を許さないので `著者=Boswell, Dustin` は成立しない
([08-プロパティ計画.md](./08-プロパティ計画.md))。普通の本文行でも全文検索には
乗るので足りる。`#book` の特性表で出版社を並べたくなったら
`publisher=` だけ足せばよい (YAGNI)。

### 刊行日は年月まで

版を見分けるのが目的なので日はいらない。形式が API ごとに揺れる
(openBD は `201206` / `20120621` / `2012-06`、NDL は `2012.6` / `1996.2` と
**月を 0 埋めしない**)。「4 桁の年 + 区切り + 1〜2 桁の月」として読み、
月は 0 埋めして `2012.06` に揃える。

数字だけを抜き出して桁数で判断する実装 (初版) だと、NDL の `2012.6` が
5 桁になって刊行年月が丸ごと落ちる。

### 書影 (cover) は入れない

外部の画像 URL を本文に置くとリンク切れする。このアプリの画像は
`/api/images/<uuid>` (DB 保存) が流儀なので、やるなら「取得して
アップロード API に流す」になり配線が増える。欲しくなってから (§6)。

## 4. 途中経過を出す。失敗しても従来の動作に戻すだけ

書誌が載らないだけで、**新規登録の導線は止めない**。

### 状況を表示する (初版で出さなかったのを覆した)

初版は「書誌が載らないことで分かる」として UI に何も出さなかった。実機で
使うと**取得に数秒かかることがあり、待ちと失敗が見分けられなかった**。
エディタの上に 1 行で状況を出す (`MemoEditor` の `BookPrefillNotice`)。

| 状態 | 表示 |
| --- | --- |
| 取得中 | 「書籍情報を取得中…」 + スピナー |
| 取得して入れた | (消す。書名が本文に出るので言わない) |
| 取得したが編集中 | 「書籍情報を取得しましたが、編集中のため反映していません」 |
| どこにも無い | 「書籍情報が見つかりませんでした」 |
| 取得に失敗 | 「書籍情報の取得に失敗しました」(赤) |

- 最初の描画から「取得中」を出す (取得開始を待つと一瞬無表示になる)
- 成功したら消すが、**失敗・未発見の知らせは消さない** (SavedToast のように
  数秒で引っ込めない)。手で書けば済む合図であり、消すと「気づかないうちに
  取得しようとして諦めていた」ことが伝わらない
- `min-h` で 1 行ぶんの高さを確保し、文言が消えてもエディタが動かない
  (打っている最中に入力欄がずれない)。`aria-live="polite"` で読み上げにも伝える
- **エディタの上**に置く。スキャン直後に目が行くのは本文の先頭で、下だと見落とす

### 「見つからない」と「失敗」を混ぜない

- **収録漏れ** … どちらの API も**はっきり「無い」と答えた**ときだけ
  `data: null` を返す。エラーではないので警告も出さない
- **片方の API の失敗** … 警告に残したうえで次の API を試す (握り潰さない)。
  1 つの API が黙り込んでも次に進めるよう、**取得ごとに 8 秒の上限**を持つ
- **失敗したまま見つからなかった** … `lookupBook` が throw し、ルートは 502。
  UI は「取得に失敗しました」と伝える。ここを `null` に混ぜると
  「見つかりませんでした」と**断定**することになるが、実際には訊けていない
  だけで、その本が無いことは分かっていない (もう一度開けば取れることがある)
- **ページ離脱** … `abort("unmount")` で中断。想定内なので警告も出さない

外部データは型を信用しない。応答が JSON/XML として何であっても
(配列でない・null・壊れた XML・HTML が来ても) パーサが `null` か部分的な
書誌を返し、例外で編集ページを壊さない。書名が無いものは `null` にする
(本文の 1 行目に置くものが無いなら、事前入力を書き換える意味がない)。

ISBN は API ルートの入り口で必ず検算する (`isIsbn`)。13 桁の数字だけを
外部 API の URL に載せることになり、書籍でないコードで NDL を叩くこともなくなる。

## 5. 打ち始めていたら書き換えない

**この機能で最悪の事故は「遅れて届いた書誌が手書きを消す」**こと。
本文の差し替えは、現在値が事前入力のままのときだけ行う
(`setMemo(prev => prev === pristine ? ... : prev)`)。

- 判定に現在値を使わず関数形の更新にするのは、`value` を effect の依存に
  すると打鍵のたびに再取得が走るため
- 差し替えは CodeMirror の履歴に 1 手として積まれるので、要らなければ
  「元に戻す」で事前入力だけの状態に戻せる

### 副作用: 未保存の確認が出るようになる

`UnsavedGuard` はマウント時の内容を「保存済み」として覚えるので、後から
届いた書誌は**未保存の変更として扱われる**。ISBN をスキャンして開いたまま
リロード・タブを閉じると確認ダイアログが出る (ページ内リンクでは出ない。
[11-アプリ的UIUX計画.md](./11-アプリ的UIUX計画.md) §2-2)。

書誌は実際に失われる内容なので、引き止めるのは筋が通っている。
`UnsavedGuard` に「取得後に基準を取り直す」配線は足さない。

## 6. 段階と見送り

- **Phase 1 (実装済み)**: openBD をクライアントから直接引いて事前入力に載せる
- **Phase 2 (実装済み)**: 実機で 4 冊中 1 冊しか取れなかったため、
  `/api/books/[isbn]` を置いて **openBD → NDL サーチ**の順に引く。
  古い本が取れるようになった (§1 の実測)
- **見送り: 書誌のキャッシュ**。同じ ISBN を 2 度スキャンすることは普通なく
  (1 度目でノートになる)、当たらないキャッシュを置いても仕方がない。
  外部 API への負荷も 1 冊につき 1〜2 リクエストで収まっている
- **見送り: Google Books** (§1)。API キーの運用が要る割に、いまの 2 つで
  洋書も引けている
- **見送り: 書影の取り込み** (§3)
- **見送り: 既存ノートの「書誌を取得」ボタン**。まず新規登録の導線だけで運用する
- **見送り: ISBN-10 (10 桁)** の手入力。スキャンで来るのは常に 13 桁
  (古い本も 13 桁で引ける。NDL 側が ISBN-10 の記録と照合してくれる)

## 7. テスト

- 純関数 (vitest):
  - `parseOpenBdResponse` … 実応答から書名・著者・出版社 / ONIX 優先と
    `summary.author` への退避 / `[null]`・空配列・配列でない応答 /
    書名なし / 文字列でない値
  - `parseNdlSearchResponse` … 実応答から書名・著者・出版社・刊行年月 /
    `dcterms:issued` 優先と `dc:date` への退避 / **10 月が 1 月に化けない** /
    生没年を落とし肩書きは残す / 同一 ISBN に 2 件あるとき / 壊れた XML・HTML
  - `formatPubdate` … `201206`・`20120621`・`2012-06`・**`2012.6`** → `2012.06`、
    年だけ・読めない値
  - `scanRegisterMemo(code, book)` … 書誌あり / なし / 項目が欠けたとき
- ブラウザでの通し確認 (Playwright MCP、実 API):
  - **1996 年の本 (openBD 未収録)** → NDL から書名・著者・出版社が入る
    (この機能の目的そのもの)
  - 2012 年の本 → 従来どおり openBD から入る
  - JAN → `/api/books/` を引かない (0 回)
  - 収録漏れの ISBN (チェックデジットは正当) → 事前入力のまま導線は生きる
  - **応答を保留 → 打鍵 → 解放** して、届いた書誌が手書きを消さないこと
    (この確認は「応答が 200 で届いた」ことまで見ないと意味がない)
- API ルート … JAN・チェックデジット違い・数字でない値はすべて 400

`useBookPrefill` に単体テストは置かない。vitest の環境が `node` で、
コンポーネントのテストは `renderToStaticMarkup` による静的描画に限られており、
effect と fetch を動かす土台 (jsdom + Testing Library) がこのリポジトリに無い。
フックの中身は「取れたら差し替える / 打っていたら差し替えない」の 2 択で、
そこは上のブラウザ確認で実物を通して見ている。

### 調査のときは NDL を叩きすぎない

収録率の調査でまとめてリクエストを投げたら **429** が返り、その後しばらく
応答が 14〜42 秒に落ちた (SRU)。NDL は具体的な上限を公開しておらず、
「継続して大量のアクセスがある場合は遮断することがある」としている。
実運用 (1 冊 1〜2 リクエスト) では問題にならないが、調査でループを回すときは
間隔を空ける。
