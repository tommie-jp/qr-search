# ISBN からの書誌自動取得の設計 (Ver2.x)

本のバーコード (ISBN) をスキャンして新規登録すると、書名も著者も自分で
打つことになる。これを openBD から自動で取り込み、**書名・著者・出版社が
入った状態でエディタが開く**ようにする。ここでは設計判断を残す。

前提: スキャン本体は [09-スキャン計画.md](./09-スキャン計画.md)、
新規登録の導線は [10-スキャン新規登録計画.md](./10-スキャン新規登録計画.md)。
10 の §6 で「見送り」としていた項目を、ここで実装に変えた。

## 1. 方針: スマホのブラウザから openBD を直接引く

導線の全体像 (太字が今回足したところ):

```text
ISBN をスキャン
  → /?q=<ISBN> が 0 件 + 「#<ISBN> を新規登録」ボタン
  → /edit/<次番号>?code=<ISBN>
  → **ブラウザが openBD を直接 fetch (サーバを経由しない)**
  → **書名・著者・出版社が入ったエディタ** (取れなければ従来どおり空)
  → 「更新」で作成 (それまで DB には何も書かない)
```

- **サーバを経由しない**。openBD は `access-control-allow-origin: *` を
  返すので CORS で弾かれず、API キーも要らない。VPS 側のコードも設定も
  増えず、アプリが落ちていない限り必ず動く
- 取得は**編集ページの 1 箇所に閉じる**。検索ページ (0 件の画面) で引いて
  ボタンに書名を出す案もあるが、書名を URL で持ち回すか二重に取得するかに
  なる。編集ページなら「ISBN で開いた新規ノート」という条件がその場で揃う
- **待たない**。ページはいつもどおり即座に開き、書誌は届いたら後から載る。
  取れなくても導線は止まらない (下の §4)

### API の選定

実測して選んだ (2026-07-16 時点)。

| サービス | 形式 | キー | CORS | 判断 |
| --- | --- | --- | --- | --- |
| openBD | JSON | 不要 | `*` | **採用** |
| Google Books | JSON | 実質必要 | 可 | Phase 2 (§6) |
| NDL サーチ | XML のみ | 不要 | **無し** | 不採用 |
| OpenLibrary | JSON | 不要 | 可 | 不採用 (和書がほぼ無い) |

- **NDL サーチ**は CORS ヘッダを返さないので、クライアントから直接引けない
  (サーバ経由にすれば使えるが、それは方針を捨てることになる)。応答も XML
- **OpenLibrary** は実測で和書が空 (`{}`)。この DB は和書が中心
- **Google Books** はサーバ IP からの匿名アクセスが 429 になり、実質 API キー
  運用が要る。openBD の収録漏れが実際に苦になってから (§6)

### Next.js の作法から外れる点

Next.js が案内するクライアント側のデータ取得は `use` API か SWR 等だが、
どちらも採らない。`use` API は「サーバで取得して promise を渡す」形なので
**サーバ経由になってしまい目的と正反対**。SWR はこの 1 箇所のために
依存を増やす価値がない。素の `useEffect` + `fetch` で引く
(実装は [src/components/useBookPrefill.ts](../src/components/useBookPrefill.ts))。

## 2. 取得した書誌はノート本文にコピーする

表示のたびに引くのではなく、**登録時に 1 回だけ本文へ書き込む**。

| | 本文にコピー (採用) | 表示時に都度取得 |
| --- | --- | --- |
| 全文検索 | **書名・著者で探せる** | ISBN でしか探せない |
| オフライン / API 停止 | 影響なし | 読めない |
| 誤マッチの修正 | 保存前にその場で直せる | 直せない |
| 正本の所在 | メモ本文 (このアプリの流儀) | 外部に分裂する |

このアプリの値打ちは「スキャン → 検索」なので、**書名・著者が全文検索に
乗ること自体が本命の便益**になる。10 で決めた「正本はメモ本文」とも揃う。

## 3. 事前入力の形

```text
リーダブルコード : より良いコードを書くためのシンプルで実践的なテクニック
Boswell, Dustin / Foucher, Trevor / 角, 征典
オーム社 (2012.06)

#9784873115658 #book
```

- **1 行目が書名**。一覧の要約 (`memoSummary`) は 1 行目から取るので、
  要約が自動で書名になる。10 の「タイトルはスキャンしたコード」が
  「タイトルは書名」に格上げされ、そのための実装は要らない
- 空行の下がタグ。空行は自分のメモ (置き場所・感想) を書く場所
- **欠けた項目は行ごと落とす** (著者不明なら著者行を作らない)。
  出版社と刊行年月は片方だけでも成立させる
- 書誌が取れなければ従来どおり `\n\n#<ISBN> #book` のまま

### 著者は ONIX から取る (summary.author は使わない)

`summary.author` は `"Boswell,Dustin Foucher,Trevor 角,征典"` と著者名を
空白で連結した文字列で、**区切り位置を復元できない** (名前自体に空白が
入りうる)。ONIX の `Contributor` は 1 人 1 要素の配列なので確実に分けられる。
Contributor が無いときだけ `summary.author` を 1 人分として使う。

### 書名が `#` で始まるとタグになる (直さない)

`#Twitter社会論` のような書名は 1 行目に置くとタグ記法に一致し、
`items.tags` にノイズのタグが入る。エスケープはしない。

- 保存前のエディタに見えていて、その場で消せる (作成は「更新」を押すまで
  起きない)。本文に `\#` と書く方が読みにくい
- 語中の `#` (`C#`) は直前が非空白なのでタグにならない
  ([06-タグ計画.md](./06-タグ計画.md))。実害が出るのは行頭 `#` の書名だけ

### プロパティ (`キー=値`) にはしない

プロパティ記法は値に空白を許さないので `著者=Boswell, Dustin` は成立しない
([08-プロパティ計画.md](./08-プロパティ計画.md))。普通の本文行でも全文検索には
乗るので足りる。`#book` の特性表で出版社を並べたくなったら
`publisher=` だけ足せばよい (YAGNI)。

### 刊行日は年月まで

版を見分けるのが目的なので日はいらない。openBD の `pubdate` は
`201206` / `20120621` / `2012-06` と形式が揺れるため、区切り文字は当てにせず
数字だけを見て `2012.06` にする。

### 書影 (cover) は入れない

外部の画像 URL を本文に置くとリンク切れする。このアプリの画像は
`/api/images/<uuid>` (DB 保存) が流儀なので、やるなら「取得して
アップロード API に流す」になり配線が増える。欲しくなってから (§6)。

## 4. 失敗したときは黙って従来の動作に戻す

書誌が載らないだけで、**新規登録の導線は止めない**。

- **収録漏れ** … openBD は版元ドットコム系が中心で、実在する ISBN でも
  `[null]` が返ることがある (実測)。エラーではないので警告も出さない
- **通信失敗・タイムアウト (5 秒)** … `console.warn` に残す。UI には出さない
  (書誌が載らないことで分かるし、出したところで手で書く以外にできることがない)
- **ページ離脱** … `abort("unmount")` で中断。これは想定内なので警告も出さない

外部データは型を信用しない。応答が JSON として何であっても
(配列でない・null・文字列が来ても) `parseOpenBdResponse` が `null` か
部分的な書誌を返し、例外で編集ページを壊さない。書名が無いものは `null`
にする (本文の 1 行目に置くものが無いなら、事前入力を書き換える意味がない)。

## 5. 打ち始めていたら書き換えない

**この機能で最悪の事故は「遅れて届いた書誌が手書きを消す」**こと。
本文の差し替えは、現在値が事前入力のままのときだけ行う
(`setMemo(prev => prev === pristine ? ... : prev)`)。

- 判定に現在値を使わず関数形の更新にするのは、`value` を effect の依存に
  すると打鍵のたびに再取得が走るため
- 差し替えは CodeMirror の履歴に 1 手として積まれるので、要らなければ
  「元に戻す」で事前入力だけの状態に戻せる

### 副作用: 未保存の確認が出るようになる

`UnsavedGuard` はマウント時の内容を「保存済み」として覚えるので、後から
届いた書誌は**未保存の変更として扱われる**。ISBN をスキャンして開いたまま
リロード・タブを閉じると確認ダイアログが出る (ページ内リンクでは出ない。
[11-アプリ的UIUX計画.md](./11-アプリ的UIUX計画.md) §2-2)。

書誌は実際に失われる内容なので、引き止めるのは筋が通っている。
`UnsavedGuard` に「取得後に基準を取り直す」配線は足さない。

## 6. 段階と見送り

- **Phase 1 (実装済み)**: openBD をクライアントから引いて事前入力に載せる
- **見送り: Google Books のフォールバック**。openBD の収録漏れ (洋書・古い本)
  が実際に苦になったら。リファラ制限付きの無料キーをクライアントに置く形になる
- **見送り: 書影の取り込み** (§3)
- **見送り: 既存ノートの「書誌を取得」ボタン**。まず新規登録の導線だけで運用する
- **見送り: ISBN-10 (10 桁)** の手入力。スキャンで来るのは常に 13 桁

## 7. テスト

- 純関数 (vitest):
  - `parseOpenBdResponse` … 実応答から書名・著者・出版社・刊行年月 /
    ONIX 優先と `summary.author` への退避 / `[null]`・空配列・配列でない応答 /
    書名なし / 文字列でない値
  - `formatPubdate` … `201206`・`20120621`・`2012-06` → `2012.06`、
    年だけ・読めない値
  - `scanRegisterMemo(code, book)` … 書誌あり / なし / 項目が欠けたとき
- ブラウザでの通し確認 (Playwright MCP、実 API):
  - ISBN → `api.openbd.jp` へ**ブラウザから直接** GET が飛び (200)、
    書名入りの本文になる
  - 収録漏れの ISBN (チェックデジットは正当) → 事前入力のまま導線は生きる
  - JAN → openBD を引かない
  - **応答を保留 → 打鍵 → 解放** して、届いた書誌が手書きを消さないこと
    (この確認は「応答が 200 で届いた」ことまで見ないと意味がない)

`useBookPrefill` に単体テストは置かない。vitest の環境が `node` で、
コンポーネントのテストは `renderToStaticMarkup` による静的描画に限られており、
effect と fetch を動かす土台 (jsdom + Testing Library) がこのリポジトリに無い。
フックの中身は「取れたら差し替える / 打っていたら差し替えない」の 2 択で、
そこは上のブラウザ確認で実物を通して見ている。
