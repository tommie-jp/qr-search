# 18. ログイン計画

ログインしなくてもヘッダ(画面上部の帯)を表示し、ユーザ名の代わりに
「ログイン」ボタンを出す。中身(ノート)は従来どおりログインしないと見えない。

実装済み (2026-07-17)。

## 1. なぜ変えたか

これまで認証はエッジ(本番=nginx、ローカル再現=Caddy)の Basic 認証だった。
サイトに触れた瞬間ブラウザの認証ダイアログが出て、通るまでアプリの HTML は
1 バイトも返らない。つまり **ヘッダを出すにはエッジの認証を外すしかない**。

外した以上、401 を返す誰かが要る。それをアプリへ移した。

## 2. 構成

```text
ブラウザ ──> nginx (認証しない / ボディ上限 / /login のレート制限)
              └─> Next.js
                   ├─ src/proxy.ts        ... 門番 (既定で閉じる)
                   ├─ src/app/login/      ... 401 + WWW-Authenticate を返す口
                   └─ src/lib/auth.ts     ... ヘッダ解析 + bcrypt 照合
```

| ファイル | 役割 |
| --- | --- |
| `src/lib/auth.ts` | Authorization ヘッダの解析と bcrypt 照合。next/headers に依存しない |
| `src/lib/session.ts` | リクエストからユーザを取る (`currentUser` / `requireUser`) |
| `src/lib/publicPaths.ts` | ログイン不要なパスの**正本** |
| `src/lib/loginRedirect.ts` | 戻り先 (`next`) の検算とログイン URL 組み立て |
| `src/proxy.ts` | 門番。Next.js 16 で middleware から改称 |
| `src/app/login/route.ts` | ログインの口 |
| `src/app/login-required/page.tsx` | 未ログイン時に本文へ出す案内 |
| `src/components/LoginButton.tsx` | ヘッダと案内に出すボタン |

## 3. ログインの流れ

1. 未ログインで `/item/ABC123` を開く
2. `proxy.ts` が `/login-required` へ **rewrite**(redirect ではない)。
   アドレス欄は `/item/ABC123` のまま、ヘッダは出て、本文だけが案内になる
3. 「ログイン」ボタン → `/login?next=/item/ABC123` へ**普通の画面遷移**
4. `/login` が 401 + `WWW-Authenticate: Basic` → ブラウザがダイアログを出す
5. 通れば 303 で `next` へ戻る

補足:

- ダイアログはブラウザ自身の画面遷移でしか出ない。`fetch` や `router.push`
  では 401 がただの失敗レスポンスになる (`LoginButton.tsx` のコメント参照)
- 保護空間は `/login` のディレクトリ = `/` 配下。一度通ればブラウザが
  サイト全体へ資格情報を自分から送る (RFC 7617)。だからログイン用の
  ページを別に持たなくてよい
- **ログアウトはない**。Basic 認証はブラウザを閉じるまで資格情報が残る

## 4. 防御は二層 (片方だけにしない)

Next.js の authentication ガイドが明示しているとおり、proxy は
「楽観的な検査」であって唯一の砦にしてはいけない。

| 層 | 場所 | 未ログインの扱い |
| --- | --- | --- |
| 門番 | `src/proxy.ts` | 画面の GET → 案内へ rewrite / 他 → 401 |
| データ側 | `requireUser()` / `denyUnlessLoggedIn()` | 投げる / 401 |

Server Action は `requireUser()`、route handler は `denyUnlessLoggedIn()`
(応答を組み立てる場所なので、投げて 500 にするより 401 を返す)。

Server Action は「画面のボタン」ではなく誰でも叩ける POST の口 (id さえ
判れば画面を通さず呼べる)。だから書き込む側でも必ず確かめる。

## 5. 公開パス

`src/lib/publicPaths.ts` が正本。かつて nginx の `auth_basic off` と
Caddy の `@pwa_public` に散らしていたものの移設先。

- PWA (`/manifest.webmanifest` とアイコン群) …
  ブラウザが Authorization なしで取りに行くため。閉じると
  「インストール可能」と判定されない
- 使い方の docs (`/docs/search`, `/docs/memo`) … ノートの中身は出ない
- ログインの口 (`/login`, `/login-required`)

アイコンの一覧は `manifest.ts` から**導出**する。手で並べると増えたときに
書き足し忘れる (実際 `icon-512.png` を落として PWA を壊しかけた)。
`publicPaths.test.ts` が「manifest が要求するものは必ず公開」を縛っている。

## 6. 落とし穴: bcrypt ハッシュを .env に生で書くと壊れる

**最重要**。実装中に踏んで原因究明に一番時間を使った。

bcrypt ハッシュは `$2b$10$…` と `$` を含む。これを `.env` に生で書くと、
Next.js の env 読み込み (`@next/env` → dotenv-expand) が `$2b` / `$10` /
`$INHJ…` を変数参照とみなして展開し、値が壊れる。

実測: **60 文字のハッシュが 6 文字になった**(末尾だけ残る)。

さらにタチが悪い点:

- シェルや docker compose で正しい値を渡していても、`.env` に同じキーが
  あるだけで壊される
- 本番 (Docker) は `.dockerignore` が `.env` を除くため**壊れない**
  → 「ローカルだけ誰もログインできない」という気づきにくい形で出る
- エラーも警告も出ない。ただ静かに 401 になる

### 採った対策: base64 で持つ

`BASIC_AUTH_HASH_B64` に **bcrypt ハッシュの base64** を入れる。
base64 の文字種に `$` は無いので、`.env`・compose・Docker のどの経路でも
同じ値がそのまま届く。

生成:

```bash
docker run --rm caddy:2-alpine caddy hash-password --plaintext '<パスワード>' \
  | tr -d '\n' | base64 -w0
```

### 採らなかった案

- **`\$` エスケープ**: compose は `\$` を解釈しないため、エスケープした
  `.env` の値がそのままコンテナへ渡り、今度は本番が壊れる。
  片方を直すと他方が壊れるので、両立しない

## 7. 設定漏れは「閉じる」側へ倒す

`BASIC_AUTH_USER` / `BASIC_AUTH_HASH_B64` が無ければ**誰もログインできない**
(全ページが「ログインが必要です」になる)。

逆に倒すと、env を書き忘れた本番が認証なしで開く。「動かない」はすぐ
気づけるが、「素通し」は気づけない。`appEnv.ts` (非本番をピンクに塗る判定)
と同じ考え方。

compose は `:?` で必須にしてあるので、設定漏れは起動時に落ちる。

## 8. bcrypt のコストとレート制限

`caddy hash-password` の bcrypt はコスト 14 で、1 回の照合に 1 秒近くかかる。

- **キャッシュ**: 一度通ったヘッダは覚える (`auth.ts`)。鍵は「設定 + ヘッダ」の
  組で、パスワードを変えれば古い資格情報は通らない。失敗は覚えない
  (当てずっぽうで上限まで埋めて正しいエントリを追い出せてしまうため)。
  上限つき (メモリを食い潰させない)
- **レート制限**: 認証がアプリへ移り、総当たりが Node と bcrypt まで届くように
  なった。nginx の `limit_req` で `/login` だけ 10r/m に絞る
