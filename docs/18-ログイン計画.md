# 18. ログイン計画

ログインしなくてもヘッダ(画面上部の帯)を表示し、ユーザ名の代わりに
「ログイン」ボタンを出す。中身(ノート)は従来どおりログインしないと見えない。

実装済み (2026-07-17)。

> **続き**: 2026-07-19 に認証手段としてパスキー (WebAuthn) を足した
> ([29-パスキー計画.md](./29-パスキー計画.md))。普段のログインはそちらへ移り、
> ここに書いた Basic 認証は「パスキー登録の門番」と「全端末のパスキーを
> 失ったときの復旧経路」として残っている。§3 のログインの流れと §8 の
> bcrypt の遅さは、いまは**初回登録と復旧のときだけ**通る道。
> §3 の「ログアウトはない」は 29 §4 のセッションで解消した。

## 1. なぜ変えたか

これまで認証はエッジ(本番=nginx、ローカル再現=Caddy)の Basic 認証だった。
サイトに触れた瞬間ブラウザの認証ダイアログが出て、通るまでアプリの HTML は
1 バイトも返らない。つまり **ヘッダを出すにはエッジの認証を外すしかない**。

外した以上、401 を返す誰かが要る。それをアプリへ移した。

## 2. 構成

```text
ブラウザ ──> nginx (認証しない / ボディ上限 / /login のレート制限)
              └─> Next.js
                   ├─ src/proxy.ts        ... 門番 (既定で閉じる)
                   ├─ src/app/login/      ... 401 + WWW-Authenticate を返す口
                   └─ src/lib/auth.ts     ... ヘッダ解析 + bcrypt 照合
```

| ファイル | 役割 |
| --- | --- |
| `src/lib/auth.ts` | Authorization ヘッダの解析と bcrypt 照合。next/headers に依存しない |
| `src/lib/session.ts` | リクエストからユーザを取る (`currentUser` / `requireUser`) |
| `src/lib/publicPaths.ts` | ログイン不要なパスの**正本** |
| `src/lib/loginRedirect.ts` | 戻り先 (`next`) の検算とログイン URL 組み立て |
| `src/proxy.ts` | 門番。Next.js 16 で middleware から改称 |
| `src/app/login/route.ts` | ログインの口 |
| `src/app/login-required/page.tsx` | 未ログイン時に本文へ出す案内 |
| `src/components/LoginButton.tsx` | ヘッダと案内に出すボタン |

## 3. ログインの流れ

1. 未ログインで `/item/ABC123` を開く
2. `proxy.ts` が `/login-required` へ **rewrite**(redirect ではない)。
   アドレス欄は `/item/ABC123` のまま、ヘッダは出て、本文だけが案内になる
3. 「ログイン」ボタン → `/login?next=/item/ABC123` へ**普通の画面遷移**
4. `/login` が 401 + `WWW-Authenticate: Basic` → ブラウザがダイアログを出す
5. 通れば 303 で `next` へ戻る

補足:

- ダイアログはブラウザ自身の画面遷移でしか出ない。`fetch` や `router.push`
  では 401 がただの失敗レスポンスになる (`LoginButton.tsx` のコメント参照)
- 保護空間は `/login` のディレクトリ = `/` 配下。一度通ればブラウザが
  サイト全体へ資格情報を自分から送る (RFC 7617)。だからログイン用の
  ページを別に持たなくてよい
- **ログアウトはない**。Basic 認証はブラウザを閉じるまで資格情報が残る

## 4. 防御は二層 (片方だけにしない)

Next.js の authentication ガイドが明示しているとおり、proxy は
「楽観的な検査」であって唯一の砦にしてはいけない。

| 層 | 場所 | 未ログインの扱い |
| --- | --- | --- |
| 門番 | `src/proxy.ts` | 画面の GET → 案内へ rewrite / 他 → 401 |
| データ側 | `requireUser()` / `denyUnlessLoggedIn()` | 投げる / 401 |

Server Action は `requireUser()`、route handler は `denyUnlessLoggedIn()`
(応答を組み立てる場所なので、投げて 500 にするより 401 を返す)。

Server Action は「画面のボタン」ではなく誰でも叩ける POST の口 (id さえ
判れば画面を通さず呼べる)。だから書き込む側でも必ず確かめる。

## 5. 公開パス

`src/lib/publicPaths.ts` が正本。かつて nginx の `auth_basic off` と
Caddy の `@pwa_public` に散らしていたものの移設先。

- PWA (`/manifest.webmanifest` とアイコン群) …
  ブラウザが Authorization なしで取りに行くため。閉じると
  「インストール可能」と判定されない
- 使い方の docs (`/docs/search`, `/docs/memo`) … ノートの中身は出ない
- ログインの口 (`/login`, `/login-required`)

アイコンの一覧は `manifest.ts` から**導出**する。手で並べると増えたときに
書き足し忘れる (実際 `icon-512.png` を落として PWA を壊しかけた)。
`publicPaths.test.ts` が「manifest が要求するものは必ず公開」を縛っている。

## 6. 落とし穴: bcrypt ハッシュを .env に生で書くと壊れる

**最重要**。実装中に踏んで原因究明に一番時間を使った。

bcrypt ハッシュは `$2b$10$…` と `$` を含む。これを `.env` に生で書くと、
Next.js の env 読み込み (`@next/env` → dotenv-expand) が `$2b` / `$10` /
`$INHJ…` を変数参照とみなして展開し、値が壊れる。

実測: **60 文字のハッシュが 6 文字になった**(末尾だけ残る)。

さらにタチが悪い点:

- シェルや docker compose で正しい値を渡していても、`.env` に同じキーが
  あるだけで壊される
- 本番 (Docker) は `.dockerignore` が `.env` を除くため**壊れない**
  → 「ローカルだけ誰もログインできない」という気づきにくい形で出る
- エラーも警告も出ない。ただ静かに 401 になる

### 採った対策: base64 で持つ

`BASIC_AUTH_HASH_B64` に **bcrypt ハッシュの base64** を入れる。
base64 の文字種に `$` は無いので、`.env`・compose・Docker のどの経路でも
同じ値がそのまま届く。

生成:

```bash
npm run hash-password
```

パスワードをシェルのコマンドラインに書かないこと。`--plaintext '<パスワード>'`
に貼る方式は、パスワードに `$` や `!` や `'` が入っているとシェルが別の
文字列に変えてしまい、**自分が思っているのとは違うパスワードのハッシュ**が
できる。しかもログインしようとするまで気づけない。
`npm run hash-password` は端末から直接読むのでシェルを通らない
(`scripts/hashPassword.mjs`)。

### 採らなかった案

- **`\$` エスケープ**: compose は `\$` を解釈しないため、エスケープした
  `.env` の値がそのままコンテナへ渡り、今度は本番が壊れる。
  片方を直すと他方が壊れるので、両立しない

## 7. 設定漏れは「閉じる」側へ倒す

`BASIC_AUTH_USER` / `BASIC_AUTH_HASH_B64` が無ければ**誰もログインできない**
(全ページが「ログインが必要です」になる)。

逆に倒すと、env を書き忘れた本番が認証なしで開く。「動かない」はすぐ
気づけるが、「素通し」は気づけない。`appEnv.ts` (非本番をピンクに塗る判定)
と同じ考え方。

compose は `:?` で必須にしてあるので、設定漏れは起動時に落ちる。

## 8. bcrypt のコストとレート制限

bcrypt はわざと遅い。コストが 1 増えるごとに所要時間はきっちり倍になる
(実測でも 2.0 倍)。**vps2 は非力で、ローカルの約 10 倍遅い**:

| コスト | ローカル | vps2 |
| --- | --- | --- |
| 14 (`caddy hash-password` の既定) | 0.68 秒 | **7.0 秒** |
| 12 (現在) | 0.17 秒 | 約 1.75 秒 |

最初コスト 14 で運用したところ、vps2 で 1 回 7 秒かかり実用にならなかった
(下の「実際に起きたこと」参照)。**コスト 12 に下げた**。12 より下げると
総当たりに弱くなるので、これ以上は下げない。

- **キャッシュ**: 一度通ったヘッダは覚える (`auth.ts`)。鍵は「設定 + ヘッダ」の
  組で、パスワードを変えれば古い資格情報は通らない。失敗は覚えない
  (当てずっぽうで上限まで埋めて正しいエントリを追い出せてしまうため)。
  上限つき (メモリを食い潰させない)。
  ブラウザは一度通れば毎リクエスト同じヘッダを送るので、遅いのは最初の 1 回だけ
- **レート制限**: 認証がアプリへ移り、総当たりが Node と bcrypt まで届くように
  なった。nginx の `limit_req` で `/login` だけ 30r/m (burst=10 nodelay) に絞る

### 実際に起きたこと (コスト 14 + 10r/m のとき)

デプロイ直後、正しいパスワードでもログインできなかった。原因は 2 つの合わせ技:

1. 1 回の照合に 7 秒
2. 1 回のログインで `/login` は **2 往復**する (未認証の 401 → 資格情報つきの
   再送)。10r/m だと 5 回のログイン試行で頭打ちになり、`limit_req` の既定
   (nodelay なし) は超過分を**待たせる**ため、数十秒無反応に見えた

「しばらくしたら入れた」= レート制限の窓が空いただけだった。
なお設定自体は正しく、データは一度も露出していない (未ログインは
`/login-required` の案内しか返らない)。

切り分けに効いたのは応答時間の測定:

| リクエスト | 時間 | 意味 |
| --- | --- | --- |
| ヘッダ無し | 0.05 秒 | 設定を見て即断 |
| `tommie` + 誤パスワード | 7.0 秒 | **bcrypt が動いている** = 配線は正常 |
| 存在しないユーザ | 0.03 秒 | 設計どおり bcrypt を回さず即断 |

「誤パスワードで 7 秒かかる」ことが、Authorization ヘッダが nginx を越えて
アプリまで届き、ユーザ名も一致し、bcrypt 照合まで到達している証拠になった。

## 9. Basic 認証は SameSite が効かない (CSRF)

**ログイン検査だけでは足りない。** ここで採った Basic 認証は Cookie を
使わないので、Cookie の `SameSite` による守りが最初から無い。ブラウザは
一度ログインすると、**どのページから出た要求でもこのオリジン宛なら認証情報を
付け直す**。つまり悪意あるページに

```html
<img src="https://qr.tommie.jp/api/books/9784873115658">
```

と書いてあるだけで、それを開いた本人のブラウザが `/api/books` を本人として
叩く。断れるのは「ログインしていない相手」だけで、**ログイン済みの本人が
第三者に動かされる**この形は素通りする。

### 何が困るのか

事前入力の 2 つの口は、ただ読むだけの口ではない。

| 口 | 動かされると起きること |
| --- | --- |
| `/api/books/<ISBN>` | 書影が `images` に溜まる (孤児)。楽天のクォータが減る |
| `/api/products/<JAN>` | Yahoo! の利用枠が減る |

`/api/books` は [19-書影取得計画.md](./19-書影取得計画.md) で**書き込みの
副作用がある GET** になったため、影響が「外部 API の踏み台」から
「DB が太る + 課金/クォータを焚かれる」に上がった。`/api/products` の
利用枠の穴は前からあった。

### Origin では見分けられない (Fetch Metadata を見る)

アップロード (POST) は `Origin` で見分けている ([uploads.ts](../src/lib/uploads.ts))
が、**GET の `<img>` は `Origin` を送らない** (単純リクエスト)。そこで
`Sec-Fetch-Site` を見る。ブラウザが自分で付けるヘッダで、`Sec-` で始まる名前は
禁止ヘッダ名なので **JS からは書き換えられない**。

| `Sec-Fetch-Site` | 誰が出したか | 扱い |
| --- | --- | --- |
| `same-origin` | 自分のページの fetch (`usePrefill`) | 通す |
| `none` | 人がアドレス欄・ブックマークで開いた | 通す (下記) |
| `cross-site` | **第三者のページの `<img>` など** | **断る (403)** |
| `same-site` | 別のサブドメイン | 断る (呼ぶのは自分のページだけ) |
| 無し | curl・古いブラウザ | 通す (下記) |

ヘッダが無いときに通すのは、**防ぎたい相手が必ずこのヘッダを送る**から。
ここを閉じても、鍵を持っている相手は自分でヘッダを付けられるので守りは
増えず、手元の curl 確認だけが不便になる。守れないのは Fetch Metadata を
送らない古いブラウザ (Safari 16.3 以前など) を使っている場合だけで、
それは承知のうえで見送る。

`none` は「開かせた元のページが無い」という意味で、**Web ページからは
作れない**。ただし仕様上ネイティブアプリ (メールや Slack) の中で押した
リンクも `none` になりうるので、「第三者に起こせない」とまでは言えない。
それでも `<img>` のように黙って大量に叩く形にはならず (人が 1 回押す)、
最悪でも外部 API 1 回と孤児 1 件なので通す側に倒す。

実ブラウザで確かめた値 (自分のページの fetch → `same-origin` /
第三者ページの `<img>` → `cross-site`)。**どちらも `Origin` は送っていない**
ので、POST と同じ Origin 検査ではこの穴は塞げない。

判定は [crossSite.ts](../src/lib/crossSite.ts)、断り方は
`denyCrossSite()` ([apiAuth.ts](../src/lib/apiAuth.ts))。§4 の二層と同じで、
ログイン検査と**並べて**両方通す:

```ts
const denied = (await denyUnlessLoggedIn()) ?? denyCrossSite(request)
if (denied) return denied
```

### 見送ったもの

- **レート制限**。クロスサイトを塞げば「悪意あるページに開かせて大量に
  焚く」経路は消える。本人が自分のブラウザで叩く分には上限は要らない
  (1 冊 1 回)。§8 のログインのレート制限 (Caddy) とは別の話
- **同一 ISBN の書影を重複保存しない冪等化**。`images` に ISBN の列が要り
  migration になる。孤児は元々許容していて (12-ゴミ箱計画.md)、
  能動的に焚く経路を塞いだ以上、増え方は手で貼る画像と同じ
- **`/api/images` の POST への Fetch Metadata 検査**。あちらは `Origin` を
  見ており、`<img>` では POST できない。二重にはしない
- **`/api/images/<name>` の GET (画像の配信) への検査**。ここは意図的に
  入れていない。副作用の無い読み出しで、外部 API も叩かず DB にも書かない。
  名前はサーバが振った UUID なので第三者のページには作れず、仮に `<img>` で
  読まれても中身はクロスオリジンでは読めない。この節が塞ぎたいのは
  「**副作用のある GET**」であって、そこに当たらない

## 10. 運用メモ

- `/etc/nginx/.htpasswd-qr` はもう読まれない。パスワードの正本は
  vps2 の `.env` の `BASIC_AUTH_USER` / `BASIC_AUTH_HASH_B64`
- **パスワードの変え方** (イメージの再ビルドもデプロイも不要):

  ```bash
  npm run hash-password          # 出力の BASIC_AUTH_HASH_B64=... 行を控える
  # ローカルの .env を書き換える
  # vps2 も同じ値に:
  D=41-QR-search/qr-search
  NEW='BASIC_AUTH_HASH_B64=<値>'
  ssh vps2 "sed -i \"s|^BASIC_AUTH_HASH_B64=.*|$NEW|\" $D/.env \
    && cd $D && docker compose up -d app"
  ```

  古い資格情報はコンテナ再作成で消える (キャッシュの鍵に設定を含めてあるため、
  仮に再作成しなくても古いパスワードは通らない)
- nginx の設定を変えたら `Caddyfile` にも入れる (ボディ上限など)。
  認証はもうどちらにも無い
- 未確認: **iOS の standalone (ホーム画面から起動) で認証ダイアログが
  出るか**。従来も初回ロードで出ていたので同じはずだが、実機確認が要る

## 11. Authorization ヘッダを見るのは /login だけ (ログアウトのため)

2026-07-19 追記。**ログアウトを成立させるための構造変更**で、§3 と §4 の
「毎リクエスト Basic 照合する」形はここで終わっている。

### 何が問題だったか

パスキー導入時、判定は「セッション Cookie → Basic ヘッダ」の順で両方を
試していた。この形だと**ログアウトが原理的に成立しない**。

Basic 認証の資格情報を握っているのはブラウザで、一度 `/login` を通ると
以後**すべてのリクエストにヘッダを自動で付け直す** (RFC 7617。realm の
保護空間が `/` 配下なのは §3 のとおり)。つまり:

1. ログアウトでセッション行を消し、Cookie も消す
2. 次のリクエストにブラウザが `Authorization` を付けてくる
3. 判定が Basic まで落ちて**そのままログイン済みに戻る**

サーバ側で何を消しても復活するので、当時は「パスワードで入っている人には
ログアウトボタンを出さない」という逃げ方をしていた。押しても効かないため。

### どう直したか

**資格情報を検証してよい場所を `/login` の 1 か所だけに絞り、通ったら
パスキーと同じセッションを発行する**ことにした。

```text
前:   毎リクエスト   Cookie → Basic の順で判定
後:   毎リクエスト   Cookie だけ
      /login だけ    Basic を検証 → セッション発行 → 303
```

ログイン手段が何であれ「ログイン中 = セッションがある」に揃うので、
**ログアウト = セッション破棄**が常に成立する。ブラウザは相変わらず
ヘッダを送り続けるが、アプリが `/login` 以外で見ないので、ただの
無視されるヘッダになる。

| ファイル | 役割 |
| --- | --- |
| `src/app/login/route.ts` | Basic を検証してよい**唯一の場所**。通ればセッション発行 |
| `src/lib/requestAuth.ts` | 判定の正本。**セッション Cookie だけ**を見る |
| `src/lib/auth.ts` | bcrypt 照合。呼ぶのは `/login` だけ |

### 副産物

- **bcrypt は 1 ログインにつき 1 回**しか回らなくなった。§8 の
  「vps2 で 1.75 秒」はログインの瞬間だけの話になり、`auth.ts` が持って
  いた照合キャッシュは存在理由が消えたので**削除した**
- §9 の「Basic は SameSite が効かない」という CSRF の穴は、Basic ヘッダが
  認証として効かなくなったことで**経路そのものが消えた**。
  `denyCrossSite()` は多層防御として残すが、依存度は下がっている
- ログアウトボタンをログイン手段で出し分ける必要がなくなった

### 限界: これは「完全なログアウト」ではない

ブラウザは Basic 認証の資格情報を記憶したままなので、同じ端末で
「パスワードでログイン」を押せば**パスワード入力なしで入り直せる**。
これは Basic 認証をブラウザに預ける方式の限界で、断ち切るにはブラウザを
閉じるしかない。共有端末で完全に切りたいときはその点に注意する。

### DB が落ちているときは入れない

以前は Basic だけで認証が通ったため、DB が死んでいてもログインはできた。
いまはセッション発行に DB 書き込みが要るので入れない。ただし DB が
落ちていればノートも画像も読めないので、実害は無いと判断した。

## 12. 認証ダイアログを閉じたときの行き止まり

「パスワードでログイン」→ 認証ダイアログをキャンセル、で**先へ進めなく
なる**という報告があった。画面には「ログインしてください。」の一行だけが
出て、リンクもヘッダも無い。

原因は `/login` の 401 のボディそのもの。ブラウザは認証ダイアログを
閉じられたとき、401 のレスポンスをそのまま画面に出す。ボディが
`text/plain` の一行だったので、戻り道の無いページに見えていた。

普段このボディは誰の目にも触れない — 資格情報を入れれば 303 で出ていく。
**見えるのはキャンセルしたときだけ**という、忘れやすい経路だった。

Chrome では途中にもう一段挟まる。保存済みの資格情報を自動入力しようとして
OS の本人確認 (Windows Hello) が出るので、キャンセルは 2 回必要になる。
2 回目を閉じたところでこの 401 に落ちる。

### 直し方: ボディを戻りリンクのある HTML にする

`cancelledPage.ts` で 401 のボディを組み、`next` (検算済み) への
「戻る」リンクを置く。保護されたページへ戻れば proxy が
`/login-required` へ rewrite するので、パスキーとパスワードが並ぶ案内に
自然に戻ってくる。

守ること:

- **自動で戻さない**。`<meta http-equiv="refresh">` などで `/login` を
  読み直すと認証ダイアログがまた出て、閉じてまた出る輪になる。
  戻るのは人がリンクを押したときだけ
- `WWW-Authenticate` ヘッダは残す。消すとダイアログ自体が出なくなる
- `next` は属性の中に置くので HTML エスケープする。`safeNextPath` は
  制御文字と外部ホストを落とすが、`"` や `<` は通り抜ける

### 自前のパスワードフォームにはしない

キャンセル経路を完全に制御したいなら Basic 認証をやめてフォームを作る手も
あるが、docs/29 でパスキーが主・パスワードは復旧用の副経路と決めている。
副経路のためにフォームと CSRF 対策を足すのは割に合わない。
401 のボディを直すだけで足りる。
