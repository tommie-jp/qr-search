# アクセス順 (最近見た順) 計画

一覧の並びに「最近見た順」を足す。

## 1. なぜ要るか

ENEX から取り込んだノート (docs/28-エクスポート計画.md §4) は作成・更新日時が
Evernote 由来 (2012〜2021 年など) なので、更新順の一覧では埋もれて探せない。

```text
item #1103 ◎うどん（白だしのみ）
  作成: 2012/04/20 17:20:09
  更新: 2021/06/05 10:13:09
```

**「取り込み時に updated_at を今にする」案は採らない。** このアプリは
「本文が変わっていないのに更新順が動くのは嘘になる」を一貫して守っており、
ゴミ箱・復元・公開トグルがわざわざ生 SQL で `@updatedAt` を避けているのは
そのため (src/lib/items.ts の trashItems / setItemPublic)。Evernote 由来の
日時は保存する価値のある情報でもある。

そこで**別の列**として「最後に開いた日時」を持つ。updated_at の正直さを
守ったまま「最近触ったものが上に来る」が手に入る。README の TODO にある
「今日の検索結果 3 件 / 過去 7 日間 / 過去 30 日間」への布石にもなる。

## 2. スキーマ — `accessed_at` を 1 列

```prisma
accessedAt DateTime @default(now()) @map("accessed_at")
```

- **既存行は `accessed_at = updated_at` で埋める** (migration に UPDATE を同梱)。
  null や now() のままだと導入直後のアクセス順が無意味になる。updated_at で
  初期化すれば**最初は更新順と同じに見え、使うほどアクセス順に育つ**
- 索引は張らない。個人規模 (580 行) では並べ替えの誤差の範囲
  (PROPS_TABLE_LIMIT と同じ判断)
- **罠**: `prisma migrate` が PGroonga 索引を DROP した前例がある。消えても
  seq scan で動いてしまい気づけないので、適用後に `\di` で `items` の
  pgroonga 索引を必ず確認する (doCopyDBfromVPS2.sh の検証部と同じ)

## 3. 記録 — クライアントから 1 発 ping

**サーバコンポーネントの描画時には記録しない。**

一覧の `<Link>` は Next.js が prefetch する。描画時に記録すると、
**画面に並んだだけのノートが「見た」ことになり、検索結果の全件が
アクセス順の先頭に来る**。prefetch はクライアント効果を実行しないので、
マウント後の `useEffect` から呼べば原理的に誤発火しない。

- `src/components/RecordAccess.tsx` … `"use client"`。itemNo を受けて
  `useEffect` で 1 回だけ server action を呼ぶ。何も描画しない
- `recordAccessAction` (src/app/actions.ts) … `requireUser()` の後、生 SQL:

```sql
UPDATE items SET accessed_at = now()
WHERE item_no = $1
  AND accessed_at < now() - interval '1 minute'
```

- **生 SQL なのは** Prisma の update が `@updatedAt` を必ず打つため
  (setItemPublic と同じ理由)。見ただけで更新順が動いては本末転倒
- **1 分ガード**はリロード連打と React StrictMode の二重発火の吸収。
  失敗は握り潰してよい (並びが 1 回進まないだけ) が、ログには残す

置き場所:

| 画面 | 置くか | 理由 |
| ------ | -------- | ------ |
| `/item/<no>` のログイン中 (ItemView) | 置く | QR シールの着地点。ここが本命 |
| `/item/<no>` の未ログイン (PublicItemView) | **置かない** | 他人やクローラに並びを書き換えられる |
| `/edit/<no>` | 置く | 開いた = 触った |
| 一覧・サムネイル | 置かない | 「見た」ではない |

## 4. 並び — Sort を 3 値にする

`Sort` は三項演算子で分岐しているため、**型を足しても全箇所が強制されない**。
grep で洗った 5 箇所を明示的に直す。

| ファイル | 変更 |
| ---------- | ------ |
| `lib/validation.ts` | `Sort` に `'accessed'` を足す。`parseSort` も |
| `lib/items.ts` | `buildOrderBy` を switch に (下の ORDER BY を追加) |
| `lib/searchUrl.ts` | 変更なし (既定以外を載せる作り)。テストで固定する |
| `components/BottomActionBar.tsx` | 2 値 → **3 値循環** (更新順→アクセス順→番号順) |
| `lib/scanResult.ts` | **変更しない**。スキャン結果は更新順でよい |

アクセス順の ORDER BY:

```sql
ORDER BY accessed_at DESC, updated_at DESC, item_no ASC
```

- タイブレークを付けるのは、同時刻の行で並びが不定になるとページ送りと
  前後ナビが読み込みのたびに揺れるため (docs/15 §2-2)
- 既定は**更新順のまま**。日常の画面を突然変えない。アクセス順は明示的に選ぶ

### 4-1. 並び順を覚える (実装後の修正)

当初は「並び順は『何を見ているか』だから URL が正」として `?sort=` だけに
置いた。**これは実際に使うと破綻した** — アクセス順に切り替えても、
`?sort=` を持たない入口から入るたびに更新順へ戻る:

| 入口 | 当時の挙動 |
| ------ | ------------ |
| ヘッダーの「QR search」 | `href="/"` なので既定へ |
| 検索フォームの送信 | `q` だけ送る GET。sort を一切持たない |
| スキャン結果 | `buildSearchUrl(value, 1, 'updated')` 決め打ち |
| タグリンク | `/?q=#tag` のみ |

皮肉なことに、これは viewMode.ts が cookie を選んだ理由として挙げていた
「スキャンやタグリンクで入るたびに既定へ戻る」そのものだった。並び順、
とくにアクセス順は「最近見た順で探したい」という**持続する好み**なので、
表示モードと同じく覚えているのが期待どおり。

そこで **URL と cookie の二段構え**にした (src/lib/sortMode.ts):

```text
URL に ?sort= があればそれが正   ← 共有リンク・ページ送り・戻り先
無ければ cookie                 ← 前に自分が選んだ並び
それも無ければ更新順
```

- 並び替えのトグルは**リンクからフォーム送信へ**変えた。cookie を書けるのは
  サーバアクションだけで、リンクでは URL しか変わらないため。アクションは
  cookie を書いてから `?sort=` 付きの URL へ redirect するので、
  「URL が正」は保たれる (ページ送りと戻り先が並びを引き継ぐ)
- cookie 一本にはしない。ページ送りと一覧への戻りは「いま見ている並びのまま」
  でなければならず、そこは URL が正でよい
- URL の指定は cookie より優先する。共有されたリンクを開いた人に、
  自分の好みを混ぜて見せない
- 循環順は viewMode の 3 値トグル (docs/32 §3) と同じ形にし、ラベルには
  現在値を出す既存方針 (docs/31 §3-4) を維持する

## 5. インポータ

ENEX 取り込みは `created_at` / `updated_at` を ENEX 由来にするが、
`accessed_at` は**取り込み時刻のまま**にする (列の default が now() なので
`applyEnexTimestamps` がそこを触らなければ自然にそうなる)。
これで取り込み直後のノートがアクセス順の先頭に並ぶ = 本計画の目的。

## 6. 見送り (YAGNI)

- **エクスポートに含めない** (docs/28 §1)。「いつ見たか」は本文ではないので、
  tags / props と同じく派生・付帯情報の扱い
- **アクセス履歴・回数** … 最終時刻 1 つで足りる。「過去 7 日間の検索結果」を
  やるときに改めて設計する
- 未ログインの公開ビューでの記録

## 7. テスト

1. `parseSort` … `'accessed'` を受ける / 不正値は `'updated'` へ倒れる
2. `buildSearchUrl` … `sort=accessed` が URL に載る / `updated` は省略のまま
3. 並び順 … ORDER BY の組み立てを純関数へ切り出して検証する
4. `recordAccessAction` … 未ログインは投げる / 1 分ガードが効く
5. **手動確認 (これが一番大事)** … 一覧をスクロールしただけで並びが動かない
   こと (prefetch 誤発火の確認)。ノートを開くと動くこと
