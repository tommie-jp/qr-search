# 画像検索の設計 (Ver2.x)

カメラで部品を映し、登録済みノートの写真と**ブラウザ内で**照合して
類似ノートをリアルタイム表示する。埋め込み (画像のベクトル化) も
類似度計算もクライアントで行い、サーバはベクトルの保管と配布だけを担う。
ここでは設計判断を残す。

前提の調査: ワークスペース側 `docs/11-画像検索調査メモ.md`
(モデル比較・リアルタイム性・検索方式・精度の勘所)。
**このメモは計画のみ。実装はまだ行わない。**(2026-07-18)

## 1. 方針: 埋め込みはクライアント、ベクトルの正本は DB

導線の全体像:

```text
検索ページの「画像検索」ボタン
  → カメラ起動 (ScannerModal と同じ getUserMedia 流儀 + ガイド枠)
  → 2〜3 fps でフレームをガイド枠でクロップして 224px に縮小
  → Web Worker で埋め込み計算 (DINOv2-small q8、ブラウザ内完結)
  → 事前取得した全登録画像ベクトルと総当たり cosine (数 ms)
  → 類似度上位のノートをカメラ下にライブ表示。タップで /item へ
```

- **カメラフレームはサーバに送らない**。送るのはベクトル取得の GET だけ
- 検索側にライブラリは使わない。**総当たり cosine の for ループを自前実装**
  (数千件規模では近似索引は総当たりに勝てない。調査メモ実測)
- pgvector は入れない。オフライン志向 (ワークスペース docs/06) とも
  クライアント照合のほうが合う

## 2. モデル: DINOv2-small q8 (Transformers.js)

- **Xenova/dinov2-small の q8 (24.5 MB / 384 次元)** を第一候補にする。
  自己教師あり系でインスタンス検索 (個体識別) に強い (採用判断)。
  速度が足りなければ **MobileCLIP-S0 q8 (11.8 MB / 512 次元)** に切替。
  どちらにするかは Phase 0 のスパイクで決める
- ライブラリは `@huggingface/transformers` (Transformers.js v4)。
  内部は onnxruntime-web で、**OCR (docs/24)・zxing-wasm と同じランタイム**
- 実行は WASM (q8) を基準性能とし、`navigator.gpu` があれば WebGPU
  (iOS 26+) を使う。**WebGPU は加速ボーナス扱い**で、無くても成立させる
- **ストック側とクエリ側で同一モデル・同一量子化を使う** (必須)。
  モデルを替えたら全件再生成 (§4 のバックフィルを再実行)

## 3. モデルの配布: ローカル同梱 + 遅延ロード

- OCR・zxing と同じく **`scripts/` のコピースクリプトで
  `public/` へローカル同梱**し、CDN (HF Hub) に依存しない。
  Transformers.js は `env.localModelPath` + `env.allowRemoteModels = false`
  で自前 URL から読める
- モデルはページロードでは読まず、**「画像検索」ボタンの初回押下で遅延ロード**。
  「準備しています (初回のみ)」表示を出し、以後はブラウザキャッシュで即時

## 4. データ: images.embedding 列 (派生キャッシュ)

```prisma
model Image {
  ...
  thumb     Bytes?   // 既存: 一覧サムネ。data から作れる派生キャッシュ
  embedding Bytes?   // 追加: Float32Array のバイト列。null = 未生成
}
```

- `thumb` と同じ流儀: **data から作れる派生キャッシュで、null = 未生成**。
  生成に失敗しても保存は成功させる (画像検索の対象から外れるだけ)
- 生成は `saveImage()` (lib/imageStore.ts) に足す。画像を作る経路は
  この関数しかない (docs/20 §1) ので、どの経路で入った画像にも付く
- サーバ側の生成は同じ Transformers.js を Node で動かす
  (onnxruntime-node が自動選択され、ブラウザの WASM より速い)。
  モデルファイルは §3 で同梱したものを読む (サーバも CDN に出ない)
- **本番 Docker は glibc ベース (node:24-slim) が必須** (2026-07-18 に判明):
  onnxruntime-node のネイティブビルドは glibc 用しか配布されておらず、
  alpine (musl) では .so が揃っていても
  `Error loading shared library ld-linux-x86-64.so.2` で読めない (実測)。
  さらに Next standalone のトレースは `onnxruntime_binding.node` は運ぶが、
  動的リンカ経由の依存 `libonnxruntime.so.1` を置いていくため、Dockerfile で
  builder から明示的に COPY して補う。この 2 段が欠けると本番だけ
  「埋め込みを生成できませんでした」が全アップロードで出る
  (アップロード自体は成功する。docs/21 のサーバログで気づける)
- **モデルキャッシュは /tmp に置く** (embedder.ts): transformers.js の既定は
  パッケージ相対の `node_modules/@huggingface/transformers/.cache` で、
  本番は /app が root 所有・実行ユーザー node のため mkdir が EACCES で落ちる
  (これも全アップロードで「埋め込みを生成できませんでした」になる)。
  Node 実行時は `env.cacheDir` を /tmp 配下に向ける
- **既存画像のバックフィル**: `scripts/backfillEmbeddings.mjs` を用意し、
  embedding が null の行を一括生成する (数千枚でも数分〜数十分)。
  モデル変更時もこれを再実行するだけ
- 画像とノートの対応は DB に無く、本文の `/api/images/<name>` 参照が正本
  (lib/memoImages.ts)。`firstImageName()` と同じ抽出規則で
  **全画像を列挙する `allImageNames()` を追加**する

## 5. API: ベクトル索引の配布

- `GET /api/image-search/index` (要ログイン。公開ノート向けには出さない):
  ゴミ箱を除く全ノートの memo から画像参照を列挙し、
  `[{ itemNo, title, imageName, embedding }]` を返す
- 大きさの目安: 384 次元 × 4 byte ≈ 1.5 KB/枚。千枚でも 1.5 MB 程度。
  まずは**モーダルを開くたびに素直に取得**する (KISS)。
  遅くなったら IndexedDB + 更新日時での差分取得へ移す (見送り §9)
- embedding が null の画像は索引から除く (未生成 = 検索対象外、黙って壊れない)

## 6. UI: ImageSearchModal

- `ScannerModal` を雛形に **`ImageSearchModal` を新設** (共通化はしない。
  読み取りループの中身が別物で、無理に束ねると両方が読みにくくなる)
- カメラ映像の上に**ガイド枠** (部品を収めさせる = 実質センタークロップ。
  精度に最も効く。調査メモ §精度の勘所)
- `requestVideoFrameCallback` + スロットリングで 2〜3 fps に絞り、
  ガイド枠内を canvas に 224px で切り出して Worker へ渡す
- 推論は **Web Worker** で回し、UI スレッドを塞がない
  (OCR の Worker 化より先にこちらで実績を作る)
- 結果はカメラ下に**上位 3〜5 件をライブ表示** (サムネ + タイトル + 類似度)。
  「1 位一発当て」を狙わない。そっくりな部品が多く、選ばせる UI が現実的
- ノートのスコアは**所属画像の最大類似度**で集約する (1 ノート複数写真の定石)
- しきい値は絶対値 + 「1 位と 2 位の差」を併用し、Phase 0 で実測して決める。
  全滅のときは「似ているノートが見つかりません」を明示する (黙らない)
- **静止画フォールバック**: シャッターボタンで 1 フレーム確定検索。
  リアルタイムが重い端末でも成立する。`<input type="file">` からの
  写真選択も同じ関数に流す (経路が 2 つ、処理は 1 つ)
- カメラを開けないときのエラー文言は `ScannerModal` の
  `ERROR_MESSAGES` を共通化して使い回す

## 7. テスト

- モデル本体はモックし、純関数をユニットテストする:
  - cosine 類似度 (Float32Array、正規化済み前提)
  - ノート集約 (画像スコア → item の最大値、上位 N 件、しきい値)
  - `allImageNames()` の抽出規則 (コードフェンス除外など firstImageName と同等)
  - 索引 API (embedding null の除外、ゴミ箱除外、未ログイン 401)
- 認識精度・速度そのものはテストではなく Phase 0 のスパイクで実機確認

### 既知の未了 (次に詰める)

- **モデルの配布と固定**: 認識モデル (DINOv2-small q8) はまだ transformers.js の
  既定で HF Hub から初回 fetch する。ort wasm は自前配布済み (copyEmbeddingWasm)。
  no-CDN を完全に満たすにはモデルも `public/models/` へ落として
  `env.localModelPath` を指す (OCR の docs/24 と同じ未了)。あわせて
  **`revision` を commit SHA に固定**して、上流が差し替わっても同じ重みを
  読むようにする (現状は main を引くので、無言で別グラフを native 実行しうる。
  セキュリティレビュー指摘)。ネットワーク制限環境では自前配布が前提。
- **サーバのモデル常駐**: `generateEmbeddingInBackground` は Next サーバ内で
  onnxruntime-node を読む。初回アップロード後はモデルが常駐する。VPS の RAM が
  気になれば、生成を saveImage から外して backfill/cron 専任にする。
- **依存の脆弱性**: transformers.js が引く onnxruntime-node に `npm audit` の
  high が出る (既存の onnxruntime-web と同系統)。更新を追う。

## 9. 見送り (必要になったら)

- pgvector・ANN 索引 (数千件の総当たりで十分。5 万件を超えたら再考)
- 索引の IndexedDB キャッシュ + 差分取得 (毎回 GET で遅くなったら)
- 背景除去 (remove-background-web)。ガイド枠クロップで足りなければ
- 複数フレームの投票 (連続 N フレームで同じ 1 位なら確定、など)
- 公開ノートへの画像検索 (索引 API を公開分だけに絞れば作れるが、用途がまだ無い)
