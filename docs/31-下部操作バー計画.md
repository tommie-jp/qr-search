# 下部操作バー計画 (Ver2.x)

スマホファーストの見直し。検索画面の主要操作 (スキャン・画像検索・表示切替・
並び順・選択) を画面下部の固定バーに集約し、片手持ちの親指で届く操作面にする。

前提はいつもどおり: 素の GET フォーム / リンク / Server Action で JS 無効でも
動く設計は壊さない (プログレッシブ拡張)。

## 1. 目的と現状

操作ボタンが 3 つの行に散っている。

- スキャン / 画像検索 … 検索窓の行 (SearchForm)
- 番号順 / 更新順 … 件数の行 ((search)/page.tsx)
- 表示 小|大 / 選択 … 一覧の直上 (ItemList)

問題は 2 つ。

1. **親指が届かない**。どれも画面上部にあり、片手持ちでは持ち替えが要る。
   届きにくいのは左右ではなく高さ (docs/11-アプリ的UIUX計画.md §8-4 で
   ハンバーガーメニューをボトムシート化したのと同じ理由)。
2. **縦幅を 3 行ぶん食う**。一覧は「1 画面に入る件数」を優先してきた画面
   (ui.ts の COMPACT_* の設計判断) なのに、操作行が件数を削っている。

下部バーに集約すれば両方が同時に解決する。移した後、元の 3 行からボタンは
**消す** (重複させない)。空いた縦幅がこの変更の主な報酬。

## 2. 何を移し、何を移さないか

移す (バーの 5 スロット):

| スロット | 種類 | 今の実装 |
| ---------- | ------ | ---------- |
| スキャン | モーダルを開く | SearchForm の button |
| 画像検索 | モーダルを開く | SearchForm の button |
| 表示 (小⇄大) | cookie を書くフォーム送信 | ViewModeToggle |
| 並び順 (番号順⇄更新順) | searchParams を変えるリンク | page.tsx の PendingLink |
| 選択 | クライアント state の切替 | ItemList の button |

移さない:

- **検索窓・検索ボタン・「+」** … 検索窓は結果 (すぐ下) と一続きに読む物で、
  上に残す。スキャン・画像検索が抜けると 320px でも 1 行に収まるはずで、
  SearchForm の flex-wrap の必要も減る。「+」をバーへ移す案は様子見 (§10)。
- **ページ送り (前へ/次へ)** … 一覧の末尾という現在地が「読み終わったら次」
  の流れに合っている。
- **件数・検索ヘルプ・ゴミ箱リンク** … 操作ではなく表示。件数の行は
  並び順が抜けて左寄せの 1 行に痩せる。

バーは **(search) ページ専用**。並び順・選択は検索ページにしか意味がなく、
layout に置いて /item や /edit を巻き込まない。

## 3. バーの形: タブバー型 5 スロット

### 3-1. テキストボタン一列は入らない (概算)

今の COMPACT_* (text-sm + px-3) のまま横に並べると:

```text
スキャン(≈82) + 画像検索(≈82) + 表示セグメント(≈76)
+ 並び順セグメント(≈132) + 選択(≈52) + gap ≈ 450px
```

375px にも 320px にも入らない。セグメントをトグル 1 個に縮めても ≈390px で
まだ溢れる。

### 3-2. 不採用にした案

- **ドラッグで左右スライド** … 画面外のボタンは存在しないのと同じ
  (スクロールできること自体に気づけない)。特に端に置かれる低頻度ボタンが
  発見不能になるため不採用と決めた。
- **2 段組み** … 入るが縦幅を食い、§1 の目的 (密度) と矛盾する。
- **低頻度ボタンを「…」でボトムシートへ** … 発見性は保てるが 2 タップに
  なる。5 個が 1 段に入る形 (3-3) があるので不要。

### 3-3. 採用: アイコン + 小ラベルの等幅スロット

iOS のタブバーと同じ形。アイコン (20〜24px) の下に 10〜11px のラベル。
320px ÷ 5 = 64px/スロットで、4 文字ラベル (10px × 4 = 40px) が収まる。

- スロット全体をタップ領域にする (44px 四方以上)。バーは「狙って押す」場所
  なので COMPACT の 36px には詰めない (ui.ts の使い分けと同じ理屈)。
- アイコンは MenuIcons.tsx と同じ流儀のインライン SVG を足す
  (スキャン=QR 枠、画像検索=写真、表示=リスト/グリッド、並び順=上下矢印、
  選択=チェック)。
- アプリ内で初めての「アイコン主体」の operation になるが、ラベルを必ず
  併記するので迷子にはならない。

### 3-4. トグルは「現在値を表示、押すと切替」

表示と並び順は 2 択なので、セグメントではなく 1 ボタンのトグルにする。
ラベルに**現在の状態**を出す (表示スロット=「小」or「大」、並び順スロット=
「番号順」or「更新順」)。

- ViewModeToggle がセグメントを選んだ理由 (現在値が常に見える) は、現在値を
  ラベルに出すことで保たれる。
- 「ラベルは現在? 押した後?」の曖昧さは残るが、2 択なのでどちらに解釈しても
  押せばトグルであり、間違えても 1 タップで戻る。
- 実装は今と同じ仕組みのまま: 表示は cookie を書くフォーム送信 (submit ボタン
  の value を「もう一方のモード」にする)、並び順は「もう一方のソート」への
  PendingLink。**どちらも JS 無効で動く**。

## 4. 配置・層・余白

- `fixed inset-x-0 bottom-0` + 中身は `mx-auto max-w-2xl` (ヘッダーと同じ幅)。
  `border-t` + `bg-white/95 backdrop-blur` (ヘッダーの帯と同じ質感)。
  非本番は `bg-pink-100/95` に塗り分ける (layout の帯と同じ判断)。
- 下端の余白は自前で持つ: `pb-[max(0.5rem,env(safe-area-inset-bottom))]`
  (HeaderMenu のボトムシートと同じパターン。standalone のホームバー対策)。
- **本文の逃げ**: ページ末尾にバーの高さぶんのスペーサを置き、最終行と
  ページ送りがバーに隠れないようにする。layout の main には触らない
  (バーは検索ページ専用のため)。
- **z-index の秩序** (確認必須):
  - モーダル (Scanner/ImageSearch, z-50) と SavedToast (z-30) より下。
  - HeaderMenu の外側タップ覆い (z-10) より**下か同層**にすること。バーが
    覆いより上だと、メニューを開いたままバーが押せてしまう。バーを z-10 に
    して DOM 順 (バーが main 内 = 覆いより前) で覆いを勝たせる想定だが、
    ここは実物で確認してから決める (docs/11 §8-4 の落とし穴 3 と同じ罠)。
- **PageTransition との関係**: バーはページ内 (= ViewTransition の中) に
  置くので、ページ遷移時はページごとスライドする。検索ページ内の遷移
  (searchParams だけ変わる) ではアニメしないので実害はない想定。fixed が
  transition 中に外れて見えないか、実機で 1 度見る。
- **iOS キーボード**: 検索窓フォーカス中、バーはキーボードの上に浮く。
  まずはそのまま出す。実害 (誤タップ・圧迫) が出たら visualViewport で
  隠す対応を検討 (最初から作り込まない)。

## 5. コンポーネント構成と状態の持ち上げ

新規 `BottomActionBar.tsx` (client component)。page.tsx (Home) から描画し、
props で view / sort / query / stickerHost とサーバーアクションを受け取る。
Suspense (HomeResults) の**外**に置くので、初回表示から即出る。

### 5-1. モーダルの持ち主が変わる

ScannerModal / ImageSearchModal の dynamic import と開閉 state を SearchForm
から BottomActionBar へ移す。「押すまで一切読み込まない」設計 (docs/09 §2,
docs/25) はそのまま維持する。SearchForm はスキャン・画像検索ボタンを失い、
検索窓 + 検索 + 「+」だけになる。

### 5-2. 選択モードの持ち上げ

selectMode は今 ItemList のローカル state。バーの「選択」から切り替えるには
共有が要る。SearchNavProvider と同じ形の小さな context
(`SelectModeProvider`: selectMode / enter / exit) を page.tsx で結果とバーの
両方に被せ、ItemList は useState をやめて context を読む。

- 選択した番号の Set は今までどおり ItemList のローカルに残す (バーは
  知らなくてよい。YAGNI)。
- 選択モード中、バーの「選択」スロットは押下状態 (aria-pressed + 色) にして
  もう一度押すと解除。BulkTagToolbar の「やめる」も今までどおり効く
  (exit を context 経由にするだけ)。

### 5-3. 選択モードとの合成 (Phase 2)

選択モード中はバーが一括操作 (タグ追加・ゴミ箱へ・全選択・解除・やめる) に
**変身する**と、iOS 写真アプリ的な一貫した操作面になる (選択→操作の親指移動
が消える)。

- BulkTagToolbar は `<form action={bulkTagAction}>` の中にある必要があるが、
  バーは form の外。`<button form="bulk-form">` の form 属性で外から同じ
  form を送信できる (JS 不要の標準 HTML)。
- ただしタグ追加の入力欄・削除チップまで下へ移すと固定バーが太る。
  Phase 2 の設計時に「下は件数 + 全選択/解除/やめる/ゴミ箱、タグ操作は上の
  ツールバーに残す」案と比較して決める。**Phase 1 では手を付けない**
  (BulkTagToolbar は現状のまま)。

- [x] JS 無効で表示トグルと並び順が動く (§8-1)
- [x] 選択モードの出入りが バー / BulkTagToolbar の両方から動く。

- [x] standalone (ホームバーあり実機) でバーが潜らない (safe-area) — §8-2

### 8-2. 実機 (iPhone / standalone) で確認したこと

**safe-area は効いている。** バーの帯は実測で約 85 CSS px あり、これは
`pt-1.5 + min-h-11` に `env(safe-area-inset-bottom)` の 34px が加わった値。
効いていなければ 54px に縮み、ラベルがホームインジケータの帯に食い込む。
ラベル下端とインジケータ領域の間の余裕は 10px 強で、詰まってはいるが
safe-area が本来与える余白そのもの。

**キーボード表示中は、止まっているか動かしているかで挙動が変わる。**

止まっている間はバーは見えない。iOS の `position: fixed` はキーボードに
押し上げられずレイアウトビューポートの下端に留まるため、キーボードの下に隠れる。

**だがスクロールするとせり上がって一覧の途中に浮く (要対処)。** iOS は
キーボード表示中にスクロールが起きると fixed 要素をビジュアルビューポート
(キーボードより上の見えている領域) の下端へ貼り直す。その結果バーの下にも
リストの中身が残り、「固定バー」ではなく「リストに紛れ込んだ何か」に見える。

一度「visualViewport での制御は不要」と書いたが**それは誤り**だった。静止状態の
スクリーンショットだけで判断したためで、指で動かした瞬間に出てくる挙動は
そこには写らない。

実害は 2 つ。見た目が壊れて見えること。そして、キーボードを出したまま結果を
流し読みする動線 (インクリメンタルサーチなので自然な使い方) で、行を押した
つもりがバーを押す誤タップが起こり得ること。バーは常に指の通り道にある。
データが壊れる類ではないので優先度は「気持ち悪いので直したい」相当。

### 8-3. 対処: スクロールし始めたらキーボードを閉じる (実装済み)

iOS 純正アプリ (メール・設定の検索) の `keyboardDismissMode = .onDrag` と同じ
作法を採った。結果を読みに行く動作に入った時点でキーボードは用済みで、閉じれば
画面が倍近く広がりバーも本来の下端に戻る。**バーの位置と「結果を見たいのに
画面半分が潰れている」の両方が同時に解決する**のが利点。

採らなかった案: `visualViewport` でキーボード中はバーを隠す。バーの見た目は
直るが、キーボードが画面を占有したままなのは変わらず体験として一段劣る。

実装は SearchForm.tsx。入力欄にフォーカスがある間だけ window の `touchmove` を
拾って `blur()` する。要点は 2 つ:

- **`scroll` ではなく `touchmove` で拾う。** キーボードが開くとき iOS は入力欄を
  見せるために自前でスクロールするので、`scroll` だと開いた直後に自分で閉じて
  しまう。`touchmove` なら必ず指が動かした合図になる。
- **入力欄の中で始まった指の動きでは閉じない。** 文字列選択やタグ候補の
  ドロップダウン操作を邪魔しないよう、`touchstart` の時点でフォームの中から
  始まったかを覚えておいて判別する。

タッチ端末として開いたブラウザで確認した 4 ケース:

| 操作 | 期待 | 結果 |
| ------ | ------ | ------ |
| 一覧をドラッグ | 閉じる | フォーカスが input → body |
| 入力欄の中をドラッグ | 閉じない | input のまま |
| タグ候補の上をドラッグ | 閉じない | input のまま・候補も開いたまま |
| 閉じた後の検索語と URL | 保たれる | `npn` / `?q=npn` のまま |

タッチイベントを持たない環境 (PC) では `touchmove` が発火しないので、挙動は
何も変わらない。

iPhone 実機でも確認済み。キーボードを閉じた直後にバーは正しい位置へ戻る
(iOS はビューポートの伸縮で稀に fixed 要素を取り残すことがあるため見ていたが、
その症状は出なかった)。**これで下部操作バーの実機確認はすべて完了**。

### 8-1. JS 無効の確認方法と、そこで判った制約

サーバーアクションの form は `method="POST"` + `$ACTION_ID_*` の hidden として
SSR される。これを curl で multipart POST すると `Set-Cookie: view=card` が
返るので、**JS を一切使わずに表示が切り替わる**ことを証明できる。
並び順は素の `<a href="/?q=npn&sort=itemNo">` なので元より JS を要らない。

一方で、**検索結果そのものは JS 無効では出ない**。`Suspense` の中にあり、
本体は後続チャンクを `$RC`/`$RS` スクリプトで差し込む方式のため、JS 無効の
クライアントは fallback (「検索結果を読み込み中…」) のまま止まる。

これは下部バーとは無関係の既存の制約 (Suspense 導入時から)。「JS 無効でも動く」
という全体方針は、検索結果については既に成り立っていない。バー自体の操作は
方針どおり JS 無しで動くので、この計画の範囲では対処しない。

## 9. 記録: この計画で決めたこと

- ドラッグスライドは**不採用** (発見性。§3-2)
- 全 5 操作を 1 段に収めるためタブバー型へ (§3-3)
- トグルは現在値表示 + 1 タップ切替 (§3-4)
- 一括操作の下部化は Phase 2 に分離 (§5-3)

### 9-1. 実装で判ったこと

- **モーダルは `<nav>` の外に置くこと** (BottomActionBar.tsx にも書いた)。
  バーは `backdrop-blur` を持ち、`backdrop-filter` のある要素は
  `position: fixed` の包含ブロックになる。中に入れると `inset-0` が
  「バーの矩形」を指し、全画面モーダルが高さ 61px に潰れる。
  HeaderMenu が覆いとシートを portal しているのと同じ理由。
- **`ViewModeToggle.tsx` は削除した**。バーが自前のトグルを持つので参照が
  無くなった。セグメント (小|大 の 2 ボタン) はバーの 1 スロットに入らない。
- **Next.js の dev オーバーレイがバーの左端スロットに重なる**。開発中だけ
  「スキャン」が押しにくい (本番には出ない)。詳細は
  `.claude/skills/verify/SKILL.md`。
- スロット幅は完全な等幅にならない (実測 375px で 67/67/59/67/67)。
  `flex-1` でも `min-width: auto` が効き、ラベルの min-content 幅が
  下限になるため。「小」は 1 文字なので他より狭い。実害はないので放置。

## 10. 見送り (YAGNI)

- 「+」(新規ノート) のバー移設 … 上の 1 行が痩せて収まるならそのままでよい
- スクロール方向でバーを自動的に隠す … 実装は簡単だが、まず常時表示で試す
- visualViewport でキーボード中に隠す … **見送り撤回**。キーボード表示中に
  スクロールするとバーが浮くため対処が要る (§8-3)。ただし採る手はこれではなく
  「スクロールでキーボードを閉じる」を推す
- 並び順の 3 択以上への拡張 (作成日など、README TODO) … 増えた時点で
  トグルをボトムシート選択に替える。今は 2 択なのでトグルで足りる

## 11. アイコンの色付け (実装済み)

現状のバーは全スロットが text-gray-700 の線画で地味。各スロットに**機能色**を
与え、ラベルを読まなくても色と形で狙えるようにする。

### 11-1. 方針: 機能色 + 二階調 (duotone)、currentColor は維持

- **1 スロット 1 色相**。線 (stroke) は各色の 600 番。アイコン内の「面」になる
  シェイプ (QR の四角・写真フレーム・チェックボックスなど) に
  `fill="currentColor" fillOpacity={0.15}` の薄い塗りを敷き、1 色相で
  二階調にする。線画のままなのでハンバーガーメニュー側 (20px 線画) と
  統一感は崩れない。
- **SVG 内は currentColor のまま**。色は BottomActionBar 側で、アイコンを
  `<span className="text-sky-600">` のように包んで与える。ラベルは
  text-gray-700 に据え置く — 0.625rem の小さい文字を 5 色に塗ると
  うるさく、読みにくくなるだけ。
- 面を持たないアイコン (並び順の矢印) はティント無しの線色のみでよい。

色の割り当て (Tailwind 600 番で統一):

| スロット | 色 | 理由 |
| ---------- | ---- | ------ |
| スキャン | sky-600 | カメラ/レンズの寒色 |
| 画像検索 | violet-600 | 埋め込み検索の「AI っぽさ」。隣の sky と離す |
| 大/小 | emerald-600 | 中央で暖色と寒色の間を取る |
| 並び順 | amber-600 | 隣の emerald / blue と識別しやすい暖色 |
| 選択 | blue-600 | 押下時の bg-blue-600 と同系。「押すとこうなる」の予告 |

選択モード中は今もスロット全体が bg-blue-600 + text-white に反転する。
色を currentColor 経由で与えていれば、包む span を
`selectMode ? "" : "text-blue-600"` にするだけで白反転がそのまま効く。

### 11-2. 不採用にした案

- **アイコンライブラリ導入** (lucide / phosphor duotone など) …
  MenuIcons.tsx 冒頭の判断 (数個のために依存とバンドルを増やさない) を
  覆す理由がない。
- **SVG グラデーション** (linearGradient) … `<defs>` の id 管理が要り、
  currentColor による白反転が効かなくなる。非本番のピンクバーとも喧嘩する。
- **塗りつぶし (filled) アイコン化** … 24px では太って見え、線画で
  揃えてきた統一感が崩れる。
- **色付き背景チップ** (iOS 設定アプリ風の丸角タイル) … 5 個並ぶと
  ボタンだらけでうるさい。あれはリスト向けの意匠。

### 11-3. 実装ステップ

1. MenuIcons.tsx … ScanIcon / ImageSearchIcon / ListViewIcon / GridViewIcon /
   SelectIcon の面シェイプにティント (`fill="currentColor"`
   `fillOpacity={0.15}`) を追加。SortIcon は線のみ。
2. BottomActionBar.tsx … 各アイコンを色クラスの span で包む。
   選択スロットだけ selectMode で条件分岐。
3. テスト … 挙動・aria は不変なので新規テストなし。既存 vitest が
   通ることを確認。
4. 検証 (Playwright + 実機) … 375px / 320px、本番 (白バー) と
   非本番 (ピンクバー) の両方、選択モード ON の白反転、active 押下色
   (bg-gray-200/70) 上の見え方を確認する。
5. README TODO へ反映し、`feat:` でコミット。

### 11-4. メニュー項目のアイコンも色付け (実装済み)

対象はハンバーガーメニューの行頭アイコン (20px 線画):
QR コード / GitHub / ログ / デバッグ / パスキー / ログイン / ログアウト。

方針は §11-1 と同じ (機能色 + 二階調、SVG 内は currentColor) だが、
**色の与え方だけ逆にする**:

- 下部バーは使用側の span で色を与える — 選択モードの白反転という条件分岐が
  使用側 (BottomActionBar) にあるため。
- メニューには反転状態が無く、使用箇所が 6 コンポーネントに散っている
  (layout.tsx / HeaderQrButton / LoginButton / LogoutButton /
  PasskeyLoginButton / DebugConsoleButton)。使用側で包むと同じ色指定が
  散らばるので、**アイコン部品側に色を焼き込む** — StrokeIcon に
  className を渡せるようにし、各アイコンが自分の色クラスを持つ。
  行ラベルは HEADER_MENU_ITEM_CLASS の text-gray-700 のまま
  (svg 側の text-\*-600 が勝つ)。

これが安全な根拠 (確認済み): 20px アイコンはどれも `variant === "menu"` の
分岐内でしか描画されない (HeaderQrButton のヘッダー表示は文字 "QR"、
Login/Logout/PasskeyLogin も menu 以外ではアイコンを出さない)。
焼き込んでもメニュー以外の文脈を巻き込まない。

色の割り当て:

| 行 | 色 | 理由 |
| ---- | ---- | ------ |
| QR コード | sky-600 | 下部バーのスキャンと同じ QR モチーフは同色で対応付け |
| GitHub | 現状維持 | Octocat は商標。単色のまま改変しない (冒頭の判断を維持) |
| ログ | teal-600 | 記録の寒色。sky / violet と並んでも識別できる |
| デバッグ | amber-600 | 虫 = 警告の黄。並び順と同色だが別画面なので衝突しない |
| パスキー | violet-600 | 鍵 / セキュリティ。KeyIcon の 2 箇所 (管理・ログイン) 共通 |
| ログイン | emerald-600 | 「入る」の緑 |
| ログアウト | rose-600 | 「出る」の暖色。破壊の red-700 より一段柔らかく |

ログアウトを red-700 にしないのは、DANGER_BUTTON_CLASS の破壊操作
(ゴミ箱へ / 永久削除) と同格に見せないため。

ティントも §11-1 と同じ fillOpacity 0.15 を面シェイプに敷く
(QR の四角・ログの書類・パスキーの鍵頭・虫の胴体)。ログイン / ログアウトは
囲いが開いたパスで塗ると崩れるため線色のみ。

実装ステップ:

1. MenuIcons.tsx … StrokeIcon に色クラスの受け口を追加し、
   Qr / Log / Key / Login / Logout に色とティントを付ける。Github は触らない。
2. DebugConsoleButton.tsx … ローカルの BugIcon はローカルのまま
   (行頭アイコンの作法を揃えるという既存コメントの判断を維持)、
   amber-600 + 胴体ティントを同じ作法で付ける。
3. grep で 20px アイコンが menu 以外で使われていないことを実装時にも再確認。
4. 検証 … メニューを開いた白シート上の見え方、hover:bg-gray-100 /
   active:bg-gray-200 の上でも沈まないこと、非本番 (ピンクヘッダー) でも
   シートは白のままなので同じ見え方になること。
5. README TODO へ反映。下部バー (§11-3) と同時に実装するなら
   コミットは 1 つでよい。

### 11-5. 実装で判ったこと

- **span の包みで高さは増えない** (実測)。バーのアイコンを
  `<span className="flex …">` で包むと 1 行ぶん伸びるのではと疑ったが、
  包む前後どちらも nav 61px / スロット 56px で同じだった。span に `flex` を
  付けているのが効いていて、inline のまま置くと svg の下にベースラインぶんの
  隙間が出る。
- **`display: contents` は「包む前」の代用にならない**。検証の途中で
  span へ `display:contents` を当てて元の高さを測ろうとしたら 49px という
  別の値が出た。svg が flex アイテムになる文脈が変わるためで、
  高さの比較は git stash で本物の変更前を測るしかない。
- **選択モードの白反転は色指定を足さないことで通る**。
  `selectMode ? "" : "text-blue-600"` とし、選択中は親の text-white を
  継がせる。青のまま置くと青地に青で沈む。実測でも選択中の svg の
  computed color は rgb(255,255,255) になっていた。
- **白反転はテストで守れない**。BottomActionBar.test.tsx に色のテストを
  足したが、守れるのは「5 スロットに色が乗っている」ことまで。選択中の
  描画は静的描画では作れず、常に text-blue-600 を当てるよう直しても
  非選択時の HTML は変わらないのでテストは通ってしまう。ここは
  HeaderMenu と同じくブラウザで確認するしかない。
- **Next.js の dev インジケータがスキャンのスロットに重なる**。実装の
  不具合に見えるがローカル限定の表示で、`nextjs-portal{display:none}` を
  当てれば消える。スクリーンショットを撮るときは先に消す。
- ログイン時のアイコン (LoginIcon) だけは目視できていない。Basic 認証が
  そのままアプリのログイン状態になるため、ローカルでは未ログインの
  メニューに到達できない。LogoutIcon の鏡像で作りは同じ。

## 12. スマホ横持ち (ランドスケープ) 対応 (実装済み)

スマホを横に持つと視界の高さが 300px 台 (iPhone の Safari では 300 前後、
standalone でも 390) まで減り、縦持ち前提の固定 UI が縦幅を食い過ぎる・
画面から溢れる問題が出る。

### 12-1. 変体 `landscape-phone`

globals.css に追加。`orientation: landscape` だけだと PC のウィンドウが
ほぼ常に該当してしまうので、`max-height: 500px` を重ねて「横向きかつ
縦の余裕がない画面」に絞る。500px は iPhone Pro Max の横持ち (視界 400px
前後) を含み、PC の通常ウィンドウを外す境目。

### 12-2. 直した場所

- **下部バー (ui.ts)** … 縦積みスロットが 52px を食う。
  横持ちではアイコン+ラベルを横並びにし min-h-9 (実測 41px) へ。
  iOS のタブバーが横向きでコンパクト形になるのと同じ作法。
- **メニューのシート (HeaderMenu)** … 項目全部 (302px) が入らず溢れる。
  max-h + overflow-y-auto + overscroll-contain を付けた。上限は
  「ヘッダーの帯の高さ」まで — シート (z-40) はヘッダー (z-20) より
  上なので、帯に被せると ✕ が押せなくなる (240px で実測)。
- **スキャン/画像検索のカメラ** … 幅 28rem の映像 (4:3 で 336px) が縦に
  溢れ、シャッターが画面外へ落ちる。max-w を `min(28rem,75dvh)` にし、
  高さ由来でも幅を縛る。幅 ≤ 75dvh なら 4:3 でも高さ ≤ 56dvh。
- **スキャンのエラー表示** … エラー文 + カメラ枠で溢れると justify-center
  が上下とも画面外に出す。器を overflow-y-auto にし、中央寄せは内側の
  my-auto へ (auto マージンは溢れると 0 に潰れるので届く)。
- **QR コードのオーバーレイ (HeaderQrButton)** … QR 240px + URL で
  300px を超える。カードに max-h-full + overflow-y-auto を付けた。

### 12-3. 実装で判ったこと

- **シートの max-h はヘッダーを避けて切る**。当初 `100dvh - 1.5rem` にしたら
  240px の横画面でシートが ✕ ボタンを覆い、タップを横取りした (Playwright の
  click が intercepts pointer events で失敗して発覚)。§4 の z の設計
  (シート z-40 > ヘッダー z-20) の帰結なので、高さ側で避けるしかない。
- **justify-center + overflow は両端が届かなくなる**。スクロール容器で
  中央寄せしたいときは、子へ my-auto を付ける (溢れたら 0 に潰れる)。
- 縦持ち (375px) は変化なしを実測で確認。バーは縦積み 49px のまま。

### 12-4. 横持ちでは器の幅上限を緩める (実装済み)

§12-2 の後、横持ちで本文の器 (max-w-2xl = 672px) の両端が遊んでいた。
max-w-2xl はメモの本文が読める行長に収めるための上限だが、横持ちは
縦が窮屈な代わりに横が余っており、行長より「スクロール量が減る」ほうが
効く。landscape-phone では上限を 4xl (896px) へ緩める。

- 揃えて緩める場所: ヘッダーの帯・main・下部バー・メニューのシートの 4 つ。
  main だけ広げると左右の端が段違いになる。
- none (無制限) にしないのは、PC の縦に低いウィンドウ (横 1200px 級) も
  landscape-phone に該当し、そこで行が伸びすぎるのを止めるため。
  スマホの横持ち (~932px) では safe-area を引くと 896px に届かず、
  実質全幅と同じ。
- 副産物: カード表示の一覧 (auto-fill minmax 20rem) が横持ちで
  2 カラムになる (672px 時代は 8px 足りず 1 カラムだった)。
- 実測 (844×390): main/帯/バー 829px、カード 2 カラム。
  縦持ち 375px・PC 1280×800 は 672px のまま変化なし。
  1280×480 は 896px で止まる。
