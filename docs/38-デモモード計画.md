# デモモード計画

隔離デモインスタンス (別スタック + 定期再シード、41-QR-search/docs/08) で
guest に書き込みを許すための前提となる `DEMO_MODE` フラグ。目的は
**「ファイル置き場 (匿名ホスティング) にされない」**ことで、手段は
①公開機能の封鎖、②設定系の封鎖、③アップロード上限の縮小、④デモで
あることの明示、の 4 点をフラグ 1 個で切り替えること。

総量クォータ・レート制限・再シードは別計画 (§8)。この計画だけでは
guest 公開の条件は揃わない。

## 1. なぜ要るか

- 公開トグルが生きたままだと、guest が画像を上げて公開 URL を配れる =
  未ログインで取得できる匿名ホスティングになる (通報リスク。
  41-QR-search/docs/08 Q2)。デモで最初に塞ぐべき穴
- 設定系 (パスキー登録・ENEX インポート・ログ閲覧) は共有アカウントの
  デモでは害しかない: 他人がパスキーを登録できる、10MB の XML パースを
  回せる、他の訪問者の操作ログが見える
- 現状コードにデモを表す仕組みは皆無 (grep 確認済み)。環境分岐は
  `APP_ENV` (本番/ローカルの見た目) だけで、機能の可否には使っていない

## 2. フラグ — `isDemoMode()` を appEnv に集約

env 判定の集約先である `src/lib/appEnv.ts` に追加する。

```ts
export function isDemoMode(): boolean {
  return process.env.DEMO_MODE === "1"
}
```

- `"1"` の完全一致だけを有効とする (`isProductionEnv()` と同じ流儀。
  `"true"` や `"yes"` の解釈揺れを作らない)
- 倒れる方向に注意: **旗の欠落は「デモ保護オフ」に倒れる**。
  `APP_ENV` (欠落→ピンクで気づける) とは逆で、デモインスタンス側で
  設定を忘れると無防備な書き込み可サイトになる。対策は §7
- デモインスタンスは `APP_ENV=production` + `DEMO_MODE=1` の組で立てる。
  ピンク [LOCAL] は「持ち主が本番と間違えない」ための印であり、
  訪問者向けのデモ告知は §6 のバナーで別に出す

## 3. 公開機能を止める — UI・書き込み・閲覧の三重封鎖

UI を隠すだけでは Server Action を直接叩けるため、サーバでも拒否する
(`requireUser` は飾りではない、の既存思想と同じ)。

- **UI**: `src/components/ItemView.tsx:93-99` の `PublicToggle` 描画を
  `isDemoMode()` で落とす (Server Component なので直読みできる)
- **書き込み**: `src/app/actions.ts:98` `setItemPublicAction` の
  `requireUser()` 直後で `isDemoMode()` なら throw
  (「デモでは公開機能は使えません」)
- **閲覧**: `src/lib/publicItem.ts` `isPublicItem()` の冒頭で
  `isDemoMode()` なら `false`

閲覧まで塞ぐのは、再シードの seed を本番 dump から作った場合などに
`publicAt` 付きの行が紛れても、未ログイン配信 (画像
`api/images/[name]` の `isPublicImageName` 経由と、公開ノート閲覧で
起動する TeX 子プロセス `PublicItemView` → `renderCircuits`) が
一括で死ぬようにするため。判定の正本 1 箇所で穴の類全体が閉じる。

## 4. 設定系を隠す — パスキー登録・ENEX インポート・ログ

隠す/塞ぐ対象と改修点 (UI はヘッダメニューのリンクを隠し、
ページは `notFound()`、API は拒否):

- **パスキー**: リンク `layout.tsx:148-154`。
  `settings/passkeys/page.tsx`、
  `api/auth/passkey/register-options`・`register-verify`、
  `api/auth/passkeys/[id]` DELETE
- **ENEX インポート**: リンク `layout.tsx:158-164`。
  `settings/import/page.tsx`、`api/import` POST
- **ログ**: リンク `layout.tsx:138-141`。`/logs` ページ、`api/logs`、
  `api/client-logs` (転送) も受けない

- route handler 用の門番は `src/lib/apiAuth.ts` に
  `denyIfDemoMode(): Response | null` を足し、既存の
  `denyUnlessLoggedIn() ?? denyCrossSite(request)` の連鎖に
  `?? denyIfDemoMode()` として繋ぐ (403 + 日本語メッセージ)
- パスキーの**ログイン**経路 (`login-options` / `login-verify`) は
  塞がない — デモに登録済みパスキーは無いので実害なし、かつ再シードで
  `WebAuthnCredential` ごと消える。登録経路だけ塞げば足りる
- `/logs` を塞ぐ理由: 共有アカウントのため、他の訪問者のエラーログ
  (URL・操作痕) が見えるのは情報漏えいに相当する

## 5. アップロード上限の縮小 — 10MB → 2MB

添付 (画像・音声・PDF・テキスト) は全部 `api/images` の 1 本なので、
上限判定を関数 1 個に寄せて切り替える。

- `src/lib/uploads.ts` に `DEMO_MAX_IMAGE_BYTES = 2MB` と
  `maxUploadBytes(): number` (デモなら 2MB、通常は
  `MAX_IMAGE_BYTES`) を追加
- 差し替え箇所: `api/images/route.ts:44` のサイズ検査、
  `uploads.ts:510` `MAX_BODY_BYTES` (モジュール定数なので
  `checkUploadRequest` 内で都度計算する形に変える)、
  `attachmentStore.ts` の既定上限、`tooLargeMessage` の表示値
- `MAX_ENEX_BYTES` は触らない (§4 でインポート自体を塞ぐ)。
  書影取得 `coverLookup.ts` も触らない (guest 操作起点ではあるが
  取得元は外部 API で、デモには API キーを渡さないため動かない)
- クライアント側の事前チェックは現状無し (サーバ 400 で足りている)。
  デモでも同じ扱いとし、追加しない

## 6. デモバナーとログイン案内

- **常時バナー**: `src/app/layout.tsx` の `<header>` (:102-211) 直後に
  Server Component で差し込む。文言:
  「これはデモです。データは毎時リセットされます。不適切な内容の
  保存は禁止します。」+ 連絡先。ピンク (ローカルの印) と混同しない
  色 (amber 系) にする
- **[DEMO] バッジ**: `layout.tsx:192-199` の [LOCAL] バッジと同じ
  位置・作りで `isDemoMode()` のときに出す
- **タイトル**: `src/lib/site.ts:13-15` `siteTitle()` に `[DEMO]`
  接頭辞を追加 (`[LOCAL]` と同じ仕組み。manifest.ts にも自動で効く)
- **ログイン案内**: 未ログイン時のバナーにデモ資格情報
  (例: `demo / …`) を明記する。Basic 認証のネイティブダイアログには
  説明を出せないため、押す前に見える場所に書くのが唯一の手

## 7. env の通し方と「旗の欠落」対策

- `.env.example` に `DEMO_MODE=` を説明コメント付きで追加
  (「デモインスタンス専用。本番・ローカルでは空のまま」)
- `compose.yaml` の `app.environment` に `DEMO_MODE: ${DEMO_MODE:-}` を
  追加 (**列挙に無い変数はコンテナに渡らない**罠の対象)
- §2 の「欠落が無防備に倒れる」対策: デモスタックを立てるときは
  compose override (例: `compose.demo.yaml`) に `DEMO_MODE: "1"` を
  **直書き**し、.env 頼みにしない。デモ用 override の整備自体は
  再シード計画 (別 doc) とセットで行い、本計画では .env 経由で
  動くところまでとする

## 8. 見送り (YAGNI)

- **総量クォータ・ノート数上限** — 別計画 (このフラグの次に必須)
- **レート制限** — nginx (エッジ) 側でやる。アプリ外
- **template DB 再シード・compose.demo.yaml** — 別計画
- **SITE_NAME の env 化** — デモ名の出し分けは [DEMO] 接頭辞で足りる
- **ゲスト専用ロール/複数ユーザ** — Ver3 の user_id 設計に含める
- **読み取り専用モード** — 段階案としては公開ノート (本番) で
  代替できており、フラグを増やす価値がない
- **DebugConsole (eruda) の無効化** — クライアント内で完結し
  サーバ資源を使わないため放置してよい

## 9. テスト

実装ファイル併置の vitest、env を触るテストは `afterEach` で原状復帰
(`appEnv.test.ts` の流儀)。

- `appEnv.test.ts`: `DEMO_MODE="1"` でだけ true、`"true"`/空/未設定は
  false
- `publicItem.test.ts`: デモ時は `publicAt` 付きでも `isPublicItem` が
  false
- `site.test.ts`: デモ時のタイトルに `[DEMO]` が付く
- `uploads.test.ts`: `maxUploadBytes()` がデモ時 2MB / 通常 10MB、
  `tooLargeMessage` が実効上限を表示する
- route 拒否 (`api/import`・`register-options` 等): 既存の route
  テストの形式に合わせ、デモ時 403 を確認
- `setItemPublicAction`: デモ時に throw し DB を触らないこと
