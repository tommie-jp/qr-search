# プロパティ機能の設計 (Ver2.x)

ノートに `hFE=208 Vf=700mV` のように部品の特性を書き、タグ検索したときに
一覧の上へ特性表として並べて比べられるようにする。
利用者向けの書き方は [メモ記法](./メモ記法.md) を参照。ここでは設計判断を残す。

土台は [タグ機能の設計](./06-タグ計画.md) とまったく同じ形で、
「正本はメモ本文・列は派生キャッシュ・保存のたびに純関数で再計算」を踏襲する。

## 1. プロパティ記法 (行全体ルール)

**行全体が空白区切りの `key=value` トークンだけで構成される行**をプロパティ行とみなす。

```text
#BJT #2sc2712-y
hFE=208 Vf=700mV
```

- 位置は自由。複数行あればマージし、同じキーは先勝ち
- `key=value` は空白を挟まない (`hFE = 208` は不可)
- キーは英字始まりの英数字・`_`・`-`。正規化は NFKC + 小文字化 (タグと同じ)。
  表示は書いた綴りのまま (`label`)
- 値は空白と区切り (`=` / `＝`) を含まない 1 文字以上
- コードフェンス・インラインコード・数式の中は対象外

### 行番号固定を採らなかった理由

「3 行目をプロパティ行とする」案は、上に 1 行足しただけで黙って壊れ、
エラーにもならず表から部品が消える。ノートを見ても規約が分からず、既存ノートとも
衝突する。行全体ルールなら位置に依存せず、散文中の `key=value`
(`実測では hFE=195 だった`) は行全体条件を満たさないので自然に除外される。
規約を覚えずに「本文中の key=value と混ざらない」が手に入る。

### 値を「数値 + 単位」に狭めなかった理由 (重要)

当初は値を `数値 + 任意の単位` (208 / 700mV) に限る案だった。実装後に実メモ
512 件へバックフィルして分かったのは、この案が**壊れる**ということ。

- 誤爆を防いでいるのは「行全体が key=value」と「キーは英字始まり」の 2 つで、
  値の狭さは安全性に寄与しない (URL `?a=1` はキー側で落ちる)
- 一方で狭めると `device=SS8050` や `pkg=TO-92` がトークン単位で不合格になり、
  **その行のプロパティが丸ごと落ちる**。エラーは出ず黙って消えるので気づけない
- 実メモには既に `device=2N5551` / `device=SS8050` / `hFE=120～200` があった。
  `device=2SC2166` は先頭が数字なので偶然通っていただけで、`SS8050` は落ちていた

そこで値は緩く受け、表示はそのまま、判断は人に委ねる。ただし区切り (`=` / `＝`) を
値に含む行は除外する。これはデータシートの引用行
(`Ｖｉ＝２Ｖｒｍｓ、Ｒ＝５ｋΩ` のような読点区切りの羅列) が
プロパティ行と誤認されるのを防ぐためで、実メモの誤爆 2 件がこれで消えた。

末尾の読点・コンマ (`hFE=440,`) は区切りの名残なので落とす
(タグが `#抵抗。` を「抵抗」で切るのと同じ考え)。

## 2. データ設計 — 派生キャッシュ列

`items.props jsonb` に `[{ key, label, value }]` を初出順で持つ。

- `key` … 正規化キー (列のマージ・比較用)
- `label` … 元の綴り (表ヘッダ表示用)
- `value` … 書いたままの値 (セル表示用)

ソート用の数値は表示時に `parsePropNumber` で導出する (派生の派生を DB に持たない)。
インデックスは張らない。WHERE で見るのは `props <> '[]'` だけで、個人規模の
テーブルなら seq scan で足りる (YAGNI)。将来 `hFE>=200` のような検索を入れるときに
GIN を検討する。

`PropEntry` を `interface` ではなく `type` にしているのは、そのまま Prisma の
Json 列 (`InputJsonValue`) へ渡せるようにするため (interface には暗黙の
インデックスシグネチャが付かず Json 型と適合しない)。

DB から読むときは `parseStoredProps` で防御的に変換する (不正な形は捨てる)。

### PGroonga インデックスとの共存 (再掲・実際に踏んだ)

`prisma migrate dev` は表現できない PGroonga 索引を毎回 drift とみなし DROP を
生成する。マイグレーションは生成後に手で編集し、DROP を除いて
`CREATE INDEX IF NOT EXISTS ... pgroonga` で張り直す (冪等)。

**この手編集は実際に一度漏れている。** `20260715080225_add_circuit_svgs` は
自動生成の `DROP INDEX "items_memo_url_pgroonga_idx"` をそのまま適用しており、
全文検索の索引はそれ以降失われていた。`&@` は索引が無くても seq scan で動くため
検索結果は出続け、遅くなるだけで気づけない (512 件では体感差も出ない)。
`20260715111735_add_item_props` で張り直して回復させた。

適用済みマイグレーションは履歴なので直さない (checksum が変わる)。
気づく手段が無いのが問題なので、疑わしいときは索引の有無を直接見る:

```bash
docker compose exec db psql -U qr -d qr -c "\di items*"
```

## 3. 表示 — タグ検索のときだけ

特性表は **クエリにタグ項が 1 つ以上あり、かつプロパティを持つヒットがある** ときだけ
出す。表は「同族の部品を並べて比べる」ビューであり、タグ検索がまさにその族の指定。
全文検索のヒットは雑多で列がばらけノイズになり、空クエリ (全件ブラウズ) では
巨大な表になる。判定は `queryHasTagTerm()` 1 関数に隔離したので、条件を変えるのは
1 行で済む。

- 列 = ヒット集合に現れるキーの和集合 (初出順・綴りは初出のもの)
- 行 = プロパティを持つノートだけ。無いノートは表に出さない
- 1 列目「部品」= `#itemNo` リンク + メモ要約。`Item` に部品名の列は無いため
  (`itemNo` / `memo` / `url` / `tags` のみ)。部品名を確実に列へ出したいときは
  `device=2SC1815` と書けば普通の列になる (実メモで既に使われていた書き方)
- 部品名は幅を抑えて省略する。抑えないと横スクロールの表で hFE などの列が画面外へ
  押し出され、「並べて比べる」目的が崩れる (特にスマホ)

### ページ送りとの独立 (LIMIT を分ける)

一覧は `PAGE_SIZE = 20` でページ送りするが、表は現ページの 20 件から作ると
2 ページ目の部品が欠ける。表は専用クエリ `searchItemProps()` で全ヒットを対象にし、
`PROPS_TABLE_LIMIT = 200` を安全弁とする。要約はサーバー側で作り、memo 全文を
クライアントへ送らない。

上限で溢れたときは件数を数えて表の下に出す (`omitted`)。黙って打ち切ると
「これで全部」と読めてしまい、部品を見落とすため。表はまさに「並べて比べて
見落とさない」ためのものなので、そこを黙るのは表の目的に反する。
判定は 1 件多く取って溢れを検知し、溢れたときだけ count を撃つ
(通常の検索でクエリを増やさない)。

### 並べ替え

ヘッダのクリックでその列を並べ替える。まず値の数値部で比べ、数値部が無い/同じなら
文字列で比べる。この 2 段構えで `hFE=400` と `hFE=208` は数値順に、
`device=2SC1815` と `device=2N2222` (どちらも数値部は 2) や `pkg=TO-92` は
名前順になる。値を持たない行は昇順・降順どちらでも末尾。

単位換算はしない。`Vf=700mV` と `Vf=0.7V` が混在すると数値順 (700 > 0.7) になる。
揃えて書けば期待どおり並ぶので、既知の限界として受け入れる。

並べ替えの本体は純関数 `sortTableRows` に置き、コンポーネントはソート列と向きの
state だけ持つ (vitest が node 環境でクリックを検証できないため)。

## 4. 実装

| ファイル | 役割 |
| --- | --- |
| `src/lib/props.ts` | 抽出・集計・ソートの純関数 (DB 非依存) |
| `src/lib/items.ts` | upsert での再計算、`searchItemProps`、SELECT 射影 |
| `src/lib/search.ts` | `queryHasTagTerm` |
| `src/components/PropsTable.tsx` | 表 (client。ソート state のみ) |
| `scripts/backfillProps.ts` | 既存ノートのバックフィル (冪等) |

書き込み経路は `upsertMemo` / `upsertItem` に集約されているので、
`derivedFromMemo()` で tags と props をまとめて作り直せば再計算漏れが起きない。

`searchItems` は生 SQL で列を明示射影しているため、**`props` を SELECT に足し忘れると
型は通るのに実行時 undefined になる**。統合テストで検知する
(`searchItems projects the props column onto the item`)。

## 5. バックフィル

```bash
npx tsx scripts/backfillProps.ts
```

冪等 (`extractProps` は純関数)。`@updatedAt` を発火させない生 SQL で更新し、
一覧の並び順 (更新日順) を保つ。いつでも再実行して整合を回復できる。
記法を変えたら再実行する。

## 6. 今回見送り

- 単位換算ソート (mV と V を揃えて比べる): SI 接頭辞の解釈が要る。必要になってから
- プロパティでの検索 (`hFE>=200`): 検索文法に比較項を足し、`props` に GIN を張る
- アイテム詳細ページでのプロパティ表示 (チップなど)
- 表示条件をタグ検索以外へ広げる (`queryHasTagTerm` の 1 行で変えられる)
