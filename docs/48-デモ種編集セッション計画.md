# デモ種編集セッション計画 (doDemoSeedEdit.sh)

qr-demo.tommie.jp の初期化ノート (= 種 `qr_seed` の中身) を気軽に直せる
ようにする。docs/40 §3 の「種の撮り直し」手動手順を **編集セッション**
という形の 2 コマンドに包む。**アプリ・スキーマの変更はゼロ**、成果物は
シェルスクリプト 1 本と docs 更新だけ。

関連: [40-デモシード計画.md](40-デモシード計画.md) /
[39-デモ公開計画.md](39-デモ公開計画.md) / `scripts/reseedDemo.sh`

## 1. なぜこの形か (却下した案)

デモの初期化ノートを「固定ノート」として持たせる案 (#demo タグ +
作成者カラム / マルチユーザ化) は却下した。理由:

- 本命の穴は「削除」ではなく **「改変」**。demo アカウントは固定ノートも
  編集できるので、選択削除方式では落書きされた固定ノートが残る。結局
  「固定ノートの内容を初期状態へ戻す」= スナップショット復元が要り、
  いまの template DB 方式に戻ってくる
- 孤児画像の GC・セッション全消し・総量クォータのリセット・件数の完全
  一致など、`createdb -T` が無料でくれている保証を個別に作り直すことになる
- マルチユーザ化は user 表・全ルートの認可・帰属・per-user クォータまで
  波及する最大級の改修で、しかも編集保護は別途 ACL が要るため上の問題は
  解決しない。デモ目的にはオーバーキル (docs/07 で単語帳ですら重いと判断)

**結論**: template DB 再シード方式を維持したまま、docs/40 §3 の手動手順を
1 スクリプトに起こすのが最も気軽で速い。「固定ノート」の実体は種 DB の
中身そのものになり、区別問題も改変問題も構造的に消える (種を撮った瞬間の
状態がすべて、それ以外はすべて毎時消える)。

## 2. 配置と前提

- スクリプト: `qr-search/scripts/doDemoSeedEdit.sh`
  - **ローカル (手元) 実行の ssh ラッパ**。reseedDemo.sh と違い人間が
    手元から叩くものなので、vps2 への配置は不要
- 接続先: `DEMO_SSH_HOST` (既定 `vps2`)、リモート dir `~/qr-demo`
  (`DEMO_DIR` で上書き可)
- 依存: リモートに reseedDemo.sh・qr-demo-reseed.timer が設置済みである
  こと (docs/40 §4)。未設置でも警告して続行する (ブートストラップ許容)

## 3. サブコマンド設計

### `doDemoSeedEdit.sh start [--keep]`

1. `systemctl --user stop qr-demo-reseed.timer` — 編集中に毎時 0 分で
   作業が消える事故を防ぐ
2. reseed を 1 回実行 (`systemctl --user start qr-demo-reseed.service`) —
   **種の状態から編集を始める**ため。guest の落書きが混じった live を
   そのまま種にしない
3. 「ブラウザで編集 → 終わったら commit」と案内を表示

`--keep` は手順 2 をスキップ (live を巻き戻さない)。用途は 2 つ:

- **migration 後の種の撮り直し** (docs/40 §3 の罠)。migration 適用後に
  既定の reseed をすると live が旧スキーマへ巻き戻って app が壊れるので、
  この場合は必ず `--keep`。定型は `start --keep` → 直後に `commit`
- **初回ブートストラップ** (種も timer もまだ無い)。種無し・timer 未設置は
  警告だけ出して続行し、docs/40 §3 の初回作業もこのスクリプトに統一する

### `doDemoSeedEdit.sh commit`

1. **ガード**: timer が active なら中断 (`start` を経ていない = 毎時境界で
   編集が消えている可能性を警告)
2. `docker compose stop app`
3. `pg_terminate_backend` で `qr` への残存接続 (healthcheck の取りこぼし)
   を切る — `createdb -T` は source に接続 1 本でも失敗するため
4. `dropdb --if-exists --force qr_seed` → `createdb -T qr qr_seed`
5. `docker compose start app` — **trap で異常終了時も必ず実行**
   (app 止めっぱなし事故の防止)
6. 検証: `qr` と `qr_seed` の items 件数一致を表示
7. `systemctl --user start qr-demo-reseed.timer`

live の `qr` は差し替えないので REINDEX 不要。種側の PGroonga 索引は複製で
壊れるが、reseedDemo.sh が毎回 live を REINDEX する既存設計でカバー済み
(docs/39 §6-2)。

### `doDemoSeedEdit.sh abort`

編集を破棄して抜ける: reseed 1 回 (live を種へ戻す) → timer 再開。

### `doDemoSeedEdit.sh status`

timer の active/inactive、`qr` / `qr_seed` の items 件数、直近の reseed
ログ 1 行。「timer 止めっぱなし」の検知手段でもある。

### `doDemoSeedEdit.sh -h` / `--help` / `help`

使い方と各サブコマンドの説明を表示して終了する。

## 4. ガード・エッジケース

| ケース | 挙動 |
| --- | --- |
| commit/abort 中の失敗 | trap で `docker compose start app` と timer 再開を保証 |
| 種が無い (初回) | start は reseed をスキップして警告、commit が初回の種作成になる |
| timer unit 未設置 | 警告して続行 (ブートストラップ順序を許容) |
| timer 止め忘れ | `status` で見える。自動監視は YAGNI で見送り (docs に注意書き) |

## 5. 実装ステップ

1. `scripts/doDemoSeedEdit.sh` 作成 (`set -euo pipefail`、remote 実行は
   `ssh "$HOST" bash -s` にヒアドキュメントを流す)
2. 静的検査: `bash -n` + shellcheck。ssh 先で叩く ops スクリプトのため
   自動テストは実施せず、§6 の実機チェックリストで代替
3. vps2 で受け入れ確認 (§6)
4. docs 更新: docs/40 §3・§4 の手動コマンド列を doDemoSeedEdit.sh 参照に
   置換、§6 の見送り「種作成の自動化スクリプト」を解除。reseedDemo.sh 冒頭
   コメントの「種の作り方」も doDemoSeedEdit.sh へ誘導

## 6. 受け入れチェックリスト

- [ ] `start` → timer が inactive、live が種の状態に戻る
- [ ] ブラウザで捨てノート追加 → `commit` → 件数一致が表示され timer が active に戻る
- [ ] 次の毎時 0 分のリセット後も追加したノートが残る (= 種に入った)
- [ ] リセット後に全文検索が効く (REINDEX 経路の確認)
- [ ] `abort` → 編集が破棄され timer 再開
- [ ] `start --keep` → live が巻き戻らないこと
- [ ] commit 途中で Ctrl-C → app が起動状態に戻っている
