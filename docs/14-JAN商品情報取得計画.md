# JAN からの商品情報自動取得の計画 (Ver2.x)

書籍以外の商品バーコード (JAN) をスキャンして新規登録しても、商品名を
自分で打つことになる。これを Yahoo!ショッピングの API から自動で取り込み、
**商品名・ブランドが入った状態でエディタが開く**ようにする。

前提: ISBN の書誌自動取得 ([13-書誌自動取得計画.md](./13-書誌自動取得計画.md))
と同じ体験を一般商品に広げるもので、構成も 13 の相似形にする。
13 で決めた設計判断 (本文へコピーする理由・失敗時の扱い・
打ち始めたら書き換えない) はそのまま踏襲し、ここでは差分だけを書く。

## 1. 方針: Yahoo!ショッピング商品検索 API v3 を 1 本だけ引く

導線の全体像 (太字が今回足すところ):

```text
JAN をスキャン
  → /?q=<JAN> が 0 件 + 「#<JAN> を新規登録」ボタン
  → /edit/<次番号>?code=<JAN>
  → **ブラウザが /api/products/<JAN> を引く**
  → **サーバが Yahoo!ショッピングを引く**
  → **商品名・ブランドが入ったエディタ** (取れなければ従来どおり空)
  → 「更新」で作成 (それまで DB には何も書かない)
```

### 他のサービスを採らなかった理由

| サービス | JAN 検索 | キー | 判断 |
| --- | --- | --- | --- |
| Yahoo!ショッピング v3 | `jan_code` (公式パラメータ) | Client ID (無料) | **採用** |
| 楽天商品検索 | `keyword` に JAN (非公式) | アプリ ID (無料) | 見送り (§5) |
| JANCODE LOOKUP | — | — | 不採用 (データ再構築中で停止) |
| Verified by GS1 | Web 画面のみ | — | 不採用 (API が無い。登録も任意で網羅性が低い) |

- **楽天**は JAN 専用の口が無く、`keyword` 検索が JAN に一致し続ける保証がない。
  Yahoo! の取りこぼしが実際に苦になってから足す (13 の Google Books と同じ扱い)
- エンドポイントは
  `https://shopping.yahooapis.jp/ShoppingWebService/V3/itemSearch`。
  `appid` と `jan_code` を付けて GET、`results=1` で 1 件だけ受け取る
- レート制限は 1 クエリ/秒。1 スキャン 1 リクエストなので届かない

### openBD/NDL と違ってキーが要る

Client ID は `YAHOO_SHOPPING_APP_ID` としてサーバの環境変数に置き、
クライアントには出さない。13 ではサーバを挟む理由が NDL の CORS だったが、
ここでは**キーの秘匿**が理由になる (ブラウザから直接引くと Client ID が
全員に見える)。ルートが 1 本 (`/api/products/[jan]`) なのは 13 と同じ。

**キー未設定なら取得せず `data: null` を返す** (起動は止めない)。
この機能が無くても新規登録の導線は従来どおり動くべきで、必須の秘密として
起動時に検証する類のものではない。ただし黙って何も起きないと設定漏れに
気づけないので、サーバの警告に 1 行残す。

### 事前準備 (ユーザー作業)

1. [Yahoo!デベロッパーネットワーク](https://developer.yahoo.co.jp/) に
   Yahoo! JAPAN ID でログイン
2. アプリケーションを登録 (種別は「サーバーサイド」) して Client ID を控える
3. 本番とローカルの `.env` に `YAHOO_SHOPPING_APP_ID=<Client ID>` を足す

## 2. 構成: books 系と 1:1 対応

| 新規 | 対応する既存 | 中身 |
| --- | --- | --- |
| `src/lib/product.ts` | `book.ts` | `ProductSummary { title, brand }` |
| `src/lib/yahooShopping.ts` | `openbd.ts` | API 境界。応答を信用せずパース |
| `src/lib/productLookup.ts` | `bookLookup.ts` | 8 秒上限。1 ソースなので薄い |
| `src/app/api/products/[jan]/route.ts` | `api/books/[isbn]/` | `isJan` を検算 |

変更するもの:

- `scanRegister.ts` … `isJan(code)` を足す
  (13 桁の数字 + EAN-13 チェックデジット + **`isIsbn` でない**)。
  `scanRegisterMemo` の見出し組み立ては書籍・商品で共用できる形に均す
- `useBookPrefill.ts` … fetch 先と文言以外は商品でも完全に同じなので、
  コード種別を受け取る形に一般化する (`usePrefill({ kind, code, ... })`)。
  フックは複製しない
- `MemoEditor.tsx` … `isbn` prop を
  `prefill?: { kind: 'book' | 'product'; code: string }` に一般化。
  取得中の文言は種別で出し分ける (「商品情報を取得中…」)
- `edit/[itemNo]/page.tsx` … `isIsbn` → 書籍、それ以外で `isJan` → 商品、の分岐

### JAN の判定は緩くてよい

- **45/49 (日本) 縛りはしない**。輸入品も国内 EC ではそのまま売られていて、
  Yahoo! でヒットする。EAN-13 として検算が通れば引く
- ISBN から除外している ISMN (9790…) は JAN 側に落ちるが、楽譜も
  EC でヒットしうるので実害がない。専用の分岐は作らない

## 3. 事前入力の形

```text
サントリー 天然水 550ml
サントリー

#4901777018686
```

- **1 行目が商品名**。一覧の要約 (`memoSummary`) が商品名になる (13 §3 と同じ理屈)
- 2 行目はブランド名。欠けたら行ごと落とす
- **価格は入れない**。変動する値で、正本 (メモ本文) に固定する意味がない
- **`#book` に相当する種別タグは付けない**。書籍には「持っている本を
  一覧したい」という自然な括りがあったが、一般商品に同等の括りはない (YAGNI)。
  欲しくなったら手でタグを足せばよい
- 画像は入れない (13 §3 の書影と同じ理由)
- 商品名が取れなければ従来どおり `\n\n#<JAN>` のまま

### 商品名は EC の飾りを含んだまま入れる

Yahoo! の商品名は「【送料無料】」のような店舗の飾り文字を含むことがある。
機械的に落とす規則は書かない (店舗ごとに揺れて、正しい削り方が決められない)。
保存前のエディタに見えているので、気になればその場で消せる —
13 の「書名が `#` で始まるとタグになる (直さない)」と同じ判断。

## 4. 失敗したとき・打ち始めていたとき

13 の §4 (黙って従来の動作に戻す)・§5 (打ち始めていたら書き換えない) を
そのまま踏襲する。実装もフックの一般化 (§2) でそのまま流用される。
差分は 1 つだけ:

- **キー未設定** … 収録漏れと同じ `data: null` に落とす (§1)。
  UI には「商品情報が見つかりませんでした」が出る

## 5. 段階と見送り

- **Phase 1 (未実装)**: EAN-13 (ISBN 以外) を Yahoo!ショッピングから引く
- **見送り: 楽天フォールバック** (§1)。Yahoo! の取りこぼしが苦になってから
- **見送り: EAN-8 (短縮 JAN)**。スキャナは読める設定だが、まず 13 桁で
  運用して必要になってから足す
- **見送り: UPC-A (12 桁)**。先頭 0 埋めで EAN-13 化できるが、来てから
- **見送り: 商品情報のキャッシュ**。13 と同じ理由 (同じコードを 2 度
  スキャンすることは普通ない)

## 6. テスト

- 純関数 (vitest):
  - `parseYahooResponse` … 実応答から商品名・ブランド / 0 件 /
    壊れた JSON・配列でない応答 / 商品名なし / 文字列でない値
  - `isJan` … 正当な JAN / **ISBN を弾く** / チェックデジット違い /
    12 桁・14 桁 / 数字でない値
  - `scanRegisterMemo` … 商品あり / ブランド欠け / 商品なし
- API ルート … ISBN・チェックデジット違い・数字でない値は 400、
  キー未設定は `data: null` の 200
- ブラウザでの通し確認 (Playwright MCP、実 API):
  - 実在の JAN → 商品名・ブランドが入る
  - ISBN → 従来どおり `/api/books/` を引き、`/api/products/` は引かない (0 回)
  - 収録漏れの JAN (チェックデジットは正当) → 事前入力のまま導線は生きる
  - 応答を保留 → 打鍵 → 解放して、届いた商品情報が手書きを消さないこと
