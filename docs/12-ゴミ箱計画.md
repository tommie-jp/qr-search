# ノート削除 (ゴミ箱) の設計 (Ver2.x)

いまのアプリにノートを消す手段はない。書き損じや二重登録を消せるように、
**二段階削除**を入れる。検索結果で選択したノートを「削除」するとゴミ箱へ
入り (検索対象外になる)、ゴミ箱で「永久削除」して初めて DB から消える。
ここでは設計判断を残す。

## 1. 方針: `deletedAt` 列によるソフトデリート

導線の全体像:

```text
検索結果 → 「選択」→ チェック → 「削除」
  → deletedAt = now() (ゴミ箱行き。検索・タグ集計から消える)
ゴミ箱 (/trash)
  → 「復元」   … deletedAt = null (元どおり検索に出る)
  → 「永久削除」… DELETE (ここで初めて DB から消える)
```

ゴミ箱は別テーブルではなく、`items` に `deletedAt DateTime?` を足すだけの
ソフトデリートにする。

- **復元が UPDATE 1 発**。行の引っ越しがないので tags / props の派生
  キャッシュも memo 内の画像リンクもそのまま生きる
- **番号の予約が自然に効く**。`nextItemNo()` は `items` 全行から空き番号を
  探すので、ゴミ箱にある間はその itemNo が新規ノートに再利用されない。
  「ゴミ箱中に同番号の新規ノートが作られて復元できない」という衝突が
  構造的に起きない (§4)
- ゴミ箱テーブル方式は、移動時の整合性管理 (2 テーブル間のコピーと削除を
  トランザクションで守る) が増えるわりに、個人規模では利点がない

## 2. スキーマとマイグレーション

```prisma
model Item {
  // ...既存列...
  // ゴミ箱。null = 通常、非 null = ゴミ箱行き日時。検索・タグ集計は
  // null の行だけを対象にする。永久削除は行 DELETE。
  deletedAt DateTime? @map("deleted_at")
}
```

`deleted_at` にインデックスは張らない。数百件規模で seq scan で足りる
(props と同じ判断)。

**PGroonga 索引の罠に注意**。全文検索インデックスは Prisma スキーマで
表現できず drift 扱いになるため、`prisma migrate dev` が `DROP INDEX` を
自動生成する (20260715080225 で実際に索引が消えた)。前回
(20260715111735_add_item_props) と同じ手順を踏む:

1. `npx prisma migrate dev --create-only` で SQL を生成
2. 自動生成された `DROP INDEX` を手で消す
3. 念のため末尾に `CREATE INDEX IF NOT EXISTS "items_memo_url_pgroonga_idx"
   ON "items" USING pgroonga ("memo", "url");` を再掲して冪等に保つ
4. 適用後に `\di` で索引の生存を確認する

## 3. 検索対象外にする範囲 (items.ts)

原則: **読み取り系は `deleted_at IS NULL` を既定にし、ゴミ箱を見るのは
ゴミ箱専用の関数だけ**。

- `buildWhere()` / `buildPropsWhere()` … ベース条件として
  `deleted_at IS NULL` を常に入れる形に組み替える。現状は空クエリで
  `Prisma.empty` を返すが、「空クエリでも WHERE は付く」構造に変える
  (一覧ブラウズにもゴミ箱を出さないため)
- `listTags()` … `WHERE deleted_at IS NULL` を足す。ゴミ箱のノートの
  タグが検索窓の補完・タグ集計に出ないようにする
- `getItem()` … **フィルタしない**。/item ページはゴミ箱のノートも
  表示してバナーを出すため、deletedAt ごと返す (§5)
- `nextItemNo()` … **変更しない**。ゴミ箱の行も `items` にあるので、
  その番号は自然に「使用中」として飛ばされる (= 予約)。意図的なので
  コメントに明記する

新規に足す関数 (すべて items.ts):

- `trashItems(itemNos)` … `updateMany` で `deletedAt = now()`
- `restoreItems(itemNos)` … `updateMany` で `deletedAt = null`
- `purgeItems(itemNos)` … `deleteMany`。**where に
  `deletedAt: { not: null }` を必ず含め**、通常ノートを直接 DELETE
  できないことをサーバ側で保証する (二段階の強制)
- `listTrashedItems()` … deletedAt 降順。要約は `memoSummary` で作り、
  memo 全文はクライアントへ送らない (特性表と同じ流儀)。個人規模なので
  ページ送りは付けない (溢れて困る規模になったら考える)
- `countTrashedItems()` … 検索ページの「ゴミ箱 (n)」リンク用
- `countTrashedMatches(query)` … `buildQueryCondition` を再利用し、
  `deleted_at IS NOT NULL` 側で数える count。0 件検索時のゴミ箱案内
  (§5) 用。通常の検索では撃たない

## 4. 番号の再利用と QR シール (docs/10 §4 の選び直し)

[10-スキャン新規登録計画.md](./10-スキャン新規登録計画.md) §4 は「欠番を
埋める採番が安全なのは削除機能がないから。削除を入れるときは前提を
選び直すこと」としていた。ここで選び直す。

- **ゴミ箱にある間、番号は予約される** (§3 のとおり `nextItemNo()` が
  飛ばす)。うっかり削除→復元の間に番号を取られることはない
- **永久削除で番号は解放され、再利用されうる**。古いシールが残っていると
  別の部品を指すため、**永久削除は「その部品もシールも処分済み」を意味する
  操作**と位置づける。ゴミ箱ページにその旨を一言添える
- 「永久削除した番号を記録して再利用しない」台帳方式は採らない。テーブルが
  1 つ増えるコストに対し、守られるのは「処分したはずのシールが手元に残って
  いた」場合だけ。個人利用では二段階の確認で十分と判断する。実害が出たら
  台帳方式へ切り替えればよい (YAGNI)

なお、ゴミ箱のノートと同じコード (JAN 等) を再スキャンすると検索は
0 件になり「新規登録」ボタンが出る。気づかず押すと別番号で二重登録に
なるため、**0 件のときはゴミ箱にも当てて案内を出す** (§5) ことで防ぐ。

## 5. UI

### 検索結果 (選択モード) に「削除」ボタン

既存の選択 UI (`ItemList` のチェックボックス + `BulkTagToolbar`) に
相乗りする。ツールバーへ「🗑 削除」ボタンを足し、
`<button formAction={trashItemsAction}>` で**同じ form のままアクション
だけ分ける** (bulkTagAction に mode 分岐を足して汚さない)。
サーバーアクションは db.ts を巻き込まないよう page.tsx から prop で
降ろす (既存の `action` prop と同じ理由)。

confirm ダイアログは**出さない**。ゴミ箱行きは復元できる操作なので、
軽さを優先する (二段階削除にした意味がここで効く)。

### 0 件のときはゴミ箱にも当てて案内する

検索が 0 件のとき、同じ条件を `deleted_at IS NOT NULL` 側で数え直す
(`countTrashedMatches`、§3)。1 件以上なら emptyState (「該当する部品が
ありません」) に **「ゴミ箱に n 件の一致があります → ゴミ箱を見る」** の
リンクを足す。復元は /trash 側の既存ボタンで行う (案内はリンク 1 本で済み、
inline 復元の部品を増やさない)。

- ゴミ箱ノートと同じコードを再スキャンしたときもこの案内が出るので、
  気づかず「新規登録」して二重登録する事故を防げる
- **「新規登録」ボタンは案内の下に残す**。同じ本の 2 冊目など、新規が
  正しい場合もある。案内が並んでいれば、復元か新規かを選ぶのは
  ユーザで足りる
- 追加クエリは count 1 発、**0 件のときしか撃たない** (registerHref の
  採番と同じく、page.tsx で 0 件判定が出た場合だけ引く)
- スキャン由来 (`isTaggableCode`) に限定しない。手入力の検索でも
  「消したはずのノートを探して 0 件」の場面で「ゴミ箱にある」と
  答えられる方が親切で、条件を分ける理由がない

### ゴミ箱ページ /trash

- deletedAt 降順の一覧。各行に itemNo・要約 (`memoSummary`)・削除日時と、
  行ごとの「復元」「永久削除」ボタン。数件しか溜まらない想定なので
  チェックボックス選択は持ち込まない (要るようになったら足す)
- 「ゴミ箱を空にする」ボタンを上部に置く (全行 purge)
- **永久削除と空にするだけ `confirm()` で確認する**。「#1234 を完全に
  削除します。シールも処分済みですか?」のように番号と §4 の運用を添える
- 検索ページの件数表示の並び (「検索ヘルプ」の隣) に「ゴミ箱 (n)」
  リンクを出す。n = 0 のときは出さない (普段は目に入らないように)

### QR シールから開いた場合 (/item)

ゴミ箱のノートも `notFound()` にせず表示する。物理シールが貼られたまま
部品が出てくる動線はあり得るため。既存の「未登録」バナーと同じ位置に
黄色バナー「このノートはゴミ箱にあります」+ 復元ボタン (form) を出す。

編集リンクは残す。ゴミ箱のノートを保存しても deletedAt は触らない
(upsert の update に deletedAt を含めない) — **復元は明示的な操作だけ**に
して、「編集したら勝手に検索へ戻ってきた」を起こさない。

## 6. サーバーアクション (actions.ts)

- `trashItemsAction(formData)` … itemNo 群 (`readItemNo` 相当の検証を
  各要素に) → `trashItems` → `revalidatePath('/')` → 元の検索 URL へ
  redirect (bulkTagAction の back と同じ持ち回し)
- `restoreItemsAction(formData)` … `/trash` と /item バナーの両方から使う。
  restore 後は呼び出し元へ戻す
- `purgeItemsAction(formData)` / `emptyTrashAction()` … `purgeItems` を
  呼ぶだけ。二段階の保証は items.ts 側 (§3) にある

## 7. スコープ外 (今回はやらない)

- **画像の孤児 GC**。永久削除しても memo が参照していた `Image` 行は残る。
  ゴミ箱中は復元に備えて消せないし、purge 時に消すには「他のノートから
  参照されていないか」の判定が要る。DB は太るが動作は正しいので、将来の
  GC スクリプト課題として切り出す
- **自動で空にする (30 日で自動 purge)**。cron 基盤がなく、個人利用では
  手動で困らない (YAGNI)
- **/trash 内の検索窓**。数件規模の一覧で目視できる (0 件検索時の
  件数案内 (§5) は付けるが、あれはリンクまでで検索 UI ではない)

## 8. テスト

- items.test.ts (DB 統合。既存の `zzft` プレフィックス流儀で後始末):
  - ゴミ箱のノートは `searchItems` / `searchItemProps` / `listTags` に
    出ない (空クエリの一覧ブラウズ含む)
  - `trashItems` → `restoreItems` で再び出る
  - `purgeItems` は deletedAt が付いた行しか消さない (通常ノートを
    渡しても残る)
  - `nextItemNo` はゴミ箱の番号を飛ばす (ゴミ箱行きにした番号が
    採番されない)
  - `countTrashedMatches` はゴミ箱の一致だけ数える (通常ノートの一致は
    数えない / タグ・全文の両方の条件で当たる)
- コンポーネント (静的描画):
  - 選択モードのツールバーに「削除」が出る / 0 件選択で disabled
  - emptyState のゴミ箱案内: ゴミ箱一致あり → リンクが出る (新規登録
    ボタンと共存する) / 一致なし → 出ない
  - /trash の一覧と復元・永久削除ボタン、ゴミ箱が空のときの表示
  - /item のゴミ箱バナー (deletedAt 付きの item を渡す)
- 実機確認: マイグレーション適用後に `\di` で PGroonga 索引の生存確認 (§2)

## 10. 実装時に分かったこと

- **PGroonga 索引の DROP は本当に出た**。`migrate dev --create-only` の生成 SQL は
  `DROP INDEX "items_memo_url_pgroonga_idx";` を含んでいた。§2 の手順どおり
  差し替えて事なきを得た。適用後の `\di` でも索引の生存を確認済み
- **ゴミ箱行き / 復元は生 SQL で書いた**。Prisma の `updateMany` は
  `@updatedAt` を必ず打つため、削除・復元しただけで更新日が動き、更新順の
  並びが変わってしまう。本文は変わっていないので嘘になる。生 SQL なら
  `updated_at` に触らずに済む (実機でも更新日が動かないことを確認)
- **DNF の条件を AND で綴じるときは括弧が要る**。検索条件は `(…) OR (…)` の
  形なので、`deleted_at IS NULL` と裸で AND すると AND が OR より強く結合して
  条件が壊れる。`buildWhereFrom` で各条件を括弧に包んでいる
- **選択フォームの解釈は itemSelection.ts へ切り出した**。一括タグとゴミ箱行きが
  同じ形のフォーム (itemNo[] + q/page/sort) を送るため。戻り先 URL は必ず
  `buildSearchUrl` で組み立て、フォームの値をそのまま redirect 先にしない
  (オープンリダイレクト防止)
- **ツールバーの文言は「ゴミ箱へ」にした**。タグを外すチップが既に「削除」を
  名乗っていて紛らわしいため、チップ側を「タグを削除:」に改め、ノートの方は
  行き先を言う「🗑 ゴミ箱へ」にした (復元できることも伝わる)
- **confirm は preventDefault で止まる**。React の form action は
  defaultPrevented なら実行されないため、キャンセルで本当に消えないことを
  実機で確認した (SubmitButton のコメントにある既存の前提と同じ)
