# 画像挿入の対応形式拡張計画 (HEIF ほか)

memo への画像挿入 (ペースト / D&D / ファイル選択) を、現行の
PNG / JPEG / GIF / WebP に加えて **HEIC/HEIF (iPhone 標準)・AVIF・TIFF**
に対応させる。ここでは設計判断を残す。
**このメモは計画のみ。実装はまだ行わない。**(2026-07-18)

## 1. 現状と課題

- 対応形式の許可リストは `src/lib/uploads.ts` に集約
  (`MIME_TO_EXT` / `IMAGE_NAME_PATTERN` / `MAGIC_BYTES`)。
  クライアント側は `MemoEditorInner.tsx` の `ACCEPTED_IMAGE_TYPES`、
  API 側は `src/app/api/images/route.ts` のエラーメッセージが連動する
- iPhone のカメラ既定は HEIC。現状はアップロードが
  「対応していない画像形式です」で弾かれる
- 判定がクライアント申告の MIME (`file.type`) 起点。iOS/Windows は
  HEIC の MIME を空で送ることがあり、MIME 起点のままでは拾えない

## 2. 方針: 「保存形式は普遍、入口で変換」

ブラウザで表示できない形式を DB にそのまま保存すると、表示・クライアント
OCR (`createImageBitmap`)・画像検索 (埋め込み) がすべて壊れる。
そこで **アップロード時にサーバでウェブ標準形式へ変換して保存**する。

| 入力形式 | 扱い | 保存形式 |
| --- | --- | --- |
| PNG / JPEG / GIF / WebP | そのまま保存 (現行どおり) | 元のまま |
| AVIF | そのまま保存 (**新規**。全主要ブラウザが表示可) | avif |
| HEIC / HEIF | **WebP (q85) へ変換**して保存 | webp |
| TIFF | **WebP (q85) へ変換**して保存 (ブラウザ表示不可のため) | webp |
| SVG | 非対応のまま (スクリプト混入リスク。現行判断を維持) | — |
| BMP | 今回は見送り (§8) | — |

- 変換先を WebP にする理由: 透過を保てる・サムネイルで実績あり
  (`thumbnail.ts` が WebP 出力)・全ブラウザ表示可。品質は q85
- 変換時は `sharp().rotate()` で EXIF の向きを焼き込む
  (sharp は既定でメタデータを落とすため、GPS 等の位置情報も消える。
  プライバシー上はむしろ好都合)
- HEIF シーケンス / アニメーション AVIF は先頭フレームのみ
  (アニメ GIF / WebP は従来どおり無変換で保存されるので影響なし)
- 10MB 上限は**変換前の受信バイト**に適用 (現行の `MAX_IMAGE_BYTES` のまま)

## 3. デコーダ: sharp + WASM libheif (実機確認済み)

sharp 0.34.5 (libvips 8.17.3、導入済み) の対応状況を実ファイルで確認した:

- **AVIF / TIFF: sharp だけでデコード可**
- **HEIC: sharp の prebuilt バイナリでは不可**。コンテナ (metadata) は
  読めるが、ピクセルのデコードで
  `Support for this compression format has not been built in` になる
  (HEVC は特許の都合で prebuilt に含まれない)
- **BMP: sharp の入力形式に無い** (§8)

HEIC のデコードは **`heic-decode` (2.1.0, 依存は `libheif-js` =
libheif + libde265 の WASM ビルドのみ)** を採用する:

```text
HEIC bytes → heic-decode.all() でコンテナ解析 (寸法のみ・画素は未確保)
           → 画素数を上限 (50MP) と照合し、超えれば確保前に拒否
           → primary.decode() で RGBA raw → sharp(raw) → .rotate().webp(q85)
```

**解凍爆弾よけは decode の前に効かせる**。既定の `heic-decode()` (one)
は先に全画素を確保してしまい寸法の先読みができない。`all()` は解析だけで
寸法を返すので、`width*height` が上限を超える入力を**メモリ確保の前に**弾ける
(数 KB の HEIC が数万×数万を名乗る攻撃で、単一プロセスを OOM させない)。

- WASM なので本番の `node:24-alpine` (musl) でもそのまま動く。
  ネイティブビルド不要
- 対抗案の `heic-convert` は同作者の上位パッケージだが、エンコードが
  純 JS (jpeg-js) で遅い。エンコードは既にある sharp に任せる方が速い
- libvips を libheif 付きで自前ビルドする案は Docker が重くなるので不採用
- `next.config.ts` の `serverExternalPackages` に `heic-decode` /
  `libheif-js` を追加 (onnxruntime と同様、WASM をバンドルさせない)

## 4. 形式判定: MIME 起点をやめ、マジックバイト起点にする

現行は `extForMime(file.type)` → `matchesMagicBytes` の順で、クライアント
申告 MIME が空だと判定できない。**先頭バイトから形式を直接判定する
`sniffImageFormat(bytes)` に置き換え**、MIME は使わない (詐称対策も兼ねて
現行より堅くなる)。

- 既存署名 (png/jpg/gif/webp) は流用
- HEIC/AVIF は ISO-BMFF なので offset 0 固定署名では判定できない。
  **`ftyp` ボックス (offset 4) を読み、major brand + compatible brands
  を見る**: `heic/heix/hevc/mif1/msf1` → heic、`avif/avis` → avif。
  major が `mif1` の場合は compatible brands で heic / avif を区別する
  (手元の Nokia サンプルは major=mif1, compat=[mif1, heic] だった)
- TIFF は `II*\0` / `MM\0*` の 2 署名

## 5. 変更点一覧

### サーバ

- `src/lib/uploads.ts` — `sniffImageFormat` 追加、`MAGIC_BYTES` に
  ftyp 判定を拡張。`IMAGE_NAME_PATTERN` に `avif` を追加
  (heic/tiff は保存名に現れないので追加しない)。`mimeForName` に avif
- `src/lib/normalizeImage.ts` (**新規**) — `normalizeImage(bytes, format)`。
  passthrough / sharp 変換 / heic-decode 変換を振り分けて
  `{ bytes, mime, ext }` を返す。`limitInputPixels: 50_000_000`
  (thumbnail.ts と同じ爆弾対策) と WebP 上限 16383px 内への縮小を含む。
  変換失敗は 400 で返す (壊れたファイルを黙って保存しない)
- `src/app/api/images/route.ts` — 判定を sniff 起点に変更し、
  `normalizeImage` を通してから `saveImage`。エラーメッセージの形式列挙を
  更新 (`png/jpg/gif/webp/avif/heic/tiff`)
- `src/lib/coverLookup.ts` — 同じ sniff / normalize を共用 (DRY)。
  書影は実質 jpeg/png だが、共通化で許可リストの二重管理をなくす
- サムネイル (`thumbnail.ts`) と埋め込み (`embedImageServer.ts`) は
  **変更不要**: `saveImage` に届く時点でデコード可能な形式になっている

### クライアント (`MemoEditorInner.tsx`)

- `ACCEPTED_IMAGE_TYPES` に `image/avif,image/heic,image/heif,image/tiff`
  と拡張子 `.heic,.heif,.tif,.tiff,.avif` を追加
  (拡張子併記は MIME を空で送る OS のファイル選択ダイアログ対策)
- `imageFiles()` のフィルタを「`image/*` **または** 対応拡張子」に緩和
  (MIME 空の HEIC を落とさないため。最終判定はサーバの sniff が行う)
- **OCR は元 File ではなく保存後の画像を使う**: 現状 `insertImages` が
  元 File を `ocrIntoDoc` に渡すが、HEIC は Chrome/Firefox の
  `createImageBitmap` でデコードできない。アップロード応答の URL を
  fetch した Blob を渡すよう変更 (常にこの経路に統一すれば、OCR が
  検索・表示と同じピクセルを見ることになり分岐も消える)

- `src/lib/normalizeImage.test.ts` (**新規**) — passthrough / AVIF・TIFF

## 7. セキュリティ確認

- 判定強化: MIME 詐称はマジックバイト起点判定で従来より堅くなる
- 新規デコーダ (libheif WASM) が信頼できない入力を読む。WASM サンドボックス
  内で完結し、解凍爆弾対策を変換経路にも適用する (TIFF は sharp の
  `limitInputPixels`、HEIC は `all()` の寸法先読み。§3 参照)
- SVG 非対応の判断は維持。保存名はサーバ生成 UUID のままで
  トラバーサル対策も現行どおり
- `sniffImageFormat` は route / coverLookup が素で呼ぶため throw しない契約。
  ISO-BMFF 解析に `byteLength < 12` の明示ガードを置き、回帰テストで固定した

### security-reviewer の結果 (2026-07-18)

- **HIGH: HEIC 解凍爆弾** — 対応済み。`all()` の寸法先読みで確保前に拒否
  (回帰テスト `normalizeImage.bomb.test.ts`)
- **MEDIUM: 正規化失敗を無ログで握り潰し** — 対応済み。route.ts で
  `console.error` を残す (thumbnail.ts と同じ流儀)
- **LOW: sniff の非 throw 保証が暗黙的** — 対応済み。明示ガード + 回帰テスト
- **積み残し (別課題)**:
  - MEDIUM: `checkUploadRequest` の Content-Length ゲートは chunked 送信で
    素通りし、`formData()` が本文を全量バッファする (coverLookup の
    `readWithLimit` 相当のストリーミング上限が無い)。本機能以前からの挙動で、
    直すにはストリーミング multipart 解析への置換が要るため別 PR とする
  - LOW: AVIF は無変換保存で、アップロード時のサーバ側画素ガードが無い
    (png/jpg/gif/webp と同じ既存挙動。原寸デコードはブラウザ側で、
    サムネ生成は fail-safe なので実害は小さい)

## 8. 見送り: BMP / JPEG XL / ICO

- **BMP**: sharp が読めず、追加デコーダ (`bmp-js`、2017 年から未更新で
  RLE 圧縮非対応) の品質リスクに対して実需が薄い。ブラウザは BMP を
  表示できるので、必要になったら「無変換で保存」の 1 行追加で足りる
- **JPEG XL**: ブラウザ普及が Safari のみで時期尚早
- **ICO / SVG**: 写真メモの用途外。SVG はスクリプト混入リスクで非対応継続
