# ノートのお絵かき (Ver2.x)

ノート編集画面から手描きの絵を作り、1 枚の画像として本文へ挿し込む。
白紙に描くほか、**既にノートに貼ってある画像を下敷きにして描ける** ——
写真に丸や矢印で注釈を入れる用途を主に想定している。実装済み (2026-07-20)。

## 1. ライブラリの選定

`fabric` (v7) + `@erase2d/fabric` を採る。別プロジェクト
(39-guruguru-avator の `src/draw-layer.jsx`) で同じ組み合わせを運用していて、
罠と対処が判っているのが決め手。

比較した候補と外した理由:

- **tldraw** — ホワイトボードとしては高機能だが、無償利用ではウォーターマークが
  付く。ノートに貼る画像に入るのは受け入れられない
- **Excalidraw** — 手描き風の作図に寄っていて、写真への注釈には過剰。
  バンドルも重い
- **react-sketch-canvas** — 軽いが消しゴムがストローク単位 (引いた線が丸ごと
  消える) で、文字も置けない。注釈用途には力が足りない

`fabric` は gzip 後 90KB 程度。お絵かきを開くまで読み込まないよう、
`next/dynamic` の `ssr: false` で遅延させる (CodeMirror と同じ流儀)。

なお `fabric` は `canvas` / `jsdom` を **optional** 依存に持つ (Node で動かす
ための実装)。ブラウザでしか使わないので実行時には要らないが、`npm ci` では
落ちてくる。Dockerfile は `npm ci --ignore-scripts` なのでネイティブビルドは
走らず、runner 段は `.next/standalone` しか運ばないため本番像には入らない。

## 2. 画面と導線

編集画面のツールバーに「お絵かき」ボタンを足した (`MemoEditorInner.tsx`)。
押すと全画面のお絵かき画面 (`DrawModal`) が開く。

- **下敷きの選び方**: 「後から OCR」と同じ `imageAtCursor` で、カーソルに
  いちばん近い自前画像 (`/api/images/...`) を探す。見つかればそれを背景に
  敷いた状態で開く。画像の上にカーソルを置いて押せばその画像に描ける
- 下敷きが要らないときは「白紙にする」で外せる。逆に外した後は「画像に描く」で
  戻せる。**器の寸法が変わって描いた線の位置が合わなくなる**ので、
  何か描いた後の切り替えは確認を挟む
- 編集フォームの中から開くため、`createPortal` で body へ逃がす。
  form の中に置くと、中のボタンが「更新」として送信されてしまう
- 道具はペン / 消しゴム / 選択 / 文字。色と太さは localStorage に持ち越す

## 3. 寸法と書き出し

**canvas の論理サイズがそのまま書き出す画像の解像度になる**。画面上の
大きさは CSS の `transform: scale()` で縮めて収める。fabric はポインタ位置を
`getBoundingClientRect` と `canvas.width` の比で補正するので、CSS で縮めても
描く位置はずれない (`fabric/dist` の `getPointer` で確認済み)。

つまり「表示に合わせて小さく作る」必要はなく、解像度として十分な論理サイズを
取ってよい。

- **下敷きあり**: 元画像の画素をそのまま使い、長辺だけ 1600px に収める
  (`canvasSizeForImage`)。小さい画像は引き伸ばさない
- **下敷きなし**: 開いた時点の表示領域の縦横比のまま、長辺を 1600px に
  伸ばす (`blankCanvasSize`)。器を決めるのは開いた 1 度きり —— 描いている
  最中に画面を回して作り直すと、それまでの線がずれる
- 書き出しは **WebP (q=0.92) を第一候補**、駄目なら PNG。写真に注釈を入れると
  PNG では 10MB の投稿上限 (`src/lib/uploads.ts`) に届きうる。`toBlob` は
  非対応の形式を黙って PNG にすることがあるので、`blob.type` を見てから決める
- 下地は白 (`#ffffff`)。透過のままだと貼った先の背景次第で線が見えなくなる

**元の画像は書き換えない**。描いたものは別の画像として増える (元のリンクは
本文にそのまま残る)。失敗しても元が失われない、という一点で非破壊にしている。

### 3-2. 消しゴムが重かった話

実装当初、消しゴムを選ぶと目に見えて重くなった。原因は 2 つあり、どちらも
**器の画素数**が効く形だった。

**(1) `EraserBrush` は生成した瞬間に canvas 1 枚ぶんのメモリを確保する。**
効果用の裏 canvas を `document.createElement('canvas')` で作り、
`canvas の論理サイズ × getRetinaScaling()` の寸法を与える。長辺 2400px の器を
DPR=2 の端末で開くと 4800×3600×4byte ≒ **69MB** (DPR=3 なら 155MB) を、
道具を選んだだけで確保していた。

**(2) 消しゴムは 1 フレームに canvas 全体を 3 回描く。**
`needsFullRender()` が常に `true` を返すため、ペンのような差分描画ができず
ポインタが動くたびに全体を再描画する。さらに `after:render` で `drawEffect()`
が**もう一枚のフルサイズ canvas に背景と非 erasable オブジェクトを描き直し**、
それを合成する。背景写真の `drawImage` もそのたびに走る。

打った手:

- **`enableRetinaScaling: false`** (これが一番効く)。既定の `true` は実バッファを
  論理サイズ × DPR にするが、ここでは論理サイズを解像度として大きく取り
  表示は CSS で縮めているので、その上に DPR を掛けても画面には 1px も
  現れない。DPR=3 の端末では 9 倍の画素を捨てるためだけに描いていたことになる。
  書き出しへの影響も無い —— `toCanvasElement` は内部で
  `enableRetinaScaling` を false にしてから描くので、出力解像度は論理サイズのまま
- **下敷きの上限を 2400px → 1600px** (白紙と揃えた)。(2) の 3 回ぶんが
  そのまま 2.25 分の 1 になる
- **ブラシを作り直すのは道具が変わったときだけ**にした。色・太さ・表示倍率の
  変更は既存のブラシの `width` / `color` を書き換えるだけで済ませる
  (`applyBrushStyle`)。以前は太さを変えるたび、さらに画面リサイズで
  `displayScale` が動くたびに (1) の確保をやり直していた
- **`EraserBrush.dispose()` を必ず呼ぶ**。裏 canvas を 0×0 にして GC に返す
  ためのもので、呼ばずに捨てるとそのぶん残る。道具を切り替えたときと
  画面を閉じたときに呼ぶ。描かない道具 (選択・文字) のときは
  `freeDrawingBrush` を持たせない

## 4. 挿入経路

描き上がった canvas を `File` にして、ファイル選択・ペーストと同じ
`insertFiles` に流す。アップロード進捗・DB 保存・サムネイル生成・画像検索用の
埋め込み・`![](url)` の挿入まで既存の仕組みがそのまま効くので、
**サーバ側の変更は無い** (WebP / PNG はどちらも `sniffImageFormat` が通す)。

`insertFiles` には `InsertFilesOptions` を足した:

- `imageAlt` — お絵かきは `![お絵かき 2026-07-20 12:34](...)` の形で入れる。
  PDF がファイル名を alt に残すのと同じ狙いで、PGroonga の全文検索から
  「お絵かき」で引ける
- `ocr: false` — 自分で描いたものを OCR に掛けない。モデルの読み込みは
  スマホでは高くつく。読ませたくなったら「後から OCR」ボタンで足せる

## 5. 取り消し履歴

`src/lib/draw/history.ts` に、fabric のシーン JSON を並べた不変の履歴を持つ。
`index` を動かすだけで戻る / やり直すので、分岐した歴史は持たない
(エディタの undo と同じ)。上限 40 手 —— シーン JSON は 1 枚が数 KB〜数十 KB に
なるため、無制限に持つとスマホで効いてくる。

消しゴムだけは `object:*` イベントを出さない (既存オブジェクトに `clipPath` を
足すだけ) ので、`EraserBrush` の `end` イベントから履歴を積む。

## 6. `@erase2d` の罠

- `import '@erase2d/fabric'` の **副作用 import が要る**。`ClippingGroup`
  (`type: 'clipping'`) を fabric の `classRegistry` に登録するのがこれで、
  忘れると消した跡を含むシーンを `loadFromJSON` で戻せない (= 取り消しが壊れる)
- `@erase2d` は **`erasable` が真のものしか消さない**。fabric v7 の既定は
  未設定なので、`object:added` で立てて回る。ただし履歴から戻した図形は
  自分の `erasable` を持っているので上書きしない
- 背景画像は `erasable: false` にする。`EraserBrush` は
  `canvas.backgroundImage.erasable` が真のときだけ背景も消すため、
  明示しておかないと将来の版で写真そのものが消えかねない
- シリアライズは `toObject(['erasable'])` —— 既定の `toObject()` では
  `erasable` が落ちて、戻したときに消せない図形になる
- `erasable` は fabric の型に無い拡張プロパティ。読むときだけ型を広げる
  (書き込みは `set` が任意のキーを受ける)

## 7. 残件

- 描いた絵を**後から編集し直す**ことはできない。`Image` テーブルは描画結果の
  バイト列しか持たないため、ストロークを残すにはスキーマの追加が要る。
  必要になってから考える (YAGNI)
- 図形 (直線・矢印・四角) の道具は無い。注釈は手描きの丸と文字で足りている
