# 挿入済み画像の 90° 回転計画

メモ本文に貼った画像を、閲覧画面の拡大表示から 90° ずつ回して保存し直す。
横倒しで貼ってしまった写真を後から直せるようにするのが目的。

## 1. 方針: 同名上書きではなく「新 UUID で保存し直し + 本文の URL 置換」

同名上書きは 2 つの理由で採らない。

- 配信 URL は `private, max-age=31536000, immutable`
  ([api/images/[name]/route.ts](../src/app/api/images/%5Bname%5D/route.ts)) で
  1 年焼かれる。同じ URL のまま中身を差し替えても、閲覧者のブラウザには旧画像が
  残る。サムネの `?v=` 版数はグローバルで、1 枚だけを割ることはできない。
- `thumb` / `embedding` は `data` からの派生キャッシュ。
  [saveImage()](../src/lib/imageStore.ts) を通せば両方とも自動で再生成される。
  上書き経路を新設すると派生の作り直しを別に書くことになる。

そこで回転済みバイト列を `saveImage()` で**新規保存**し、本文中の旧 URL を
新 URL に置換する。旧画像行は消さない — 画像 GC (docs/20、未実装の sweep) が
いずれ回収する設計そのままで、undo や開きっぱなしのタブが旧 URL を指していても
画像が割れない。

## 2. UI: ZoomableImage の拡大オーバーレイに回転ボタン

- 拡大表示 (黒背景) の隅に「↻」ボタンを置く (`stopPropagation` で閉じと分離)。
- **押すたびに即 CSS `transform: rotate()` で回す** — サーバ往復を待たせない。
- 連打対策: 押すごとに投げると回数分の孤児画像ができる。約 800ms の
  デバウンス後に累計角度 (`n×90 % 360`) を **1 回だけ** POST する。
  0° に戻っていたら送らない。
- 成功したら新 URL を差し替え、CSS 回転を 0 に戻す。小さい表示側の `<img>` も
  同じ state で新 URL に追従させる。
- **0° へのリセットはアニメーションしない** (transition を切って瞬時に戻す)。
  差し替え先は既に回転済みの新画像なので、そこから 90°→0° をアニメーション
  すると「もう一回回った」ように見える (初版で実際に起きた)。アニメーション
  するのは押した瞬間の +90° だけ。
- 確定後に `router.refresh()` でサーバ描画を取り直す。回転は本文の URL 置換
  なので、ページ (とくに**まだ開いていない編集タブ** — MemoPanel は開くまで
  マウントを遅延する) を新しい本文に追随させる。開きっぱなしの CodeMirror は
  refresh でも古い本文のまま (クライアント状態は保持される) — そこから保存
  すると旧 URL に戻るが、旧画像は残っているので割れない (§5 の既知の制約)。
- 失敗時はボタン付近にエラーを出し、CSS 回転を元へ戻す (握り潰さない)。
- ボタンは `MarkdownView` の `allowRotate` prop が true のときだけ出す。
  [ItemView.tsx](../src/components/ItemView.tsx) からのみ true を渡し、
  公開ビュー・docs ページには出さない。

## 3. API: `POST /api/images/[name]/rotate`

body は `{ angle: 90 | 180 | 270 }`。手順:

1. ログイン + CSRF (`checkUploadRequest` 相当) を検査。
2. 名前を `isValidImageName` で検算。**gif は対象外** (アニメ GIF は別扱いが
   要るため 400)。動画・音声・PDF も当然対象外。
3. 行を取得し、`sharp(data).autoOrient().rotate(angle)` で回して同一形式へ
   再符号化。**`autoOrient()` が必須** — passthrough 保存の JPEG は EXIF の
   向きを持ったままなので ([normalizeImage.ts](../src/lib/normalizeImage.ts) で
   jpg/png 等は素通し)、EXIF を残したまま画素だけ回すとブラウザ側で二重回転する。
   sharp は既定でメタデータを落とすので、autoOrient 後の出力は画素の向きが正になる。
4. `saveImage()` で新規保存 (thumb + embedding が自動でつく)。
5. 本文書き換え: `/api/images/<旧名>` を含む item を全件 (ゴミ箱内も) 探して
   置換し、`upsertMemo` 相当で保存 (派生列 tags/props も再計算)。検索は
   `position(url IN memo) > 0`。**LIKE は PGroonga に乗っ取られるので使わない**。
   URL 文字列だけの置換なので `![alt|200](…)` の幅記法・alt は無傷。
6. 新 URL を返す。旧行は消さない (GC に委ねる)。

デモモードは既存アップロードと同じ門番 (`checkDemoUploadQuota`) を通す —
実質ストレージが 1 枚分増えるため。

## 4. 実装フェーズ (TDD)

1. `src/lib/rotateImage.ts` — sharp で autoOrient + rotate。縦横入替・EXIF 二重
   回転が起きないこと・gif 拒否をテスト。
2. `rewriteImageReference()` — 本文の純関数置換 + `position()` での対象列挙・
   一括更新 (トランザクション)。
3. API route — 認証・名前検算・404・非対応 400・成功時の新 URL 返却。
4. UI — ZoomableImage にボタン + デバウンス確定。回転の状態計算は純関数
   (`rotationState.ts`) に切り出してテスト (連打 3 回 → angle=270 を 1 回)。

## 5. 承知しておく制約

- **JPEG/WebP/AVIF は再符号化で世代劣化する** (90° 自体は可逆だが再エンコードが
  挟まる)。3 回押して 270° でも符号化は 1 回にまとまるのがデバウンスの利点。
  PNG は無劣化。
- 別タブで同じノートを編集中に回転すると、そちらの保存が旧 URL を書き戻しうる
  (旧画像は残るので割れない。向きが戻るだけ)。
- 画像 GC 未実装の現状、旧画像は DB に残り続ける (GC 実装時に回収)。
- 検索の画像モード ([ImageMasonry.tsx](../src/components/ImageMasonry.tsx)) の
  タイルは新名のサムネ URL に自然に追従するが、タイル上に回転ボタンは置かない
  (初版はノート閲覧の拡大表示のみ)。
