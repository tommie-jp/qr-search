# デモ公開計画 (残機能)

デモインスタンス (qr-demo.tommie.jp) を guest 書き込み可で公開するまでに
必要な残りの全機能。DEMO_MODE フラグ (docs/38) は実装済みで、この計画は
その §8 で見送った分 + 運用側の仕組みを 1 つにまとめたもの。
検討の経緯は 41-QR-search/docs/08。

公開条件: **§2〜§7 がすべて揃うまで guest 書き込み可にしない** (docs/38
冒頭の約束)。§8 (シード内容) は公開する体験そのものなので同時に用意する。

## 1. 全体像 — 何が何を守るか

| 節 | もの | 守るもの |
| --- | --- | --- |
| §2 | 総量クォータ (アプリ) | ディスク (再シード間の書き溜め) |
| §3 | robots.ts (アプリ) | デモが検索結果に載らないこと |
| §4 | ログイン案内 (アプリ) | 訪問者が入れること |
| §5 | デモスタック (compose) | 本番との資源・データ隔離 |
| §6 | 再シード (systemd timer) | データの TTL、消し忘れゼロ |
| §7 | nginx (エッジ) | 連打・巨大ボディ |

アプリ側 (§2〜§4) を 1 コミット、運用側 (§5〜§7) を 1 コミットの
2 段で進める。

## 2. 総量クォータ — アップロード総量とノート数

再シード間隔 (毎時) の間でも数 GB 書けるので、上限を 2 つ入れる。
定数は `src/lib/demoLimits.ts` に新設して集約する:

- `DEMO_MAX_TOTAL_UPLOAD_BYTES = 200MB` — 添付の総量
- `DEMO_MAX_ITEMS = 500` — ノート数

### 2-1. 添付の総量

`/api/images` POST (唯一のアップロード口) で、デモのときだけ
保存前に合計を見る:

```sql
SELECT COALESCE(SUM(octet_length(data)), 0) FROM images
```

`$queryRaw` で引く (Prisma の aggregate は bytea 長を取れない)。
超過は 507 Insufficient Storage +「デモの保存容量が いっぱいです。
定期リセットをお待ちください」。判定は `denyUnlessLoggedIn` →
`checkUploadRequest` の後 (DB を見るので順序は最後)。

厳密な直列化はしない。並行アップロードで数 MB はみ出ても、
守りたいのは「GB 単位の書き溜め」であり、毎時の再シードで消える。

### 2-2. ノート数

新規作成の口は `upsertMemo` / `upsertItem` (items.ts:137-156) の
create 経路 2 つだけ (ENEX インポートはデモでは封鎖済み)。
デモのときだけ upsert の前に存在を確認し、**新規になる場合のみ**
`prisma.item.count()` を見て上限なら throw する
(既存ノートの更新は数に依らず通す — 使い勝手を殺さない)。

競合で 500 を数件超えるのは許容 (単一 guest とは限らないが、
守りたいのは桁違いの量産)。

## 3. robots.ts — デモは全 UA 拒否

`src/app/robots.ts` を新設し `isDemoMode()` で分岐:

- デモ: `disallow: "/"` (全 UA)
- 本番/ローカル: 全許可 (rules は allow)

**本番にも robots.txt が新たに生える**ことに注意。全許可なので挙動は
実質変わらず、公開ノートの「検索エンジンに載せない」は従来どおり
ページ側の noindex metadata が担う (robots.txt で crawl を止めると
noindex を読んでもらえなくなるため、本番は許可が正しい)。

## 4. バナーにログイン案内 — DEMO_LOGIN_HINT

Basic 認証のネイティブダイアログには説明を出せないため、押す前に
見える場所 = DemoBanner に資格情報を書く (docs/38 §6 の宿題)。

- env `DEMO_LOGIN_HINT` (自由文。例: `ログイン: demo / <パスワード>`)
- 設定されているときだけ DemoBanner の末尾に 1 行足す。
  値はデモスタックの .env にだけ置き、リポジトリには入れない
- layout が env を読み props で降ろす (site.ts と同じ流儀)
- compose.yaml の environment に `DEMO_LOGIN_HINT: ${DEMO_LOGIN_HINT:-}`
  を配線 (列挙に無い変数は届かない罠の対象)

guest のパスワードは公開する前提なので強度は不要だが、bcrypt cost は
12 のままでよい (照合 1.75 秒は nginx の枠 §7 が守る)。

## 5. デモスタック — 同一イメージ 2 セット目

docs/17 C 案の流儀で、vps2 に完全な別スタックを立てる。DB ごと分ける
(同一 Postgres 内の別 DB にしない) — guest が DB を膨らませたとき
本番とディスク・メモリを共有させないため。

- vps2 に `~/qr-demo/` を作り、compose.yaml + `compose.demo.yaml`
  (どちらもリポジトリの本番と同一物を配置) + デモ専用 .env
- `.env` に 2 行:
  - `COMPOSE_FILE=compose.yaml:compose.demo.yaml` — 素の
    `docker compose` で override が効く (`-f` の打ち忘れが起きない)
  - `COMPOSE_PROJECT_NAME=qr-demo` — コンテナ/volume の接頭辞が分かれ、
    本番と資源・データが混ざらない
- override (`compose.demo.yaml`) の中身 (実装済み):
  - `DEMO_MODE: "1"` を**直書き** (docs/38 §2 の「旗の欠落は保護オフに
    倒れる」対策。.env 頼みにしない)
  - `mem_limit` / `cpus` を app (1g / 1.0) と db (512m / 0.5) に付ける
    (本番 compose には無い。デモだけの上限)
- デモ .env の要点: `APP_ENV=production` (ピンクにしない。目印は DEMO
  バッジ/バナー) / `DEMO_LOGIN_HINT=…` (§4) /
  `QR_BASE_URL=https://qr-demo.tommie.jp` (忘れるとデモで印刷した QR が
  本番を指す) / `APP_PORT=3100` / `DB_PORT=5433` / `WEBAUTHN_*` は空 /
  外部 API キーは置かない / `BASIC_AUTH_*` は guest 用
- **DB 名は本番と同じ `qr` のまま**でよい。デモは別コンテナ (別 Postgres)
  なので `qr` 同士でも衝突しない。DB 名を変えると compose.yaml の
  `POSTGRES_DB` / `DATABASE_URL` を override する羽目になるので、隔離は
  コンテナの層に任せる (compose.yaml を触らずに済む)
- イメージは本番と同じ `qr-search-app:latest` を共用。本番デプロイの
  `docker save/load` で届いたものをそのまま使う
- デプロイ: doDeploy.sh に `DEPLOY_DB_PORT` (migrate トンネルのリモート側
  DB ポート) と `DEPLOY_APP_PORT` (ヘルスチェック先) を足した (どちらも
  既定は本番の 5432 / 3000 で後方互換)。デモは別ポートを渡す:
  `DEPLOY_REMOTE_DIR=qr-demo DEPLOY_DB_PORT=5433 DEPLOY_APP_PORT=3100 ./doDeploy.sh`
  (**APP_PORT を渡さないと本番 app (3000) を叩いて誤って成功と判定する**)

## 6. 再シード — template DB + systemd timer

「DB を定期的に消す」の実装 (`scripts/reseedDemo.sh` +
`deploy/systemd/qr-demo-reseed.{service,timer}`、実装済み)。ノート単位の
TTL ではなく DB ごと初期状態へ戻す (消し忘れが構造的に起きない。
41-QR-search/docs/08 Q2)。対象は live `qr` と 種 `qr_seed` (§5 の通り
DB 名は本番と同じ `qr` のまま)。

### 6-1. 種 (seed) の作り方

1. デモスタックを立て、guest でログインしてショーケースノート (§8) を
   作り込む
2. `docker compose exec -T db createdb -U qr -T qr qr_seed` — この瞬間の
   `qr` が種 `qr_seed` になる (添付は bytea なので画像・音声ごと 1 DB に閉じる)

種を撮り直すときは `dropdb -U qr qr_seed` してから同じ手順。

### 6-2. 毎時のリセット

reseedDemo.sh を **ユーザ systemd timer** (`systemctl --user`、毎時 0 分)
から実行する。root 不要 (docker は vps2 ユーザが docker グループで叩ける)。
設置と有効化は service ファイルのヘッダに書いた (`loginctl enable-linger`
を含む)。スクリプトの動き:

1. 種 `qr_seed` の存在確認 (無ければ §6-1 未了として明確に落ちる)
2. `docker compose stop app` — 接続を確実に断つ。template コピーは
   source/target への接続が 1 本でもあると失敗するため、
   `pg_terminate_backend` で競走するより止めるほうが固い
3. `dropdb --if-exists --force -U qr qr` → `createdb -U qr -T qr_seed qr`
   (`--force` は healthcheck の取りこぼし接続まで切る保険)
4. `docker compose start app`

数十秒の瞬断はデモには許容。セッションも消え全員ログアウトする
(「定期的に削除」の建て付けと整合)。ログバッファも消えるが、
デモでは /logs 自体を閉じている。

### 6-3. 罠 — migration を当てたら種も作り直す

`prisma migrate deploy` は live の `qr` にしか当たらない。
**種 `qr_seed` は古いスキーマのまま**なので、次の毎時リセットで
migration 前の DB に巻き戻り、app が起動できなくなる。
**新しい migration を含むデプロイをデモに当てたら、直後に §6-1 の
手順で種を作り直すこと** (reseedDemo.sh の冒頭コメントにも書いた)。

## 7. nginx — qr-demo 専用 server block

`deploy/nginx/qr-demo.tommie.jp.conf` を新設 (本番 conf には触らない、
実装済み)。doDeployNginx.sh は conf パス (local/remote) とヘルスチェック先を
`DEPLOY_SITE` から導くようにした (既定 `qr.tommie.jp` は従来と同一)。反映:
`DEPLOY_SITE=qr-demo.tommie.jp ./doDeployNginx.sh`。

- `server_name qr-demo.tommie.jp;` → `proxy_pass 127.0.0.1:3100`
- `client_max_body_size 3M` (2MB 上限 + multipart 余白。本番は 12M の
  まま。Caddyfile はローカル用なので触らない)
- `limit_req` 2 枠 (zone 定義は http コンテキスト = ファイル先頭に置く):
  - ログイン枠 `qr_demo_login`: 本番と同じ 30r/m を**別 zone**で。
    zone を分けるのは、デモへの総当たりで本番のログインまで
    絞られないようにするため
  - **書き込み枠 (デモだけの追加)** `qr_demo_write`: `location /` に
    120r/m + burst 30。通常操作 (検索・閲覧・保存) は当たらず、
    スクリプトでの量産だけ絞る線。実測して調整する
- DNS A レコード + certbot は手作業で先に (conf の ssl_* が指す証明書が
  無いと nginx -t が落ちるため。conf ヘッダに初期設定手順を書いた)

## 8. シード内容 — ショーケースノート (機能カタログ)

41-QR-search/docs/08 Q1/Q3 の具体化。種 (§6-1) に入れるもの:

- 部品ノート風 (2SC1815 など実運用の姿) — 核の見せ場
- 記法ショーケース: KaTeX / CircuiTikZ / mermaid / お絵かき /
  プロパティ表 (1 枚に詰めず数枚に分ける)
- 検索プレイグラウンド: `#bjt !(#npn OR #pnp)` などの例をコピペで
  試せる形で本文に列挙
- 音声・PDF 添付入りのノート各 1 枚 (閲覧側の挙動が見える)
- **サンプル QR**: デモノートの /print で刷った QR 画像をノート自身と
  README に貼る — 訪問者が自分のスマホで読んで飛ぶ、核の動線の体験
- 書誌自動取得はデモでは動かない (API キーを渡さない) ので、GIF を
  README に置く (本番側の作業)

## 9. 見送り (YAGNI)

- **guest 専用ロール / 複数デモユーザ** — Ver3 の user_id 設計に含める
- **blue/green 再シード (瞬断ゼロ)** — 数十秒の stop で足りる
- **画像 GC・ゴミ箱パージの自動化** — 再シードが兼ねる
- **CAPTCHA・アクセス解析** — 反応を見てから
- **クォータの厳密な直列化** — §2-1 の通り、はみ出しは再シードが拭く

## 10. テスト

- `demoLimits`: 定数と判定の純関数部分 (超過/未満/ちょうど)
- `items`: デモ時、新規 create が上限で断られ、既存 update は通る
  (count と findUnique を差し替え)
- `robots.ts`: デモは disallow "/"、非デモは許可
- DemoBanner: `DEMO_LOGIN_HINT` があるときだけ案内が出る
- `/api/images`: デモ時、総量超過で 507 (SUM を差し替え)
- reseedDemo.sh は自動テストしない。手動検証手順 (種を撮る → ノートを
  荒らす → timer を手で 1 回叩く → 初期状態に戻る) を script の
  コメントに書く
