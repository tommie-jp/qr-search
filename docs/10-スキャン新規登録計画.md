# スキャンからの新規登録の設計 (Ver2.x)

スキャンで DB にないコード (書籍の JAN、部品の DataMatrix など) を読むと、
検索が 0 件で「該当する部品がありません」と出て行き止まりになる。
ここに「新規登録」ボタンを足し、押すと**スキャンしたコードがタグ
(`#9784873115658`) として本文に入った新規ノートの編集ページ**が開き、
そのまま書き始められるようにする。ここでは設計判断を残す。

前提: スキャン本体の設計は [09-スキャン計画.md](./09-スキャン計画.md)。
スキャンしたコードは `/?q=<コード>` の検索に落ちてくる。

## 1. 方針: コードはタグとしてメモ本文に置く

導線の全体像:

```text
未登録コードをスキャン
  → /?q=<コード> が 0 件 + 「#<コード> を新規登録」ボタン
  → /edit/<次番号>?code=<コード>
  → 本文に #<コード> が入った空ノートのエディタ (フォーカス済み)
  → 「更新」で作成 (それまで DB には何も書かない)
```

- **正本はメモ本文** (タグ・プロパティと同じ)。コードをタグとして本文に
  置けば、`items.tags` の派生キャッシュ・タグリンク・一覧のタグチップが
  すべて既存の仕組みでそのまま動く
- 再スキャンすると `/?q=<コード>` の全文検索が本文中の `#<コード>` に
  ヒットし、導線が閉じる。ノート側からタグをタップすれば同じコードを
  持つノートを一覧できる (同じ本を 2 冊登録した場合など)
- itemNo は従来どおりの数値を自動採番する。4 桁の採番体系に 13 桁の
  コードが混ざらず、シール印刷 (`/print` の `#番号`) も短いまま

### itemNo をコードにする案 (この文書の初版) を差し替えた理由

初版は「itemNo = スキャンしたコード」だった。実装は最小 (0 件のとき
`/edit/<コード>` へのリンクを出すだけ) だが、比べると分が悪い。

- データ (コード) を識別子の列に押し込む形で、採番体系が汚れる
- itemNo は 20 文字まで。タグは長さ無制限で文字種も同等以上なので、
  対応できるコードの範囲が狭い (部品 DataMatrix の長い英数字が入らない)
- タグ案で増える実装は採番と事前入力の小部品 2 つだけ。それで設計の
  一貫性 (正本はメモ本文) と対応範囲を買える

## 2. タグにできるコードの範囲

タグ名に使える文字は `\p{L}\p{N}\p{M}` と `_` `-`、長さ無制限
(tags.ts の `TAG_INNER`)。

- JAN / ISBN (13 桁数字) … 入る
- 英数字の DataMatrix (20 文字超も) … 入る
- 記号 (`/` `.` 空白など) を含むコード … タグにならないので
  ボタンを出さない (§6 の見送り)

正規化は NFKC + 小文字化なので、英字コードは tags キャッシュ上は
小文字になるが、本文の綴りは保たれ、検索も NormalizerAuto が大小を
同一視するので実害はない。

## 3. ボタンを出す条件と場所

「該当する部品がありません」(`ItemList` の emptyState) の直下に出す。

- 条件: **検索 0 件 かつ クエリが単一トークンでタグ文字のみ**。
  判定は純関数 `isTaggableCode(query)` を lib に置く (`TAG_INNER` を
  再利用し、tags.ts と判定がずれないようにする)
- 文言は **「#<コード> を新規登録」** と、何が作られるかを見せる。
  作成は編集ページで「更新」を押すまで起きないので、押し間違えても
  実害はない
- 外部 URL の QR (`:` `/` を含む) や複数語の検索 (空白を含む) では
  条件を満たさないので出ない
- スキャン由来かどうかはマークしない。検索ページは来歴を知らないし、
  手入力した未登録コードから作れて困ることもない (`?scan=1` のような
  印はリロードや URL 共有で嘘をつく)

## 4. 採番 — 1000 以上で未使用の最小番号

リンク先の `/edit/<次番号>` に使う番号は、0 件かつ §3 の条件が成立した
ときだけ `page.tsx` で引いて埋める。

**欠番を埋める**。`max + 1` だと番号が増える一方だが (実データの最大は
6000)、511 件が 1〜6000 に散らばっていて隙間は大量にある。番号は
シールに印刷して部品に貼るものなので、短いほど扱いやすい。

**下限は 1000**。実データで 1000 未満は `1 / 4 / 5 / 6 / 100` の 5 件だけで、
実質の運用は 1000 番台から始まっている。ここを埋めにいくと既存の番号帯
から外れるので触らない。1000 以降の最初の欠番は 1011 (実測)。

判定は純関数 `firstUnusedNo(usedAsc, min)` に切り出す (items.ts は
db.ts 経由で DATABASE_URL を要求するため、そこに置くとテストできない)。
`item_no_num >= 1000` を昇順で引き、最初の隙間を返すだけ。511 件規模なら
SQL の gap 検索まで持ち込む必要はない (index 済みの列で、全件引いても安い)。

### 欠番の再利用が危なくない理由 (今は)

普通「未使用の最小番号」で怖いのは**削除済み番号の再利用**で、古いシール
が別の部品を指してしまう。だが**このアプリに削除機能はない**ので、
「未使用 = 一度も使われていない」が成立しており、この危険が存在しない。
将来ノート削除を入れるときは、この前提が崩れるので
「欠番は再利用しない (max+1 へ戻す)」か「削除済み番号を記録する」かを
選び直すこと。

### 設定で選べるようにしない

README にあった「設定で選択できるようにする?」は採らない。単一ユーザの
アプリで採番方針は一度決めれば終わりの選択であり、設定 UI と保存先を
作るコストに見合わない (YAGNI)。方針を変えたくなったら
`firstUnusedNo` の呼び出しを差し替えれば済む。

### 競合

競合 (別タブで先に同番号が作られる) は単一ユーザで実質起きない。
万一起きても、編集ページは既存ノートなら**その本文を表示する**
(事前入力しない) ので、開いた瞬間に気づける。予約テーブルのような
仕組みは持たない (YAGNI)。

この採番は README の「"＋"でノート追加」と同じ部品。ヘッダに「＋」を
足すときは `code` なしで `/edit/<次番号>` を開くだけになる (副産物)。

## 5. 編集ページの事前入力

`/edit/[itemNo]?code=<コード>` を受け、**未登録の itemNo のときだけ**
`MemoEditor` の defaultValue に入れる。既存ノートでは無視する
(既存本文を差し替えない)。

事前入力の形は **空行 2 つ + タグ**:

```text
(空行)   ← カーソルはここ (文書先頭 = CodeMirror の既定位置)
(空行)
#9784873115658
```

- 一覧の要約 (`memoSummary`) は空行を飛ばすので、書き込む前の要約は
  `#<コード>` になる (=「タイトルはスキャンしたコード」がここで満たされる)
- 1 行目に書名・部品名を書けば要約がそれに差し替わり、タグは下に残る。
  カーソル位置の制御を実装しなくても、既定の文書先頭がそのまま
  「タイトルを書く位置」になる
- `code` パラメータは `isTaggableCode` を通し、通らなければ黙って無視する
  (通常の導線では起きない。URL を手で触った場合に変な初期値を入れないため)

### ISBN のときは #book も付ける

コードが ISBN なら `#<コード> #book` にする。書籍は「持っている本を
一覧したい」が自然に起きる括りで、タグにする価値がある。

- **判定はコードの中身だけで完結する**。978/979 で始まる 13 桁は
  Bookland (書籍用に予約された EAN-13 の接頭辞) なので、読み取った
  数字だけで書籍と分かる。チェックデジットも検算して、名前どおり
  「ISBN である」ことを保証する (2 行で済む)
- 979 帯のうち **9790 だけは ISMN (印刷楽譜の番号) で書籍ではない**ので
  接頭辞で外す。チェックデジットは正しいので、ここを見ないと楽譜に
  #book が付く (部品 DB で楽譜を読む場面はまずないが、`isIsbn` と
  名乗る以上は正しくしておく)
- **#ISBN と #book を両方付けない**。ISBN を持つノート = 書籍なので
  2 つのタグは完全に同じ集合を指し、タグチップとタグ一覧のノイズが
  倍になるだけで検索能力は増えない
- 誤判定しても実害はない。編集ページが開いてカーソルが載っている状態
  なので、要らなければその場で消せる

### JAN / QR / DataMatrix の種類タグは付けない

- **括りとして弱い**。JAN は食品にも部品の外装にも付いていて、
  「小売バーコードで登録したもの一覧」を見たい場面がない。QR /
  DataMatrix に至っては読み取り手段の記録でしかない
- **コストも高い**。ISBN と違い種類はコードの中身から分からず、
  読み取ったフォーマットが要る。それはスキャン → 検索 URL → 登録ボタン
  → 編集ページと 3 ホップ持ち回す配線になる (今は `/?q=<コード>` に
  落とした時点で捨てている)

## 6. 段階と見送り

- **Phase 1 (実装済み)**: 採番・事前入力・ボタン + テスト (§7)。JAN・ISBN・
  英数字 DataMatrix のスキャン → 新規登録 → 再スキャンでヒットまで通る
- **見送り: 記号入りコード**。タグにならないのでボタンを出さない。
  必要になったら「タグではなく本文プレーン記載」(全文検索には乗る) を
  検討する
- **見送り: ISBN からの書誌自動取得** (openBD 等)。09 で見送った判断の
  継続。まず「読んで登録して手で書く」で運用し、苦になったら検討する

## 7. テスト

- 純関数 (vitest):
  - `isTaggableCode` … JAN 13 桁 / 英数字 21 文字 → true、
    URL / 空白入り / 記号入り / 空文字 → false
  - 事前入力の組み立て (`code` → `\n\n#<コード>`)、既存ノートでは
    適用しない判定
- `ItemList` (静的描画): 0 件 + コード → `/edit/<次番号>?code=…` の
  リンクが出る / URL・複数語・1 件以上 → 出ない
- E2E はフェイクカメラの手法 (09 参照) で通しまで確認できる:
  未登録 JAN を読む → 0 件 + ボタン → 編集 (タグ入り・フォーカス済み) →
  「更新」 → 同じ JAN を再スキャン → 1 件ヒット

`nextItemNo` に単体テストは置かない。items.test.ts の DB 統合テストは
`zzft` プレフィックスの (= item_no_num が null の) ノートで後始末する作りで、
最大値を動かす数値ノートを入れると後始末の枠から外れる。上の E2E で
実データ (最大 6000) に対し 6001 が採番されることを確認しており、
そちらで担保する。

### 実装時に分かったこと

- **保存すると本文の改行が CRLF になる**。HTML のフォーム送信仕様
  (application/x-www-form-urlencoded は改行を CRLF へ正規化する) によるもので、
  この機能とは無関係の既存挙動 (実データ 512 件中 459 件が CRLF を含む)。
  `memoSummary` は `/\r?\n/` で分割し、タグ抽出は `\s` で拾うので実害はない
- **英字コードの再スキャンも通る**。tags キャッシュは小文字化されるが、
  全文検索の NormalizerAuto が大小を同一視するため `ABC` で `#abc` に当たる
  (PGroonga で実測)
