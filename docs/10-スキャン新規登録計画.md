# スキャンからの新規登録の設計 (Ver2.x)

スキャンで DB にないコード (書籍の JAN、部品の DataMatrix など) を読むと、
検索が 0 件で「該当する部品がありません」と出て行き止まりになる。
ここに「新規登録」ボタンを足し、押すと編集ページが開いて、
タイトルはスキャンしたコード・エディタにフォーカスが当たった状態で
書き始められるようにする。ここでは設計判断を残す。

前提: スキャン本体の設計は [09-スキャン計画.md](./09-スキャン計画.md)。
スキャンしたコードは `/?q=<コード>` の検索に落ちてくる。

## 1. 土台はほぼ全部すでにある

調べた結果、この機能の中身は既存の仕組みで揃っていた。

- **`/edit/:itemNo` は未登録の itemNo でも空のエディタを表示する**
  (Ver1 互換の挙動)。見出しは `edit #<itemNo>`
- **保存 (`updateItemAction`) は upsert** で、未登録なら新規作成になる
- **`MemoEditor` は `autoFocus` 済み**。「編集状態にする」は編集ページを
  開くだけで満たされる
- **検索は itemNo の前方一致 + 全文検索** (`searchItems`)。コードを
  itemNo にして登録すれば、同じコードを次にスキャンしたとき 1 件ヒットする
  (再スキャンの導線が閉じる)

つまり残る実装は **「検索 0 件のときに `/edit/<コード>` へのリンクを
1 つ出す」だけ**。新しい画面もサーバアクションも要らない。

## 2. 「タイトルはスキャンしたコード」= itemNo をコードにする

このアプリに独立したタイトル欄はない。ノートの見出しに当たるものは
itemNo (`#4951` の形で一覧・編集・表示の見出しに出る) と、一覧の要約に
使われるメモ先頭行 (`memoSummary`) の 2 つ。**itemNo をスキャンした
コードにする** (例: `#9784873115658`)。

- 編集ページの見出しが `edit #9784873115658` になり、
  「タイトルはスキャンしたコード」がそのまま満たされる
- `ITEM_NO_PATTERN` は英数字・`_`・`-` の 1〜20 文字。JAN/ISBN の 13 桁は
  適合する。英数字の itemNo は Ver1 実データ (`100x`) で許容済み
- 次回スキャン → `/?q=<コード>` → itemNo 前方一致でヒット

### メモ先頭行に入れる案を採らない理由

- 一覧の要約はメモ先頭行から作られる。コードを先頭行に入れると要約が
  数字の羅列で埋まり、肝心の内容 (書名・部品名) が一覧に出なくなる
- itemNo にコードがあれば、メモへ重複して書く理由がない (検索は
  itemNo でヒットする)。採番の仕組みも要らない
- ただしこの案は「コードが itemNo になれる」ことが前提。なれないコードは
  §4 の Phase 2 で扱う

## 3. ボタンを出す条件と場所

「該当する部品がありません」(`ItemList` の emptyState) の直下に出す。
`ItemList` は `query` を既に prop で持っているので、配線は足さなくてよい。

- 条件: **検索 0 件 かつ `isValidItemNo(query)`**
- 文言は **「#<コード> を新規登録」** と、何が作られるかを見せる。
  誤操作対策は出し分けではなくこの文言で行う (作成は編集ページで
  「更新」を押すまで起きないので、押し間違えても実害がない)
- リンク先は `/edit/<encodeURIComponent(コード)>`

### スキャン由来かどうかはマークしない

「スキャンから来たときだけ出す」には `/?q=…&scan=1` のような印が要るが、
採らない。リロードや URL 共有で印だけが残って嘘をつくし、手入力した
未登録番号から新規登録できて困ることもない。むしろ README の
「"＋"でノート追加」の部分的な実現になる。

isValidItemNo の条件で自然に絞られるものがある:

- 外部 URL の QR → `:` や `/` を含むので出ない
- 複数語の検索 (`bjt 2sc`) → スペースを含むので出ない
- 21 文字以上のコード → 出ない (Phase 2 へ)

なお `2SC2712` のような部品名の 0 件検索でもボタンは出る。部品名を
itemNo にしたノートが作れてしまうが、文言に `#2SC2712` と出るので
何が起きるかは見えるし、単一ユーザのアプリなので運用で選べばよい。

## 4. 段階

- **Phase 1**: emptyState にリンクを足す + テスト。これだけで
  JAN・ISBN・数字系 DataMatrix のスキャン → 新規登録が通る
- **Phase 2 (必要になったら)**: itemNo になれないコード (記号入り・
  21 文字以上。DataMatrix の中身は任意文字列) への対応。自動採番した
  itemNo で作り、コードはメモ先頭行へ入れる形になる。自動採番は
  「"＋"でノート追加」(README の未着手項目) と同じ部品なので、
  そちらを作るときに一緒に設計する
- **見送り**: ISBN からの書誌自動取得 (openBD 等)。09 で見送った判断の
  継続。まず「読んで登録して手で書く」で運用し、苦になったら検討する

## 5. テスト

- `ItemList.test.tsx` (静的描画) に emptyState の分岐を足す:
  - 0 件 + `9784873115658` → `href="/edit/9784873115658"` が出る
  - 0 件 + 外部 URL / スペース入り / 空文字 → 出ない
  - 1 件以上 → 出ない
- E2E はフェイクカメラの手法 (09 参照) で通しまで確認できる:
  未登録 JAN を読む → 0 件 + ボタン → 編集 → 保存 →
  同じ JAN を再スキャン → 1 件ヒット
