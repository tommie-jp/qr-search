# スキャン内蔵の設計 (Ver2.x)

アプリ内にカメラスキャナを載せ、部品シールの QR に加えて書籍 ISBN などの
バーコードも読めるようにする。ここでは設計判断を残す。

動機は 2 つ。

- **他コード対応**: 書籍 ISBN (EAN-13) などを読んで検索できるようにする
- **iOS の導線の穴埋め**: iOS は OS カメラで QR を読むと常に Safari で開き、
  ホーム画面の PWA が起動しない (41-QR-search/docs/05-PWA計画.md で実機確認済み)。
  アプリ内スキャンなら「PWA を開いてから読む」形になり、OS カメラを経由しない

## 1. ライブラリ選定

**@yudiel/react-qr-scanner** (v2.6.0 時点) を採用する。

- peer deps が `react ^17 || ^18 || ^19` で React 19 に対応済み
- 内部は Sec-ant/barcode-detector (BarcodeDetector ポリフィル) →
  zxing-wasm (zxing-cpp の WASM ビルド) の構成。QR と EAN-13/ISBN を
  1 ライブラリでカバーでき、zxing-wasm は活発にメンテされている
  (2026-07 時点で v3.1.1)
- カメラ起動・フレームループ・デバイス選択を含む React コンポーネントが
  そのまま使え、getUserMedia まわりを自作しなくてよい

ポリフィル固定 (ネイティブ BarcodeDetector に委譲しない) なのはむしろ利点。
Safari は BarcodeDetector をデフォルト無効のままなので iOS では WASM が必須で、
どのブラウザでも対応フォーマットと挙動が完全に同一になる。

### 採らなかった案

- **qr-scanner (nimiq)** … QR 専用。ISBN が読めず要件を満たさない
- **quagga2** … 1D 専用。QR が読めない
- **@zxing/library + @zxing/browser** … JS 移植版。メンテが停滞気味で、
  1D の読み取り精度も zxing-cpp (WASM) に劣る
- **barcode-detector + 自前ループ** … 依存は 1 枚薄くなるが、カメラ選択や
  タブ非表示時の停止などを自作することになる。react-qr-scanner が
  停滞したらこの形へ退避できる (§8) ので、今は取らない (YAGNI)

## 2. UI / 導線

検索窓 (`SearchForm`) の「検索」ボタンの隣に「スキャン」ボタンを置き、
押すと全画面モーダルでカメラビューを開く。

- モーダルは `ScannerModal` コンポーネントに閉じ、`next/dynamic` の
  `ssr: false` で遅延ロードする。wasm (約 1 MiB) とコンポーネントは
  ボタンを押すまで一切ロードされない
- カメラ非対応・許可拒否・非 secure context ではモーダル内に
  理由を表示する (黙って何も起きないのが最悪)
- **エラーは Scanner を差し替えず、上に重ねて出す。** ライブラリは
  カメラ起動の失敗もトーチ切り替えの失敗も同じ `onError` に流すので、
  差し替える作りにすると「ライトを点けようとしただけでカメラが死ぬ」
  ことになる (レビューで発見)
- 読み取りに成功したら振動 (`navigator.vibrate`、対応環境のみ) して
  モーダルを閉じ、§3 の判定に従って遷移する

## 3. 読み取り後の処理 — 純関数で判定

判定は `lib/scanResult.ts` の `resolveScanPath(rawValue, itemHosts)` に切り出す
(純関数・TDD 対象)。出力は遷移先の path か null。

| 読んだもの | 判定 | 遷移 |
| --- | --- | --- |
| 部品シールの URL | itemHosts + `/item/:itemNo` | `/item/:itemNo` へ `push` |
| 外部 URL の QR | それ以外の URL | `/?q=<URL>` で検索 |
| ISBN / JAN などのバーコード | URL ではない | `/?q=<コード値>` で検索 |
| 任意テキストの QR | URL ではない | `/?q=<テキスト>` で検索 |
| 空文字・空白のみ | — | 何もしない (読み直させる) |

- **外部 URL を開かない**のは意図的。カメラにたまたま写った QR で
  勝手に外部サイトへ飛ぶのは危険なので、検索に落として人に見せる
- ISBN を検索に落とすだけで済むのは、メモ本文に ISBN を書いておけば
  全文検索がヒットするから。「ISBN 列」のような専用データは持たず、
  タグ・プロパティと同じく正本はメモ本文とする

### 「自サイト」の判定 (実装時に 2 度直した)

当初は「same-origin なら部品ページへ」と書いていたが、実装して実際に
読ませたところ 2 点とも成り立たなかった。

**1. スキームは見ない。** Ver1 のシールに焼かれているのは
`http://qr.tommie.jp/item/<itemNo>` (41-QR-search/docs/02-Ver1調査.md)。
origin はスキームを含むので、https 化した Ver2 から見ると
`http://qr.tommie.jp` ≠ `https://qr.tommie.jp` となり、**貼り替えられない
既存シールが全部読めなくなる**。ホスト名だけで判定する。
なお当初案にあった「旧パス形式の読み替え」は不要だった。Ver1 も
`/item/:itemNo` で、パス互換は最初から保たれている。

**2. 現在のホストだけでは足りない。** シールに焼かれるホストは
`QR_BASE_URL` (既定 `qr.tommie.jp`) 固定である一方、確認や作業では
localhost や LAN の IP でアプリを開く。現在ホストだけで判定すると
両者が食い違い、**シールが黙って検索に落ちて「0 件」になる** (実際に
localhost の実測でこうなった)。そこで判定に使うホストは
`[QR_BASE_URL のホスト, いま開いているホスト]` の 2 つとする。

`QR_BASE_URL` は印刷 (`/print`) が QR に焼く値そのものなので、
`lib/site.ts` の `qrBaseUrl()` を唯一の出どころにして印刷側と共用する
(ずれると「印刷した QR を自分で読めない」が起きる)。`process.env` は
`NEXT_PUBLIC_` 以外クライアントへ渡らないため、値はサーバコンポーネント
(`app/page.tsx`) から props で降ろす。

`QR_BASE_URL` が壊れていても (scheme 忘れ・`QR_BASE_URL=` と書いて空文字)
`qrStickerHost()` は投げずに既定へ倒し、警告だけ残す。トップページは検索の
ためのページで印刷設定とは関係がなく、設定ミスの道連れで 500 にする理由が
ないため。なお `??` は空文字を素通しするので既定へ倒すには `||` が要る。

### ホスト名は完全一致で見る

`endsWith` だと `qr.tommie.jp.evil.com` を自サイトと誤認する。また
遷移先は読み取った path をそのまま使わず、`isValidItemNo` を通した
itemNo から組み立て直す (他人の作った QR に遷移先を仕込ませない)。

## 4. 対応フォーマット

`formats` は定数 `SCAN_FORMATS` として 1 箇所で管理する。初期セット:

```text
qr_code  micro_qr_code  data_matrix          (2D: 部品シール・工業系)
ean_13   ean_8  upc_a  upc_e                 (小売系: ISBN/JAN はここ)
code_128 itf                                  (工業系 1D)
```

全フォーマット対応 (`any`) にしない理由: `code_39` や `codabar` など
チェックデジットの弱い 1D コードは、カメラノイズを短いコードとして
誤検出することがある。使う予定のあるものだけ有効にし、増やすときは
定数に 1 行足す。なお `maxi_code` は zxing-cpp がカメラ映像からの
位置特定に対応しておらず、有効にしても実質読めない。

## 5. wasm のセルフホスト

zxing-wasm は既定で .wasm を jsDelivr CDN から取得する。tikzjax フォント
(`scripts/copyTikzFonts.mjs`) と同じ理由・同じ流儀で自前配信に切り替える。

- `scripts/copyZxingWasm.mjs` を作り、ビルド時に
  `node_modules/zxing-wasm/dist/reader/zxing_reader.wasm` を
  `public/zxing/` へ複製する (正確なパスは導入時に確認)。
  リポジトリに wasm を抱え込まず、パッケージ更新に自動追随する
- barcode-detector が re-export する `prepareZXingModule` の
  `locateFile` で取得先を `/zxing/` に向ける (API 詳細は導入時に
  barcode-detector v3 README で確認)
- **バージョン一致が重要**: JS グルーと wasm はペアなので、CDN 固定 URL や
  リポジトリ同梱にすると更新時にずれる。node_modules から複製する形なら
  ずれようがない
- Basic 認証は問題にならない。wasm の fetch は same-origin なので
  ブラウザが認証済みの Authorization ヘッダを自動で付ける
  (manifest と違い、認証除外の nginx 設定は不要)

## 6. 前提: secure context (https)

`getUserMedia` は secure context (https または localhost) でしか動かない。

- 開発は `localhost` なのでそのまま動く
- 本番は https 化 (41-QR-search/docs/05-PWA計画.md) が前提。
  `http://qr.tommie.jp` のままではカメラが起動できない
- 非 secure context では `navigator.mediaDevices` 自体が undefined になる。
  その場合スキャンボタンは隠さず、押したら「https が必要」と表示する
  (隠すと「機能が消えた」ように見えて原因を追えない)

## 7. テスト

- `lib/scanResult.ts` (判定純関数) … vitest で TDD。シール URL /
  旧シールの http / 外部 URL / 紛らわしいホスト / path traversal /
  ISBN / 任意テキスト / 空文字を網羅
- `ScannerModal` はテストしない。このリポジトリのコンポーネントテストは
  `renderToStaticMarkup` の静的描画だけで、操作を伴うテストの土台
  (jsdom / testing-library) を持たない。カメラを持つモーダルは
  静的描画では何も確かめられないので、**判定を純関数へ追い出して
  モーダル側は配線だけ**にしてある (当初案のモック方式は、その土台を
  入れる話になるので採らない)

### フェイクカメラでの実測 (実装時に実施)

カメラは自動化できないと書いていたが、Chrome のフェイクカメラに
実物のコード画像を流せば、デコードから遷移まで通しで確認できた。

```bash
# QR 画像 → y4m にして Chrome に食わせる (720p。既定制約が height>=640 必須のため)
ffmpeg -loop 1 -i qr.png -t 4 -r 15 \
  -vf "scale=560:560,pad=1280:720:(ow-iw)/2:(oh-ih)/2:white,format=yuv420p" qr.y4m
google-chrome --use-fake-device-for-media-stream \
  --use-file-for-fake-video-capture=qr.y4m
```

これで確認できたこと:

- 旧シール相当の `http://qr.tommie.jp/item/1003` の QR → `/item/1003` へ遷移
- 実在 ISBN (978-4-87311-565-8) の EAN-13 → `/?q=9784873115658` で検索
- wasm が自前配信 (`/zxing/zxing_reader.wasm`) から 200 で取れている

ISBN のバーコード画像は `zxing-wasm/writer` で生成できる (読み取りと
同じ zxing-cpp なので、別途ツールを入れずに済む)。

### 実機でしか確認できない分

フェイクカメラでは代えられない (カメラ許可・実際のレンズとピント・
iOS の WebKit 実装) ので、実機で残す:

- [x] iPhone で QR が読める (2026-07-16 実機確認)
- [x] iPhone で JAN (EAN-13。ISBN も同じ) が読める (同上)
- [x] iPhone で DataMatrix が読める (同上)
- [ ] iOS ホーム画面 PWA でも読める (起動ごとに許可を再要求される癖がある。
  仕様として割り切る)
- [ ] Android Chrome / PWA で読める

## 8. 計画に無かったが実装時に要った分

- **検索窓が空のままになる既存バグを直した** (`app/page.tsx` の
  `key={query}`)。検索窓の値は `useState` の初期値なので、クライアント遷移
  では再初期化されない。スキャンで `/?q=9784873115658` へ飛ぶと結果は
  新しいのに窓だけ空、という状態になり、読み取った値を絞り込めなかった。
  タグリンクでも前から同じことが起きていた (スキャンで目立っただけ)
- **`decodeURIComponent` を try で囲った**。`/item/%E0%A4%A` のような
  壊れた %シーケンスは `new URL()` を通ってしまい、そのあと URIError を
  投げる。スキャンのコールバックは他人の作った QR で呼ばれるので投げさせない
- **型は `barcode-detector` から取らない** (`lib/scanFormats.ts`)。あれは
  スキャナの依存であってこちらの直接依存ではなく、npm のフラットな巻き上げに
  頼る形になる。`IScannerProps['formats']` から導けば相手とずれない

## 9. 段階と見送り

- **Phase 1 (実装済み)**: 依存追加・wasm セルフホスト・スキャンボタン +
  モーダル・判定純関数。フォーマットは初めから §4 のセットで入れた
  (ISBN 対応は定数に含まれるだけでコスト差がない)
- **Phase 2 (必要になったら)**: フォーマット追加、トーチ (ライト) や
  ズーム制御、連続スキャンモード
- **見送り**: ISBN から書誌情報の自動取得 (openBD 等)。まず「読んで
  検索」で運用し、書籍ノートが増えて手入力が苦になったら検討する

### リスクと退避先

@yudiel/react-qr-scanner は個人メンテのラッパーで、停滞する可能性はある。
そのときは依存の本体である barcode-detector + 小さな自前ループ (§1 の
採らなかった案) へ乗り換える。判定ロジックは純関数、UI は `ScannerModal`
に閉じているので、乗り換えの影響はモーダル内部だけに収まる。
