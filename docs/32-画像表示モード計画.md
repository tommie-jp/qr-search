# 画像表示モード計画 (Ver2.x)

表示モードに 3 つ目の「画像 (image)」を足す。ページ内のノート本文に貼られた
自前画像だけをピンタレスト風の masonry で敷き詰め、写真からノートを探す
入口にする。1 ノートに複数枚あれば全部並べ、画像の無いノートは出さない。

前提はいつもどおり: 素のフォーム / リンク / Server Action で JS 無効でも動く
設計は壊さない。切替は既存の cookie フォーム送信のまま
(docs/23-検索結果表示モード計画.md §4-5)。

## 1. masonry は CSS multi-column で組む

`columns-[10rem] gap-2` + タイルに `break-inside-avoid mb-2`
(ImageMasonry.tsx)。JS の masonry ライブラリは足さない — 必要なのは
「縦横比を保ったまま隙間なく敷き詰める」ことだけで、そのために依存と
バンドルを増やす釣り合いが取れない (MenuIcons がアイコンライブラリを
足さなかったのと同じ判断)。

- **列数は指定しない**。列幅 10rem を基準に画面が決める — card グリッドの
  auto-fill と同じ思想 (docs/23 §1)。スマホで 2 列、PC の広幅で 5〜6 列。
  幅指定の columns は器が狭ければ自動で 1 列に畳まれるので、card で踏んだ
  「minmax の横スクロール」(ItemList.tsx) と同種の問題は起きない。
- 10rem はサムネの解像度予算でもある。サムネは長辺 320px (thumbnail.ts) で、
  列幅 160px × 2x DPR ≈ 320px とほぼ等倍。これより広い列にすると高 DPR で
  甘くなる。必要になったら thumbnail.ts に大きい変種を足す (将来課題)。
- multi-column は**左の列を上から下へ**埋める。Pinterest の JS 実装
  (横方向に配って高さを揃える) とは並び順が違うが、1 ページ 20 ノート分を
  眺める用途では困らないので許容。
- `column-span: all` の Tailwind ユーティリティは無いので、全幅に渡したい
  案内 (「このページには画像がありません」) は columns の器の**外**に置く。

## 2. タイルは「画像 1 枚 = 1 リンク」。ItemRow は使わない

描画は ItemList から ImageMasonry へ丸ごと委譲する。ItemRow を拡張しない
のは、あちらの仕掛け (stretched link とタグリンクの共存、選択モードの
チェックボックス差し込み) がタイルには一切不要で、共通化すると両方が
読めなくなるため。

- 画像の列挙は `allImageNames()` (memoImages.ts)。画像検索の索引づくりと
  同じ抽出規則 (出現順・重複除去・コードフェンス除外・自前画像のみ) を
  そのまま使う。URL モードのノートは memo が空なので自然に落ちる。
- タップでノート詳細 `/item/<itemNo>` へ。画像は入口であって、その場で
  拡大するビューアではない (拡大は詳細ページの ZoomableImage が持っている)。
- タイル左下に `#番号` のバッジを焼き込む。同じノートの画像が並ぶと
  押す前に行き先を見分けられないため。
- **next/image は使わない** — 画像 API はログイン必須で、optimizer が
  Cookie 無しで取りに行き 401 になる (docs/23 §2.1、ItemRow.tsx)。素の
  `<img loading="lazy">` + `?thumb=1`。原寸 (数 MB) は引かない。
- 寸法は DB に無いので width/height 属性を出せず、読み込みで高さが決まる
  (layout shift)。`bg-gray-100` の地色と lazy で緩和して許容。気になったら
  Image テーブルに寸法列を足して保存時に埋める (将来課題)。

## 3. 切替は 1 スロットの循環トグル (小→大→画像→小)

3 値になってもセグメントには戻さず、下部バーの 1 スロットで循環する
(BottomActionBar.tsx)。5 スロット構成を崩さないため。ラベルとアイコンには
**現在の値**を出す原則 (docs/31 §3-4) はそのまま。行き先は aria-label
(「押すと◯に切替」) に乗せる。

- アイコンは ImageViewIcon (高さ不揃いのタイル 4 枚 = masonry の形) を
  新設。GridViewIcon (均等 2×2) との差は高さの不揃いだけに抑えて「表示」
  スロットの仲間だと判るようにする。ImageSearchIcon の流用は不可 — 隣の
  「画像検索」スロットと同じ絵になり、色と形で狙えなくなる (docs/31 §11-1)。
- `parseViewMode` は 'image' を受けるだけで、知らない値→compact の畳み込み
  (docs/23 §4) は変えない。**setViewModeAction に revalidatePath を
  足さないこと** (docs/23 §5.1 の EACCES の罠) も変わらず。
- ItemRow には 'image' が流れない。`RowViewMode = Exclude<ViewMode, "image">`
  で型ごと締め出してある (ItemRow.tsx)。

## 4. ページ割りはノート単位のまま

1 ページ = ノート 20 件は変えず、サーバのクエリも触らない。画像モードは
「いま見ているページの画像を敷き詰める」だけで、検索状態 (何を見ているか)
には手を入れない — 表示モードが cookie であって URL でない理由 (docs/23 §4)
と同じ線引き。

その代償として、ページ内の 20 件に画像持ちが 1 件も無いページが起こり得る。
そこでは「このページには画像がありません」を出す (ページ送りは残るので
次のページへは進める)。画像持ちノートだけでページを詰め直す案は、クエリの
分岐とページ番号の意味が他モードとずれる (同じ 3 ページ目が別の 20 件を
指す) ので採らなかった。

0 件 (検索ヒットなし) は別物で、該当なしの案内と新規登録/ゴミ箱の導線
(docs/10, docs/12) を画像モードでも失わない。

## 5. 選択モード中は compact の一覧に畳む

画像モードで下部バーの「選択」に入ると、masonry ではなく compact の一覧に
チェックボックスを出す (ItemList.tsx の rowView)。タイルは画像単位で
ノート単位ではなく、画像の無いノートがタイルに出ない — 「このページの
ノートから選ぶ」操作の対象と画面が食い違うため。選択を抜ければ masonry に
戻る。BulkTagToolbar / form の仕掛けもそのまま使い回せる。
