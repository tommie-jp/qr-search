# 29. パスキー計画 (WebAuthn)

普段のログインを Face ID / Touch ID (パスキー) にする。パスワードと
ブラウザの認証ダイアログを日常から消す。

Basic 認証は**消さない**。「パスキーを登録するための門番」と
「全端末のパスキーを失ったときの復旧経路」として残す。

前提は [18-ログイン計画.md](./18-ログイン計画.md)。あちらが認証の土台で、
この計画はその上に認証手段を 1 つ足すもの。

## 1. なぜ変えるか

現状の Basic 認証には、直しようのない不便が 3 つある。

| 困りごと | 原因 |
| --- | --- |
| ログインのたびにパスワード入力 | ブラウザの認証ダイアログしか口がない |
| vps2 で 1 回 1.75 秒待たされる | bcrypt コスト 12 (18 §8)。下げると総当たりに弱くなる |
| ログアウトできない | Basic 認証はブラウザを閉じるまで資格情報が残る |

パスキーはこの 3 つを同時に解く。iPhone なら Face ID だけで入れて、
検証は署名 1 回 (ミリ秒) で終わり、セッションを持つので失効させられる。

## 2. Basic 認証を残す理由

パスキーの秘密鍵は端末の中にあり、サーバからは作れない。つまり
**全端末のパスキーを失うと、サーバ側に入り直す手段が要る**。

案は 3 つあった。

| 案 | 復旧経路 | bcrypt |
| --- | --- | --- |
| **1. Basic を裏口として残す** (採用) | いつでもパスワードで入れる | 残る |
| 2. 初回登録だけ Basic、以後は削除 | SSH で DB の行を消す | 消せる |
| 3. 使い捨てセットアップトークン | SSH で env を書き換える | 最初から不要 |

2 と 3 は「復旧に SSH が要る」。単一ユーザの道具にそこまでの厳しさは
要らないし、出先の iPhone から復旧できないのは実害が大きい。
**1 を採る。bcrypt とパスワードはそのまま残る。**

残るとはいえ、通るのは登録時と復旧時だけになる。毎リクエストの照合が
消えるので、18 §8 の「vps2 で 1.75 秒」はログインの瞬間だけの話になった。
`auth.ts` の検証キャッシュは存在理由が消えたため、のちに**削除した**
(18 §11)。

## 3. 構成

```text
ブラウザ ──> nginx (ボディ上限 / /login と /api/auth のレート制限)
              └─> Next.js
                   ├─ src/proxy.ts              ... 門番 (セッション Cookie を見る)
                   ├─ src/lib/requestAuth.ts    ... 判定の正本
                   ├─ src/app/login/route.ts    ... Basic を検証する唯一の場所
                   ├─ src/lib/auth.ts           ... Basic 照合 (/login からのみ)
                   ├─ src/lib/sessionStore.ts   ... セッションの発行と照合
                   ├─ src/lib/passkeys.ts       ... 公開鍵の保存と照合
                   └─ src/app/api/auth/...      ... 登録・ログイン・ログアウトの口
```

| ファイル | 役割 |
| --- | --- |
| `src/lib/webauthnConfig.ts` | rpID / origin を env から読む。欠けたら機能ごと無効 |
| `src/lib/sessionToken.ts` | トークンの生成・ハッシュ・Cookie 名と有効期限 |
| `src/lib/sessionStore.ts` | セッション行の発行・照合・延長・破棄 |
| `src/lib/webauthnChallenge.ts` | 使い捨てチャレンジの記憶 (プロセス内) |
| `src/lib/passkeys.ts` | 登録済みパスキーの一覧・保存・削除 |
| `src/lib/requestAuth.ts` | 判定の**正本**。セッション Cookie だけを見る (18 §11) |

## 4. 一番大きい変更: セッション層ができる

Basic 認証は「ブラウザが毎回ヘッダを送ってくる」ことがセッションの代わり
だった。パスキーは**署名がログイン時の 1 回きり**なので、その後の
リクエストを誰と結びつけるかを自前で持つ必要がある。

- ログイン成功時に 256bit の乱数トークンを発行し、Cookie に入れる
  (`HttpOnly` / `Secure` / `SameSite=Lax` / `Path=/`)
- DB には **sha256(トークン) だけ**を保存する。生のトークンは Cookie に
  しか存在しない。pg_dump が漏れてもセッションを乗っ取れないようにする
  (パスワードのハッシュを env に持つのと同じ「生の秘密を置かない」考え方)
- 有効期限は **90 日のスライディング**。使うたびに 90 日先へ延ばす

### 延長は 1 日に 1 回まで

毎リクエスト `expiresAt` を書き換えると、ページを開くたびに UPDATE が飛ぶ。
**残り 89 日を切っていたら** 90 日へ延ばす、とすることで書き込みは
1 日 1 回に収まる。

延長を行うのは `proxy.ts` だけ。Cookie を貼り直す必要があり、
Server Component (`currentUser()`) からは Cookie を書けないため。

### ログアウトができるようになる

18 §3 の「ログアウトはない」はここで解消する。セッション行を消せば
その端末は入れなくなる。

> **追記 (2026-07-19)**: 当初はパスキーで入ったときだけログアウトできた。
> パスワードで入っている間はブラウザが `Authorization` を送り続けるため、
> セッションを消しても次のリクエストで復活してしまったため。
> その後 **Basic を検証するのは `/login` だけ**に絞り、通ったらセッションを
> 発行する形にしたので、いまはログイン手段によらずログアウトできる
> (18 §11)。下の「ログアウト: セッションで入っているときだけ出す」も
> 同様に古い記述で、いまはログイン中なら常に出す。

## 5. DB (Prisma に 2 モデル追加)

```prisma
model WebAuthnCredential {
  id         String   @id            // credential ID (base64url)
  publicKey  Bytes                   // 公開鍵。秘密鍵は端末から出てこない
  counter    BigInt                  // クローン検知用
  transports String[] @default([])
  label      String                  // 「iPhone」など。一覧と削除のための名前
  createdAt  DateTime @default(now())
  lastUsedAt DateTime?
}

model Session {
  tokenHash  String   @id            // sha256(トークン)
  user       String
  createdAt  DateTime @default(now())
  expiresAt  DateTime
}
```

チャレンジ (登録・ログインの使い捨て乱数) は **DB に置かない**。
TTL 5 分・上限つきのプロセス内 Map で持つ (`auth.ts` の検証キャッシュと
同じ流儀)。再起動で消えても「もう一度ボタンを押す」だけで済む。
触るのは route handler だけなので、proxy が別バンドルになる問題も踏まない。

> ⚠️ **migrate 時の注意**: `prisma migrate` は PGroonga の索引を DROP する
> (18 の頃から続く既知の罠)。消えても seq scan で動いてしまい気づけないので、
> 既存の migration と同じく `CREATE INDEX IF NOT EXISTS` を末尾に足し、
> 適用後に `\di` で残っていることを確認する。

## 6. 口 (API)

| 口 | 認証 | 役割 |
| --- | --- | --- |
| `POST /api/auth/passkey/register-options` | `requireUser()` | 登録チャレンジの発行 |
| `POST /api/auth/passkey/register-verify` | `requireUser()` | 検証 → 公開鍵を保存 |
| `POST /api/auth/passkey/login-options` | **公開** | ログインチャレンジの発行 |
| `POST /api/auth/passkey/login-verify` | **公開** | 検証 → セッション Cookie 発行 |
| `POST /api/auth/logout` | ログイン中 | セッション破棄 + Cookie 削除 |
| `GET/DELETE /api/auth/passkeys[/<id>]` | `requireUser()` | 一覧・削除 |

登録の門番は既存の `requireUser()` に任せる。Basic で入っていても
パスキーで入っていても通るので、**初回は Basic で登録、2 台目以降は
パスキーで入ったまま追加登録**が同じ口で成立する。

公開にするのはログインの 2 口だけ。`publicPaths.ts` の一覧に明記する
(「公開したいものだけを書く = 既定が閉じている」を維持)。

## 7. rpID と origin は env で固定する (リクエストから組まない)

検証には rpID (`qr.tommie.jp`) と expected origin (`https://qr.tommie.jp`) が
要る。これを**リクエストから組み立ててはいけない**。

アプリは nginx の内側に居るので、自分では `http://0.0.0.0:3100` しか
見えていない。`login/route.ts` が絶対 URL の redirect をやめたのと同じ罠で、
あちらは実測で `http://0.0.0.0:3100/...` が出た。組み立てれば必ず外す。

```bash
WEBAUTHN_RP_ID=qr.tommie.jp
WEBAUTHN_ORIGIN=https://qr.tommie.jp
```

未設定なら**パスキー機能ごと無効に倒す** (`appEnv.ts` と同じで、迷ったら
閉じる側へ)。Basic 認証は生きているので締め出しにはならない。

ローカルは `localhost` が WebAuthn の特例で HTTP でも動くため、
`.env` に `localhost` / `http://localhost:3000` と書けばよい。

**`127.0.0.1` では動かない。** rpID は「ドメイン名」でなければならず、
IP アドレスは指定できないため (ループバックが安全な文脈として扱われるか
どうかとは別の話)。

厄介なのは、VS Code (WSL2) のポート転送通知の「Open in Browser」が
**必ず 127.0.0.1 を開く**こと。設定では変えられない既知の制限:

- [vscode#304355](https://github.com/microsoft/vscode/issues/304355)
- [vscode-remote-release#2711](https://github.com/microsoft/vscode-remote-release/issues/2711)

毎回アドレス欄を直すのは忘れるので、**非本番のときだけアプリ側が
127.0.0.1 を localhost へ 307 で送り返す** (`src/lib/loopbackRedirect.ts`)。

> ⚠️ 判定は **Host ヘッダ**で行うこと。`request.nextUrl` は Next.js が
> ホストを `localhost` に正規化してしまい、127.0.0.1 で開いても見分けが
> つかない (実測で確認)。
>
> 本番では絶対に転送しない。アプリの前に nginx が居て自分では 0.0.0.0 しか
> 見えておらず、host を信じて絶対 URL を組むと §7 冒頭と同じ罠を踏む。

## 8. 画面

- **ヘッダ / ログイン案内**: 「パスキーでログイン」を主ボタンにする。
  従来の Basic ログインは「パスワードでログイン」として小さく残す
  (パスキー未登録の端末でも締め出されないため)
- **`/settings/passkeys`**: 登録ボタン、登録済み一覧 (名前・最終使用)、削除。
  `requireUser()` で保護する
- **ログアウト**: ログイン中は常にヘッダに出す (当初はセッションで入って
  いるときだけだった。18 §11 の追記を参照)

登録の手順は 4 歩:

1. パスワード (Basic) でログインする
2. `/settings/passkeys` を開く
3. 「この端末を登録」→ Face ID
4. 以後はパスキーで入れる

## 9. セキュリティ上の決めごと

- **チャレンジは使い捨て + TTL 5 分**。検証の成否によらず消費する。
  **知らない credential ID で早期に断る経路でも消す** — そこを抜かすと、
  でたらめな ID を送りつけるだけで同じチャレンジを 5 分間生かしておける
  (`consumeChallengeFromClientData`)
- **セッション Cookie の名前は `__Host-` で始める**。ブラウザはこの接頭辞の
  Cookie を「Secure + Path=/ + Domain 属性なし」でなければ受け取らない。
  効くのは Domain の条件で、隣のサブドメイン (同じ `tommie.jp` 配下の
  別アプリなど) が乗っ取られても `Domain=.tommie.jp` でこのアプリに
  セッションを差し込めなくなる (cookie tossing → セッション固定)
- **`userVerification: 'required'`**。端末を持っているだけでは通さず、
  Face ID / PIN を必ず要求する
- **署名カウンタは記録するが、巻き戻りで即失効はさせない**。
  iCloud キーチェーンで同期されたパスキーはカウンタが常に 0 のことがあり、
  硬く倒すと正規の利用者が締め出される (SimpleWebAuthn の既定に従う)
- **`attestation: 'none'`**。認証器の真正性は要求しない。使うのは自分だけ
- **CSRF**: Cookie 認証になったので `SameSite=Lax` が効くようになる。
  18 §9 の `denyCrossSite()` は Basic 認証のために引き続き必要 (あちらは
  Cookie を使わないので SameSite が効かない)。両方残す
- **レート制限**: nginx の `/login` と同じ枠を `/api/auth/` にも掛ける。
  総当たりというより、チャレンジ発行の乱打を防ぐため

## 10. 実装フェーズ

1. **セッション層** — Prisma 2 モデル + migration、`sessionToken` /
   `sessionStore` / `requestAuth`、`proxy.ts` と `session.ts` の判定順、
   ログアウト。ここまでは WebAuthn 抜きで完結し、単体テストが書きやすい
2. **登録の儀式** — `register-options` / `register-verify`、`/settings/passkeys`
3. **ログインの儀式** — `login-options` / `login-verify`、`publicPaths` 追記、
   ヘッダと案内の改修
4. **仕上げ** — 実機 (iPhone) 確認、nginx のレート制限、この文書の追記

## 11. やらないこと

- **多ユーザ化**。利用者は従来どおり `BASIC_AUTH_USER` の 1 名。
  credential も session もその 1 名に紐づく
- **Basic 認証の削除** (§2)
- **パスキーだけでの新規ユーザ登録**。登録には必ず既存のログインが要る

## 12. 運用メモ

- **本番で有効にするには `.env` と `compose.yaml` の両方を更新する**
  (イメージの再ビルドは不要):

  ```bash
  D=41-QR-search/qr-search
  ENV_LINES='WEBAUTHN_RP_ID=qr.tommie.jp\nWEBAUTHN_ORIGIN=https://qr.tommie.jp\n'
  ssh vps2 "printf '$ENV_LINES' >> $D/.env"
  ./doDeploy.sh --send-compose.yml   # compose.yaml も送る (下記)
  ```

  ⚠️ **`.env` に書くだけでは届かない。`compose.yaml` の転送が要る。**
  `doDeploy.sh` は既定ではイメージしか送らず、**`compose.yaml` は転送しない**
  (サーバに配置済みであることが前提)。env を `environment:` に渡す行は
  `compose.yaml` にあるので、そちらが古いと `.env` に正しい値を書いても
  コンテナのプロセスには何も渡らない。`--send-compose.yml` がそのための
  フラグで、内容が同じなら転送は省略される。

  実際にこれで詰まった: `.env` は正しいのに画面は「WEBAUTHN_RP_ID と
  WEBAUTHN_ORIGIN が未設定」のままだった。切り分けは
  `docker exec qr-search-app-1 printenv | grep -i webauthn` が速い
  (`docker ps -qf name=app` は **他プロジェクトの `kick-app` などにも当たる**
  ので、コンテナ名は `qr-search-app-1` と直に書くこと)。

  有効になったかは外からも確かめられる。`login-options` が
  **503 なら無効、404 (まだ登録されていません) なら有効**:

  ```bash
  curl -s -X POST https://qr.tommie.jp/api/auth/passkey/login-options \
    -H 'Content-Type: application/json' -H 'Sec-Fetch-Site: same-origin' -d '{}'
  ```

- **`WEBAUTHN_RP_ID` は後から変えない**。変えると登録済みのパスキーが
  すべて無効になる(認証器は rpID に紐づけて鍵を持っているため)
- nginx のレート制限は `deploy/nginx/qr.tommie.jp.conf` に入れた
  (`/api/auth/` を `/login` と同じ枠で絞る)。反映は `./doDeployNginx.sh`
- **パスキーを全部失ったとき**: パスワードでログインし、
  `/settings/passkeys` で古い鍵を消して登録し直す。SSH は要らない

## 13. 自動ログイン (2026-07-19 追加)

パスキーの実績がある端末では、「ログインが必要です」の案内を出した時点で
**押さなくてもログインを試みる**。

### 判定は「このブラウザの実績」で行う (サーバの登録有無ではない)

**サーバに聞いてはいけない。** サーバが知っているのは「どこかの端末に鍵が
ある」ことだけで、いま画面を開いているブラウザに鍵があるかは分からない。
鍵を持たない端末で自動発火させると、いきなり「他のデバイスで QR を
読み取ってください」のハイブリッド画面が出て、初見では事故に見える。

そこで **localStorage に「この端末でパスキーが使えた」印**を置き、
それがあるときだけ自動発火する (`src/lib/passkeyHint.ts`)。

| 出来事 | 印 |
| --- | --- |
| パスキーのログイン成功 / 登録成功 | 立てる |
| `login-options` が 404 (鍵が 1 つも無い) | 消す |

印は**認可の判断には一切使わない**。偽造されても起きるのは「本人の Face ID
ダイアログが出る」だけで、検証は従来どおりサーバのチャレンジと署名で行う。
だから localStorage に置いてよい。置く値も `'1'` だけで、利用者名は入れない。

### 出す場所は「ログインが必要です」の案内だけ

ヘッダにも同じボタンがあるが、**あちらでは自動発火させない**。ヘッダは
公開ノート (docs/22) にも出るため、共有リンクを開いただけの人に Face ID
ダイアログが出てしまう。「保護されたページを開こうとした = ログインの
意思がある」場面に絞る。

### 取り消しはタブ単位で覚える

自動発火を断ったら、そのタブでは以後出さない (`sessionStorage`)。
保護ページを開くたびにダイアログが出るのは不快なため。恒久的に無効化しない
のは「一度断ったら二度と自動で出ない」を避けるためで、タブを開き直せば戻る。

### 失敗しても赤字を出さない

**自動発火の失敗は画面に出さない** (console には残す)。利用者はページを
開いただけで何も頼んでいないので、そこへ赤いエラーが出ると「壊れた」ように
見える。実際、パスキーを全部消した状態で「まだ登録されていません」が出て
しまい、案内としても的外れだった。黙って通常の案内 (ボタン 2 つ) に戻し、
**押されたときだけ本当の理由を出す**。

この設計のおかげで、iOS がジェスチャ無しの `credentials.get()` を拒む場合も
**自動が黙って「ボタンにフォーカスが当たるだけ」に退化する** (壊れない)。

### 手動の導線は残す

「パスキーでログイン」「パスワードでログイン」の両方をそのまま残す。
自動はあくまで先回りで、失敗・取り消しの後の受け皿が要る。

### 将来

Chrome 系には `mediation: 'immediate'` (端末に鍵があるときだけモーダルを
出す) が入りつつある。普及すれば localStorage の印ごと置き換えられる。
判定を `passkeyHint.ts` に閉じ込めてあるのはそのため。
