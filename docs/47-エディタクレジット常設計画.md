# 47 エディタクレジット常設計画 (Yahoo クレジットを編集画面下部に)

[46-クレジット表記計画.md](46-クレジット表記計画.md) §5 の Phase 2。
Yahoo の「アプリケーションの下部に配置」を厳格に満たすため、JAN 商品情報を
実際に使う編集画面 (`/edit/[itemNo]`) の下部にも規定クレジットを常設する。

関連: [46-クレジット表記計画.md](46-クレジット表記計画.md) /
[14-JAN商品情報取得計画.md](14-JAN商品情報取得計画.md)

## 1. なぜエディタか

Yahoo!ショッピング API の結果 (商品名・ブランド) が画面に現れるのは、
JAN スキャン後の**編集画面の事前入力**だけ (`usePrefill` → `MemoEditor`)。
「API を利用する画面の下部」を最も素直に満たす場所がここになる。

- `/new` は採番して `/edit/<番号>` へ redirect するだけの薄い入口なので対象外。
  エディタ画面は実質 `/edit/[itemNo]` の 1 枚だけで、常設先もそこだけで足りる。
- 表示画面 (`/item/`) に残るのは保存済み本文 (ユーザーのデータ) であって
  API の応答そのものではないため、対象にしない。

## 2. 前提の確認 (46 §2-1 との違い)

- **下部操作バーはエディタに無い。** `BottomActionBar` は検索ページ
  (`(search)/page.tsx`) 専用なので、46 §2-1 で挙げた競合はここでは起きない。
- 代わりに **sticky の「更新」ボタン** (`STICKY_ACTIONS_CLASS`) が form の
  末尾で画面下に貼り付く。ただし sticky は**自分の親 (form) の中でだけ**
  貼り付くので、form の外・ページ末尾に置いたクレジットとは重ならない
  (form を過ぎれば「更新」は所定の位置に戻る)。実装時に実機で目視確認する。

## 3. 変更点 (実装時の当たり所)

### 3-1. 規定 HTML を共有コンポーネントへ抽出

`about/page.tsx` に定数で持っている `YAHOO_ATTRIBUTION_HTML` を共有化し、
二重管理をなくす (46 §5 の方針どおり)。

- 新規: `src/components/YahooAttribution.tsx`
  - 規定 HTML の文字列定数と、それを `dangerouslySetInnerHTML` で描く
    小さなコンポーネントを 1 ファイルに同居させる。
  - **規定 HTML は一字一句そのまま** (リンク先・文言・span の inline style)。
    改変禁止の注意コメントと再確認 URL
    (<https://developer.yahoo.co.jp/attribution/>) も移す。
  - 見た目: リンクと分かる標準の作法 (下線 + リンク色) だけ与える。
    これは Tailwind の preflight がリンク色を inherit に潰すのを戻すためで、
    規定 HTML 自体には手を入れない。`text-xs` などの縮小はしない
    (「極端に小さくしない」の趣旨)。
- `about/page.tsx` は定数と div を消し、`<YahooAttribution />` に置き換える。

### 3-2. 編集画面: `src/app/edit/[itemNo]/page.tsx`

- ページ末尾 (「表示へ / 一覧へ」リンクの後、`space-y-4` の最後の子) に
  `<YahooAttribution />` を置く。
- **無条件で常設する** (JAN 由来のノートかどうかで出し分けない)。
  - 出し分けると「この画面は API を使ったか」の判定 (prefill の有無だけでは
    足りない — 保存後の再編集もある) が要り、複雑さに見合わない (KISS)。
  - デモでも出す。デモは JAN 取得が無効 (docs/45) だが、使っていない API の
    クレジットを出すことは規約違反ではなく、分岐を増やさないほうを取る。

### 3-3. about ページとの関係

/about は総覧 (楽天・openBD・NDL を含む帰属の集約) として残す。
エディタ下部は Yahoo の規定クレジット**だけ**を出す — 義務があるのは
Yahoo だけで、エディタに全社分を並べる理由はない。

## 4. テスト観点

- `YahooAttribution` コンポーネント: 規定 HTML (href・文言) が一字一句
  そのまま出ること。既存の `about/page.test.tsx` の文字列一致テストは
  コンポーネント側へ移し、about 側は「含まれること」だけ見る形に整理してよい。
- `about/page.test.tsx`: 置き換え後も規定 HTML が出続けること (回帰)。
- 編集画面: `logs/page.test.tsx` の流儀 (サーバページを直接描く) に倣うなら
  `getItem` のモックが要る。ページテストが重ければ、コンポーネントのテスト +
  実機目視 (§5) で足りると判断してもよい。

## 5. 実機確認

- 編集画面の最下部までスクロールし、クレジットが読める大きさで出ること。
- sticky「更新」ボタンとクレジットが重ならないこと (§2)。
- スマホ実機 (縦・横) で、safe-area 内に収まること (`main` の `pb-safe` が
  効いているはず)。
