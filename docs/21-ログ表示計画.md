# サーバログの画面表示 (Ver2.x)

書影が出ない・書誌が入らないといった失敗の原因は、サーバの警告ログにしか
出ない (設計上そうしてある — 導線を止めず、失敗は `console.warn` /
`console.error` に残す)。ところがスキャン運用はスマホで、`ssh` +
`docker compose logs` は開けない。**ログイン済みならヘッダの「ログ」から
直近のサーバログを見られる**ようにする。

実例: 書影が本番で出なかったとき、原因 (compose.yaml の環境変数の列挙漏れ)
はログの 1 行が言い当てていた。クライアントからは「書影が無い」ことしか
分からず、この 1 行に届く手段が ssh しかなかった。

```text
RAKUTEN_APP_ID / RAKUTEN_ACCESS_KEY / RAKUTEN_APP_ORIGIN が未設定のため、…
```

## 1. 方針: console を包んでメモリに控え、ページで見せる

```text
起動時 (instrumentation.ts)
  → console.warn / console.error を包む (logBuffer.ts)
  → 呼ばれるたび、元の console へ流しつつリングバッファ (200 件) に控える
ログイン済みでヘッダの「ログ」→ /logs
  → Server Component が同一プロセスのバッファを直接読む
```

- **既存コードを 1 行も変えない**。失敗はすべて `console.warn/error` に
  集約済みで、文言も「どの env が欠けているか」「楽天が何と言ったか」を
  名指しする形にしてある (docs/19 §3)。console を包めば全部拾える。
  各所に logger を配る改修はしない
- **元の console にも必ず流す**。`docker compose logs` (ssh での一次調査) は
  引き続き使えるままにする。画面はその補助
- **info / log は拾わない**。診断に要るのは警告と失敗だけで、
  リクエストログ等のノイズで 200 件が流れると肝心の警告が消える

### API ルートは作らない

当初は `/api/logs` + クライアント fetch を考えたが、やめた。ページは
ログインの門番 (proxy + `requireUser`) を既に通り、Server Component は
同一プロセスのバッファを直接読める。API・fetch・ローディング状態が
丸ごと要らない。読み直しはブラウザの再読み込みで足りる。

副作用の無い読み出しなので、`Sec-Fetch-Site` の検査 (docs/18 §9) も
要らない — あれは「副作用のある GET」を守るもの。

## 2. リングバッファの形

```ts
interface LogEntry {
  at: number        // epoch ms。表示のときに Asia/Tokyo で整形する
  level: 'warn' | 'error'
  text: string      // 引数を整形して連結。1 件 2000 文字まで
}
```

- **200 件**で古いものから捨てる。1 件 2000 文字 × 200 = 最大 400KB 程度
- **プロセスが再起動すると消える**。診断用途 (「いま何が失敗しているか」)
  には十分で、恒久保存が要るなら `docker compose logs` がある。
  DB 保存・pino 等の構造化ログ基盤はこの規模には過剰 (YAGNI)
- バッファは `globalThis` に持つ (db.ts と同じ)。dev のホットリロードで
  モジュールが再評価されても、包み直し・取りこぼしが起きない
- `Error` は `message` を採り、それ以外の非文字列は `JSON.stringify`。
  循環参照などで文字列化に失敗した値は `String()` に落とす (拾う側の失敗で
  ログ自体を落とさない)

### 秘密が映らないこと

鍵・パスワードがログに乗らないことは**書く側で**担保済み
(楽天の鍵を例外メッセージに載せないテスト、URL をログに出さない方針。
docs/19 §3)。ここでは何もフィルタしない — 乗ってしまう設計ミスを画面側の
黒塗りで隠すより、乗らないことをテストで固定するほうが確実。
表示はログイン済み本人だけ。

## 3. 画面

- ヘッダのユーザー名の右に「ログ」(ログイン中だけ表示。未ログインでは
  リンク自体を出さない — 見えても 401 だが、押せない物を見せない)
- `/logs` は新しい順に等幅で並べる。error は赤、warn は琥珀のバッジ
- 空のときは「ログはありません (起動後にサーバ側の警告・エラーはまだ
  発生していません)」— 「壊れて出ない」と見分けが付く文言にする
- 時刻は `Asia/Tokyo` で整形 (サーバの TZ に依存させない)

## 4. 段階と見送り

- **Phase 1 (この計画)**: warn / error のリングバッファ + `/logs`
- **見送り: 自動更新 (ポーリング・SSE)**。再読み込みで足りる
- **見送り: レベル切り替え・検索**。200 件なら目で追える
- **見送り: リクエストログ**。診断は警告・失敗だけで足りている
- **見送り: 永続化**。再起動をまたぐ調査は `docker compose logs`

## 5. テスト

- `logBuffer` (vitest):
  - warn / error を控え、元の console にも流れる
  - 200 件で古い順に捨てる / 2000 文字で切る
  - `Error`・オブジェクト・文字列化に失敗する値の整形
  - 二重 install しても包みが 1 重のまま
- `/logs` ページ: バッファの中身が HTML に出る (renderToStaticMarkup)。
  ログイン検査はページ側の `requireUser` が持ち、既存の仕組みのテストに任せる
- 実機: 本番でわざと失敗させる必要はない — 書影の warn は
  収録なしの本では出ないので、`RAKUTEN_APP_ID` を一時的に落とすまでは
  しない。デプロイ後にヘッダから開けて空表示が出れば配線は生きている
