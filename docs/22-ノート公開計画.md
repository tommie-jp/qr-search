# 22. ノート公開計画

QR シールから開いた `/item/4518` を、**ログインしていない人にも見せられる**
ようにする。既定は今までどおり非公開。ログイン中に「公開」トグルを押した
ノートだけが、URL を知っている人に開ける。

README の Ver2.x「公開/非公開に対応：ログインしなくても一部見える」の実装。

## 1. なぜ「パスの一覧」では足りないか

ログイン不要なパスの正本は `src/lib/publicPaths.ts` にある (docs/18 §5)。
だがそこへ `/item/4518` を書き足すわけにはいかない。**公開かどうかは
パスではなくデータで決まる**からで、トグルを押すたびに `publicPaths.ts` が
書き換わる作りにはできない。

そこで判定を二段に分ける。

| 段 | 問い | 答える場所 |
| --- | --- | --- |
| パス | このパスは誰でも開いてよいか | `isPublicPath()` (静的な一覧) |
| パス | このパスは**自分で判定するか** | `isSelfGuardedPath()` (静的な一覧) |
| データ | この行は公開されているか | `isPublicItem()` + ページ / route handler |

`proxy.ts` は 2 つめに当たったとき **読み取り (GET/HEAD) だけ素通しし、
公開か否かの判定はページに委ねる**。委ねる先は一覧に書いてあるので、
「新しいページを足したら黙って公開されていた」は起きない
(= docs/18 の「既定が閉じている」は保たれる)。

書き込み (POST = Server Action) は素通ししない。門番で 401 のまま止める。
公開ノートは読み取り専用であって、誰でも書ける口にはしない。

これは docs/18 §4 の二層防御をそのまま延長したもの。proxy は相変わらず
「楽観的な検査」で、本当の判定はデータに触る場所にある。

## 2. スキーマ: `publicAt DateTime?`

```prisma
model Item {
  // ...既存列...
  // 公開 (docs/22-ノート公開計画.md)。null = 非公開、非 null = 公開した日時。
  publicAt DateTime? @map("public_at")
}
```

`isPublic Boolean` ではなく **nullable なタイムスタンプ**にする。

- `deletedAt` と同じ形になる。「null = 既定、非 null = その状態になった日時」
  という読み方がスキーマ内で 1 通りに揃う
- いつ公開したかが残る。Boolean では「うっかり公開したのはいつか」を
  後から追えない
- 既存行は `NULL` = 非公開で入る。**移行で誰かのノートが公開に倒れることがない**
  (既定を閉じる側へ倒すのは docs/18 の `readBasicAuthConfig()` と同じ考え方)

索引は張らない。数百件規模で seq scan で足りる (`deletedAt` / `props` と同じ判断)。

**PGroonga 索引の罠**: 全文検索インデックスは Prisma スキーマで表現できず
drift 扱いになり、`prisma migrate dev` が `DROP INDEX` を自動生成する。
DROP は採用せず、20260716035008 と同じく「無ければ張り直す」を migration に
書いて冪等に保つ。`&@` は索引が無くても seq scan で動いてしまうため、
消えても気づけない。適用後に `\di` で確認すること。

## 3. 公開の判定 (正本は 1 か所)

```ts
// src/lib/publicItem.ts
export function isPublicItem(item: PublicCheckable | null): boolean {
  if (item === null) return false        // 未登録
  if (item.deletedAt !== null) return false  // ゴミ箱
  return item.publicAt !== null
}
```

**ゴミ箱のノートは公開しない**。`/item` はゴミ箱の行も本文を見せる仕様
(docs/12 §5) だが、それは「シールが貼られたままの部品が出てくる」持ち主の
ための救済であって、外の人に見せる理由はない。捨てたものが公開され続ける
ほうが驚きが大きい。復元すれば公開状態は戻る (`public_at` は消さない)。

`publicItem.ts` は DB に触らない純粋な層にする (auth.ts / session.ts と
同じ流儀)。ページ・route handler・テストのどこからでも同じ判定を呼べる。

## 4. 画面の分岐

```text
/item/4518 の GET
  ├─ ログイン中        → 従来の画面 + 公開トグル
  ├─ 未ログイン & 公開 → 公開ビュー (読み取り専用)
  └─ 未ログイン & 非公開/未登録/ゴミ箱 → 「ログインが必要です」の案内
```

最後の行が要点。**未登録と非公開を同じ応答にする**。分けると
`/item/1`, `/item/2`, … を順に叩くだけでノートの存在が判る
(itemNo は連番。§8 も参照)。

案内は `/login-required` と同じもの。従来は proxy が rewrite で差し替えて
いたが、素通しした以上ページが自分で出す。中身は
`components/LoginRequiredNotice.tsx` に切り出して両者で共有する
(2 か所に書くと文言がずれる)。

### 公開ビューで削るもの・残すもの

| 要素 | 公開ビュー | 理由 |
| --- | --- | --- |
| 本文 (markdown / テキスト) | ○ | これを見せるための機能 |
| URL 欄・タイムスタンプ | ○ | ノートの一部 |
| **QR ボタン** | ○ | シールを見た人がその場で刷り直せる。§5 |
| タグ | △ 文字だけ | リンク先のタグ検索は非公開。押して 401 になる物を出さない |
| 編集リンク・編集タブ | × | `/edit` は非公開。Server Action も門番で閉じている |
| 記法リンク | × | 書く人のための説明。読むだけの人には要らない |
| ゴミ箱バナー・復元 | × | ゴミ箱は公開しない (§3) ので、そもそも出る場面がない |
| 公開トグル | × | 持ち主だけの操作 |

「押しても 401 になる物は出さない」は、ヘッダの「ログ」リンクを未ログイン時に
隠しているのと同じ判断 (layout.tsx)。

公開ビューにはログインボタンも置く。自分のノートを未ログインのブラウザで
開いた持ち主が、そのまま編集へ入れるようにするため。

## 5. QR ボタンを公開ビューにも出す

`/print/4518` は QR 画像と itemNo しか出さない。だが**素通しはしない**。
`/item` と同じ判定を通す (公開ノートなら誰でも、そうでなければ案内)。

QR の中身は itemNo から誰でも作れるので、ここを開けても秘密は漏れない。
それでも `/item` と規則を揃えるのは、「公開ノートに関する読み取りだけが
公開」という一文で全体を説明できるようにするため。例外を 1 つ作ると、
次に足す人が「印刷は素通しだから他も…」と読む。

## 6. メモに貼った画像 (これを忘れると公開が半分壊れる)

`/api/images/<uuid>.png` は `denyUnlessLoggedIn()` で閉じている。
このままだと**公開ノートを開いた人には画像だけが割れて表示される**。

未ログインのときのフォールバックを足す: その画像名が**公開中のノートの
本文に含まれているか**を DB に訊く。

```sql
SELECT 1 FROM items
WHERE public_at IS NOT NULL
  AND deleted_at IS NULL
  AND position($1 IN memo) > 0
LIMIT 1
```

**`LIKE` は使えない**。このアプリの DB には PGroonga が入っており、`LIKE` の
挙動を乗っ取っている。部分一致は `position()` で判定すること。

名前が UUID であることは根拠にしない。既存のコメント (`route.ts`) が書いて
いるとおり「当てにくさは認証の代わりにならない」。

### `Cache-Control` を `public` から `private` へ

ついでに直す。画像は今 `public, max-age=31536000, immutable` で配信して
いる。`public` は共有キャッシュ (プロキシ) が保存してよいという意味で、
**非公開の画像にこれが付いているのは元から筋が悪い**。未ログインでも
取れる口ができた以上、なおさら曖昧にしておきたくない。

`private` にしてもブラウザのキャッシュは効く (`immutable` も効く)。
個人利用のこのアプリに共有キャッシュの利点はないので、失うものがない。

**残る穴 (承知のうえ)**: 一度公開した画像は、閲覧者のブラウザに 1 年
キャッシュされる。非公開に戻しても、既に見た人の手元からは消えない。
`immutable` を外して都度確認させる手もあるが、公開を取り消したところで
「相手が既に見た」事実は変えられない。実効のない厳密さのために全員の
表示を遅くはしない。

## 7. トグル

```ts
export async function setItemPublicAction(formData: FormData): Promise<void> {
  await requireUser()
  // ...
}
```

- **希望する状態をフォームが送る** (`public=1` / `public=0`)。DB を読んで
  裏返す作りにすると、二重送信や戻るボタンで意図と逆に倒れる。
  「1 にせよ」なら何回送っても 1
- `updated_at` は触らない。本文は変わっていないのに更新順が動くのは嘘になる
  (`trashItems` / `restoreItems` と同じ理由)。Prisma の `update` は
  `@updatedAt` を必ず打つので生 SQL で書く
- 既に公開中のノートへもう一度「公開」しても `public_at` を上書きしない
  (`WHERE public_at IS NULL` で守る)。公開した日時が押し直しで進むのは嘘
- 未登録のノートにはトグルを出さない。まだ公開する中身がない

UI は**状態を文で書く**。トグルの絵だけだとどちらが今なのか読み取れず、
公開の事故は取り返しがつかない。

```text
🔒 非公開 — ログインした人だけが見られます。      [公開する]
🌐 公開中 — この URL を知っていれば誰でも見られます。[非公開にする]
```

## 8. 承知している穴

- **itemNo は連番**。`/item/1`, `/item/2`, … と順に叩けば、公開ノートは
  総当たりで見つけられる。「URL を知っていれば」は響きほど強くない。
  §4 で未登録と非公開の応答を揃えたのは、せめて**公開していないノートの
  存在**は漏らさないため。本当に隠したいなら推測できない公開トークン
  (`/p/<random>`) を別に発番することになるが、印刷済みシールの URL 互換を
  最優先する今の設計 (schema.prisma の itemNo の項) とは別の話なので、
  必要になってから考える
- **検索エンジン**: 公開ページには `noindex` を付ける。「URL を知っている人に
  見せる」であって「web に公開する」ではない。クロールされたくなったら
  外せばよいが、既定は狭いほうへ倒す
- **ログアウトがない** (docs/18 §3)。持ち主のブラウザで公開ビューを確認
  するには、別ブラウザかプライベートウィンドウを使う

## 9. 実装順

1. migration (`public_at` 追加 + PGroonga 索引の張り直し)、`\di` で確認
2. `publicItem.ts` (純粋な判定) + テスト
3. `publicPaths.ts` に `isSelfGuardedPath()` + テスト、`proxy.ts` を素通しへ
4. `LoginRequiredNotice` 切り出し → `/login-required` と共用
5. `/item` の分岐と公開ビュー (読み取り専用)、`/print` の判定
6. トグル (`setItemPublic` + Server Action + UI)
7. 画像配信の公開フォールバック
