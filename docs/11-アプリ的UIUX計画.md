# アプリ的な操作感 (UI/UX) の計画 (Ver2.x)

PWA としてホーム画面から起動できるようになったが、中身の挙動はまだ
「Webページ」で、押しても無反応な時間・ぶつ切りの画面切り替え・
二重送信の余地が残っている。ここを段階的に「アプリ的な動き」へ寄せる
計画をまとめる。

**Phase 1〜5 は実装済み** (実装しながら分かったことは §8 に追記した)。
この文書は計画時点の設計判断と、実装で変わった点の記録。

前提: 素の GET フォーム / リンク / Server Action で JS 無効でも動く現在の
設計は壊さない。JS が効く環境で体感だけを上乗せする (プログレッシブ拡張)。

## 0. 現状の「Webページ的」な点 (棚卸し)

全ページ `force-dynamic` で毎回 DB を引くため、操作からサーバ応答までの
待ちが構造的に発生する。その待ちが可視化されていないのが体感の大半を占める。

- 「更新」を押しても無反応のまま止まって見える。連打で二重送信もできる
- リンク (編集 / QR / タグ / ページャ) を押しても遷移完了まで何も起きない
- 画面切り替えがぶつ切り (アニメーションなし)
- 編集中にリンクを踏む・タブを閉じると未保存の本文が黙って消える
- 保存後のフィードバックがない (いきなり /item に戻るだけ)
- undo は Ctrl+Z で効く (CodeMirror basicSetup の history) が、
  モバイルには押すボタンがない
- standalone 起動時はブラウザの戻るボタンがなく、ジェスチャー頼み
- 操作系が text-sm の下線リンクで、タップターゲットが小さい

## 1. Phase 1: 押した感と待ちの可視化 (最小の変更で効果最大)

体感の問題は「遅い」ではなく「反応がない」。まず全操作に即時の反応を付ける。

### 1-1. 更新ボタンの pending 表示 (README の TODO)

`useFormStatus` を使う `<SubmitButton>` を components/ に切り出し、
/edit と /item の編集フォームで使う。送信中は「更新中です…」+ `disabled` に
して、二重送信防止も同時に済ませる。

- form の子コンポーネントでないと `useFormStatus` が効かないため、
  ボタンだけ client component に分ける (ページは server component のまま)
- MemoEditorInner の「アップロード中は送信ブロック」とは独立に共存する
  (あちらは submit イベントの preventDefault、こちらは action 実行中の表示)

### 1-2. リンク遷移の pending 表示

Next 16 の `useLinkStatus` で、押したリンクにスピナー (または不透明度) を
出す。対象は操作頻度の高い 編集 / 一覧へ / 表示へ / ページャ。
`loading.tsx` (1-3) と重なるので、両方入れて過剰なら片方に絞る。

### 1-3. loading.tsx でスケルトン表示

`/` `/item/[itemNo]` `/edit/[itemNo]` に `loading.tsx` を置く。
force-dynamic なので遷移時に即座に骨組みが出て、体感の「止まってる時間」が
消える。スケルトンは凝らず、見出し + 枠のグレー矩形程度 (KISS)。

### 1-4. タップの視覚反応とターゲットサイズ

- ボタン・リンクに `active:` スタイル (押した瞬間に沈む/色が変わる)
- 操作系リンク (編集 / QR / 記法 など) をボタン風の見た目にし、
  タップターゲットを 44px 相当に広げる

## 2. Phase 2: 編集画面をアプリ的に

### 2-1. 更新ボタンの下部固定 (README の TODO)

編集フォームのボタン行を `position: sticky; bottom: 0` + `pb-safe` で
画面下に固定する。長文でもスクロールせずに保存できる。
1-1 の pending 表示と同じボタンを固定するだけで、実装は独立。

### 2-2. 未保存変更の離脱ガード

本文が defaultValue から変わっていたら `beforeunload` で確認を出す
(タブ閉じ・リロード・シール再スキャンによる離脱)。
Next のクライアント遷移 (Link) には組み込みのガードがないため、
まず beforeunload だけ入れる。Link 遷移のガードはコストの割に
編集ページ内のリンクが少なく、見送り (§6)。

### 2-3. 保存後のトースト

更新 → `/item` リダイレクトの直後に「保存しました」を 2 秒ほど出す。
redirect 先 URL に `?saved=1` を付け、クライアントで表示後に
`history.replaceState` で消す (リロードや共有で再表示されないように)。
flash cookie より単純で、サーバ状態も持たない。

### 2-4. undo / redo ボタン (README の TODO「編集：undo」)

CodeMirror の history は既定で有効なので、エディタ下のツールバー
(「画像を挿入」の並び) に undo / redo ボタンを足して
`undo(view)` / `redo(view)` を呼ぶだけ。新規の状態管理は不要。

## 3. Phase 3: 検索をその場更新に (README の TODO「インクリメンタルサーチ」)

- 入力を debounce (300ms 程度) して `router.replace(/?q=…)` +
  `useTransition`。URL が正なのは今と同じで、共有・戻るも壊れない
- `isPending` 中は結果一覧を薄くしてスピナーを出す
- GET フォームはそのまま残す (JS 無効・IME 確定前の Enter 検索の経路)
- タグ補完ドロップダウンとの競合 (入力ごとの遷移で候補が閉じないか) を
  実装時に確認する

## 4. Phase 4: 画面遷移アニメーション (View Transitions)

`next.config.ts` の `experimental.viewTransition` + React の
`<ViewTransition>` で、一覧 → 詳細をフェード/スライドさせる。

- 非対応ブラウザではアニメーションなしで普通に動く (劣化なし)
- **experimental なので最後に薄く入れる**。Next のバージョンアップで
  壊れたら外せる程度の使い方 (共有要素モーフィングまではやらない) に留める

## 5. Phase 5: standalone (ホーム画面起動) の作法

- `@media (display-mode: standalone)` のときだけヘッダに「←戻る」を出す
  (`history.back()`)。ブラウザで開いたときは出さない
- ヘッダを `position: sticky; top: 0` で固定し、深くスクロールしても
  検索・ホームに戻りやすくする

## 6. ヘッダーのハンバーガーメニュー (2026-07-19 追加)

ヘッダー右側の項目をハンバーガーメニューに畳んだ
(`src/components/HeaderMenu.tsx`)。

### なぜ

ログイン (docs/18) とパスキー (docs/29) を足した結果、ログイン中は
QR / GitHub / ユーザー名 / ログ / パスキー / ログアウト の 6 項目が
横に並び、**iPhone の幅で 1 文字ずつ縦に折り返れて崩れた**。
項目は今後も増える側なので、畳める入れ物にする。

### 画面幅で出し分けない

「狭いときだけメニュー」にするとコードパスが 2 本になり、片方だけ壊れる。
実際この崩れもデスクトップでは見えていなかった。全幅で同じ形にして、
壊れ方を 1 通りに保つ。

**ユーザー名だけはメニューの外に残す。** 「誰で入っているか」は一目で
確かめたい情報で、押す物でもない。1 項目ならヘッダーは崩れない。

置き場所は**ヘッダーの左端**(ユーザー名は右端のまま)。片手持ちの親指が
届く側に開閉ボタンを置く。パネルも合わせて左寄せにすること —
`right-0` のままだと画面の外へはみ出す。

### 閉じる条件は 3 つ

項目タップ / 外側タップ / Escape。項目タップはパネルでバブリングを
1 か所で受ける (項目が増えても閉じ忘れない)。

### 踏んだ落とし穴 2 つ

#### 1. backdrop-filter は position:fixed の包含ブロックになる

外側タップ用の覆いを `fixed inset-0` でヘッダーの中に置いたところ、
覆いが**ヘッダーの矩形にしか広がらなかった**。ヘッダーには
`backdrop-blur` が掛かっており、`backdrop-filter` を持つ要素は
fixed 子孫の包含ブロックになるため。メニューを開いたまま本文のリンクを
押せてしまい、閉じずに画面が遷移した。**覆いは body へ portal する。**

z-index は `z-10` (ヘッダーの `z-20` より下)。本文より上なので外側タップは
拾えるうえ、ヘッダー自身の操作 (✕) を邪魔しない。

#### 2. 自前の UI 状態を持つ項目は unmount で道連れになる

「項目タップで閉じる」をそのまま適用すると、押した部品ごと unmount される。
`HeaderQrButton` は開いたはずの QR オーバーレイが即座に消え、
`PasskeyLoginButton` は失敗しても赤字を出す先が消える。
該当する部品は自分の `onClick` で `stopPropagation()` し、
メニューを開いたままにする。

### 項目の並びは画面の下端から出す (2026-07-19 変更)

当初は開閉ボタンの直下に垂れるドロップダウンだった。これは PC の作法で、
このアプリの主戦場であるスマホでは
**「上端のボタンを押す → さらに上に出た項目を押す」**となり、
片手持ちの親指が 2 回とも届かない。届きにくいのは左右ではなく**高さ**で、
開閉ボタンを左端に寄せただけでは解けていなかった。

項目をボトムシート (画面の下端に貼り付く板) にすると、開いた後の選択が
親指の範囲で完結する。ハンバーガーそのものをやめる案もあるが、
中身は QR / GitHub / ログ / パスキー / ログアウトという**月に数回触るか**の
管理項目だけで、主要動線 (検索・スキャン・画像検索・新規) はすべて
画面に出ている。畳む対象の選び方は正しいので、畳み方だけを直す。

シートに関わる決めごと:

- 幅は `mx-auto max-w-2xl` で本文の器と揃える。全幅に伸ばすと PC で
  画面を横切る帯になり間延びする (画面幅で出し分けない方針は維持)
- 下端の余白は自前で持つ (`pb-[max(0.5rem,env(safe-area-inset-bottom))]`)。
  `main` の `pb-safe` は `fixed` には効かず、ホームバーに潜る
- 開くときだけ下からスライドさせる (`animate-sheet-up`)。閉じる側の
  アニメは付けない — 閉じ終わるまで unmount を待つ状態管理が要るわりに、
  得られるのは見た目の余韻だけ
- 開いている間は `document.body` の `overflow` を `hidden` にする。
  暗くした背景が指で動くと「触れないのに動く」矛盾になる

#### 3. 覆いを暗転させてもヘッダーは暗くならない (できない)

シート化に合わせて覆いを透明から `bg-black/40` にしたが、
**暗くなるのは本文だけでヘッダーの帯は明るいまま**である。これは妥協ではなく
避けられない制約で、理由は落とし穴 1 の z-index と同じ構造にある:
ヘッダーは `z-20` で積み重ね文脈を作るため、その中にある開閉ボタンの
`z-40` は「ヘッダーの中での 40」でしかない。覆いをヘッダーより上
(`z-30` 以上) へ出すと ✕ が覆いの下に潜り、落とし穴 1 で避けたはずの
「押した物と動いた物が食い違う」状態に戻る。
**画面全体を暗くするより ✕ が正しく効くほうを取る。**

#### 4. portal はタブ順を DOM の末尾へ動かす

パネルをヘッダーの中に描いていた頃は、DOM の並びがそのままタブ順だったので
開閉ボタンの次に項目へ入れた。body へ portal するとシートは文書の末尾に付き、
Tab がヘッダー → 本文 → シートの順になる。**開いたらシートへフォーカスを移す**
(`tabIndex={-1}` + `focus()`)。項目の外へ出た後まで閉じ込める処理
(focus trap) は持たない — Escape と外側タップで出られるので行き止まりに
ならない。

### テストの範囲

`HeaderMenu.test.tsx` は**閉じている状態の描画**しか確かめない
(開閉ボタンの aria 属性、閉じている間は項目を描かないこと)。
docs/09 §7 と同じ理由で、このリポジトリは操作を伴うテストの土台

## 9. 実装して分かったこと (計画からの変更点)

計画どおりにいかなかった点と、その場で決めたことの記録。

### 8-1. loading.tsx はルートグループに入れる

`app/loading.tsx` に置くと、自前の loading を持たない **すべての**ページ
(/print・/docs) にまで検索の骨組みが出てしまう。`app/(search)/page.tsx` +
`app/(search)/loading.tsx` のルートグループに移した (URL は `/` のまま)。

### 8-2. useLinkStatus と loading.tsx は「重ねない」

計画では両方入れて過剰なら削るとしたが、住み分けができた。

- 骨組みを出せるページ (`/` `/item` `/edit`) … loading.tsx。押した瞬間に
  骨組みが出るので、リンク側のスピナーは出番がない (プリフェッチ済みなら
  pending 状態自体が飛ばされる、と Next のドキュメントにもある)
- 骨組みのないページ (`/print`) と、同じルートで searchParams だけ変わる遷移
  (ページ送り・並び替え) … `PendingLink` (useLinkStatus)。ここは押しても
  loading.tsx が出ないため、リンク側で反応を返すしかない

### 8-3. `<ViewTransition>` は layout ではなくページに置く

layout に置くと **startViewTransition が一度も呼ばれない** (実測)。layout の
要素は遷移しても unmount されず、enter / exit が起きないため。
`components/PageTransition.tsx` に切り出し、各ページの本体を包んだ。

副作用として都合がよかった点: ページ単位なので、検索で searchParams だけが
変わる遷移では動かない。打つたびに一覧がスライドしたら鬱陶しい。

型は `@types/react` (19.2) がまだ `ViewTransition` を持たないので
`src/types/react-view-transition.d.ts` で補っている。実体は App Router が使う
React canary 側にあり実行時には存在する。@types/react が追いついたら消す。

### 8-4. 検索窓の追従は key ではなくフォーカスで決める

`page.tsx` の `key={query}` は外した (1 文字ごとに作り直されるとフォーカスも
キャレットも飛ぶ)。代わりに SearchForm が URL の変化を見て、

- 窓にフォーカスがある = 入力中 … 反映しない。自分が投げた検索の応答が返る頃
  には続きを打っていることがあり、URL で上書きすると打った文字が消える
- フォーカスがない … URL が正として反映する (スキャン・タグリンク・戻る)。
  打ち終わっていれば最後の応答に必ず追いつく

「最後に投げた語を覚えて比べる」案は、応答が順に返ると古い応答を外部遷移と
誤判定するため採らなかった。

### 8-5. CRLF が undo ボタンを壊していた (直した)

DB には Ver1 由来の CRLF の本文がある。CodeMirror は行末を LF で扱うので、
hidden input (CRLF) とエディタの中身 (LF) が初手から食い違い、
@uiw/react-codemirror が差を埋めようと dispatch する。この dispatch が履歴に
1 手積まれるので、**何も編集していないのに「元に戻す」が押せ、押すと本文が
LF に化けて未保存扱いになる**状態だった。

MemoEditor で最初に LF へ揃えて解決 (どのみち 1 文字でも打てば LF に正規化
されて保存される)。undo ボタンを足さなければ気づかないままだった。

### 8-6. 離脱ガードは「入力があったら」ではなく「内容を比べる」

マウント時のフォーム内容 (FormData) を覚えておき、離脱時に比べる方式にした。
入力イベントで dirty を立てる方式だと、書いて元に戻しても引き止めてしまう。
比較方式なら「打つ → 元に戻す → 離脱」で黙って通る (実機相当で確認済み)。

### 8-7. 実機相当の確認結果 (Chromium)

- インクリメンタルサーチ: 打つと URL が `?q=` に変わり結果が追従、リロードなし、
  フォーカス維持。応答が返る最中に打ち続けても入力は消えない
- 更新ボタン: 「更新中です…」+ disabled (二重送信防止) → 保存 →
  `?saved=<時刻>` へ → 83ms でトースト表示 → URL から印が消える → 2 秒で消滅
- View Transitions: 一覧→詳細で `nav-forward`、編集→表示へで `nav-back` が
  渡ることを確認。骨組み→本文の差し替え (Suspense) は種類なし = 無効
- 離脱ガード: 編集後の離脱で確認ダイアログ、元に戻した後は出ない
- standalone: CSS (`@media (display-mode: standalone)`) の出力は確認したが、
  display-mode のエミュレートが効かず実際の表示は未確認。**実機で要確認**
