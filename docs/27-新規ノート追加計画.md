# 「+」ボタンからの新規ノート追加の設計 (Ver2.x)

現状、新規ノートを作る道は 2 つしかない。

- スキャン・手入力した未登録コードの「新規登録」ボタン
  ([10-スキャン新規登録計画.md](./10-スキャン新規登録計画.md))
- `/edit/<番号>` の URL 直打ち (未登録番号なら空のエディタが開く)

コードも番号も無しで「思いついたことを書き始める」導線がない。
検索ページに **「+」ボタン** を足し、押すと**空き番号を採番した
空ノートのエディタ**が開くようにする。ここでは設計判断を残す。

## 1. 導線の全体像

```text
「+」を押す
  → GET /new (サーバで採番: 1000 以上で未使用の最小番号)
  → /edit/<番号> へ redirect
  → 空ノートのエディタ (フォーカス済み)
  → 「更新」で作成 (それまで DB には何も書かない)
```

- 編集ページは未登録番号を開くと空のエディタになる (upsert 前提の
  Ver1 由来の作り)。今回増えるのは入口の「+」と採番の `/new` だけで、
  編集・保存 (`updateItemAction` → `upsertItem`) は既存のまま
- **保存するまで DB に行を作らない** (スキャン新規登録と同じ)。
  押し間違えて戻ってもゴミの空ノートは残らない

## 2. 採番 — クリック時に「1000 以上で未使用の最小番号」

採番は既存の `nextItemNo()` (src/lib/items.ts) をそのまま使う。
設計は [10-スキャン新規登録計画.md](./10-スキャン新規登録計画.md) §4 で
済んでいる。

- `MIN_ITEM_NO = 1000` 以上で、**欠番を埋める** (max+1 ではない)。
  番号はシールに印刷して貼るものなので短いほどよく、実データは
  1〜6000 に隙間が大量にある
- 純関数 `firstUnusedNo` (src/lib/itemNo.ts) はテスト済みで、
  重複表記 ("1000" と "01000") も既に考慮されている
- ゴミ箱のノートも `items` に行が残るので、その番号は自然に飛ぶ
  (= 復元まで予約。[12-ゴミ箱計画.md](./12-ゴミ箱計画.md) §4)

### 採番のタイミングはページ描画時ではなくクリック時 (`/new`)

対抗案は「検索ページの描画時に採番して `/edit/<番号>` を直リンク」。
これは 2 点で分が悪い。

- すべての検索描画で余計な DB クエリが増える (今の採番は
  0 件検索のときだけ引いている。(search)/page.tsx のコメント参照)
- タブを開きっぱなしにすると番号がどんどん古くなる

`/new` は redirect するだけの薄いサーバページで、押した瞬間に採番する
ので常に最新。**DB には書かないので GET でよい** (副作用のない読みだけ)。

- リンクは **prefetch を切る** (`prefetch={false}`)。切らないと
  viewport に入るだけで採番クエリが飛ぶうえ、prefetch 済みの古い
  redirect 先へ飛んでしまう
- 副産物として「常に新規ノートを開く URL」ができる。PWA の
  manifest shortcuts (ホーム画面アイコン長押し → 新規ノート) を
  あとから足せる (§6 で見送り)

## 3. 「+」の置き場所 — 検索フォームのボタン列の端

SearchForm の「スキャン・画像検索・検索」の並びに足す。

- ノートへの入口 (検索・スキャン・画像検索・新規) が一列に揃う
- 0 件検索などの条件なしに、いつでも見えていつでも押せる
- `<a>` (Link) は form の中に置けるので、GET フォームとの入れ子問題が
  ない。server action の `<form>` を足す案は form の入れ子になり
  HTML として不正 — これも `/new` (リンク) 案を選ぶ理由のひとつ
- ラベルは **「+」だけ**にして最小幅に抑える。意味は
  `aria-label="新規ノート"` と `title` で補う
- 押しても画面が変わるまで何も起きないので、素の Link ではなく
  **PendingLink** でスピナーを出す。`/new` は force-dynamic かつ
  `loading.tsx` を持たない = 骨組みが出ない導線で、PendingLink が
  まさにそのために用意されている (docs/11 §1-2)

### ボタン列は折り返す (実測して判った)

当初は「『+』だけなら細いので一列のまま入る」と踏んでいたが、**実測すると
そもそも余白が無かった**。375px でボタン 4 つ (スキャン 90 + 画像検索 90 +
検索 80 + `+` 58) を一列に置くと、入力窓が **26px** まで潰れ、さらに
横スクロールが 12px 出る (「+」を入れる前でも入力窓は 44px しかなかった)。

そこで折り返す作りにする。

- フォームを `flex-wrap`、入力窓を `min-w-40 flex-1` にして、
  **入りきらなければボタン列を次の行へ送る**
- ボタンは 1 つの `div` にまとめて塊として折り返す。個別に折り返すと
  「スキャンだけ入力窓と同じ行に残る」散らかった並びになる
- 塊の中でも `flex-wrap` する。320px ではボタン 4 つが一列に収まらず、
  折り返せないと横スクロールが出るため

結果 (実測): 375px 以上で入力窓 328px・はみ出しなし、PC は従来どおり一列。

> 320px にはヘッダ帯 (QR/GitHub/ユーザ名/ログ) 由来のはみ出しが別途
> 残っている。これは以前からあるもので、この計画の範囲外。

却下した置き場所:

- 結果ヘッダ行 (件数・検索ヘルプ・ゴミ箱の行) … テキストリンクに
  なって目立たない。「作る」は主要操作なのでボタンの列に置く
- 右下フローティングボタン (FAB) … モバイルアプリの常道ではあるが、
  既存 UI に浮遊要素がなく (sticky の更新ボタンは面で貼り付く)、
  これだけ浮くと不揃い。ページ送りとも重なる

## 4. 競合 — 同じ番号を同時に掴んだ場合

2 タブ・2 端末がほぼ同時に「+」を押すと同じ番号の `/edit` が開き、
後から保存した方が upsert で上書きする。これはスキャン新規登録で
織り込み済みの既知のリスクと同じもの。

- クリック時採番 (§2) なので、番号が古いまま保存に至る窓は狭い
- 編集ページは既存ノートがあれば**その本文を初期値に出す**ので、
  先を越された場合は開いた時点で気づける (空のはずのエディタに
  相手の本文が見える)
- 行の事前確保や楽観ロックは足さない。単一ユーザー運用で多重タブの
  同時作成はまれ。マルチユーザ化 (Ver3) では採番ごと見直す

## 5. 認証

`/new` は publicPaths (src/lib/publicPaths.ts) に**足さない** = 既定で
ログイン必須のまま。未ログインで開けば既存の仕組みで login-required に
流れる。編集ページと更新 action も従来どおり守られているので、
この機能で新しく開く口はない。

## 6. 見送り (YAGNI)

- **番号を選んで作る UI** … `/edit/<好きな番号>` の直打ちで従来どおり
  できる。選ばせる UI は「空いているか」の検査と競合の面倒が増えるだけ
- **編集ページの見出しを新規用に変える** … 「edit #<番号>」のまま。
  スキャン新規登録も同じ見た目で通っており、困っていない
- **manifest shortcuts** (§2 の副産物) … PWA まわりは実機確認が残って
  いる項目があり、その作業のときに併せて検討する
- **作成直後に一覧へ反映する工夫** … 保存後は `/item/<番号>` へ飛ぶ
  (savedHref) ので、一覧に出す必要がそもそもない

## 8. 確認したこと (ローカル end-to-end)

- `/new` → `302/307` で `/edit/1018` へ。ゴミ箱の 1015〜1017 を予約として
  正しく飛ばし、最初の空き番号を返した
- `/edit/1018` は**空**のエディタでフォーカス済み。DB には行が増えない
- 「更新」で `items` に 1018 が作成され、`/item/1018` へ遷移。その後の
  `/new` は 1019 を返す (採番が進む)
- 未ログインの `/new` はログイン必須ページへ差し替わる (rewrite)。
  `publicPaths` に載せていないので既定で閉じている
- 320 / 375 / 414 / 768 / 1280px でレイアウト実測 (§3)。タグ補完の
  ドロップダウンは折り返し後も入力窓の直下・左揃えで出る
- `npm test` 793 件通過、`tsc --noEmit` / `eslint` エラーなし

> App Router の `prefetch={false}` は viewport でも **hover でも**
> 先読みしない (Pages Router は hover で先読みする点が違う)。
> `/new` の採番が押す前に走らないことは、この仕様に依っている。
