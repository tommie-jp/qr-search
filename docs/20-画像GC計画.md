# 画像の孤児 GC 設計 (Ver2.x)

memo に貼った画像は `images` テーブル (bytea) に入るが、いま**消す手段が
何もない**。[12-ゴミ箱計画.md](./12-ゴミ箱計画.md) §7 で「将来の GC スクリプト
課題」として切り出したものを、ここで設計する。まだ実装はしていない。

## 1. 前提: 参照は本文中の文字列でしかない

`Image` は `Item` への外部キーを持たない。参照は memo 本文に書かれた
`/api/images/<UUID>.<ext>` という**文字列だけ**である
([prisma/schema.prisma](../prisma/schema.prisma))。この一点が設計の制約を
ほぼ全部決めている。

- 1 枚の画像を複数のノートが参照しうる (本文をコピペすれば起きる)
- 同じノートの中に同じ画像が 2 回現れうる
- 「このノートの画像」という関係は DB に存在せず、本文を読まないと分からない

画像を作る経路は 2 つあり、どちらも `saveImage()`
([src/lib/imageStore.ts](../src/lib/imageStore.ts)) を通る。

- エディタからの貼り付け / D&D / 画像ボタン (`POST /api/images`)
- サーバが取ってきた書影 ([19-書影取得計画.md](./19-書影取得計画.md))

**どちらも「アップロード時点では本文から参照されていない」**。参照が生まれる
のは、ユーザが「更新」を押して memo が保存された後である。この時間差が §3 の
猶予期間の理由になる。

## 2. 検討したが採らなかった案: イベント駆動の即時削除

最初に考えたのは、削除の瞬間に消す 3 つのルールだった。

1. ノートを永久削除したときに、そのノートの画像も消す
2. 画像を挿入したまま「更新」を押さずに画面遷移したら、その画像を消す
3. 画像を消して「更新」を押したときに、その画像を消す

**方向は正しいが、これだけでは取りこぼす。** 単独では正本にできない。

### 2.1 「画面遷移」は起きるとは限らない (ルール 2)

タブを閉じる・ブラウザが落ちる・端末の電源が切れる・PWA が OS に kill される・
通信が切れる — このどれでも遷移イベントは飛ばない。`beforeunload` / `pagehide`
と `sendBeacon` の組み合わせは best effort であって保証がなく、モバイル Safari
では特に落ちる。**孤児を減らす施策にはなるが、なくす保証にはならない**。

### 2.2 逆に消しすぎる (ルール 2 の危険側)

「このエディタセッションでアップロードした画像」を消す実装にすると、
`挿入 → 更新 (保存済み) → さらに編集 → 更新せず離脱` で**保存済みの画像を
消す**。消す対象は「セッション中にアップロードしたもの」ではなく
「**保存済みの本文が参照していないもの**」でなければならない。

### 2.3 生きている参照を壊す (ルール 1・3)

§1 のとおり複数参照がありうるので、「そのノートの画像を消す」を素朴に実装すると
他方のノートが画像切れになる。削除前に「他に参照が残っていないか」の確認が要る。
確認するなら、結局それは §3 の参照集合の計算である。

### 2.4 エディタを通らない経路がある

pg_dump からの復元、Ver1 からの移行、スクリプトでの一括更新、オフライン同期での
本文上書き、アップロード成功後に Ctrl+Z で挿入を取り消す、複数枚ペーストして一部
だけ消す、書影を取ったが登録を完了しない — いずれも「更新ボタン」を通らない。
イベント駆動 GC はこれらを一切カバーしない。

## 3. 方針: mark & sweep を正本にする

**参照集合を毎回作り直して、参照されていない画像を消す**。

```text
定期ジョブ:
  1. items 全行の memo を走査し、/api/images/<UUID>.<ext> を全部拾って
     参照集合を作る (deleted_at の行も含める ← §3.2)
  2. images のうち、参照集合になく、かつ created_at が猶予期間より古い
     ものを DELETE (← §3.1)
```

この 1 本で §2 の 3 ルールが狙っていたケースと、§2.1〜2.4 の取りこぼしを
まとめて吸収できる。複数参照 (§2.3) も、参照集合ベースなので自然に正しい。

数百件規模なので memo の全行走査でよい。参照の逆引きテーブルを持つ・`Image` に
FK を張るといった正規化は、本文が正本である以上いつでも壊れうる派生でしかなく、
壊れたときに直す手段は結局この全走査になる。持たない。

### 3.1 猶予期間が肝

アップロード直後の画像は**定義上どこからも参照されていない** (まだ「更新」を
押していない、§1)。猶予がないと**編集中の画像を消す**。24 時間程度あれば、
書きかけのまま寝て翌朝続ける、くらいの使い方でも巻き込まない。

### 3.2 ゴミ箱のノートも参照元として数える

`deleted_at != null` の行が参照する画像を消すと復元できなくなる
(docs/12 §1 は「復元は UPDATE 1 発、memo 内の画像リンクもそのまま生きる」を
ソフトデリートの利点として挙げている。ここで画像を消すとその前提が崩れる)。
参照スキャンは**ゴミ箱を含む全行**を対象にする。読み取り系は
`deleted_at IS NULL` を既定にする (docs/12 §3) が、**GC はその原則の例外**で
あることをコードに明記する。

### 3.3 自動 purge との関係

ゴミ箱の自動 purge (一定日数で永久削除) は**予定はあるが未実装**
(docs/12 §7 は cron 基盤がないことを理由に YAGNI としていた)。

順序を間違えないこと。**自動 purge を入れるなら、画像 GC が先か同時**である。
自動 purge だけ入れると、ノートは自動で消えるのに画像は永久に残る — つまり
「放っておくと DB が太り続ける」状態が自動化される。

逆に GC が先にあれば、purge が入った瞬間から追加の実装なしで画像も回収される
(purge で本文が消える → 次の sweep で参照集合から外れる → 猶予後に消える)。
GC は purge の実装を待つ必要がない。

## 4. 実行方法

cron 基盤がないので、まずは**手で叩くスクリプト** (`scripts/` に置く) から始める。

- `--dry-run` を既定にする。消す前に「何枚・何 MB が対象か」を出す。GC は
  取り返しがつかないので、既定は数えるだけにしておく
- dangling 参照 (本文は参照しているのに `images` にない = 画像切れ) も
  同じ走査で数えてログに出す。**GC のバグに気づく唯一の手段**がこれになる。
  消す側のバグは静かに進行するので、検知は明示的に持つ
- 定期実行は後から考える (compose に 1 コンテナ足すか、VPS の cron か)。
  手で年に数回叩くだけでも、個人規模なら容量は困らない

## 5. 即時削除を足すとしたら

§2 の 3 ルールは、正本ではなく**「ゴミ箱を空にした直後に容量が戻る」体感の
ための最適化**として位置づけるなら足してよい。ただし:

- 正しさを即時削除に依存させない (落ちても sweep が拾う、を常に成立させる)
- 削除判定は必ず「保存済み本文の参照集合」で行う (§2.2)
- 他ノートからの参照確認を必ず入れる (§2.3)

**最初は入れない**。sweeper だけで運用して、容量が体感で困ってから足す
(YAGNI)。

## 6. スコープ外

- **`circuit_svgs` の GC**。同じく孤児になるが、消えても本文から再描画できる
  派生キャッシュで実害がない (schema.prisma のコメントのとおり)。同じ走査で
  ついでに消せる形にはしておくが、優先度は低い
- **参照の逆引きテーブル / `Image` への FK** (§3 の理由)
- **画像の重複排除** (同じバイト列を 1 行にまとめる)。ハッシュで引ける形に
  すれば効くが、GC の参照計算が「名前 → 実体」の 2 段になって複雑になる。
  容量が本当に困ってから

## 7. テスト

DB 統合テスト (既存の `zzft` プレフィックス流儀で後始末):

- 参照されている画像は消えない
- 参照されていない画像は、猶予期間より古ければ消える
- **猶予期間より新しい孤児は消えない** (アップロード直後の編集中を守る、§3.1)
- **ゴミ箱のノートが参照する画像は消えない** (§3.2)
- **2 つのノートが参照する画像は、片方を消しても残る** (§2.3)
- 同じノートに 2 回貼った画像は、1 つ消しても残る
- dangling 参照が数えられる (§4)
