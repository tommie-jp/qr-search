# 44 アクセスキー方式の検討 (サーバ固定 / クライアント設定 / デモ開放)

書誌・商品情報の取得に要る外部 API キーを、いまのサーバ固定から
**ユーザがクライアントで設定する方式 (BYOK: Bring Your Own Key)** に
変えられるか、規約・実装・デモ開放の 3 面から検討したメモ。

関連: [14-JAN商品情報取得計画.md](14-JAN商品情報取得計画.md) /
[19-書影取得計画.md](19-書影取得計画.md) /
[39-デモ公開計画.md](39-デモ公開計画.md)

## 1. 前提: キーが要るのはどこか

キーが要るのは 2 系統だけ。書誌そのもの (書名・著者) はキー不要。

| 取得物 | 取得元 | キー | 実装 |
| ------- | ------- | ---- | ---- |
| 書誌 (書名・著者) | openBD → NDL サーチ | **不要** | `bookLookup.ts` |
| 書影 (カバー画像) | openBD cover → 楽天ブックス | openBD は不要 / 楽天は要 | `coverLookup.ts` |
| JAN 商品情報 | Yahoo!ショッピング | 要 (Client ID) | `yahooShopping.ts` |

- **書誌**: openBD (`fetchFromOpenBd`) と NDL サーチ (`fetchFromNdl`) の 2 段。
  どちらもキー不要。
- **書影**: openBD の応答に入っている cover URL (キー不要・ホワイトリスト版元のみ・
  実測 1/11) をまず使い、外れたら楽天 (`RAKUTEN_APP_ID` /
  `RAKUTEN_ACCESS_KEY` / `RAKUTEN_APP_ORIGIN` の 3 点) にフォールバック。
- **JAN**: Yahoo (`YAHOO_SHOPPING_APP_ID`) のみ。

つまり **キーが絡むのは「楽天書影」と「Yahoo JAN」だけ**。

## 2. BYOK は規約上問題ないか

結論: **BYOK 自体は「譲渡・貸与」に当たらず筋が良い**。ただし無条件 OK ではなく
注意が 4 点。

### 2-1. キーの名義人 = 利用主体、が一致する (むしろ改善)

両社ともキーの第三者開示・譲渡は禁止だが、それは「名義人が他人に渡すな」の話。
ユーザ自身が開発者登録し、自分の利用のために自分のキーを設定するのは
名義人本人の利用なので禁止に当たらない。蔵書管理系アプリの
「ご自身の楽天アプリ ID を入れてください」は広く行われている方式。
運営者キー 1 本を全ユーザに使わせる現行より、レート制限も分散して健全。

### 2-2. 楽天・第 10 条(10): 限定公開環境での利用 (現行運用にも共通するグレー)

楽天ウェブサービス規約の禁止事項に次の号がある。

> Using the Web Services in an environment where only specific people can
> access (excludes the cases where the Company explicitly grants permission)

本アプリは Basic 認証の非公開サービスなので、**キーを誰の名義にしても
この条項は解消しない**。BYOK 固有の問題ではなく、サーバ固定の現状でも
同じくグレー。個人の私的利用で実害が出る条項ではないが、
「規約上問題ないか」への最も正直な引っかかりはここ。

### 2-3. 楽天は登録作業が重い (Origin 3 点セット)

楽天は `applicationId` / `accessKey` に加え、**アプリ登録時のサイト URL と
`Origin` ヘッダの一致検査**がある (不一致は 403 `HTTP_REFERRER_NOT_ALLOWED`、
`rakutenBooks.ts` に実測記録)。ユーザに「楽天でアプリ登録し、サイト URL 欄に
このサービスの URL を入れ、3 値を設定」を踏ませることになり、
他人運営ドメインを自分のアプリとして登録させる点もややグレー。導入障壁は高い。

### 2-4. Yahoo はクレジット表示義務 (キー名義と無関係)

Yahoo!デベロッパーネットワークの API を使うサイト・アプリは、規定 HTML の
クレジットを下部に表示する義務がある (改変・極端な縮小も禁止)。
キーの名義と無関係にかかるので、BYOK 化とは別に対応が要る (→ §4)。

### 2-5. 呼び出しはサーバ経由のまま

BYOK にしてもブラウザから外部 API を直接叩く構成には変えられない
(Yahoo は CORS で通らず、楽天は Origin 検査)。正しい形は
「クライアントで入力 → ユーザごとにサーバへ保存 → 既存の
`/api/books/`・`/api/products/` がそのユーザのキーで代理呼び出し」。
サーバは**他人の秘密を預かる**ので、DB では secret 扱い (ログに出さない —
既存の「URL をログに載せない」方針を維持)、設定 API はキー本体を返さず
「設定済みか否か」だけ返す。`denyUnlessLoggedIn` + `denyCrossSite` の防御も、
焚かれる枠が運営者からユーザ本人に変わるだけで引き続き必要。

## 3. まとめ (キー系統ごと)

- **Yahoo (JAN)**: BYOK 化は規約上問題なし。クレジット表示だけ対応すればクリーン。
- **楽天 (書影)**: BYOK 自体より、非公開サービスでの利用 (第 10 条(10)、現行共通) と
  3 点セット登録の導入コストがネック。
- 副次メリット: デモ版でも「自分のキーを入れれば使える」開放が可能になる。

## 4. クレジット表示はどこに出すか

### 4-1. 義務の中身 (Yahoo)

- 対象: Yahoo!デベロッパーネットワーク提供 API を使う**すべて**のサイト・アプリ。
- 位置: 「アプリケーションの下部に配置」。
- HTML: 規定のソースのみ。改変・CSS での色変更・極端な縮小は禁止。
- 楽天も第 13 条で「会社が要求する場合、Product Information が Group によって
  供給されている旨を表示」とあり、要求があれば同様の帰属表示が要る。

### 4-2. 本アプリでの推奨: ハンバーガーメニュー →「クレジット / About」ページ

推奨は **専用の「アプリについて / クレジット」ページを作り、ハンバーガーメニュー
(`HeaderMenu.tsx`) から辿らせる**。理由:

- 既存の IA に合う。ログ・パスキーも同じくハンバーガーから開く**ページ**で、
  クレジットも同じ作法に乗せられる。
- 本アプリはスマホ主戦場で下部操作バー (`BottomActionBar.tsx`) があり、
  常時表示のグローバルフッターは操作バーと競合する。専用ページなら衝突しない。
- ページ本体で Yahoo の規定 HTML を**そのまま・読める大きさ**で出せば、
  「極端に小さくしない」の趣旨を満たせる。楽天・openBD・NDL の帰属もここに集約。

補足 (要検証): Yahoo の「下部に配置」を厳格に読むと、API 結果が出る画面
(エディタ) に常時クレジットを置く解釈もありうる。ハンバーガー内ページは
「常時表示」ではないため、心配なら結果表示画面のフッターにも規定 HTML を
出す二段構えが安全。まずは About ページで開始し、必要なら足す。

### 4-3. 表示すべき帰属 (集約先の 1 ページに全部)

- Yahoo: 規定 HTML (改変不可)。
- 楽天ウェブサービス: 帰属表示 (Rakuten Developers のバナー/文言)。
- openBD / NDL サーチ: 義務は緩いが出典として併記しておくと親切。

## 5. デモ版でキー不要の書籍情報を実行可能にする

**可能。** 書誌 (openBD/NDL) はキー不要で、書影フォールバックも
キー未設定を握り潰して null を返す (throw しない) 設計になっている。

### 5-1. いまデモで止まっている理由

`/api/books/[isbn]` と `/api/products/[jan]` は、ルート冒頭で `isDemoMode()` を見て
`demoDisabled` を即返している。このゲートが lookupBook より**手前**にあるため、
キー不要の openBD/NDL 書誌取得まで一律で止まっている。

### 5-2. 開放の方針

- **書誌 (`/api/books`)**: 冒頭の一律 `demoDisabled` を外し、`lookupBook`
  (openBD/NDL、キー不要) を走らせる。書影は `saveCoverImage` →
  `lookupCover` が楽天キー未設定を自動でスキップ (`fetchCoverUrl` が warn して
  null)、openBD cover はキー不要でそのまま効く。→ **書誌 + openBD 書影がデモで動く。**
- **JAN (`/api/products`)**: Yahoo キーが本質的に要るので `demoDisabled` のまま。
- **楽天書影**: キーが要るのでデモでは出ない (openBD 書影のみ)。

### 5-3. 注意

- デモ開放は `/api/books` を **NDL/openBD への代理踏み台**として外へ開くことでもある。
  すでに `denyUnlessLoggedIn` + `denyCrossSite` + デモ用クォータ (`demoQuota.ts`) の
  下にあるので濫用は限定的だが、開放時にデモクォータの対象に含める。
- クライアントの `demoDisabled` 文言 (`MemoEditor.tsx` の `PREFILL_DEMO_MESSAGE`) は、
  「書誌は取れる / JAN は不可」に食い違わないよう見直す。
