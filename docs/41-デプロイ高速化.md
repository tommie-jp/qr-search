# デプロイ高速化 (レジストリ差分転送)

`doDeploy.sh` の転送を、イメージ全体 (631MB) の丸ごと転送から、変わった
レイヤーだけの差分転送に変えた記録。実測で**転送は毎回 ~55MB(圧縮)** に
なり、旧方式 (全体を毎回 gzip 転送) から 3〜4 倍削減した。

## 何が遅かったか

旧 5/8 は次のように毎回イメージ全体を送っていた。

```bash
docker save qr-search-app:latest | gzip | ssh vps2 'gunzip | docker load'
```

631MB の内訳は、ほとんどが**毎回は変わらない**もの。

| 中身 | サイズ | 変わる頻度 |
| ---- | ---- | ---- |
| node:24-slim ベース | 約227MB | ほぼ不変 |
| public/ のモデル群 (onnx/embedding/paddle) | 188MB | ほぼ不変 |
| .next standalone | 88MB | 毎デプロイ |
| .next static | 93MB | 毎デプロイ |
| onnxruntime .so | 35MB | 不変 |

実際に変わるのは `.next` の standalone + static (計 ~180MB, 圧縮 ~55MB) だけ。
残り 400MB 超を毎回送り直し、しかもシングルスレッド gzip で圧縮していた。

## 何をしたか

vps2 に私設レジストリ (`registry:2`) を常駐させ、SSH トンネル越しに push する。
レジストリは同じダイジェストのレイヤーを「既に存在」で受け取らないので、
変わっていないレイヤーは転送されない。

- ローカルでビルドし、トンネル `127.0.0.1:15000 → vps2:5000` 越しに push
- vps2 は自分の `127.0.0.1:5000` から pull し、`qr-search-app:latest` に付け替える
- `compose.yaml` は無変更 (参照するタグ名は同じ)

## 詰まった点: レイヤーの再現性

素朴にやると、中身が同じ public (188MB) も**毎回再送**された。原因は COPY 層の
tar に入る**親ディレクトリ `app/` のエントリの mtime がビルド時刻**になること。
中身が同一でも `app/` の 1 行が変わるだけでレイヤーダイジェストが変わり、全再送になる。

- Dockerfile 内で `find public -exec touch` しても、`public/` 配下は固定できるが
  親の `app/` は固定できず、効果なし (実測で確認)。
- `docker compose build` / `docker build` に `SOURCE_DATE_EPOCH` を渡しても、
  この環境の既定 exporter はタイムスタンプを書き換えず無効だった。

解決は **buildx の `rewrite-timestamp=true` + `SOURCE_DATE_EPOCH` 固定**。
全レイヤーの mtime を固定エポックに書き換えるので、`app/` を含めて再現性が出る。

```bash
export SOURCE_DATE_EPOCH=1704067200
NAME=127.0.0.1:15000/qr-search-app:vX
OUT="type=image,name=$NAME,push=true"
OUT="$OUT,rewrite-timestamp=true,registry.insecure=true"
docker buildx build --builder qr-host --provenance=false --sbom=false \
  --output "$OUT" .
```

- ビルダーは **host ネットワーク** (`--driver-opt network=host`) にする。既定の
  docker-container ビルダーは別 netns で、SSH トンネル (127.0.0.1) に届かない。
- `registry.insecure=true` は 127.0.0.1 の平文レジストリ (トンネル越し) を許すため。
- `SOURCE_DATE_EPOCH` はデプロイ間で**一定**にする。変えると一度だけ全再送になる。

## 実測 (2 回目の push)

```text
再利用 (転送不要): 8 layers, 148.2 MB(圧縮)   ← ベース + public 188MB + onnx 35MB
再送:              static + standalone のみ, 計 54.8 MB(圧縮)
```

## 使い方

初回だけレジストリを設置する (一度きり)。

```bash
./deploy/setupRegistry.sh
```

以降は従来どおり。

```bash
./doDeploy.sh            # patch
./doDeploy.sh minor
./doDeploy.sh -h         # 説明を表示
```

## 本番とデモを同じ版で配る (--no-version-up)

`doDeploy.sh` は毎回 `doVersion.sh` で版を上げる。デモは薄いラッパ
`doDeployDemo.sh` が本体を呼ぶだけなので、本番→デモと続けて無印で呼ぶと**版が
ずれる** (同じコードなのに 2 回上がり、リリースコミットも 2 つできる)。

`--no-version-up` を使うと版を上げず、現行 `package.json` の版をそのまま配る。
レジストリに同版のイメージがあれば**ビルドも lint/test も飛ばして再利用**するので、
本番と**ビット単位で同一**のイメージがデモに載る。

```bash
./doDeploy.sh patch                 # 本番: v0.19.20 に上げて配る
./doDeployDemo.sh --no-version-up   # デモ: 同じ v0.19.20 を再利用して配る
```

- レジストリに同版が無ければ (初回や GC 後)、版据え置きのままビルドして push する。
- 再利用時に配られるのは「その版を push した時点のイメージ」。手元の未コミット
  変更は含まれない (同版=同一ビットを保証する仕様上の正しい挙動。ログで注意を出す)。
- `--no-version-up` と `patch|minor|major` の同時指定は矛盾なので usage エラー。

古い版が溜まったら掃除する (既定は dry-run、`--apply` で実削除)。

```bash
./deploy/registryGc.sh           # 対象表示のみ
./deploy/registryGc.sh --apply   # 5 版残して削除 + GC
```

buildx ビルダー (`qr-host`) は `doDeploy.sh` が無ければ自動生成する。

## 副産物

- レジストリに版ごとのタグ (`vX.Y.Z`) が残るので、ロールバックはリモートで
  旧タグを `qr-search-app:latest` に付け替えて再作成するだけでよい。
- SSH を ControlMaster で 1 本に束ね、全 `ssh`/`scp` で使い回すようにした
  (旧はトンネル時だけ多重化。ハンドシェイク往復を削減)。
