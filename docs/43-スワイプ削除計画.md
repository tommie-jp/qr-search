# スワイプ削除計画 (Ver2.x)

スマホアプリでよくある「行を左へスライドすると削除ボタンが現れる」操作を
検索結果の**小表示 (compact)** に入れる。押すとゴミ箱へ
(docs/12-ゴミ箱計画.md の二段階削除。復元できる)。ここでは設計判断を残す。

## 1. スコープ

- 対象は**小表示のみ**。カード表示・画像表示・選択モードには入れない
  - 選択モードには既に一括削除 (BulkTagToolbar) がある。スワイプが使えない
    環境 (キーボード操作・スクリーンリーダー) の削除導線もこれが兼ねる
- 検索一覧・アクセス順は同じ `ItemList` → `ItemRow` を通るので、小表示なら
  どちらでも効く (追加実装なし)
- ゴミ箱 (/trash) 側のスワイプ復元は今回やらない (§8 の将来案)

## 2. 方針: ライブラリは使わず Pointer Events で自前実装

候補を比較した:

| 候補 | 判断 |
| ------ | ------ |
| react-swipeable-list | DOM とスタイルに独自の前提。React 19 での実績が読めない |
| @use-gesture/react | 汎用ジェスチャ全部入り。本件は「横 1 軸 + 方向ロック + スナップ」だけで過剰 |
| 自前 (Pointer Events) | 判定ロジックは 100 行弱。純関数に切り出せばテストも書ける |

自前にする。要件が単純で、依存を増やすリスク (canary な Next との相性、
バンドル増) に見合わない。Pointer Events で書けばタッチとマウスドラッグの
両方が 1 系統で動く。

## 3. 操作仕様 (状態遷移)

```text
idle ──pointerdown──▶ tracking (始点を記録。まだ何も動かさない)
tracking ──|dx| > 8px かつ |dx| > |dy|──▶ dragging (setPointerCapture)
tracking ──|dy| が勝つ──▶ idle (縦スクロールに譲る)
dragging ──pointermove──▶ offset = clamp(dx, -(BUTTON_W + ゴム余白), 0)
dragging ──pointerup──▶ open (offset = -BUTTON_W) または closed (0)
```

- **方向ロック**: 横 8px (スロップ) を超え、かつ横が縦に勝ったときだけ
  ドラッグ開始。それまでに縦が勝てば手を引き、ブラウザの縦スクロールに任せる
- **開く判定**: 離した位置がボタン幅の半分より深い、**または**左向きの速度が
  十分 (目安 0.3px/ms) なら open へスナップ。それ以外は closed へ戻す
- **右スワイプでは開かない** (offset は 0 が上限)。open 状態からの右スワイプは
  閉じる操作
- **開くのは常に 1 行だけ**。別の行をドラッグし始めたら既存の open は閉じる
- **開いている間のタップは「閉じる」だけ**。行本体を押してもノートへ飛ばない
  (iOS 標準の作法。誤操作でノートが開くのを防ぐ)
- ページをスクロールしたら open は閉じる

### stretched link との衝突

行は「どこを押してもノートが開く」ように `::after` でリンクの当たり判定を
枠いっぱいに広げてある (ItemRow.tsx)。ドラッグが発生した pointer 操作の
直後には click イベントも飛んでくるため、**dragging に入ったらフラグを立て、
capture 段の click で preventDefault + stopPropagation して遷移を握りつぶす**。
これをしないと「スワイプしたつもりがノートが開く」誤爆になる。

### 定数

- `BUTTON_W = 72px` … 削除ボタン幅。44px (タップ最小) + 文字が収まる幅
- `SLOP = 8px` … 方向判定のしきい値

## 4. DOM / CSS

```text
<li class="relative overflow-hidden">
  <button 背面: absolute inset-y-0 right-0 w-[72px] bg-red-600 text-white>
    削除
  </button>
  <div 前面: 既存の行コンテンツ + bg-white, style={translateX(offset)}>
    ...
  </div>
</li>
```

- 前面レイヤーに `bg-white` が必須。今の行は背景が透明なので、そのまま
  ずらすと背面のボタンが透ける
- 行に `touch-action: pan-y` を指定。縦スクロールはブラウザに任せ、横だけ
  JS が取る (これが無いとドラッグ中に画面ごとスクロールして喧嘩する)
- ドラッグ中は transition なしで指に追従、スナップ時だけ
  `transition-transform` を付ける
- 削除ボタンは閉じている間 `tabIndex={-1}` + `aria-hidden` (Tab 移動で
  見えないボタンに止まらないように)

## 5. コンポーネント構成

| ファイル | 役割 |
| ------ | ------ |
| `src/lib/swipeRow.ts` (新規) | 状態遷移の純関数群。**テストはここに集中** |
| `src/components/SwipeToTrashRow.tsx` (新規) | "use client" の `<li>` ラッパー |
| `ItemRow.tsx` (改修) | compact 分岐だけ包む。card はそのまま |
| `ItemList.tsx` (改修) | `trashAction` と「開いている itemNo」を降ろす |

- swipeRow.ts は pointer 座標列 → `{phase, offset, open}` を返す純関数
- SwipeToTrashRow が pointer ハンドラ・open 管理・削除ボタン・削除実行を持つ
- スワイプ有効条件は `view === 'compact' && !selectMode` (ItemList が判定)

「1 行だけ開く」の調整は ItemList の `openItemNo` state で行う (ItemList は
既に client なので追加コストなし)。Context を切るほどの深さではない。

## 6. 削除の実行と画面反映

- サーバーアクションは既存の **`trashItemsAction` をそのまま再利用**
  (FormData に `itemNo` 1 件)。confirm は出さない — 復元できるので、
  BulkTagToolbar の一括削除と同じ判断 (docs/12 §5)
- 呼び出しは `useTransition` + `trashAction(formData)`。pending 中は
  ボタンを disabled にして二度押しを防ぐ
- 成功の見た目: 押した瞬間に行を高さ 0 へ collapse (ローカル state。
  リスト配列は触らない)。`revalidatePath` でサーバから消えた一覧が届いたら
  DOM ごと消える。collapse を挟まないと、サーバ反映までの間「押したのに
  残っている」ように見える
- **失敗時は collapse を戻してエラーを見せる** (静かに握りつぶさない)
- (任意) 「ゴミ箱に移動しました」トースト + /trash リンク。SavedToast の
  流儀 (key で作り直し) を流用。まずは無しで出し、欲しくなったら足す

## 7. テスト (TDD)

1. `swipeRow.test.ts` … 純関数を先に RED → GREEN
   - 縦優勢で手を引く / 横優勢でドラッグ開始 (方向ロック)
   - 右方向は 0 で clamp、左は -(72+余白) で clamp
   - 半分未満で閉じ、半分以上 or 速度で開く
   - ドラッグ後は click 抑止フラグが立つ
2. `SwipeToTrashRow.test.tsx` … pointer イベント発火で open/close、
   削除ボタンで action が itemNo 付きで呼ばれる、pending 中 disabled
3. `ItemRow.test.tsx` 追記 … compact だけラップされる (card にボタンが無い)
4. 実機確認 … Playwright のタッチエミュレーション + iPhone 実機。
   特に「縦スクロールと喧嘩しないか」「stretched link の誤爆が無いか」

## 8. 注意点・将来案

- **iOS の「進む」ジェスチャ**: 画面右端発の左スワイプは履歴の「進む」と
  衝突しうる (進む先があるときだけ)。行の右端はサムネ領域なので実害は
  小さい見込みだが、実機で確認する
- **デモモード**: `trashItemsAction` は選択モードの一括削除で既に通っている
  導線なので新たな制限は不要のはず。デモ環境でも一応動作確認する
- 将来案: /trash での右スワイプ復元、フルスワイプ (深く払って即ゴミ箱)、
  カード表示対応。いずれも今回は入れない (YAGNI)
