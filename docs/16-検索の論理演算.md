# 検索の論理演算 (文法の正本)

検索窓のクエリ文法 — 語の種類、AND / OR / NOT、引用、優先順位 — の
意味論を 1 枚にまとめた正本。実装 (`search.ts` が解析、`items.ts` が SQL) と
利用者向けヘルプ ([05-全文検索の使い方.md](./05-全文検索の使い方.md)、
`/docs/search` にそのまま表示される) はここに従う。

**NOT (`!`) は計画段階で未実装**
([15-単語帳向け改善計画.md](./15-単語帳向け改善計画.md) §1)。
各表に状態を明記する。実装したらこのメモの「計画」を「実装済み」に更新する。

## 1. 語 (オペランド)

クエリはトークン列に分解され、各トークンは 3 種類の語のどれかになる:

| 語 | 書き方 | 探す場所 |
| --- | --- | --- |
| テキスト語 | `抵抗` | memo・url の全文一致 (PGroonga `&@`) と itemNo 前方一致 |
| タグ語 | `#npn` (引用なし) | items.tags の完全一致 (`@>`) |
| リテラル語 | `"or"` `"#npn"` | テキスト語と同じ (演算子・タグに昇格しない) |

- テキスト語 1 つは memo / url / itemNo の 3 フィールドを**暗黙の OR** で探す。
  「1 語でもすでに OR がある」ことは NOT の意味を考えるときに効く (§3)
- 表記ゆれの正規化 (全半角・大小・バイグラム) は
  [05-全文検索の使い方.md](./05-全文検索の使い方.md) §1 に譲る

## 2. 演算子

| 演算子 | 書き方 | 意味 | 状態 |
| --- | --- | --- | --- |
| AND | 空白 (半角/全角) | 両方を含む | 実装済み |
| OR | `OR` (大小無視) または `\|` | どちらかを含む | 実装済み |
| NOT | 語の先頭に `!` または `！` | 含まない | **計画** |

- 全角の同一視: `＃` → `#` と同じく、`！` → `!` と扱う
  (日本語キーボードからの入力ゆれの吸収)
- `\|` は空白なしでも区切りになる (`A\|B` = `A OR B`)
- 括弧によるグループ化は**ない** (§3 の DNF が全て)

## 3. 優先順位と意味論 (DNF)

結合の強さ: **NOT > AND (空白) > OR**。

- `!` は直後の語 1 つだけに付く
- クエリ全体は「OR で区切られた AND グループの列」= DNF (選言標準形) になる。
  NOT はグループ内の項の否定であり、グループやクエリ全体は否定できない

```text
#英検準1 !#覚えた OR #英検1級
  → (tags に 英検準1 AND NOT (tags に 覚えた)) OR (tags に 英検1級)

抵抗 !1608 OR コンデンサ
  → ((抵抗) AND NOT (1608)) OR ((コンデンサ))
```

SQL への写像 (`items.ts`)。語 → 条件、`!` → `NOT (...)`、
グループ内 AND・グループ間 OR、の素直な対応になる:

```sql
-- #英検準1 !#覚えた 抵抗  (1 グループ、3 項)
WHERE (deleted_at IS NULL) AND (
  (    tags @> ARRAY['英検準1']
   AND NOT (tags @> ARRAY['覚えた'])                      -- 計画
   AND (memo &@ '抵抗' OR url &@ '抵抗' OR item_no ILIKE '抵抗%'))
)
```

- テキスト語の否定は 3 フィールドの OR 全体を `NOT` で包む。
  つまり「**どのフィールドにも**含まない」(ド・モルガン)
- ゴミ箱の除外 (`deleted_at IS NULL`) はクエリ文法の外で常に AND される

## 4. 引用の抑止規則

**引用 (`"`) を含むトークンは、演算子・タグ・否定のどれにも昇格しない。**
リテラル語になる。これが文法の一貫した逃げ道で、記号そのものを
検索したいときは引用すればよい:

| 書き方 | 解釈 | 状態 |
| --- | --- | --- |
| `"or"` | 語 or (OR 演算子ではない) | 実装済み |
| `"A\|B"` | 語 A\|B | 実装済み |
| `"#npn"` | 語 #npn (タグ検索ではなく全文検索) | 実装済み |
| `"!foo"` | 語 !foo (否定ではない) | 計画 |
| `!"foo"` | 語 !foo (トークンが引用を**含む**ので否定にならない) | 計画 |

最後の 2 つが同じ結果になるのは、規則が「先頭が引用か」ではなく
「トークンに引用を含むか」だから。`"#tag"` と同じ判定 (`quotedHere`) を
そのまま使う。

## 5. エッジケースの早見表

| 入力 | 解釈 | 状態 |
| --- | --- | --- |
| `#` / `＃` 単独 | 捨てる | 実装済み |
| 空引用 `""` | 捨てる | 実装済み |
| 未閉じ引用 `"foo` | 語 foo (末尾までがリテラル) | 実装済み |
| 連続 OR (`A OR OR B`) | 空グループは落ちる (`A OR B` と同じ) | 実装済み |
| `!` / `！` 単独 | 捨てる (`#` 単独と同じ) | 計画 |
| `!#` | 捨てる (否定つきの空タグ) | 計画 |
| `!!foo` | 「`!foo` という語」を含まない (剥がす `!` は 1 個だけ) | 計画 |
| `!or` | 否定つきの語 or (OR 演算子に昇格しない) | 計画 |
| `foo !foo` | 矛盾グループ。弾かず常に 0 件 | 計画 |
| `!#覚えた` 単独 | 否定だけのクエリ。**許可** (全ノートから除外) | 計画 |

- 否定だけのクエリを許可するのは「タグ X が付いていないノート」の棚卸しが
  普通に要るから。全件走査になるが個人規模 (数百〜数千件) では問題ない
  (関連度ソートを捨てたときと同じ判断。
  [04-全文検索計画.md](./04-全文検索計画.md) §3-4)

## 6. 上限と畳み込み

解析 (`parseSearchQuery`) は SQL を組む前に次を行う:

- グループ内の**重複語**、同一内容の**重複グループ**を畳む。
  同一判定のキーは kind + value (NOT 実装後は否定の有無も区別する —
  `!foo` と `foo` は別の語)
- 語の総数を `MAX_SEARCH_TERMS` (= 10) で頭打ちにする
  (WHERE 肥大の安全弁。超過分は黙って捨てる)。否定つきの語も 1 語と数える
