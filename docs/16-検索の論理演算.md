# 検索の論理演算 (文法の正本)

検索窓のクエリ文法 — 語の種類、AND / OR / NOT、括弧、引用、優先順位 — の
意味論を 1 枚にまとめた正本。実装 (`search.ts` が解析、`items.ts` が SQL) と
利用者向けヘルプ ([05-全文検索の使い方.md](./05-全文検索の使い方.md)、
`/docs/search` にそのまま表示される) はここに従う。

文法は Lucene (Elasticsearch) / Obsidian 系のサブセットに揃えてある。
NOT・括弧まで含めて**すべて実装済み** (2026-07-17)。

## 1. 語 (オペランド)

クエリはトークン列に分解され、各トークンは 3 種類の語のどれかになる:

| 語 | 書き方 | 探す場所 |
| --- | --- | --- |
| テキスト語 | `抵抗` | memo・url の全文一致 (PGroonga `&@`) と itemNo 前方一致 |
| タグ語 | `#npn` (引用なし) | items.tags の完全一致 (`@>`) |
| リテラル語 | `"or"` `"#npn"` | テキスト語と同じ (演算子・タグに昇格しない) |

- テキスト語 1 つは memo / url / itemNo の 3 フィールドを**暗黙の OR** で探す。
  「1 語でもすでに OR がある」ことは NOT の意味を考えるときに効く (§3)
- 表記ゆれの正規化 (全半角・大小・バイグラム) は
  [05-全文検索の使い方.md](./05-全文検索の使い方.md) §1 に譲る

## 2. 演算子

| 演算子 | 書き方 | 意味 |
| --- | --- | --- |
| AND | 空白 | 両方を含む |
| OR | `OR` (大小無視) または `\|` | どちらかを含む |
| NOT | `!` | 続く 1 つを含まない |
| グループ化 | `(` `)` | 中を 1 つの被演算子にする |

**演算子はすべて全角でも書ける。** `＃` → `#` と同じ入力ゆれの吸収で、
日本語キーボードの全角モードのまま打っても効く:

| 半角 | 全角 | 判定の仕方 |
| --- | --- | --- |
| 空白 | `　` | 文字クラスで比較 |
| `\|` | `｜` | 文字クラスで比較 |
| `!` | `！` | 文字クラスで比較 |
| `(` `)` | `（` `）` | 文字クラスで比較 |
| `OR` | `ＯＲ` `ｏｒ` | NFKC + 小文字化してから比較 (タグ名と同じ規則) |

- 全角の記号を文字として検索したいときは引用する (`"（株）"` / `"A｜B"`)
- `\|` と括弧は空白なしでも区切りになる (`A\|B`、`#bjt(!#npn)`)
- `!` は**トークンの先頭でだけ**演算子。語中の `!` は文字のまま (`a!b`)
- 否定に Google 式の `-` を使わないのは、`-40℃` `TO-92` `#part-a` のように
  ハイフン始まり/ハイフン入りの語がこのドメインに頻出して曖昧なため
  ([15-単語帳向け改善計画.md](./15-単語帳向け改善計画.md) §1-1)

## 3. 優先順位と意味論 (AST)

結合の強さ: **NOT > AND (空白) > OR**。SQL・Lucene と同じ。

```text
expr    := orExpr
orExpr  := andExpr ( ("OR" | "|") andExpr )*
andExpr := unary+                    ← 空白の並置 = 暗黙 AND
unary   := ("!" | "！")* primary     ← 重ねがけは打ち消し
primary := "(" expr ")" | 語
```

- `!` は続く被演算子 1 つ (語または括弧の中身) だけに付く
- クエリ全体は**木 (AST)** になる。括弧で任意の深さに入れ子でき、
  グループやクエリ全体も否定できる

```text
#英検準1 !#覚えた OR #英検1級
  → (tags に 英検準1 AND NOT (tags に 覚えた)) OR (tags に 英検1級)

#bjt !(#npn OR #pnp)
  → (tags に bjt) AND NOT ((tags に npn) OR (tags に pnp))

(抵抗 OR コンデンサ) 1608
  → ((抵抗) OR (コンデンサ)) AND (1608)
```

SQL への写像 (`items.ts` の `exprCondition`) は木の再帰そのもの。
語 → 条件、`!` → `NOT (...)`、AND/OR → 同名の SQL 演算子で、
各ノードを括弧で包むので木の入れ子がそのまま優先順位になる:

```sql
-- #英検準1 !#覚えた 抵抗
WHERE (deleted_at IS NULL) AND (
  (    tags @> ARRAY['英検準1']
   AND NOT (tags @> ARRAY['覚えた'])
   AND (memo &@ '抵抗' OR url &@ '抵抗' OR item_no ILIKE '抵抗%'))
)
```

- テキスト語の否定は 3 フィールドの OR 全体を `NOT` で包む。
  つまり「**どのフィールドにも**含まない」(ド・モルガン)
- NULL の罠はない: `tags` は `@default([])`、`memo` / `url` は `@default("")`
  で、どの列も NULL にならない (`schema.prisma`)。NULL が混じると
  `NOT NULL` → `NULL` で行が黙って落ちるので、列を足すときは注意
- ゴミ箱の除外 (`deleted_at IS NULL`) はクエリ文法の外で常に AND される
- **DNF (選言標準形) には展開しない**。括弧と NOT の組み合わせで項が
  指数的に増えうるため、木のまま SQL にする (入力サイズに比例した WHERE で済む)

## 4. 引用の抑止規則

**引用 (`"`) を含むトークンは、演算子・タグのどちらにも昇格しない。**
リテラル語になる。これが文法の一貫した逃げ道で、記号そのものを
検索したいときは引用すればよい:

| 書き方 | 解釈 |
| --- | --- |
| `"or"` `"ＯＲ"` | 語 or / ＯＲ (OR 演算子ではない) |
| `"A\|B"` `"A｜B"` | 語 A\|B / A｜B |
| `"#npn"` | 語 #npn (タグ検索ではなく全文検索) |
| `"!foo"` | 語 !foo (否定ではない) |
| `"("` `"（株）"` | 語 `(` / 語 `（株）` (グループ化ではない) |
| `!"or"` | **語 or の否定** (引用は `!` の外なので否定は生きる) |

**引用符だけは半角 `"` のみ** (§2 の全角同一視から漏れている既知の非対称)。
日本語 IME の全角モードは `”` を出すので、`”or”` は引用にならず語 `”or”` に
なる。逃げ道の逃げ道が要る場面は稀なため現状維持だが、直すなら
`"` `＂` `“` `”` をまとめて引用符として扱うことになる (開き/閉じの対応を
どうするかは要検討)。

最後の 1 行が要点。`!` はトークンの一部ではなく独立した演算子なので、
引用は `!` を抑止しない。むしろ `!"or"` が**演算子語を否定する唯一の書き方**
になる (素の `or` は常に演算子に昇格するため)。`"!or"` (語 `!or`) と
`!"or"` (or の否定) が別物なのはこのため。

## 5. エッジケースの早見表

壊れた入力は例外にせず「それらしく」解釈する (未閉じ引用と同じ思想)。
検索窓でエラーを出すより、打ちかけでも何か返るほうが UX に合う。

| 入力 | 解釈 |
| --- | --- |
| `#` / `＃` 単独 | 捨てる |
| 空引用 `""` | 捨てる |
| 未閉じ引用 `"foo` | 語 foo (末尾までがリテラル) |
| 連続 OR (`A OR OR B`) | 空の被演算子は落ちる (`A OR B` と同じ) |
| `!` / `！` 単独 | 捨てる (`#` 単独と同じ) |
| `!#` | 捨てる (否定つきの空タグ) |
| `!or` / `A !or B` | `!` を捨てる (`A OR B`)。OR は被演算子にならない |
| `!!foo` | 二重否定は打ち消して `foo` |
| `foo !foo` | 矛盾。弾かず常に 0 件 |
| `!#覚えた` 単独 | 否定だけのクエリ。**許可** (全ノートから除外) |
| 閉じ忘れ `(A B` | 末尾で自動クローズ (`(A B)`) |
| 余った `)` (`A) B`) | その `)` を捨てる (`A B`) |
| 空括弧 `()` / `!()` | 捨てる |
| `(A)` | 冗長な括弧は畳む (`A`) |

- 否定だけのクエリを許可するのは「タグ X が付いていないノート」の棚卸しが
  普通に要るから。全件走査になるが個人規模 (数百〜数千件) では問題ない
  (関連度ソートを捨てたときと同じ判断。
  [04-全文検索計画.md](./04-全文検索計画.md) §3-4)
- `NOT (memo &@ ...)` は PGroonga の索引が効かず seq scan になるが、
  同じ理由で実害なし

## 6. 上限と畳み込み

解析 (`parseSearchExpr`) は SQL を組む前に次を行う:

- 同一の**部分式**を畳む。同一判定のキーは構造 + kind + value なので、
  `!foo` と `foo` は別、`(A OR B) (A OR B)` は 1 つに畳まれる
- 被演算子が 1 つだけになった AND/OR ノードは中身に畳む (`(A)` → `A`)
- 語の総数を `MAX_SEARCH_TERMS` (= 10) で頭打ちにする
  (WHERE 肥大の安全弁。超過分は黙って捨てる)。否定つきの語も 1 語と数える。
  予算切れで `!` の被演算子が消えたら否定ごと捨てる
  (「全件除外」に化けるより、条件が無いほうが安全)

## 7. 旧計画 (docs/15 §1) からの変更

[15-単語帳向け改善計画.md](./15-単語帳向け改善計画.md) §1 は NOT を
「トークン先頭の `!` を 1 個剥がして `SearchTerm.negated` を立てる」
前置修飾子として設計していた。括弧の要望を受けて**独立した演算子**へ
改めた (2026-07-17)。前置修飾子モデルでは `! #npn` のように `!` と語の間に
空白を置けず、`!(...)` も書けないため。

この変更で、旧計画から意図的にずれた規則が 4 つある:

| 入力 | 旧計画 | 現在 |
| --- | --- | --- |
| `! #npn` (空白あり) | `!` を捨てて タグ npn | **npn の否定** |
| `!"foo"` | 語 `!foo` (引用が否定を抑止) | **foo の否定** |
| `!!foo` | 語 `!foo` の否定 | **`foo`** |
| `!or` | 語 `or` の否定 | **`!` を捨てる** (`or` は演算子のまま) |

理由:

- `! #npn` — 変更の動機そのもの
- `!"foo"` — `!` がトークンの外にある以上、引用で抑止しようがない。
  結果としてこれが演算子語を否定する唯一の手段になる (§4)
- `!!foo` — 演算子なら二重否定が打ち消すのが SQL・Lucene と一貫する。
  語 `!foo` を否定したいなら `!"!foo"` と書ける
- `!or` — 素の `or` は常に演算子 (`"or"` と書かねば語にならない) 以上、
  `!or` だけが語 `or` を指すのは元の規則と不整合。否定は `!"or"`

`SearchTerm` に `negated` は足していない (否定は AST の `not` ノード)。
それ以外 (`-` ではなく `!`、全角同一視、否定だけのクエリを許可、
`queryHasTagTerm` は肯定タグだけ、`MAX_SEARCH_TERMS` の数え方) は
旧計画のまま。
