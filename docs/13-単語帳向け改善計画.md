# 単語帳向け改善 (NOT 検索・前後ナビ・タグトグル) の設計 (Ver2.x)

きっかけは「英単語 1000 語を 1 語 1 ノートで入れ、覚えていない単語だけを
順に表示し、覚えたら次から出さない」というフラッシュカード運用。
ただし以下の 3 機能はどれも単語帳専用ではなく、部品ノートにもそのまま効く
**汎用の検索・閲覧改善**として設計する。ここでは設計判断を残す。
**実装はまだ**。計画のみ。

想定する単語帳の運用 (この 3 機能が揃ったときの姿):

```text
登録:   1 行目 = 英単語、2 行目 = #英検準1 タグ、3 行目以降 = 和訳・例文
出題:   検索「#英検準1 !#覚えた」→ 一覧は 1 行目 (英単語) だけ見える
確認:   行をタップ → /item で和訳を見る (これが「和訳ボタン」)
記録:   /item の「#覚えた」トグルをワンタップ → 出題対象から外れる
次へ:   /item の「次へ →」で次の未習得語へ (一覧に戻らない)
```

3 機能はこの順に依存が薄い → 濃い なので、**§1 → §2 → §3 の順で実装**する。
3 つとも**スキーマ変更なし** (コードのみ)。したがって prisma migrate を
走らせないので、PGroonga 索引が DROP される罠 (docs/12 §2) も踏まない。

実装前の注意: この repo の Next.js は学習データと差分がある。
`node_modules/next/dist/docs/` の該当ガイドを読んでから書く (AGENTS.md)。

## 1. NOT 検索 (`!語` / `!#タグ`)

README (親 repo) の TODO「検索の論理演算に論理否定(!)を追加」の実装。

### 1-1. 文法

- **語の先頭に `!` を付けると否定**になる: `!#覚えた` `!抵抗`
- 半角 `!` と全角 `！` を同一視する (`＃` を全角で受けるのと同じ理由。
  日本語キーボードからの入力ゆれを吸収する)
- `"!foo"` と引用したものはリテラル (否定にならない)。引用が演算子解釈を
  抑止する既存ルールの自然な延長
- `!` 単独 (語が続かない) は捨てる (`#` 単独と同じ扱い)。
  文字どおり検索したいときは `"!"`
- 先頭の `!` を 1 個だけ剥がす。`!!foo` は「`!foo` という語を含まない」
- `!or` は否定つきリテラル語 "or" (OR 演算子に昇格しない。引用と同じ理屈で、
  修飾された語は演算子にしない)
- 結合の強さ: `!` は語 1 つだけに付く。空白 (AND)・OR との優先順位は不変

意味は DNF の各 AND グループ内での否定:

```text
#英検準1 !#覚えた
  → (tags に 英検準1) AND NOT (tags に 覚えた)

抵抗 !1608 OR コンデンサ
  → ((抵抗) AND NOT(1608)) OR ((コンデンサ))
```

否定だけのクエリ (`!#覚えた` 単独) も**許可する**。「タグ X が付いていない
全ノート」は棚卸しに普通に使いたい。全件走査になるが数百〜数千件の
個人規模では問題ない (関連度ソートを捨てたときと同じ判断。docs/04 §3-4)。

`空白区切りの語を ! で修飾する`案のほか「`-語` (Google 風)」も考えたが、
`-` はタグ名 (`#part-a`) や型番 (`TO-92`) に頻出して曖昧なため不採用。
README の記述どおり `!` にする。

### 1-2. 実装 (search.ts)

- `SearchTerm` に `negated: boolean` を足す:

  ```ts
  export type SearchTerm =
    | { kind: 'text'; value: string; negated: boolean }
    | { kind: 'tag'; value: string; negated: boolean }
  ```

- `tokenize()` の `flush()` 冒頭で、引用を含まないトークンの先頭 `!`/`！` を
  1 個剥がして `negated` を立てる。剥がした残りで従来どおり
  OR 判定 → タグ判定 → テキスト、の順に処理する。ただし **`negated` が
  立った語は OR 演算子に昇格させない** (引用と同じ抑止)
- `termKey()` に否定を含める (`!text:foo` と `text:foo` は別の語)。
  `foo !foo` という矛盾グループは弾かず 0 件のまま通す (特別扱いしない)
- `queryHasTagTerm()` は**肯定のタグ項だけ** true にする。特性表は
  「同族の部品を並べる」ビューであり、`!#npn` は族の指定ではないから

### 1-3. 実装 (items.ts)

`termCondition()` で `negated` なら全体を `NOT (...)` で包むだけ:

```ts
const cond = /* 既存の tag / text 条件 */
return term.negated ? Prisma.sql`NOT (${cond})` : cond
```

- NULL の罠はない: `tags` は `@default([])`、`memo` / `url` は
  `@default("")` で、どの列も NULL にならない
- `buildQueryCondition` 以下 (検索・特性表・ゴミ箱一致数) は自動で追従する
- 0 件時の新規登録ボタンは `isTaggableCode(query)` 判定なので、
  `!` を含むクエリでは元々出ない。変更不要

### 1-4. ドキュメント

- docs/05-全文検索の使い方.md に「NOT 検索」の節を追加
  (文法・優先順位・`"!"` のリテラル化・否定だけのクエリの挙動)
- 検索ヘルプページ (/docs/search) も同期する

### 1-5. テスト (先に書く)

- search.test.ts: `!#tag` / `!語` / `！全角` / `"!foo"` リテラル /
  `!` 単独破棄 / `!!foo` / `!or` / `A !B OR C` の DNF /

## 2. アイテム画面の「前へ / 次へ」(検索結果内ナビ)

一覧に戻らずカードをめくる。**URL が正**の原則 (docs/11 §3) を守り、
検索コンテキストをクエリパラメータで持ち回る:

```text
/item/4951?q=%23英検準1+!%23覚えた&sort=itemNo
```

- 共有・再読込・戻るがそのまま動く。QR シールから開いた素の
  `/item/4951` (パラメータなし) では**ナビを出さない** (従来どおり)
- sessionStorage に一覧を覚えるクライアント方式は、直リンク・再読込で
  文脈が消えるため不採用

### 2-1. 入口 (一覧側)

- `ItemRow` の詳細リンクに、検索コンテキスト (`q` / `sort`) を付ける。
  props で受け取り、`/item/${itemNo}?q=...&sort=...` を組み立てる
  (組み立ては `searchUrl.ts` に `itemHref(itemNo, query, sort)` を足して
  1 箇所に集約)
- 空クエリ (「すべて」ブラウズ) でもコンテキストは付けてよい
  (`q=` なしで `sort` だけ、など。全件の中で前後できる)

### 2-2. サーバ側 (items.ts)

隣接ノートは SQL のタプル比較では書かない。現行コードの流儀
(`nextItemNo()` が全件引いて JS で隙間を探す) に合わせ、
**該当クエリの itemNo を並び順どおり全件引いて JS で前後を求める**:

```ts
// 検索ヒット全件の itemNo を一覧と同じ並びで返す (ナビ用の軽い射影)。
export async function listSearchItemNos(
  query: string,
  sort: Sort,
): Promise<string[]>
```

- `SELECT item_no FROM items ${buildWhere(query)} ${buildOrderBy(sort)}
  LIMIT ${NAV_LIMIT}` (NAV_LIMIT = 5000 程度の安全弁。超えたらナビに
  「以降省略」と正直に出す — 黙って打ち切らない)
- 単語 1000 語 + 部品数百件の規模で、文字列 1 列 × 数千行は誤差

**前提修正**: `buildOrderBy` の更新順に同値タイブレークがない
(`ORDER BY updated_at DESC` のみ)。同時刻の行で並びが不定になり、
ページ送りもナビも揺れるため、`, item_no ASC` を足して全順序にする。
これは既存のページ送りにも効く独立したバグ修正なので最初にやる。

### 2-3. アイテム画面 (page.tsx)

`searchParams` の `q` / `sort` を読み、あれば `listSearchItemNos` を
1 回引いて現在位置を探し、ヘッダ下 (またはメモの下の固定バー) に出す:

```text
← 前へ    12 / 987 件    次へ →
```

- リンク先も同じ `q` / `sort` を引き継ぐ (`itemHref` を共用)
- 「n / 総数」の表示が単語帳の進捗感を兼ねる
- 一覧へ戻るリンクは `buildSearchUrl(query, 該当ページ, sort)`
  (該当ページ = `floor(index / PAGE_SIZE) + 1`。戻ったとき同じページが開く)
- 遷移は `PendingLink` + view transition (前へ = nav-back、次へ =
  nav-forward) で docs/11 の作法に合わせる

### 2-4. 現在ノートが検索から外れているとき

§3 のトグルで `#覚えた` を付けた直後など、開いているノートが
もう検索条件に一致しないことがある。このときこそ「次へ」が要る
(覚えた → 次の未習得語、が単語帳の主動線):

- 前後の計算は純関数に切り出す:

  ```ts
  // list に current がいれば通常の前後。いなければ挿入位置で前後を返す。
  export function neighborsOf(
    list: string[],
    current: string,
    compare: (a: string, b: string) => number,
  ): { prev: string | null; next: string | null; index: number | null }
  ```

- **番号順**: itemNo の比較は SQL と同じ順序
  (`item_no_num ASC NULLS LAST, item_no ASC`) を JS で再現できるので、
  外れていても挿入位置から前後が正しく出る。単語帳はこの並びを推奨
- **更新順**: itemNo だけでは位置を再構成できない (updated_at が要る) うえ、
  外れる操作 (タグ編集) 自体が updated_at を動かすので位置の意味が薄い。
  外れていたらナビを隠し「検索条件から外れました」+ 一覧へ戻るリンクだけ出す。
  凝った復元はしない (YAGNI)

## 3. タグトグル (アイテム画面のワンタップ「覚えた」)

現状タグを付け外しするには、編集画面で本文を書き換えるか、一覧に戻って
選択モード (BulkTagToolbar) を使うしかない。/item にワンタップの
トグルを足す。

### 3-1. どのタグのトグルを出すか

固定の「覚えたボタン」は単語帳専用になるので作らない。
**§2 で持ち回る検索コンテキスト (`q`) に含まれるタグ項**からトグルを導出する:

```text
q = #英検準1 !#覚えた で開いた /item
  → トグルチップ: [☐ #覚えた] [☑ #英検準1]
```

- 否定タグ (`!#覚えた`) はまさに「絞り込みに使っているフラグ」なので
  **先頭に出す** (主ボタン)。押す → タグが付く → 出題対象から外れる
- 肯定タグも出す。逆タグ運用 (`#未` を外す = 覚えた) や部品の棚卸し
  (`#未整理` を外す) がこれでできる
- `q` がない・タグ項がないときは出さない (QR から開いた通常の部品閲覧を
  汚さない)。将来「よく使うタグを常設」したくなったら別途考える (YAGNI)
- 設定画面・専用スキーマは作らない。**タグの正本はメモ本文** (docs/06) の
  原則を崩さず、トグルは本文書き換えの近道にすぎない

### 3-2. サーバーアクション (actions.ts)

```ts
// /item のタグトグル。付いていなければ追加、付いていれば外す。
export async function toggleTagAction(formData: FormData): Promise<void>
```

- 入力: `itemNo` (isValidItemNo 検証) と `tag` (parseTagToken で正規化。
  通らなければ throw)。フォーム由来でも境界で必ず検証する
- 処理は既存の純関数を丸ごと再利用: `extractTags(memo)` で現状を判定 →
  `addTagToMemo` / `removeTagFromMemo` (tagEdit.ts) → 変化したときだけ
  `upsertMemo` (bulkTagAction と同じ流儀)
- `revalidatePath('/')` と `revalidatePath('/item/<itemNo>')`。
  リダイレクトはしない (同じページに留まり、`q` 付き URL が保たれ、
  §2 のナビが「外れた」状態を描き直す)

### 3-3. UI

- 置き場所: 既存のタグ一覧 (チップの列) の並び、またはその直上の行
- 見た目: `☐ #覚えた` (未付与) / `☑ #覚えた` (付与済み)。押すと
  `SubmitButton` の pending 表示 (docs/11 Phase 1 の作法) で待ちを見せる
- **外せないタグの扱い**: `removeTagFromMemo` は「タグだけの行」からしか
  消さない (文章中のタグを壊さないため)。付与済みなのに
  `removeTagFromMemo(memo, tag) === memo` のタグは、トグルではなく
  編集画面へのリンクにして「本文中にあるため編集で外す」と伝える。
  判定はサーバ側の純関数 `canRemoveTag(memo, tag)` で行い、
  黙って no-op にしない (silent failure を作らない)
- 削除は復元可能 (もう一度押せば付く) なので confirm は出さない
  (ゴミ箱と同じ判断。docs/12)

### 3-4. §2 との連携 (単語帳の主動線)

1. `#英検準1 !#覚えた` の 12 / 987 件目を開いている
2. `☐ #覚えた` をタップ → タグが付き、ノートは検索条件から外れる
3. ナビは §2-4 の挿入位置ロジック (番号順) で「次へ」を出し続ける
4. 「次へ」→ 13 件目だった単語が繰り上がって 12 / 986 件目として開く

## 5. 単語帳としてなお残る制約 (この計画のスコープ外)

- 和訳をノート内で隠す記法 (spoiler) はない。一覧の 1 行目 =
  英単語だけが「表」であり、/item を開けば全文が見える
- 出題順は番号順/更新順のみ。シャッフルも SRS (間隔反復) もない。
  そこまで要るなら Anki 系の専用アプリが向く
- 1000 語の一括登録は UI にはない。scripts/migrateFromVer1.ts と同じ要領の
  投入スクリプトを別途書く。itemNo は文字列主キーなので、単語は `w0001` の
  ような別プレフィックスにすると部品の番号空間 (採番・QR シール) を汚さない
