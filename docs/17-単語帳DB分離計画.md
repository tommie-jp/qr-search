# 単語帳の DB 分離 (2 インスタンス構成) の計画 (Ver2.x)

単語帳 ([15-単語帳向け改善計画.md](./15-単語帳向け改善計画.md)) を
qr-search の中で運用すると、1000 語のノートが部品の検索・タグ補完・
「すべて」ブラウズに混ざり続ける。これを **DB ごと分けて**解決する。

方針: アプリに空間の概念は足さない。**同じ Docker イメージを単語帳用に
もう 1 台起動し、同じ Postgres インスタンスの中の別データベースを指す**。
単語帳アプリを「独立させる。ただし 1 から作らない — 実体は qr-search
そのもの」という位置づけ。

## 1. 案の比較

| 案 | 番号の隔離 | 追加コード | 判断 |
| --- | --- | --- | --- |
| A. 接頭辞 (`w0001`) | 採番は無事 | ほぼゼロ | 不採用 |
| B. アプリ内に space 列 + 切替 | 完全 | 最大 | 不採用 |
| C. 同一イメージ 2 台 + 別 DB | 完全 | ほぼゼロ | **採用** |

- **A**: `nextItemNo()` は `itemNoNum` が数値の行しか見ないので、`w0001` は
  部品の採番を壊さない (確認済み)。だが全文検索・タグ補完・一覧に単語が
  混ざり続け、NOT 検索で毎回 `!#英単語` を打つのは対症療法
- **B**: `itemNo` が主キーで URL (`/item/4951`) にも QR シールにも刻まれて
  いる以上、複合キー化と URL 設計が全域に波及する。cookie で空間を切り替える
  変形は**同じ URL が違うノートを指す**ことになり、「URL が正」の原則
  (docs/11) を壊すので論外
- **C**: 番号空間・タグ・検索が DB ごと完全に分離する。単語側は素直に
  1〜1000 で採番でき、接頭辞も要らない。docs/15 の 3 機能
  (NOT 検索・前後ナビ・タグトグル) は**両インスタンスが同時に恩恵を受ける**

### B の再検討: app_id 列 + URL プレフィックス (2026-07-17)

B の真面目な変形として「`app_id` 列で部品と単語帳を分ける」案を検討した。
cookie ではなく URL に空間を刻む (`/w/item/4951`) 形なら「URL が正」は守れる。
それでも不採用。影響範囲を実測した:

- items を引くクエリ **14 箇所** (items.ts に集中) すべてに
  `WHERE app_id = ?` を通す。1 箇所漏れると単語が部品検索に混ざるが、
  漏れはエラーにならず**気づけない** (静かな正しさバグの類)
- `/item/` 系 URL を組む箇所 **14 箇所** + ルーティング再設計。
  QR シールの `/item/4951` は部品を指し続ける必要があり、
  部品 = 無印・単語帳 = `/w/` の非対称プレフィックスになる
- 主キーを `(app_id, itemNo)` の複合へ変更 → prisma migrate が走り、
  **PGroonga 索引 DROP の罠 (docs/12 §2) を踏む** (C はスキーマ変更ゼロ)
- Image テーブルは itemNo への FK を持たない (UUID 名で独立) ので無傷

C に勝る点は本物 — デプロイ・バックアップ・migrate が 1 系統、
アプリ内切替、3 つ目以降の空間が行の値で増える、Ver3 の user_id と同じ
機械の予行演習になる。だが「おまけ程度でよい」の要件に対し、
隔離という**結果は C と同じ**で、値段だけがアプリ史上最大の横断リファクタ。

**乗り換える合図** (§5 と同じ): 空間が 3 つ以上・アプリ内で行き来したい・
Ver3 着手。そのときは app_id と user_id を別々に 2 回作らず、
「所有スコープ」として 1 回で設計する。

## 2. 構成

Postgres は 1 コンテナのまま、その中にデータベースを 2 つ持つ
(インスタンス = クラスタは複数データベースを持てるのが Postgres の標準設計)。
増えるのは軽い app コンテナ 1 つだけ:

```yaml
  # 単語帳。部品 (app) と同じイメージ・別データベース
  app-words:
    image: qr-search-app:latest
    restart: unless-stopped
    environment:
      DATABASE_URL: postgresql://qr:${POSTGRES_PASSWORD}@db:5432/qr_words
    ports:
      - "127.0.0.1:${WORDS_PORT:-3001}:3000"
    depends_on:
      db:
        condition: service_healthy
```

- **DB 作成 (1 回だけ)**: docker の entrypoint が初期化時に作るのは
  `POSTGRES_DB=qr` だけなので、2 つ目は手で作る:

  ```bash
  docker compose exec db psql -U qr -d qr -c 'CREATE DATABASE qr_words OWNER qr'
  ```

- **マイグレーション**: migration に `CREATE EXTENSION IF NOT EXISTS pgroonga`
  が入っている (確認済み: add_item_tags / add_item_props) ので、
  `qr_words` に `prisma migrate deploy` を当てるだけで PGroonga も有効になる。
  拡張の**バイナリ**はイメージ共通・**有効化**はデータベース単位、が要点
- **doStart.sh / doDeploy.sh**: `DATABASE_URL` を差し替えて
  `migrate deploy` を DB の数だけ回すループにする (小改修)
- **入口**: サブドメインで分ける (例: words.qr.tommie.jp → :3001)。
  パス分け (`/words/…`) は Next.js の basePath が絡むので不採用
- **見分け**: `SITE_NAME` を環境変数で上書き可能にし、ヘッダと `<title>` で
  「QR search」「単語帳」を出し分ける (数行)。開いているタブを取り違えて
  部品 DB に単語を登録する事故を防ぐ。これが唯一のコード変更

## 3. Ver3 マルチユーザとの関係 — この分離は布石ではない

「単語帳で DB を分けたなら、マルチユーザも 1 ユーザ 1 DB か？」— **しない**。
1000 ユーザに 1000 DB は非現実的で、Ver3 は**単一 DB + `user_id` 列**
(行レベルの所有) にする。理由:

- **マイグレーションが N 倍**: スキーマ変更のたび 1000 DB に migrate を回す。
  1 個でも失敗すると「ユーザによってスキーマが違う」状態になり、
  アプリは全バージョンのスキーマを相手にすることになる
- **接続が N 倍**: Prisma のコネクションプールはデータベース単位。
  1000 DB = 1000 プール分の接続をたった 1 プロセスで抱えることになる
- **PGroonga 索引が N 倍**: 索引もデータベースごとなので、
  メモリ・ディスクのオーバーヘッドが件数ではなく**ユーザ数**に比例する
- **横断機能が作れない**: Ver2.x の TODO「公開/非公開 (ログインなしでも
  一部見える)」は他ユーザのノートを 1 クエリで引く機能で、
  DB を跨いだ SELECT はできない
- **作成・削除・バックアップが運用作業になる**: ユーザ登録のたびに
  `CREATE DATABASE` + migrate、退会のたびに DROP。pg_dump も 1000 本

DB 分割 (今回の C) が向くのは「**固定・少数・完全隔離**」の空間だけ。
単語帳はまさにそれ (2 個・増えない・混ざってほしくない)。
ユーザは「動的・多数・一部は見せ合う」ので正反対 —
単一 DB で全クエリに `WHERE user_id = ?` を通す
(締めを固くするなら Postgres の Row Level Security を重ねる) のが定石。

Ver3 に進むとき、この単語帳インスタンスはそのまま個人用として残してよい
(マルチユーザ版に単語を持ち込むなら、その時点で user_id 付きで
インポートすればよい)。

### 単語帳もマルチユーザにするなら (2026-07-17 追記)

分かれ目は機能ではなく**時期**。鍵は、Ver3 の user_id 改修が
app_id 案 (§1 の B') の値段を丸ごと含んでいること — ユーザごとに
1〜N で採番するなら URL はユーザの名前空間 (`/u/…/item/4951`) になり、
主キーは複合になり、全クエリが所有でスコープされる。
**同じ 3 点**を app_id も要求していた。したがって:

- **マルチユーザが近い・本気** → C を飛ばし、`(user_id, app_id, itemNo)`
  の所有スコープを 1 回で設計する (C を挟むと、組んで・溜めて・
  統合して・畳む、の回り道)
- **Ver3 が遠い・未確定** → C で始める。捨て代がほぼゼロ
  (compose 1 サービス + CREATE DATABASE 1 回 + SITE_NAME 数行) で、
  統合時は単語 1000 行を app_id 付きでインポートするスクリプト 1 本
  (migrateFromVer1 と同じ要領)。番号はスコープ後なら 1〜1000 のまま衝突しない

**C → 統合は安い一方通行、app_id の先行は高い賭け**という非対称が
あるので、Ver3 の着手が具体化するまでは C。具体的なマルチユーザ需要
(例: 単語帳を家族と共有) が見えた時点で、所有スコープ設計へ直行する。

## 4. 運用

- **バックアップ**: pg_dump の対象が 1 DB 増える (qr と qr_words)
- **リソース**: Next.js 1 プロセス分 (100〜200MB 程度) 増える。DB は共有
- **単語の一括投入**: docs/15 §5 のスクリプト案のまま。番号は素の 1〜1000
  でよく、`w0001` 接頭辞は不要になった
- **migrate の罠**: 新しい migration を当てるときは、**両方の DB** で
  PGroonga 索引の生存を `\di` で確認する (docs/12 §2 の罠は DB ごとに踏み得る)

見送り:

- **アプリ内の DB 切替 UI**: 同じ URL が違うノートを指すことになり
  「URL が正」に反する (§1 の B の変形)。ブックマーク 2 つで足りる
- **横断検索**: 部品と単語を同時に探したいことがない (分けたい、が動機)
- **3 つ目以降の空間**: 欲しくなったら compose に 1 サービス足すだけだが、
  数が増えて管理が苦になったらそれは B (space 列) を再検討する合図
