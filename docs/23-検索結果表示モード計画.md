# 検索結果の表示モード (Ver2.x)

検索結果の見せ方を「小 (compact)」「大 (card)」の 2 つから選べるようにする。
小は今までどおりの 2 行の一覧、大はサムネと本文プレビューを持つカード。

## 1. なぜ 3 つではなく 2 つか

当初案は「小 (1 カラム) / 中 (2 カラム) / 大 (1 カラム)」の 3 択だった。
これは採らなかった。

- **中と大は表示内容が同じ**である。どちらも「タイトル / ハッシュ / 本文 3 行 +
  5 行分のサムネ」で、違いはカラム数だけだった。ユーザに選ばせているのは
  「情報の密度」ではなく「カラム数」ということになる。
- **カラム数は画面が決めるべきもの**である。スマホ幅 (375px) で 2 カラムにすると
  1 カードが約 180px になり、タイトルもハッシュも本文 3 行も読めない。「中」を
  選べるようにすると、選んだ人の画面が壊れる。

そこで、ユーザが選ぶのは密度 (小 / 大) だけにし、カラム数は CSS の `auto-fill`
に任せた。

```css
grid-template-columns: repeat(auto-fill, minmax(20rem, 1fr));
```

これでスマホは 1 カラム (旧「大」)、タブレット・PC は 2 カラム以上 (旧「中」) が
自動で出る。3 択でやりたかったことは、選択肢を 1 つ減らして達成できている。

下限を 20rem (320px) にしているのは、これより狭いカードでは本文 3 行が読めず、
密度を上げた意味が無くなるため。

## 2. サムネイル (images.thumb)

**この機能の一番の技術課題はレイアウトではなく画像配信である。**

`images.data` はアップロードされた原寸のバイト列で、`/api/images/<name>` は
それをそのまま配っていた。実データで測ると 1 枚 2〜3MB のスマホ写真が並んでいる
(バックフィル時のログ: 10 枚で 12.9MB)。検索結果 1 ページは 20 件なので、
カード表示は数十 MB の読み込みになり一覧として実用にならない。**CSS で小さく
見せてもバイト数は減らない。**

採った方式は「**保存時にサムネを作って `images.thumb` に持つ**」。

- 画像は UUID 名で内容が変わらない (`route.ts` が `immutable` で配っている) ので、
  保存時に 1 度作れば作り直す理由がない
- リクエスト時に縮小してキャッシュを持つ方式より単純で、
  [20-画像GC計画.md](./20-画像GC計画.md) とも干渉しない (行が消えればサムネも消える)
- 既存分は [scripts/backfillThumbs.ts](../scripts/backfillThumbs.ts) の 1 回実行で埋まる

実測: 2854KB → 11KB、12.9MB → 140KB (約 1/90)。

生成は [src/lib/thumbnail.ts](../src/lib/thumbnail.ts)。長辺 320px の webp
(カード枠 96px の 3 倍 DPR まで足りる)。`.rotate()` で EXIF の向きを画素に焼く —
スマホ写真は横倒しで保存され向きが EXIF にしかないため、これを忘れると一覧だけ
倒れて出る。

生成の入口を `saveImage()` の 1 か所にしているのは、画像を作る経路がそこしか
ないため ([20-画像GC計画.md](./20-画像GC計画.md) §1)。

### 2.1 next/image は使えない

画像 API はログイン必須で、Next の image optimizer がサーバ側から取りに行くとき
Cookie / Authorization が付かず 401 になる。素の `<img>` に `loading="lazy"` と
`width`/`height` を付けて使う。縮小は保存時に済んでいるので optimizer は要らない。

### 2.1.1 sharp を実行時依存にした

sharp はこれまで devDependency で、アイコン生成
([scripts/genIcons.mjs](../scripts/genIcons.mjs)) にしか使っていなかった。今回
アプリの実行時に使うので `dependencies` へ移した。あわせて package-lock.json も
同期してある — 移動しただけでは lock は sharp を dev のまま記録し続け、
`npm ci --omit=dev` を使った瞬間に本番から消える。

**ユーザが投げたバイト列を sharp に渡すのは今回が初めて**である。そのため
`limitInputPixels` を明示した (50MP)。アップロードの 10MB 制限
(`MAX_IMAGE_BYTES`) は**展開後の大きさを縛らない** — マジックバイト検査
([uploads.ts](../src/lib/uploads.ts)) が見るのは先頭数バイトだけなので、
単色なら 12000x12000 (144MP) が数十 KB に収まり、10MB 制限をすり抜けて
libvips に GB 単位のメモリを掴ませられる。sharp の既定 (約 268MP) は事故よけの
値で、悪意ある入力には緩い。

### 2.2 thumb が null のとき

`thumb` は NULL 可である。バックフィル前の行と、壊れた画像で生成に失敗した行が
ありうる。このとき `?thumb=1` は**原寸で代替する** — 一覧は重くなるが絵は出る。
サムネが作れないことでユーザの画像アップロードを失敗させる理由はない。

**代替で返すときのキャッシュを `immutable` にしてはいけない。** 1 年 immutable で
焼くと、後からバックフィルしても閲覧者のブラウザは数 MB の原寸を掴んだままになる。
短い `max-age=60` にして、次の訪問でサムネへ入れ替わるようにしている。

この分岐 (認可・代替・キャッシュの出し分け) はこの機能で一番間違えやすい所なので、
実 DB を使ったルートのテストで固定してある
([images.test.ts](../src/app/api/images/images.test.ts) の `RUN_DB_TESTS=1` の節)。

### 2.2.1 承知の弱点: 全部失敗しても静かに動き続ける

この代替には**代償**がある。sharp が丸ごと壊れた場合 (本番は alpine/musl なので
ネイティブの入れ違いは起こりうる)、**全部の画像が黙って原寸で配られ続ける**。
一覧が重いだけで絵は出るので、壊れて見えない。「1 枚が壊れている」ときと
区別が付かないのが厄介な点である。

いまの備えは 2 つ。どちらも能動的な監視ではないので、弱点として書いておく。

- 失敗ログに画像名とバイト数を添えてある。ログはヘッダの「ログ」から読める
  ([21-ログ表示計画.md](./21-ログ表示計画.md))。全アップロードで出ていれば
  sharp 側、特定の 1 枚だけならその画像の問題、と切り分けられる
- バックフィルは失敗が 1 件でもあれば終了コードを 1 にする。cron や deploy から
  呼んだとき「全部失敗したのに exit 0」で素通りさせない

数を見るなら以下で数えられる。

```bash
docker compose exec db psql -U qr -d qr -c \
  "SELECT count(*) FROM images WHERE thumb IS NULL;"
```

### 2.3 どの画像を出すか

参照は本文中の `![](/api/images/<UUID>.<ext>)` という文字列だけで、
「このノートの画像」という関係は DB にない ([20-画像GC計画.md](./20-画像GC計画.md) §1)。
そのため本文から抽出する ([src/lib/memoImages.ts](../src/lib/memoImages.ts))。
検索は `memo` を丸ごと引いているので追加の問い合わせは要らない。

遅くなったら `tags` / `props` と同じく「保存時に計算する派生キャッシュ列」
(`items.firstImageName`) へ移せる。そのときもこの関数が抽出規則の正本として残る。
今は YAGNI で持たない。

## 3. 本文プレビュー (memoPreview)

カードの 3 行に「もう他の場所で見えているもの」を流すと枠が無駄になるので、
次を落として本文だけを残す ([src/lib/memoPreview.ts](../src/lib/memoPreview.ts))。

| 落とすもの | 既にどこに出ているか |
| ------------ | ---------------------- |
| 1 行目 | タイトル (memoSummary と同じ行) |
| タグだけの行 | 2 行目のタグ |
| プロパティ行 (key=value) | タグ検索の特性表 (PropsTable) |
| 画像 | カード右端のサムネ |

散文に混じったタグ (`これは #npn の…`) や key=value (`実測では hFE=195 だった`) は
本文なので残す。「行全体が key=value か」の線引きは
[props.ts](../src/lib/props.ts) の `isPropLine()` に寄せ、2 か所に書き分けない。

**行数は抽出側で数えない。** Markdown 上の 1 行は画面では折り返して 2 行にも 3 行にも
なるので、抽出側で 3 行に切っても画面の行数とは一致しない。残った行を空白で連結して
返し、3 行に収めるのは CSS (`line-clamp-3`) の仕事にしている。これなら長い行でも
短い行でも枠がきれいに埋まる。

200 字で打ち切るのは、`line-clamp` が隠すのは見た目だけで、本文が 50KB あれば
その 50KB が DOM とサーバ→クライアントのペイロードに乗るため。1 ページ 20 件ぶん
積まれるので効く。

## 4. モードの持ち回し: cookie

**URL (searchParams) ではなく cookie に持つ。**

`sort` やページ番号は「何を見ているか」なので URL が正
([11-アプリ的UIUX計画.md](./11-アプリ的UIUX計画.md) §3) だが、表示モードは
「どう見たいか」という端末ごとの好みで、検索状態ではない。URL に混ぜると:

- 共有したリンクに自分の好みが漏れ、相手の好みを上書きする
- スキャンやタグリンクで入るたびに既定へ戻る
- `buildSearchUrl` の呼び出し全部に引数が増える

cookie ならサーバコンポーネントが描画前に読めるので、**初回描画から正しい
レイアウトで出る**。localStorage は SSR で読めず、一度描いてから跳ねるので採らない。

既定は `compact` (今までの見た目)。この機能が入っても、何もしていない人の画面は
変わらない。cookie は利用者が書き換えられる外部入力なので `parseViewMode()` で
畳む (`parseSort` と同じ流儀)。

これはこのアプリで最初の cookie である (ログインは Basic 認証でヘッダー方式)。

## 5. 切替 UI

「選択」の反対側に置いたセグメント切替 (`小 | 大`)。ドロップダウンにしないのは、
選択肢が 2 つしかないので開いて選ぶ 2 手より 1 タップのほうが早く、いま何が
選ばれているかも常に見えているため (並び替えの「番号順 / 更新順」と同じ佇まい)。

cookie を書くのでサーバアクション。Server Component の描画中は `Set-Cookie` を
出せないため。フォームの `action` に置くことで**クライアント JS は要らない**。

`setViewModeAction` は他のアクションと違い `requireUser()` を呼ばない。書き換わる
のは呼び手自身のブラウザに載る見た目の好みだけで、DB にも他人にも触れないため。

### 5.1 踏んだ罠: revalidatePath('/', 'layout') が manifest を巻き込む

最初このアクションの末尾に `revalidatePath('/', 'layout')` を置いていた。これは
**`/` 配下の全ルートを無効にする**ため、静的な `/manifest.webmanifest`
([manifest.ts](../src/app/manifest.ts)) まで再生成の対象になり、Next がその
prerender キャッシュを書き直そうとする。ところがコンテナは `node` ユーザーで動く
のに `.next` は root 所有 (Dockerfile の `COPY --from=builder` は root:root で
入り、その後 `USER node` する) なので書けない。結果、**表示モードを切り替える
たびにサーバログへ警告が出た**。

```text
Failed to update prerender cache for /manifest.webmanifest
EACCES: permission denied, open '/app/.next/server/app/manifest.webmanifest.body'
```

見た目は正常に切り替わり、警告以外に症状が無いので気づきにくい。ログを見て
初めて判った ([21-ログ表示計画.md](./21-ログ表示計画.md) が効いた例)。

**そもそも不要だった。** 一覧は `force-dynamic` でサーバ側にキャッシュが無く、
フォームの `action` から呼ばれたサーバアクションは、その場のページを Next が
描き直す (Router Cache も一緒に更新される)。外しても切り替えは効く。

再現・確認の手順 (ホストでは root 所有ではないので書き込みは成功する。
警告は出ないが、**書き直された事実は mtime に出る**):

```bash
npm run build && npx next start -p 3003
stat -c '%y' .next/server/app/manifest.webmanifest.body   # 切り替え前
# 画面で 小/大 を切り替える
stat -c '%y' .next/server/app/manifest.webmanifest.body   # 変わっていたら巻き込んでいる
```

なお `.next` が root 所有で書けないこと自体は Dockerfile の元からの作りで、
この機能とは独立している。ここでは「静的ルートを再生成させない」側で直した。

## 5.2 枠内どこでも押せるようにする

当たり判定はタイトルの文字の上だけでは狭い。指で狙うには枠全体が押せるほうがよい
(min-h-11 と同じ考え方。[11-アプリ的UIUX計画.md](./11-アプリ的UIUX計画.md) §1-4)。

**行全体を `<a>` で包むことはできない。** タグは別の行き先 (タグ検索) を持つので
リンクの入れ子になり、HTML として不正なうえ挙動も壊れる。そこで **stretched link**
にした — リンクはタイトルの 1 つに保ったまま、その `::after` を
`absolute inset-0` で枠いっぱいに広げ、当たり判定だけを大きくする。器の側に
`relative` を置いて基準にする。

この形なら `href` を持った本物のリンクが 1 つ残るので、中クリックで新しいタブ、
右クリックで URL コピーも今までどおり効く (`onClick` で飛ばす実装だと壊れる)。

上に出したい物 (タグ) は `relative z-10` で膜より前に出す。これが無いとタグを
押してもノートが開いてしまう。

**選択モードでは敷かない。** 膜がチェックボックスを覆って押せなくなるうえ、
選んでいる最中に枠へ触れるたびノートへ飛んでしまう。選択モードの当たり判定は
今までどおり (チェックボックスとタイトル)。

承知の代償: 膜が覆う部分の文字はドラッグで選択できない。一覧は「開く」ための
画面で、本文を読む・拾うのはノート側の仕事なので、当たり判定の広さを優先した。

## 6. やらなかったこと

- **中 (2 カラム固定)**: §1 のとおり。auto-fill が同じことを画面幅に応じて行う。
- **items.firstImageName の派生キャッシュ列**: §2.3。今は本文から都度抽出で足りる。
- **リクエスト時リサイズ + キャッシュ**: 保存時生成のほうが単純で、
  無効化を考えなくてよい (画像は immutable)。
- **複数サイズのサムネ**: 小 (40px) と大 (96px) は 320px の 1 枚でまかなえる。
  枠は CSS が決め、絵は縮む側に強い。
