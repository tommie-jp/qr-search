# 全ノートのエクスポート・インポート計画 (Ver2.x)

全ノートをクライアント (スマホ / PC) からダウンロードして手元に保全し、
同じファイルを戻せる (インポート) ようにする。ここでは**ファイル形式の
設計判断**を残す。実装はまだ (§6 のタスクは未着手)。

完全バックアップは既に pg_dump が担っている (画像を DB に置いた設計理由が
まさにそれ。schema.prisma の Image コメント参照)。この機能の目的は
バックアップではなく**ユーザー自身による持ち出し・可搬性**。人間がそのまま
読める形式であることを最優先にする。

## 1. 形式 — ZIP 1 ファイル (Markdown + frontmatter + images/)

```text
qr-search-export-2026-07-18.zip
├── notes/
│   ├── 1000.md        ← itemNo をファイル名に
│   ├── 1001.md
│   └── ...
└── images/
    └── <uuid>.jpg     ← Image.data をそのままファイル化
```

各 `.md` は YAML frontmatter にメタデータ、本文は memo をそのまま:

```markdown
---
itemNo: "1042"
mode: memo
url: ""
created: 2025-11-02T10:30:00+09:00
updated: 2026-07-01T08:12:00+09:00
public: false
---
hFE=208
2SC1815 のストック #トランジスタ
![](../images/abc-123.jpg)
```

選定理由:

- memo はもともと Markdown 系の記法 ([メモ記法.md](./メモ記法.md)) なので
  変換コストゼロ・ロスレス。circuitikz フェンスもソースのまま残る
- 人間がそのまま読める。テキストエディタや Obsidian でそのまま開ける
  (frontmatter + 相対パス画像は Obsidian 互換で、vault に置くだけで
  閲覧できる)
- frontmatter が構造化メタデータを持つので機械的な再インポートにも耐える
- 画像は base64 化せず生ファイルで同梱。サイズ膨張 (base64 は約 +33%) が
  なく、画像単体でも取り出せる。本文中の `/api/images/<name>` リンクは
  `../images/<name>` に書き換えるだけ
- ZIP はサーバでストリーミング生成して 1 リクエストで返せる。iOS の
  「ファイル」アプリも Android も標準で展開でき、スマホで「ファイルが
  数百個バラバラに降ってくる」事態を避けられる

エクスポートに**含めない**もの — memo から再生成できる派生キャッシュ:
`tags` / `props` / `Image.thumb` / `Image.embedding` / `CircuitSvg`。

## 2. 却下した形式

- **JSON / JSONL 単一ファイル** … 機械復元には最適だが、画像を base64 で
  抱えてサイズが膨れ、人間が読めない。「完全バックアップ」用途は pg_dump が
  担うので、ユーザー向けでこれを選ぶ動機は薄い。復元保証を強めたければ
  ZIP に `meta.json` (全ノートのメタデータ一覧) を 1 枚同梱する折衷で足りる
- **PDF / HTML** … 見た目は保てるが編集・再利用できず、エクスポートと
  いうより「印刷」。用途が違う
- **CSV** … 複数行 Markdown + 画像に根本的に不向き

## 3. インポート — 同じ ZIP をそのままアップロード

エクスポートと同一形式をラウンドトリップさせる。専用フォーマットを
別に作らない (KISS)。

- スマホからは `<input type="file" accept=".zip">` で「ファイル」アプリ /
  Files から選ぶだけ。iOS・Android とも ZIP のまま選択できるので、
  端末側で展開させる必要はない
- サーバ側で展開し、`notes/*.md` の frontmatter + 本文を upsert、
  `images/*` を `Image.name` (UUID.拡張子) で upsert。本文中の
  `../images/<name>` は `/api/images/<name>` に書き戻す
- 派生キャッシュ (tags / props / thumb / embedding / CircuitSvg) は
  既存の保存経路を通して再生成する。エクスポートに含めない (§1) のと
  対になる設計
- 手書きの Markdown (エクスポート由来でないファイル) も、frontmatter の
  必須項目さえ満たせば取り込める副産物がある

### 検証 (インポートは書き込み境界なので必須)

- frontmatter の必須項目・型 (itemNo の形式、mode の enum) を検証し、
  不正なファイルは**そのファイルだけスキップして結果レポートに載せる**
  (全体を止めない)
- ZIP 爆弾対策: 展開後サイズ・ファイル数・1 ファイルの上限を設ける
- 画像は mime と拡張子の整合を確認。パス名は `notes/` `images/` 直下のみ
  許可 (パストラバーサル対策)
- 要ログイン。公開側 (publicPaths) には一切出さない

## 4. ENEX (Evernote) インポート

Evernote からの乗り換え取り込みもやりたい。ENEX は XML で、
1 ファイルに複数ノートが入る:

- `<content>` … ENML (XHTML 系) の CDATA。Markdown ではない
- `<resource>` … 添付 (画像など) の base64 + mime。本文からは
  `<en-media hash="MD5">` で参照される
- `<title>` `<created>` `<updated>` `<tag>` … メタデータ

方針: **ENEX を §1 の内部形式に変換する前段コンバータ**として作り、
upsert・検証・派生キャッシュ再生成は §3 の経路をそのまま通す。
インポート経路を 2 本持たない。

変換の対応表:

| ENEX | 本アプリ |
| ------ | --------- |
| `<title>` | memo の 1 行目 (title 列は無い) |
| `<content>` (ENML) | Markdown へ変換 (turndown 系。ENML 固有タグは前処理で除去) |
| `<tag>` | memo の 2 ブロック目に `#タグ` 行 (tags は memo からの派生のため) |
| `<resource>` | base64 を decode して Image 行に。name は新規 UUID を発番 |
| `<en-media hash>` | resource の MD5 と突き合わせて `![](/api/images/<name>)` に置換 |
| `<created>` `<updated>` | createdAt / updatedAt に反映 |
| itemNo | 無いので `nextItemNo()` で採番 (衝突なし = §5 のポリシー対象外) |

- ENML → Markdown はロスが出る前提 (表・整形・チェックボックスなど)。
  変換できない断片は素の HTML のまま残すか落とすかを実装時に決める
- 既存 OSS (evernote2md など、ENEX → Markdown 変換器) の変換ルールを
  参考にする。丸ごと採用できれば前段はそれで済む可能性もある
- ENEX も ZIP と同じ `<input type="file">` から拡張子で振り分ける
  (`.zip` → §3、`.enex` → 本節)

### 実装結果 (Ver2.x)

ENEX の取り込みだけ先に実装した (エクスポートと ZIP インポートは未着手)。
入口は `/settings/import` の `<input type="file" accept=".enex">` で、
**変換はすべてサーバ側** (`POST /api/import`)。クライアントは端末の
ファイルを選んで送るだけで、認証・CSRF・大きさの作法は `/api/images` に揃えた。

構成 (純関数を分けてテストできるようにする):

| ファイル | 役割 |
| ---------- | ------ |
| `lib/enex/parseEnex.ts` | ENEX (XML) → ノート配列。fast-xml-parser (既存依存) |
| `lib/enex/enmlToMarkdown.ts` | ENML → Markdown。turndown + gfm プラグイン |
| `lib/enex/buildMemo.ts` | 題名・タグ行・本文を memo 1 枚に組む |
| `lib/enex/importEnex.ts` | 繋ぎ役。保存は既存の `upsertItem` を通す |
| `lib/attachmentStore.ts` | 添付の形式判定と保存。`/api/images` と共有 |

決めたこと:

- **変換ライブラリは turndown + turndown-plugin-gfm**。HTML → Markdown の
  実質の標準で、Evernote 移行ツール (yarle) も内部で使っている = ENEX が吐く
  実際の HTML に対する実績がある。yarle / evernote2md を丸ごと使う案は
  不採用 — どちらも「ファイルシステムに .md を書き出す CLI」設計で、
  添付を DB に置く本アプリには嵌まらない
- **フォント・文字サイズは捨てる**。memo は Markdown なので持てない。
  turndown が span/font の殻を落とす既定の挙動がそのまま要件に一致する
- **リンクは残す**。`evernote:///` の内部リンクはそのまま (解決には
  ノート間の GUID 対応表が要る。複数ファイル対応時の課題)
- **`<div>` は段落ではなく改行**にする。Evernote は 1 行を 1 つの div で
  包むため、turndown の既定 (未知のブロック要素 = 前後に空行) だと行間が
  倍に開く。このアプリの表示は remark-breaks なので改行 1 つで元の形になる
- **ENML の空要素は img / input へ寄せてから** turndown に渡す。ENML は
  XHTML で `<en-media/>` と自分で閉じるが、turndown の HTML パーサ (domino)
  は HTML の規則で読むため未知のタグが「開いたまま」になり、後続の本文を
  丸ごと飲み込む (実測: チェックボックス直後の文字が消えた)
- **表は 1 行目を `<th>` に仕立ててから**変換する。turndown-plugin-gfm は
  見出し行の無い表を `keep` して**生 HTML のまま**残すが、このアプリの表示は
  生 HTML 無効なので**表がまるごと消える**。しかも Evernote の表は `<td>` しか
  使わないため、実物の ENEX はまるごとこの穴に落ちる。表の中の `<en-media>` も
  変換されず、保存した添付が参照されないまま残る二次被害もあった。
  自前で表を組むより入力を寄せて既存の実装に乗せるほうが安全
- **表のセルの中では div を改行にしない**。Evernote はセルの中身も div で
  包むが、GFM の表は 1 行 1 レコードなので改行を入れると崩れる
  (実測: `| 型番 |` が 3 行に割れた)
- **変換にかける前に大きさと入れ子の深さで断る**。turndown も domino も木を
  再帰で歩くため、`<div>` を延々と入れ子にした ENML は失敗するまでに数十秒の
  同期処理を回す (実測: 660KB / 6 万段で 23.5 秒)。Node は 1 本の
  イベントループなのでその間アプリ全体が止まる。上限は 512KB / 200 段で、
  門番を通せば同じ入力が 1ms で片付く
- **添付のファイル名は Markdown の記号を殺してからリンクの表示文字にする**。
  ファイル名は ENEX の書き手が自由に書けるので、`資料.pdf](https://evil…)[`
  のような名前で意図したリンクを閉じて偽装リンクを差し込める
  (rehype-sanitize は `javascript:` は消すが http(s) の偽装先は止めない)
- **タグ行は本文より前**に置く。後ろに置くと、コードフェンスで終わる本文の
  ときにフェンスの内側へ落ちて `extractTags` に拾われない
- Evernote のタグは空白・記号を含められるので `-` に寄せる
  (「電子 工作」→ `#電子-工作`)。潰して「電子工作」にすると別タグと衝突する
- **申告 MIME は信用しない**。`<resource><mime>` は書き出し元の申告でしかなく、
  形式は既存の先頭バイト判定 (`uploads.ts`) で決め直す
- 添付は既存の保存経路を通すので、**サムネイル・画像検索の埋め込みも自動で付く**
- 本文が `MAX_TEXT_LENGTH` (10000 字) を超えるノートは切り詰めずに飛ばし、
  **保存済みの添付も消す** (本文が入らない以上どこからも参照されないため)
- 1 回の取り込みは 500 ノートまで (採番を延々と回されないための安全弁)

#### 取り込み口は 2 つある (Web と CLI)

実データの書き出しが **40.2MB** あり、これを Web の口へ投げると本番 VPS
(RAM 2GB / swap 常用。docs/09) には重い。そこで**大きいものはローカル (WSL) で
変換する**。イメージをローカルでビルドして送る `doDeploy.sh` と同じ考え方で、
「重い処理は手元、成果物だけ本番へ」に揃える。

| 口 | 使いどころ | ファイル上限 | 添付 1 件の上限 |
| ---- | ------------ | ------ | ------ |
| `/settings/import` (Web) | スマホから小さい .enex を入れる | 10MB | 10MB |
| `./doImportEnex.sh` (CLI) | 本命。40MB 級をローカルで変換して本番 DB へ | 実質なし | 50MB |

**添付 1 件の上限を CLI で分けている**のが要点。10MB (`MAX_IMAGE_BYTES`) は
エッジのボディ上限 12MB に収まる大きさとして決めた **HTTP アップロードの
都合**であって、DB に置ける大きさの上限ではない。iPhone の写真は普通に超える
(手元の書き出しでは添付 10 枚のうち 3 枚が 11〜12MB) ので、そのまま CLI に
持ち込むと移行のたびに写真が虫食いになる。実際 1 枚目の取り込みで
12.1MB の写真が落ちた。

縮小はしない。移行は一方通行なので、元の解像度を黙って落とすより、そのまま
入れて必要になってから間引くほうが取り返しがつく。

判定は**変換前のバイト列**に対して行う点に注意 (HEIC は保存時に WebP へ縮むが、
その前に大きさで弾かれる)。

**中身は同じ `importEnex()` を呼ぶ**。変換規則・レポート・巻き戻しを 2 か所に
持たない (§4 の「インポート経路を 2 本持たない」はここにも効く)。

CLI は `doDeploy.sh` の「6/8 マイグレーション」と同じく SSH トンネル
(`localhost:15433` → vps2 の `127.0.0.1:5432`) 越しに書く。vps2 の Postgres を
外へ公開する必要はない。**採番が本番に対して行われる**のがこの形の勘所で、
ローカル DB に取り込んでから行を移す案だと itemNo が本番と衝突する。

ローカルは潤沢なので CLI は既定で**画像検索の索引まで作る** (`embedImages`)。
Web の口は逆に作らない (下の表のとおりモデル読み込みだけで +475MB)。

Web の上限を 10MB に抑えているのは、エッジ (Caddyfile / deploy/nginx) の
ボディ上限 12MB をそのまま使うため。**片方だけ上げると事故る** — エッジが
低いとアプリに届く前に 413 でボディを読み捨てられ、ブラウザには
「Load failed」としか出ない (サーバの JSON エラーは届かない)。
上限超過はクライアント側でも送信前に検査し、CLI へ誘導する。

手順の推奨:

1. `./doCopyDBfromVPS2.sh` — 本番のコピーをローカルに作る (採番も本番と同じ)
2. `./doImportEnex.sh f.enex --local` — 下見。レポートと画面を確認する
3. `./doImportEnex.sh f.enex` — 本番へ。**実行中はアプリを触らない**
   (`nextItemNo()` は空き番号の最小値を返すだけで予約しない)

本番へ書く前に DB のバックアップを取る (`doImportEnex.sh` が自動で取る)。
数百行を本番へ書く操作で、取り消す手段はそのダンプしかない。

#### 複数ノート・複数ファイル・重複 (Evernote の実運用)

Evernote で複数ノートを選択して書き出すと、**1 ファイルに複数ノート**が入る。
パーサもインポータも最初からノート配列で扱うので、これはそのまま取り込める
(そこは変更不要)。実運用で足したのは次の 4 点:

- **重複はスキップ (既定)**。手で選択してバッチを作ると、選択の重なりや
  同じファイルの再実行で二重取り込みが現実に起きる。「同じ created_at
  (ENEX 由来・秒精度) + 同じ題名 (memo 1 行目)」で既存を照合し、あれば
  入れない。この組はほぼ一意で、同時刻・同題名の別内容ノートは実用上ない。
  **日時の無いノートは判定対象外** (題名だけでは同名の別ノートを取り違える)。
  `--force` で入れ直せる。判定は添付を保存する前に撃つのでゴミが残らない。
  これにより「途中で失敗 → 同じファイルを流し直す」が安全になった
  (再実行しても増えない)
- **500 ノート上限は CLI では撤廃**。`MAX_NOTES_PER_IMPORT` は Web の口の
  安全弁 (細工ファイルで採番を回されない) で、自分のファイルを渡す CLI には
  不要。撤廃しないと 500 超のバッチで再実行時に先頭 500 件が二重になる
- **`#evernote` を全ノートに必ず付ける**。由来の印で、移行に不満が出たとき
  「#evernote で全選択 → ゴミ箱」でやり直せる。要らなくなれば一括タグ削除で
  外せる (付けて後悔は消せるが、付けずに後悔は手遅れ、の非対称性)。
  `--tag レシピ` でノートブック名の代わりの分類タグを足せる (役割が違う:
  #evernote は由来、--tag は分類)
- **複数ファイルを一度に渡せる**。`./doImportEnex.sh a.enex b.enex`。
  本番へのバックアップは最初に 1 回、SSH トンネルも張りっぱなしで使い回す
  (ファイルごとに 4GB 級のダンプを繰り返さない)

大きすぎるファイルは読む前に断る。Node は 1 つの文字列を 512MB
(`buffer.constants.MAX_STRING_LENGTH`) までしか持てず、`readFileSync` は
ファイル全体を 1 文字列にするため、超えると意味不明な `ERR_STRING_TOO_LONG`
で落ちる。余白を取った 400MB (`MAX_CLI_ENEX_BYTES`) で断り、
「選択を分けて書き出し直す」よう案内する (docs/13-EVERNOTE全ノート移行メモ.md)。

本番 VPS は RAM 2GB で swap を常用している (docs/09)。50MB の ENEX
(4000x3000 の写真 4 枚) を素直に実装したときの実測は **RSS +2610MB** で、
そのままでは確実に落ちる。内訳を測って 3 つ直し、**+288MB** まで下げた:

| 原因 | 実測 |
| ------ | ------ |
| 巨大な base64 を XML パーサへ渡していた | 48MB で **+1219MB** / 2.2 秒 |
| 復号したバイト列を全件ぶん抱えていた | 30MB 前後が生き続ける |
| 画像検索の埋め込みを画像の数だけ撃っていた | モデル読み込みで **+475MB** |

対処:

- 解析の前に `<data>` の中身を抜いて目印に差し替える
  (`liftDataPayloads`)。抜いた中身は V8 の部分文字列なので複製は起きない
- `base64` のまま持ち、保存の直前に 1 件ずつ復号する
  (`decodeResourceData`)。md5 のために 2 回復号することになるが、
  CPU を少し払ってピークを下げる取引
- 一括取り込みでは埋め込みを作らない (`deferEmbedding`)。派生キャッシュなので
  `npm run backfill:embeddings` が後から埋める。後回しにしたことは画面で知らせる

参考: サムネイル生成 (sharp) は +2MB で無視してよい。`XMLValidator.validate`
も 86ms / +7MB と安い (壊れたファイルを弾く価値のほうが大きい)。

測り直すときは、検証スクリプト自身の画像生成分を基準から外すこと
(`--expose-gc` で GC してから基準 RSS を取る)。取らないと 2 倍近く多く見える。

ロスが出るもの (レポートに載せる): 対応外形式の添付、base64 として読めない
添付、参照先の見つからない `<en-media>`、タグに書けない文字だけのタグ、
長すぎる本文、大きすぎる/深すぎる本文、`<en-crypt>` (端末の鍵でしか開けない)。

**「入らなかったもの」は必ずレポートに出す**のがこの実装の背骨。本文に跡を
残すだけだと、利用者は全ノートを目視するまで欠落に気づけない。逆に、行が
できた後で転んだノートは「取り込み済み」として報告する — 入っているのに
「入らなかった」と伝えると、取り込み直して二重に作ってしまう。

## 5. 衝突ポリシー (未決)

itemNo が既存ノートと衝突したときの扱い。案:

1. **上書き (upsert)** … 「戻す」用途には自然。ただし手元の編集を
   黙って潰すリスク
2. **スキップ + レポート** … 安全側。デフォルト候補
3. インポート時にどちらかを選ばせる (ラジオボタン)

updated の新旧比較で自動判定する案もあるが、時計依存の複雑さが増すので
Ver2.x では見送り。実装時に 2 をデフォルト + 1 をオプションで選べる形を
第一候補として検討する。

## 6. 見送り (YAGNI)

- **選択エクスポート** (タグ・検索結果単位) … まず全件のみ。欲しく
  なったら検索条件を流用して足す
- **ゴミ箱 (`deletedAt` 非 null) の同梱** … デフォルト除外。含める
  オプションは需要が出てから
- **差分インポート・マージ UI** … §5 のとおり単純ポリシーで始める
