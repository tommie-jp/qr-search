# 47 引っ張って更新 (pull-to-refresh) 計画

一覧の先頭で下へ引くと再読み込みする、iPhone アプリでおなじみの操作を足す。

## 1. なぜ自前実装か

`globals.css` の `html { overscroll-behavior: none }` で、iOS のラバーバンドと
Android のネイティブな引っ張り更新をわざと止めている (docs/11)。よって
ブラウザ任せの更新は出ない。ジェスチャは自分で判定し、`router.refresh()` を
呼ぶ。スクロールは window (内側スクローラなし) なので、リスナーも window に張る。

## 2. 何を「更新」とするか

- Server Component 前提の `force-dynamic` ページなので、`router.refresh()` で
  Server Component を再取得すれば最新の一覧になる。
- 検索状態はすべて URL (`q` / `page` / `sort`) と cookie (`view` / `sort`) に
  あるので、`refresh()` はスクロール位置もクライアント state も保ったまま
  再取得する。累積 `page` も維持されるため、読み込み済みの件数はそのまま。

## 3. 判定 (純関数 `lib/pullToRefresh.ts`)

DOM も React も触らない状態遷移に切り出し、テストをここに集中させる
(swipeRow.ts と同じ方針)。

- `idle` … 触れていない/縦上・横に譲った後。
- `tracking` … 先頭で触れたが、縦下向きかまだ決まっていない。
- `pulling` … 縦下向きと確定し、指に追従して引き量を伸ばす。

しきい値・スロップ・上限・抵抗:

| 定数 | 値 | 意味 |
| ---- | ---- | ---- |
| `PULL_THRESHOLD` | 64px | これを超えて離すと更新 |
| `PULL_SLOP` | 8px | 縦/横を決めるまでの遊び (swipeRow と揃える) |
| `PULL_MAX` | 96px | 引ける上限 |
| 抵抗 | 0.5 | 指の移動量に掛けて「重い」手応えを出す |

方向ロックで、横がスロップを超えたら横スワイプ (行削除, docs/43) に譲り、
上向きが勝てば上スクロールに譲る。互いに喧嘩しない。

## 4. 配線 (`components/PullToRefresh.tsx`)

- タッチ端末 (`(pointer: coarse)`) だけで有効化。
- `touchstart` … 更新中・複数指 (ピンチズーム)・スクロール可能な入れ子
  (モーダル本文) の中では掴まない。先頭 (`scrollY <= 0`) のときだけ tracking。
- `touchmove` … `passive:false` で登録し、引っ張り確定後だけ `preventDefault`。
- `touchend` … しきい値超えで `startTransition(() => router.refresh())`。
  完了 (`useTransition` の `isPending` 落ち) でインジケータを畳む。

## 5. インジケータ

- ヘッダー (sticky, z-20) より前 (z-30) に出し、引くと上から降りてくる。
- `pt-safe` で standalone のステータスバー下から出す。light/dark 両対応。
- `pointer-events-none` で一覧のタップを邪魔しない。
