# ISBN からの書影取得の設計 (Ver2.x)

ISBN をスキャンして新規登録すると、書名・著者・出版社は自動で入る
([13-書誌自動取得計画.md](./13-書誌自動取得計画.md))。ここに**書影**を足す。

13 の §3・§6 では「書影は入れない」としていた。外部の画像 URL を本文に
置くとリンク切れするし、このアプリの画像は `/api/images/<uuid>` (DB 保存)
が流儀なので配線が増える、というのが理由だった。**配線を足すほうを選んで
覆した**が、13 を書いた時点とは前提が変わっている (下の §1)。

## 1. 前提: 2023-2026 年に無料の書影ソースが枯れた

13 が「欲しくなってから」と書いた当時 (openBD + NDL サーチが健在) と違い、
**JPRO の規約改定を軸に無料の書影配信が次々止まった**。2026-07-17 の実測:

| ソース | 状態 | 書影の実測 |
| --- | --- | --- |
| openBD `summary.cover` | 生存。ただし**ホワイトリスト版元のみ** | **1/11** |
| NDL 書影 API | **2026-03-31 終了** | 実質 0 |
| 版元ドットコム (`img.hanmoto.com`) | 動くが公開 API ではない | 10/11 |
| 楽天ブックス書籍検索 API | 生存。要 applicationId | (第三者検証で 286/440) |
| Google Books | 匿名は 429 | 引けない |

書誌 (13) と違い、**書影は「新刊なら openBD」が成立しない**。上の 11 冊は
2022 年の本もマンガも新潮文庫も openBD の cover が空で、当たったのは
作品社の 1 冊だけだった。

### NDL 書影 API は終了している (使わない)

[書影APIのサービス終了について](https://ndlsearch.ndl.go.jp/news/20251217)
のとおり、**データ提供機関である JPRO の利用規約改定に伴い 2026-03-31 に
終了**した。以後は JPO に直接問い合わせる案内になっている。

終了後も口自体は生きていて、**`Referer: https://ndlsearch.ndl.go.jp/` を
付けたときだけ 200 で JPEG が返る** (実測)。Referer 無し・自サイト・
localhost・無関係な外部はすべて 403 (CloudFront の `Request blocked`)。

```text
Referer なし                        -> 403 text/html 919B
Referer: https://ndlsearch.ndl.go.jp/ -> 200 image/jpeg 6300B
Referer: https://qr.tommie.jp/...     -> 403
```

つまり **Referer を詐称すれば今でも取れる**が、やらない。提供者が
「終了した」と告知したうえで自サイト以外を遮断しているものを、ヘッダを
偽ってすり抜ける形になる。規約に反するうえ、いつ塞がれてもおかしくない
土台の上に本文 (= 正本) を作ることになる。

この遮断は**案 A (ブラウザから外部 URL を直接表示) が成立しない**ことも
意味する。ブラウザは自サイトの Referer を送るので 403 になる。

### 版元ドットコムは当たるが、使わない

`https://www.hanmoto.com/bd/img/<ISBN>.jpg` は 301 で
`img.hanmoto.com` に飛び、**11 冊中 10 冊**で実書影 (200x256 前後の JPEG)
を返す。Referer も鍵も要らない。カバー率だけなら断然これ。

それでも採らない。**これは公開 API ではなくサイトの画像パス**で、
利用条件の裏付けが無い。しかも版元ドットコムは openBD プロジェクトの
運営者 (カーリル・版元ドットコム) そのもので、**openBD が API で
ホワイトリスト版元の書影しか返さないのは、意図的に絞っているから**。
同じ組織の同じ書影をサイト側から取るのは、その制御を迂回することになる。

なお「書影なし」は **HTTP 404 + 176 バイトのプレースホルダ JPEG** で返る
(`Content-Type: image/jpeg`)。マジックバイトだけで判定すると
プレースホルダを掴む。採るときはステータスを見ること。

## 2. 方針: openBD cover → 楽天ブックスの順に、サーバが取得して DB に保存

書誌 (13) と同じ形。**性格の違う 2 つを順に試し、サーバ側で引く**。

```text
ISBN をスキャン
  → /edit/<次番号>?code=<ISBN>
  → ブラウザが /api/books/<ISBN> を引く            … 13 のまま
  → サーバが openBD → NDL の順に書誌を引く          … 13 のまま
  → **サーバが openBD cover → 楽天の順に書影を引く**
  → **書影を images テーブル (bytea) に保存**
  → **書名・著者・出版社 + 書影が入ったエディタ**
```

- **openBD cover** … 書誌の応答 (`summary.cover`) に URL が入っている。
  書誌を引くついでなので**追加リクエストは画像本体の 1 回だけ**。
  当たるのはホワイトリスト版元だけ (実測 1/11) だが、
  **唯一の「明文の利用条件があるタダの書影」**なので先に引く
- **楽天ブックス書籍検索 API** … 鍵が要る代わりに実際に当たる。
  書影は `largeImageUrl` (200x200)

### 楽天は新しい口 + 3 点セットでないと通らない

2026 年の仕様変更で、口も鍵も変わっている。実測で確かめた通し方:

| 要るもの | どこに | 無いとどうなるか |
| --- | --- | --- |
| 口 `openapi.rakuten.co.jp` | — | 旧 `app.rakuten.co.jp` は**何を送っても 400** |
| `applicationId` | クエリ | 400 |
| `accessKey` (`pk_` 始まり) | クエリ | 400 `accessKey must be present ...` |
| **`Origin` ヘッダ** | ヘッダ | 403 `REQUEST_CONTEXT_BODY_HTTP_REFERRER_MISSING` |

- **旧い口は使えない**。`app.rakuten.co.jp` は新しい形式の鍵 (UUID 風の
  applicationId・`pk_` 始まりの accessKey) を知らず、正しい鍵を送っても
  `specify valid applicationId` を返す。ここに嵌ると鍵を疑い続けることになる
- **`Origin` はアプリ登録の「サイトURL」**。`Referer` では通らない (実測。
  エラー名は REFERRER だが、効くのは `Origin`)。登録と違うドメインを送ると
  `HTTP_REFERRER_NOT_ALLOWED` になるので、値は `RAKUTEN_APP_ORIGIN` で持つ
  (QR シールの `QR_BASE_URL` とは別物。楽天側の登録に合わせる値であり、
  たまたま同じでも意味が違う)
- レート制限は厳しめで、続けて叩くと 429
  (`Rate limit is exceeded. Try again in 1 seconds.`)。実運用は 1 冊 1 回なので
  当たらないが、調査でループを回すときは間隔を空ける (13 §7 の NDL と同じ)

エラーは**どれが原因か教えてくれない**ので、`fetchCoverUrl` は引く前に 3 つ
揃っているか見て、欠けているものを名指しで警告する。それでも通らないときの
ために、応答の `errorMessage` は例外に載せる (鍵は載せない)。

書影の判定は**書誌とは独立**に行う。「openBD で書誌が取れたから書影も
openBD」ではない (書誌が取れても cover が空なほうが多数派)。

### なぜ本文にコピーするのか (外部 URL を置かない)

13 §2 と同じ理由に加えて、書影では**外部 URL が実際に表示できない**。

| | DB に保存 (採用) | 外部 URL を本文に置く |
| --- | --- | --- |
| 表示 | いつでも出る | NDL は 403、楽天は規約次第 |
| バックアップ | `pg_dump` だけで揃う | 画像は入らない |
| リンク切れ | 起きない | 提供終了で消える (§1 が実例) |
| オフライン | 出る | 出ない |

「`pg_dump` だけでメモと画像が一緒にバックアップできる」は手動アップロード
画像の設計 ([メモ記法.md](./メモ記法.md)) で、書影もそこに揃える。

### 事前入力の形

```text
リーダブルコード : より良いコードを書くためのシンプルで実践的なテクニック
Boswell, Dustin / Foucher, Trevor / 角, 征典
オーム社 (2012.06)
![書影|120](/api/images/<uuid>.jpg)

#9784873115658 #book
```

- **1 行目は書名のまま**。一覧の要約 (`memoSummary`) が書影の Markdown に
  なっては困る。書影は見出し (書名〜出版社) の**下**、空行より上に置く
- `|120` は独自の幅指定記法 (`MarkdownView` の `imgWithWidth`)。原寸は
  200x200 前後で、本文に原寸で出ると邪魔になる。**画像自体は無改変**で、
  表示幅だけを指定する (openBD の「改変禁止」に触れない)
- 書影が取れなければ**行ごと落とす**。13 §3 の「欠けた項目は行ごと落とす」と同じ
- alt を「書影」にするのは、`![|120]` だと読み上げで無名の画像になるため

## 3. 取れなくても書誌を道連れにしない

**書誌が本体、書影はおまけ**。この順位を崩さない。

- 書影の取得に失敗・タイムアウトしても、応答は成功のまま
  `coverImageUrl` を付けずに返す。書名・著者は今までどおり入る
- `RAKUTEN_APP_ID` 未設定は「書影なし」に落とす (エラーにしない)。
  Yahoo!ショッピングのキー未設定と同じ扱い
  ([14-JAN商品情報取得計画.md](./14-JAN商品情報取得計画.md) §1)。
  ただし黙ると設定漏れに気づけないので警告を 1 行残す

### 書影の上限は書誌より短く、取得ぜんぶで 5 秒

`/api/books/<ISBN>` は書影の保存まで済ませてから応答する。**書影を待つ時間は
書名が出るまでの時間にそのまま乗る**ので、上限が要る。

書誌の流儀 (13 §4 の「取得ごとに 8 秒」) をそのまま真似ると、取得元が
2 つ (openBD の画像 + 楽天の API と画像) あるぶん、全部が黙り込んだときに
**書名すら出ないまま 20 秒以上待たせる**。「1 つの API が黙り込んでも導線を
吊るさない」の趣旨に反するので、**書影は取得ぜんぶで 5 秒**に切る
(`COVER_TIMEOUT_MS`)。書影はおまけで、書誌ほどの粘りは要らない。

実測は書影ありで 19ms、無しで 0ms なので、まともに動いている限り当たらない。

書影だけ後から届ける (書誌を先に返し、クライアントが 2 回目を引く) 案は採らない。
ルートとクライアントの状態が増えるうえ、13 §5 の「遅れて届いた内容が手書きを
消す」危険を 2 つに増やす。5 秒で切れば足りる。

### 「画像が返った」を鵜呑みにしない

書影 API は「無い」を素直に 404 で返すとは限らない (§1 の版元ドットコムは
404 + JPEG のプレースホルダ)。**3 つとも通ったものだけ**を書影として扱う。

1. HTTP 200 であること
2. `Content-Type` が対応する画像形式であること
3. **先頭バイトが画像の署名と一致**すること (`matchesMagicBytes`)

3 は手動アップロードと同じ検査 ([uploads.ts](../src/lib/uploads.ts))。
アップロード経路の検査水準を、サーバが取ってきた画像でも下げない。

### 外部 API 由来の URL をそのまま fetch しない (SSRF)

openBD の `summary.cover` も楽天の `largeImageUrl` も**外部 API の応答に
書かれた URL** で、そのまま fetch すると、応答が壊れた・乗っ取られたときに
このサーバが任意の URL を叩く踏み台になる。境界で縛る:

- `https:` のみ。**ホストは許可リスト**
  (`cover.openbd.jp` / `thumbnail.image.rakuten.co.jp` など) に一致するものだけ
- **リダイレクトを追わない** (`redirect: 'error'`)。許可リストを通した URL が
  別ホストへ飛ばされたら意味がない
- サイズ上限 (`MAX_IMAGE_BYTES` = 10MB)。`Content-Length` と実読みの両方で見る
- そもそも `/api/books/[isbn]` はログイン必須 (`denyUnlessLoggedIn`) で、
  ISBN も入口で検算済み (13 §4)

大きさの上限は **Content-Length に頼らない**。ヘッダは送り手の申告でしかなく、
chunked では付かない。それでいて `arrayBuffer()` は申告と無関係に本文を最後まで
メモリへ読み切るので、「読み切ってから大きさを見る」形にすると上限を宣言している
意味がない。読みながら数えて、超えた時点で受信をやめる (`readWithLimit`)。

### 残っている穴 (承知のうえで見送ったもの)

- **DNS rebinding**。許可リストはホスト名しか見ておらず、`fetch` が解決した
  実 IP は検査していない。`cover.openbd.jp` などが一時的に内部 IP を指せば、
  許可リストを守ったまま内部に到達しうる。取得先が第三者の実在ドメインで
  攻撃の敷居が高いため、解決先 IP の検査 (SSRF-safe fetch) までは作らない
- ~~**GET に書き込みの副作用が付いた**~~ → **塞いだ**
  ([18-ログイン計画.md](./18-ログイン計画.md) §9)。この計画で
  `/api/books/<ISBN>` は「外部 API を引くだけ」から「images に INSERT し、
  楽天のクォータを使う」GET になった。このアプリの認証は Basic 認証なので
  **Cookie の SameSite が効かず**、ログイン済みのブラウザが悪意あるページを
  開くと `<img src="https://.../api/books/978...">` だけで孤児の書影と
  楽天クォータを焚かれた。`Sec-Fetch-Site` を見て、自分のページ
  (`same-origin`) と人が直接開いた場合 (`none`) 以外を 403 で断る。
  同じ穴が前からあった `/api/products/<JAN>` (Yahoo! の利用枠) にも同時に入れた

## 4. 保存は事前入力のときに 1 回。孤児は許容する

書影は**エディタを開いた時点で保存**する。ノートを保存せず離脱すると
`images` に参照されない行が残るが、許容する。

- 手動アップロードも同じ挙動 (アップロード後に保存しなければ孤児)。
  [12-ゴミ箱計画.md](./12-ゴミ箱計画.md) の「画像の孤児 GC」で既知の
  見送り事項として整理済み。書影だけ特別扱いしない
- 書影は 6〜65KB (実測) で、孤児のコストが小さい
- 同じ ISBN の再スキャンは既存ノートに当たるので新規登録に来ない。
  重複保存が起きるのは「登録を破棄してもう一度スキャン」の稀な場合だけ

保存時ではなく事前入力時にするのは、保存時にすると本文中の外部 URL を
保存の瞬間に書き換える変換層が要るため。事前入力なら**本文に入る URL は
最初から `/api/images/<uuid>.jpg`** で、正本 (本文) が常に自己完結する。

## 5. 段階と見送り

- **Phase 1 (この計画)**: openBD cover → 楽天。`RAKUTEN_APP_ID` が未設定の
  間は openBD のぶん (ホワイトリスト版元) だけが入る
- **見送り: 版元ドットコム** (§1)。カバー率は最も高いが公開 API ではなく、
  openBD のホワイトリストを迂回する形になる
- **見送り: NDL 書影 API** (§1)。2026-03-31 終了。Referer 詐称はしない
- **見送り: Google Books** (13 §1 と同じ)。匿名 429 は 2026-07 でも再現した
- **見送り: 書影のリサイズ・再圧縮**。200x200 前後で 6〜65KB なら DB に
  そのまま入れて困らない。openBD の「改変禁止」とも相性が悪い
- **見送り: 既存ノートへの遡及取得**。まず新規登録の導線だけで運用する
- **見送り: JP-e コード** での書影取得。NDL 書影 API 固有の話で、その API を
  使わない以上いらない

## 6. 利用条件

- **openBD** … 書影は「本の販促・紹介目的に限り使用できます」。**改変禁止**
  (§2 のとおり表示幅の指定だけで画像自体はいじらない)。著作権者からの
  削除要請には従う
- **楽天** … `applicationId` の登録が要る。画像の利用条件は楽天ウェブサービスの
  規約に従う
- いまは私的な蔵書ノートなので上のどちらにも収まるが、
  [17-単語帳DB分離計画.md](./17-単語帳DB分離計画.md) /
  [18-ログイン計画.md](./18-ログイン計画.md) の先でノートを公開する形に
  進めるときは、openBD の「販促・紹介目的」との整合を見直すこと

## 7. テスト

- 純関数 (vitest):
  - `parseOpenBdResponse` … `summary.cover` を拾う / 空文字 / 文字列でない値
  - `parseRakutenBooksResponse` … `largeImageUrl` を拾う / `items` が空 /
    画像 URL が空 / 配列でない応答
  - `isAllowedCoverUrl` … 許可ホスト / 許可外ホスト / `http:` / URL でない値 /
    **許可ホストを含むだけの偽 URL** (`https://evil.example/cover.openbd.jp/x.jpg`)
  - `scanRegisterMemo` … 書影あり / なし / 書誌が無いとき
- 取得 (fetch をモック):
  - `lookupCover` … openBD で当たる / openBD が空で楽天に落ちる / 両方外す /
    404 は書影なし / **200 でも中身が画像でなければ書影なし** /
    リダイレクトを追わない / サイズ超過 / キー未設定は警告して null
- ブラウザでの通し確認 (Playwright MCP、実 API):
  - ホワイトリスト版元の本 (作品社 9784861827754) → 書影が入る
  - 書影の無い本 → 書誌だけ入り、画像行は出ない (導線は止まらない)
  - 書影の取得が失敗しても書名・著者が入る
