# 全文検索 (Ver2.1) 実装計画 — PostgreSQL 本格全文検索

## 1. ゴールと現状のギャップ

現状の検索 (`searchItems`, `src/lib/items.ts`) は `contains` / ILIKE の部分一致。
メモ本文は検索できるが、以下が未対応。

| 要件 | 現状 (`contains`/ILIKE) | 実装後 |
| --- | --- | --- |
| 複数語 AND 検索 (`抵抗 1608`) | ✗ 文字列そのまま検索 | ✓ 空白区切りで AND |
| 表記ゆれ (全角/半角・大小文字) | 大小文字のみ | ✓ `２ＳＣ１８１５`→`2sc1815` も一致 |
| 日本語トークン化 | ✗ | ✓ TokenBigram |
| 関連度順ランキング | ✗ 更新順/番号順のみ | 見送り (§3-4 の実測参照) |

## 2. 技術選定: PGroonga を採用

- 標準 tsvector — 日本語トークナイザが無く日本語 ✗
- pg_trgm — 2 文字の日本語語彙に弱い、ランキング実質 ✗
- pg_bigm — ILIKE 高速化のみでランキング無し、公式 Docker イメージ無し
- **PGroonga** ✓ — 日本語 (バイグラム)・ランキング・正規化 (NormalizerAuto が全半角/大小を同一視)
  を満たし、公式イメージ `groonga/pgroonga:4.0.6-alpine-16` が現行 `postgres:16-alpine` の
  ドロップイン置き換え (同 PG16・同 alpine/musl なのでデータ volume そのまま)

個人利用・数千件規模では性能は主目的ではなく、**AND 検索・表記ゆれ吸収**が実益。

## 3. 設計

### 3-1. インフラ (`compose.yaml`)

`db.image` を差し替える。

```yaml
db:
  image: groonga/pgroonga:4.0.6-alpine-16   # postgres:16-alpine から差し替え
```

### 3-2. DB マイグレーション (手書き SQL)

`prisma migrate dev --create-only` で空マイグレーションを作り SQL を記述。

```sql
CREATE EXTENSION IF NOT EXISTS pgroonga;
CREATE INDEX items_memo_url_pgroonga_idx ON items USING pgroonga (memo, url);
```

- トークナイザ/ノーマライザは既定 (TokenBigram / NormalizerAuto) で開始
- `schema.prisma` は変更なし (Prisma は pgroonga インデックスを表現できないが
  migration ファイル経由なら drift しない)

### 3-3. 検索クエリ設計 (`searchItems` 改修)

検索語を空白 (半角・全角) で分割し、語ごとに
「memo OR url の全文一致、または itemNo 前方一致」、語間は AND。

```sql
SELECT item_no AS "itemNo", ... FROM items
WHERE (memo &@ $1 OR url &@ $1 OR item_no ILIKE $2)
  AND (memo &@ $3 OR url &@ $3 OR item_no ILIKE $4)
ORDER BY updated_at DESC
LIMIT 20 OFFSET ...
```

設計上のポイント:

- `&@` (単一キーワード一致) を使い `&@~` (クエリ構文) は使わない
  → ユーザー入力の構文エスケープ不要・構文エラーの心配なし。AND は SQL 側で組む
- itemNo は現行の前方一致 (`ILIKE 'prefix%'`) を維持。セマンティクスは現行と同じ
- Prisma は `&@` を query builder で書けないため `$queryRaw` + `Prisma.join` で構築。
  SQLi はプレースホルダで防御。列は `AS "camelCase"` で射影し既存 `Item` 型に合わせる
- 空クエリ (一覧ブラウズ) も同じ raw パスで WHERE 無しとして扱う (単一経路)
- 語分割・LIKE エスケープは純関数として `src/lib/` に切り出し (将来のタグ/日付検索でも再利用)

### 3-4. ランキング (実測の結果: 見送り)

`pgroonga_score(tableoid, ctid)` は **PGroonga インデックススキャンが選ばれた時のみ** 値が入り、
seq scan では 0 になる。実データ 512 行で検証したところ、`ANALYZE` 後のプランナは
(正しく) seq scan を選ぶため **スコアは常に 0** になった。異なる列にまたがる OR
(`memo/url` と `item_no`) では `enable_seqscan = off` でもインデックスに乗らない。

したがって **ランキングは本リリースでは見送り**、並び順は現行の更新順/番号順を維持する。
AND 検索・全半角/大小の正規化・日本語トークン化という主要な実益は seq scan でも
そのまま得られる。将来データ量が増えプランナがインデックスを選ぶようになれば
`pgroonga_score` の再導入を検討する。

### 3-5. ソート UI (`validation.ts`, `page.tsx`)

- 3-4 によりランキングは見送るため **`Sort` は現行の `'itemNo' | 'updated'` のまま変更なし**
- `page.tsx` のプレースホルダ文言のみ全文検索向けに更新 (任意)

## 5. デプロイ手順 (vps2)

1. バックアップ: `docker compose exec db pg_dump ...` を先に取得
2. **vps2 の compose.yaml を手動更新** — `doDeploy.sh` は compose.yaml を同期しない
3. vps2 で `docker compose pull db && docker compose up -d db`
4. `./doDeploy.sh` — 4/6 の `prisma migrate deploy` が `CREATE EXTENSION` +
   インデックス作成を適用 (接続ユーザー `qr` は superuser なので拡張作成可)
5. `https://qr.tommie.jp` で実機確認 → README の `[ ] 全文検索` にチェック

## 6. リスクとロールバック

- ロールバック: `DROP INDEX` → `DROP EXTENSION pgroonga` → 旧イメージへ戻す
  (拡張を残したまま素の postgres に戻すと pg_dump 等が失敗するので順序厳守)
- 以後の dump のリストア先も pgroonga イメージが必要 (運用メモに残す)
- debian 系タグへは変えない (musl→glibc の collation 差でインデックス破損リスク)
